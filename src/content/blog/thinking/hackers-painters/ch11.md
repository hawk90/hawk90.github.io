> **Hackers and Painters** Essay 11 요약

## 핵심 주장

**100년 후에도 쓰일 프로그래밍 언어를 상상해보라.** 그것은 오늘날 우리가 설계해야 할 언어의 방향을 알려준다.

## 왜 100년인가?

### 장기적 관점

100년을 생각하는 이유:
- 근시안적 선택 피하기
- 본질적인 것에 집중
- 유행과 본질 구분

### 기술 발전 예측

100년 후:
- 컴퓨터는 훨씬 빨라짐
- 메모리는 훨씬 커짐
- **효율성보다 표현력**이 중요해짐

## 언어 진화의 방향

### 추상화의 증가

역사적 추세:
- 기계어 → 어셈블리 → C → Python
- 점점 더 인간에 가까워짐
- 더 적은 코드로 더 많이

### 병렬성

미래의 요구:
- 멀티코어가 기본
- 병렬 처리 필수
- 언어가 이를 지원해야

## 미래 언어의 특성

### 1. 간결함

- 코드가 짧아짐
- 보일러플레이트 감소
- 의도만 표현

### 2. 유연성

- 다양한 스타일 지원
- 제약 최소화
- 프로그래머 자유

### 3. 효율성 덜 중요

- 하드웨어가 보완
- 개발 속도가 더 중요
- **프로그래머 시간 > 컴퓨터 시간**

### 4. 쓰레기 수집 (GC)

- 수동 메모리 관리 사라짐
- GC가 기본
- 안전성 우선

## Lisp의 위치

### 왜 Lisp인가

Graham의 주장:
- Lisp는 이미 미래에 가까움
- 극도의 유연성
- 매크로 = 언어 확장

### 문법의 미니멀리즘

Lisp의 문법:
- 거의 없음
- 괄호만
- **코드 = 데이터**

이것이 강점:
- 메타프로그래밍 쉬움
- 언어 자체를 확장
- 무한한 가능성

## 언어 설계 원칙

### 핵심만 유지

좋은 언어:
- 작은 코어
- 강력한 확장성
- 불필요한 것 없음

### 점진적 타이핑

미래의 타입 시스템:
- 필요할 때만 타입
- 선택적 정적 타이핑
- 유연성 + 안전성

### 기본값의 중요성

언어 설계:
- 기본값이 중요
- 대부분의 코드는 기본값 사용
- 좋은 기본값 = 좋은 언어

## 오늘날의 시사점

### 무엇을 배워야 하나

100년 언어를 생각하면:
- 유행 언어에 집착하지 마라
- 기본 원칙을 배워라
- Lisp류 언어를 알아둬라

### 무엇을 만들어야 하나

언어 설계자에게:
- 간결함 추구
- 유연성 중시
- 하드웨어에 덜 의존

## 예측의 한계

### 틀릴 수 있다

Graham의 인정:
- 예측은 어렵다
- 하지만 방향은 알 수 있다
- 생각하는 것 자체가 가치

### 확실한 것

변하지 않을 것:
- 추상화 증가
- 간결함 추구
- 인간 중심

## 시사점

1. **장기적 사고**: 유행을 넘어서 생각
2. **간결함**: 미래 언어는 더 짧다
3. **유연성**: 제약 없는 표현
4. **효율성 < 표현력**: 개발자 시간이 더 중요

## 인용

> "I think a lot of the best languages are converging on Lisp."

> "A hundred years from now, people will be programming in something like Lisp."

> "Inefficient software isn't gross. What's gross is a language that makes programmers do needless work."
