코덱을 이해하기 전에 먼저 "비디오"가 무엇인지부터 잡고 가야 한다. 픽셀, 프레임, 색 공간, 중복성, 정수 연산 — 이 다섯 가지가 앞으로 AV1을 읽어 나갈 토대다.

## 0.1 픽셀, 프레임, 해상도

### 픽셀 — 디지털 이미지의 최소 단위

**Pixel** = **Pic**ture **El**ement.

- 하나의 점에 색상값 하나가 저장된다 (예: R=200, G=150, B=80)
- 모니터의 **물리 픽셀**과 영상의 **논리 픽셀**은 다를 수 있다 (예: 4K 모니터에 1080p 영상 → 업스케일링)

```
┌────┬────┬────┬────┐
│ P₁ │ P₂ │ P₃ │ P₄ │   ← 각 칸이 하나의 픽셀
├────┼────┼────┼────┤
│ P₅ │ P₆ │ P₇ │ P₈ │   4×2 해상도 = 8 픽셀
└────┴────┴────┴────┘
```

### 해상도

가로 × 세로 픽셀 수다.

| 이름 | 해상도 | 화소 수 | 용도 |
|------|--------|---------|------|
| SD | 720×480 | ~35만 | DVD |
| HD (720p) | 1280×720 | ~92만 | 웹 스트리밍 |
| **FHD (1080p)** | **1920×1080** | **~207만** | **TV, 유튜브** |
| 4K (2160p) | 3840×2160 | ~830만 | 넷플릭스, UHD TV |
| 8K (4320p) | 7680×4320 | ~3,320만 | NHK 방송 |

<img src="/images/blog/av1/chapter0/fig-0-1-resolution.svg" alt="해상도 비교" style="max-height: 280px; margin: 0 auto; display: block;" />

SD에서 8K까지 화소 수가 약 100배 차이가 난다.

### 프레임과 프레임레이트

**프레임**은 특정 시각의 정지 이미지 한 장이다. 비디오는 이 프레임을 빠르게 연속 재생하는 플립북 원리다.

**프레임레이트(fps)**는 초당 프레임 수다.

| fps | 용도 | 특징 |
|-----|------|------|
| 24 | 영화 | 시네마틱 느낌 |
| 30 | TV, 일반 영상 | 가장 보편적 |
| 60 | 게임, 스포츠 | 부드러운 움직임 |
| 120+ | 슬로모션, VR | 고속 촬영 |

인간 시각의 한계는 대체로 60fps 부근이다. 그 이상은 차이를 구분하기 어려워진다.

### Progressive vs Interlaced

**Progressive (p)** — 매 프레임 전체 라인을 한 번에 그린다.

```
프레임 1:        프레임 2:
──────────      ──────────
──────────      ──────────
──────────      ──────────
(전체 갱신)     (전체 갱신)
```

1080**p** = Progressive. 현대 표준이고 화면이 깔끔하다.

**Interlaced (i)** — 홀수/짝수 줄을 번갈아 그린다.

```
필드 1(홀):     필드 2(짝):
──────────
                ──────────
──────────
(홀수줄만)      (짝수줄만)
```

1080**i** = Interlaced. 구형 TV의 유산이다.

> **AV1은 Progressive만 지원한다.** Interlaced는 아예 다루지 않는다.

## 0.2 색의 표현 — RGB

### RGB 색 모델

빛의 삼원색 **Red, Green, Blue**를 혼합해 색을 표현한다.

- 각 채널은 0~255 범위다 (8-bit 기준)
- 1 픽셀 = 3 채널 × 8 비트 = **24 비트**
- $256^3 = 16{,}777{,}216$ 가지 색 (약 1,677만 색)

모니터/디스플레이의 물리적 구조(RGB 서브픽셀)와 정확히 일치한다.

<img src="/images/blog/av1/chapter0/fig-0-2-rgb-cube.svg" alt="RGB 색 큐브" style="max-height: 280px; margin: 0 auto; display: block;" />

### 비트 깊이

채널당 몇 비트를 쓰는가의 문제다.

| 비트 깊이 | 채널당 단계 | 총 색상 수 | 용도 |
|-----------|-----------|-----------|------|
| 8-bit | 256 | 1,677만 | 일반 영상, 웹 |
| **10-bit** | **1,024** | **10.7억** | **HDR, AV1 기본** |
| 12-bit | 4,096 | 687억 | 영화, 아카이브 |

비트 깊이가 높을수록 그래디언트가 부드럽다. 8-bit에서 그라데이션이 계단처럼 보이는 현상을 **밴딩(banding) 아티팩트**라 한다.

```
8-bit  그래디언트:  ████████████████████  (계단이 보인다)
10-bit 그래디언트:  ▓▓▓▓▒▒▒▒░░░░░░░░░░░  (부드럽다)
```

AV1은 8/10/12-bit를 모두 지원하며, 프로파일에 따라 결정된다.

### RGB의 한계 — 왜 RGB를 쓰지 않는가

문제는 RGB가 인간의 색 인지와 맞지 않는다는 데 있다.

인간 눈의 특성은 다음과 같다.

- 밝기 변화에 **매우 민감**하다
- 색상 변화에 **상대적으로 둔감**하다

그런데 RGB는 구조적으로 다음과 같은 문제를 가진다.

- 밝기와 색상이 세 채널에 **섞여** 있다
- "눈이 덜 민감한 부분"을 분리해서 줄이기가 어렵다

```
RGB에서 "밝기"만 바꾸려면?
  R: 200 → 100  (세 채널을
  G: 150 → 75    전부 바꿔야 한다)
  B:  80 → 40
→ 밝기와 색을 분리할 수 없으니 압축에 불리하다
```

해결책은 밝기와 색을 분리하는 **YCbCr** 색 공간이다.

## 0.3 YCbCr — 비디오의 진짜 색 공간

### 인간 시각 시스템(HVS)의 핵심

밝기와 색을 왜 분리해야 할까. 답은 눈의 감광 세포 구조에 있다.

- **간상세포(Rod)** — 밝기 감지, 약 1억 2천만 개
- **원추세포(Cone)** — 색 감지, 약 600만 개
- 간상세포가 원추세포보다 **20배** 더 많다

결론적으로 눈은 밝기에는 민감하고 색차에는 둔감하다. 색 정보를 덜 정밀하게 저장해도 잘 모른다.

### YCbCr 색 공간

밝기와 색차를 분리해 저장한다.

| 성분 | 의미 | 특성 |
|------|------|------|
| **Y** (Luminance) | 밝기 | 흑백 이미지와 동일 |
| **Cb** (Chrominance Blue) | 파란색 방향 색차 | 색 정보만 |
| **Cr** (Chrominance Red) | 빨간색 방향 색차 | 색 정보만 |

<img src="/images/blog/av1/chapter0/fig-0-3-ycbcr-channels.svg" alt="YCbCr 채널 분리" style="max-height: 280px; margin: 0 auto; display: block;" />

### RGB → YCbCr 변환 (BT.709)

가장 일반적인 HD 비디오 표준이다.

$$
\begin{aligned}
Y  &= 0.2126 \cdot R + 0.7152 \cdot G + 0.0722 \cdot B \\[4pt]
Cb &= \frac{B - Y}{2 \cdot (1 - 0.0722)} + 128 \\[4pt]
Cr &= \frac{R - Y}{2 \cdot (1 - 0.2126)} + 128
\end{aligned}
$$

Y에서 초록(G)의 비중이 가장 큰 이유는 인간 눈이 초록에 가장 민감하기 때문이다. G의 가중치 0.7152 ≫ R의 0.2126 ≫ B의 0.0722.

다른 표준으로 BT.601(SD)과 BT.2020(4K/HDR)도 있다. 계수만 다르고 구조는 같다.

### 실습 — Y, Cb, Cr 채널 분리

같은 이미지를 세 채널로 분리해 보면 직접 확인할 수 있다.

```python
import cv2
import numpy as np

img = cv2.imread("photo.jpg")
ycrcb = cv2.cvtColor(img, cv2.COLOR_BGR2YCrCb)
Y, Cr, Cb = cv2.split(ycrcb)

cv2.imwrite("Y.png", Y)    # 선명한 흑백 이미지
cv2.imwrite("Cb.png", Cb)  # 흐릿한 색 패턴
cv2.imwrite("Cr.png", Cr)  # 흐릿한 색 패턴
```

- **Y만 보면** 선명한 흑백 이미지다 — 디테일이 다 살아 있다
- **Cb, Cr만 보면** 흐릿한 색 패턴이다 — 디테일이 거의 없다

"색 정보를 줄여도 된다"가 눈으로 확인된다.

## 0.4 크로마 서브샘플링

코덱 이전 단계의 첫 번째 압축이다.

### J:a:b 표기법

4픽셀 단위로 크로마 샘플링 비율을 표현한다.

| 표기 | 크로마 해상도 | 절약량 | 용도 |
|------|-------------|--------|------|
| **4:4:4** | 루마와 동일 | 0% | 전문가, 후반작업 |
| **4:2:2** | 수평 ½ | 33% | 방송 |
| **4:2:0** | 수평 ½ + 수직 ½ | **50%** | **가장 일반적!** |
| 4:0:0 | 없음 (흑백) | 66% | Monochrome |

<img src="/images/blog/av1/chapter0/fig-0-4-chroma-subsampling.svg" alt="크로마 서브샘플링" style="max-height: 280px; margin: 0 auto; display: block;" />

YouTube, Netflix 등 거의 모든 소비자 영상이 4:2:0을 쓴다.

### 4:2:0 메모리 레이아웃 (Planar)

1920×1080 FHD 기준이다.

```
┌─────────────────────────────────┐
│          Y  평면                 │  1920 × 1080 = 2,073,600 바이트
│        (전체 해상도)              │
└─────────────────────────────────┘
┌────────────────┐
│    Cb  평면     │  960 × 540 = 518,400 바이트
│  (½ × ½ 해상도) │
└────────────────┘
┌────────────────┐
│    Cr  평면     │  960 × 540 = 518,400 바이트
│  (½ × ½ 해상도) │
└────────────────┘
──────────────────────────────────────
총합: 3,110,400 바이트 ≈ 3 MB/프레임
```

### 50% 절약의 원리

**RGB** (서브샘플링 없음)는 다음과 같다.

$$1920 \times 1080 \times 3 = 6{,}220{,}800 \text{ 바이트} \approx 6 \text{ MB / 프레임}$$

**YCbCr 4:2:0**은 다음과 같다.

$$\underbrace{1920 \times 1080}_{Y} + \underbrace{960 \times 540}_{Cb} + \underbrace{960 \times 540}_{Cr} \approx 3 \text{ MB / 프레임}$$

일반화하면 명확하다.

$$\text{4:4:4} \rightarrow 1 + 1 + 1 = 3 \quad (\text{채널})$$

$$\text{4:2:0} \rightarrow 1 + 0.25 + 0.25 = 1.5 \quad (\text{채널}) \rightarrow \textbf{절반}$$

크로마 서브샘플링은 코덱 이전 단계의 압축이다. 코덱은 여기서 **추가로** 압축한다.

## 0.5 비디오에 숨겨진 두 가지 중복성

비디오 코덱이 수백 배의 압축을 달성할 수 있는 이유는 비디오에 두 가지 중복성이 존재하기 때문이다.

### 공간 중복성 (Spatial Redundancy)

한 프레임 안에서 인접 픽셀이 비슷한 값을 가진다.

```
하늘 영역의 실제 픽셀값 (Y 채널):

200 201 200 202 201 200 201 200
201 200 201 200 202 201 200 201
200 201 200 201 200 201 202 200
         ↓ 거의 같다
"200 근처 값이 16개 연속" → 예측 가능
```

- 하늘, 벽, 도로 같은 넓은 영역은 거의 같은 색이다
- 텍스처가 있어도 인접 픽셀 간 변화는 작다
- "옆 픽셀이 200이면 이 픽셀도 200 근처일 것이다"

나중에 다룰 **Intra Prediction** (Ch8)의 기반이다.

### 시간 중복성 (Temporal Redundancy)

연속된 프레임이 거의 같다.

```
Frame 1 (t=0.000s)          Frame 2 (t=0.033s)

┌─────────────────┐         ┌─────────────────┐
│ 하늘             │         │ 하늘             │
│                  │         │                  │
│     🚗          │    →    │       🚗        │
│                  │         │                  │
│ 도로             │         │ 도로             │
└─────────────────┘         └─────────────────┘

30fps = 33ms 간격 → 차이가 거의 없다
오직 차가 살짝 오른쪽으로 이동한 것뿐!
```

- 카메라가 고정되어 있으면 99%의 픽셀이 동일하다
- 물체가 이동해도 위치만 바뀌고 픽셀값 자체는 유사하다

나중에 다룰 **Inter Prediction** (Ch12)의 기반이다.

### 실습 — 차이 이미지

두 연속 프레임의 차이를 계산해 보자.

```python
import cv2
import numpy as np

frame1 = cv2.imread("frame001.png", cv2.IMREAD_GRAYSCALE)
frame2 = cv2.imread("frame002.png", cv2.IMREAD_GRAYSCALE)

diff = cv2.absdiff(frame1, frame2)
cv2.imwrite("diff.png", diff)
```

거의 검은 이미지에 움직임이 있는 부분만 밝게 보인다. **"차이만 보내면 된다"** — 시간 중복성을 활용하는 핵심 아이디어다.

## 0.6 비디오 데이터량의 현실

왜 압축이 **필수**인지 숫자로 보자.

### 비압축 비디오 데이터량

$$
\text{바이트/프레임} = W \times H \times 1.5 \quad (\text{4:2:0, 8-bit 기준})
$$

$$
\text{바이트/초} = \text{바이트/프레임} \times \text{fps}
$$

| 해상도 | fps | 바이트/초 | 비트레이트 |
|--------|-----|----------|-----------|
| 720p | 30 | 41.5 MB/s | 332 Mbps |
| **1080p** | **30** | **93.3 MB/s** | **746 Mbps** |
| 1080p | 60 | 186.6 MB/s | 1.49 Gbps |
| 4K | 30 | 373.2 MB/s | 2.99 Gbps |
| **4K** | **60** | **746.5 MB/s** | **5.97 Gbps** |

비압축 1시간 4K/60fps는 $746.5 \times 3600 \approx$ **2.6 TB**가 된다.

### 200:1 — 코덱이 만들어내는 기적

비교는 단순하다.

- **비압축 4K/30fps**: $2.99 \text{ Gbps}$
- **Netflix 4K 스트리밍**: $\approx 15 \text{ Mbps}$
- **압축률**: $\frac{2{,}990}{15} \approx \textbf{200:1}$

<img src="/images/blog/av1/chapter0/fig-0-6-bitrate-chart.svg" alt="비트레이트 비교" style="max-height: 280px; margin: 0 auto; display: block;" />

비디오 코덱은 이 200배를 만들어내는 기술이다. 구성 요소별로 보면 다음과 같다.

- 크로마 서브샘플링: ~2:1
- 공간 예측 + 변환 + 양자화: ~10:1
- 시간 예측 (모션 보상): ~10:1
- 엔트로피 코딩: ~2:1
- 총 $2 \times 10 \times 10 \times 2 \approx$ **400:1** (이론적 최대)

## 0.7 고정소수점 연산

마지막으로 비디오 코덱이 왜 부동소수점 대신 고정소수점 정수 연산을 쓰는지 짚어 두자. 앞으로 AV1 스펙을 읽을 때 반복해서 등장하는 핵심 패턴이다.

### 왜 정수만 쓰는가

**부동소수점의 문제**는 CPU마다 미세한 오차가 생긴다는 점이다.

```
인코더 (Intel CPU):  0.7152 × 200 = 143.0400000001
디코더 (ARM CPU):    0.7152 × 200 = 143.0399999999
                                      ↑ 다르다!
→ 한 픽셀의 오차가 예측에 전파 → 프레임 전체가 깨진다
```

**고정소수점의 해결**은 정수 연산으로 모든 CPU에서 동일한 결과를 보장하는 것이다 (bit-exact).

```
인코더 (Intel CPU):  (200 * 183) >> 8 = 142
디코더 (ARM CPU):    (200 * 183) >> 8 = 142
                                        ↑ 항상 같다
```

AV1 스펙의 **모든 연산**은 정수로 정의되어 있다.

### 고정소수점의 원리 — Q 포맷

아이디어는 실수를 정수로 스케일링하는 것이다.

$$
\text{고정소수점 값} = \text{실수} \times 2^n \quad (\text{Q}n\text{ 포맷})
$$

| 표기 | 소수부 비트 | 스케일 | 예시: 0.75 |
|------|-----------|--------|-----------|
| Q4 | 4 | $\times 16$ | $0.75 \times 16 = 12$ |
| Q8 | 8 | $\times 256$ | $0.75 \times 256 = 192$ |
| Q12 | 12 | $\times 4096$ | $0.75 \times 4096 = 3072$ |

연산 규칙은 다음과 같다.

- **덧셈** — 같은 Q 포맷이면 그냥 정수 덧셈이다  
  $Q8(0.5) + Q8(0.25) = 128 + 64 = 192 = Q8(0.75)$ ✓
- **곱셈** — 결과의 Q가 두 배가 되므로 시프트로 복원한다  
  $Q8(0.5) \times Q8(0.25) = 128 \times 64 = 8192$, $8192 \gg 8 = 32 = Q8(0.125)$ ✓

### AV1의 핵심 패턴 — Round2

AV1 스펙에서 가장 자주 등장하는 연산이다.

```cpp
// Round2: 반올림 후 시프트 (AV1 스펙 §7.11.3.4)
int Round2(int x, int n) {
    return (x + (1 << (n - 1))) >> n;
}
```

왜 `(1 << (n-1))`을 더할까? 반올림 때문이다.

$$
\text{Round2}(x, n) = \left\lfloor \frac{x + 2^{n-1}}{2^n} \right\rfloor = \left\lfloor \frac{x}{2^n} + 0.5 \right\rfloor
$$

예를 들어 보면 차이가 명확하다.

```
Round2(100, 3) = (100 + 4) >> 3 = 104 >> 3 = 13
단순 시프트:     100 >> 3 = 12  (버림 → 오차 누적)
Round2:          (100 + 4) >> 3 = 13  (반올림 → 정확)
```

이 패턴은 보간 필터, 역변환, Loop Restoration, 모션 벡터 스케일링 등 코덱 전반에 반복해서 등장한다. 앞으로 코드에서 자주 보게 될 것이다.

## 정리

- **픽셀과 프레임**은 비디오의 물리적 단위이며, 해상도와 프레임레이트가 데이터량의 두 축이다
- **RGB**는 모니터 친화적이지만 인간 시각과 어긋나 압축에 불리하다
- **YCbCr**은 밝기와 색차를 분리해 압축의 발판을 마련한다
- **크로마 서브샘플링**(4:2:0)은 코덱 이전 단계에서 이미 50%를 절약한다
- **공간·시간 중복성**이 코덱이 200:1 압축을 달성하는 근거다
- **고정소수점 + Round2**가 AV1 모든 연산의 기본 패턴이다
