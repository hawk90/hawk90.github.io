## Amdahl의 비관론

Amdahl의 법칙은 비관적이다:

- 10%가 순차면 → 최대 10배 스피드업
- 1%가 순차면 → 최대 100배 스피드업
- 1000코어? 순차 부분이 지배할 뿐

하지만 1988년, John Gustafson이 **다른 관점**을 제시했다.

---

## 가정의 전환

### Amdahl의 가정

> "문제 크기가 **고정**되어 있다."

1000개 데이터를 처리하는 시간을 줄인다.

### Gustafson의 가정

> "실행 **시간**이 고정되어 있다."

주어진 시간에 **더 많은** 데이터를 처리한다.

---

## 실전 예시

### 시나리오

```
단일 코어: 1시간에 1GB 처리
8코어: 1시간에 얼마나 처리할 수 있나?
```

**Amdahl 관점** (문제 크기 고정):
- 1GB를 더 빨리 처리
- 순차 부분 때문에 한계

**Gustafson 관점** (시간 고정):
- 1시간 동안 8GB 처리
- 코어가 많을수록 더 큰 문제 해결

### 현실에서

- 영화 렌더링: 더 높은 해상도, 더 많은 프레임
- 과학 시뮬레이션: 더 정밀한 그리드, 더 긴 시간
- 머신러닝: 더 큰 모델, 더 많은 데이터

## Gustafson의 법칙

### 정의

N개 프로세서에서 실행 시간을 기준으로:

- **s**: 순차 부분의 비율 (병렬 실행 시간 기준)
- **p**: 병렬 부분의 비율 (p = 1 - s)

**Scaled Speedup**:

$$
\text{Speedup}(N) = s + p \cdot N = s + (1-s) \cdot N
$$

### 직관적 이해

```
8코어 병렬 실행:
총 시간: 100초
순차 부분: 10초 (10%)
병렬 부분: 90초 (각 코어가 90초씩 일함)

단일 코어로 같은 일을 하면:
순차 부분: 10초
병렬 부분: 90초 × 8 = 720초
총 시간: 730초

스피드업: 730 / 100 = 7.3배
```

거의 **선형 스피드업**!

## Amdahl vs Gustafson

### 수식 비교

| | Amdahl | Gustafson |
|--|--------|-----------|
| 고정 | 문제 크기 | 실행 시간 |
| 관점 | 단일 코어 → 병렬 | 병렬 → 단일 코어 |
| 스피드업 | 1 / (s + p/N) | s + p·N |
| 한계 | 1/s | 무한 (선형) |

### 같은 예시, 다른 해석

**상황**: 순차 10%, 병렬 90%, 8코어

**Amdahl**:
```
스피드업 = 1 / (0.1 + 0.9/8) = 1 / 0.2125 ≈ 4.7x
```

**Gustafson**:
```
스피드업 = 0.1 + 0.9 × 8 = 0.1 + 7.2 = 7.3x
```

**왜 다른가?**

- Amdahl: 고정된 문제를 병렬화 → 순차 부분이 병목
- Gustafson: 코어 수에 맞게 문제 확장 → 병렬 부분이 지배

## Strong Scaling vs Weak Scaling

### Strong Scaling (Amdahl)

- 문제 크기 **고정**
- 코어 추가 → 실행 시간 감소
- 목표: **얼마나 빨리?**

```
문제: 1000×1000 행렬 곱
1코어: 100초
8코어: 15초 (6.7x 스피드업)
64코어: 3초 (33x 스피드업) ← 한계
```

### Weak Scaling (Gustafson)

- 코어당 문제 크기 **고정**
- 코어 추가 → 더 큰 문제 해결
- 목표: **얼마나 크게?**

```
1코어: 1000×1000 행렬 100초
8코어: 2000×2000 행렬 100초 (8배 큰 문제)
64코어: 4000×4000 행렬 100초 (64배 큰 문제)
```

### 어떤 스케일링이 맞는가?

| 상황 | 적합한 스케일링 |
|-----|---------------|
| 실시간 응답 필요 | Strong (빨리!) |
| 배치 처리 | Weak (많이!) |
| 과학 시뮬레이션 | 보통 Weak |
| 웹 서버 응답 | Strong |
| 데이터 분석 | 보통 Weak |

## 현실의 제약

### Gustafson도 만능이 아니다

**1. 메모리 한계**

문제를 키우면 메모리도 필요:

```
1코어: 1GB 데이터
64코어: 64GB 데이터? → 메모리 부족
```

**2. 통신 오버헤드**

큰 문제 → 더 많은 코어 간 통신:

```
코어 수 N이 커지면:
- 동기화 오버헤드 증가
- 캐시 일관성 비용 증가
- 메모리 대역폭 포화
```

**3. 알고리즘 한계**

모든 알고리즘이 무한히 확장되지 않는다:

```cpp
// 병합 정렬: log N 단계의 순차 부분
// 문제가 커져도 순차 단계는 줄지 않음
```

## 실전 조언

### 1. 목표를 명확히

```
"이 워크로드는 Strong인가 Weak인가?"

Strong: 응답 시간 줄이기
Weak: 처리량 늘리기
```

### 2. 두 스케일링 모두 측정

```bash
# Strong scaling test
for cores in 1 2 4 8 16; do
    run_benchmark --cores=$cores --size=fixed
done

# Weak scaling test
for cores in 1 2 4 8 16; do
    size=$((base_size * cores))
    run_benchmark --cores=$cores --size=$size
done
```

### 3. 효율성 추적

```
Strong Scaling 효율 = Speedup / N
Weak Scaling 효율 = T_1 / T_N  (같은 코어당 문제 크기)
```
