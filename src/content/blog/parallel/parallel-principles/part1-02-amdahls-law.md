## 병렬화의 꿈과 현실

4코어 CPU가 있으면 프로그램이 4배 빨라질까?

**대부분 아니다.**

Gene Amdahl이 1967년에 이 한계를 수학적으로 설명했다.

---

## Amdahl의 법칙

### 정의

프로그램의 일부만 병렬화할 수 있다고 가정하자.

- **P**: 병렬화 가능한 비율 (0 ≤ P ≤ 1)
- **S**: 순차적으로만 실행 가능한 비율 (S = 1 - P)
- **N**: 프로세서(코어) 수

**스피드업(Speedup)**:

$$
\text{Speedup}(N) = \frac{1}{S + \frac{P}{N}} = \frac{1}{(1-P) + \frac{P}{N}}
$$

### 직관적 이해

```
원래 실행 시간: 100초
순차 부분: 20초 (20%)
병렬 부분: 80초 (80%)

코어 4개 사용:
순차 부분: 20초 (병렬화 불가)
병렬 부분: 80초 / 4 = 20초

총 시간: 20 + 20 = 40초
스피드업: 100 / 40 = 2.5배
```

4코어인데 2.5배밖에 안 빨라진다!

---

## 스피드업 계산

### 예시: 90% 병렬화

| 코어 수 | 계산 | 스피드업 |
|--------|------|---------|
| 1 | 1 / (0.1 + 0.9/1) | 1.0x |
| 2 | 1 / (0.1 + 0.9/2) | 1.82x |
| 4 | 1 / (0.1 + 0.9/4) | 3.08x |
| 8 | 1 / (0.1 + 0.9/8) | 4.71x |
| 16 | 1 / (0.1 + 0.9/16) | 6.40x |
| 64 | 1 / (0.1 + 0.9/64) | 8.77x |
| ∞ | 1 / (0.1 + 0) | **10x** |

**90%를 병렬화해도 최대 10배가 한계!**

### 예시: 99% 병렬화

| 코어 수 | 스피드업 |
|--------|---------|
| 10 | 9.17x |
| 100 | 50.25x |
| 1000 | 90.99x |
| ∞ | **100x** |

99%를 병렬화해야 100배 스피드업에 근접.

## 그래프로 보는 한계

```
스피드업
    ^
100 |                              ......... 99% 병렬
    |                    .........
 50 |           .........
    |     ......
 20 |  ...                         ......... 95% 병렬
 10 | .                    ........
    |.              .......
  5 |         ......               ......... 90% 병렬
    |    .....
  2 | ...
  1 |.________________________________> 코어 수
    1    10    100   1000  10000
```

**수확 체감(Diminishing Returns)**: 코어를 추가해도 스피드업이 점점 더디게 증가.

## 순차 부분이 지배한다

### 핵심 통찰

$$
\lim_{N \to \infty} \text{Speedup}(N) = \frac{1}{S} = \frac{1}{1-P}
$$

**무한 개의 코어가 있어도** 순차 부분(S)이 스피드업을 제한한다.

### 실전 예시

```cpp
void process_data(std::vector<int>& data) {
    // 순차 부분: 데이터 로드 (10%)
    data = load_from_disk();

    // 병렬 부분: 처리 (85%)
    #pragma omp parallel for
    for (int i = 0; i < data.size(); i++) {
        data[i] = expensive_computation(data[i]);
    }

    // 순차 부분: 결과 저장 (5%)
    save_to_disk(data);
}

// 최대 스피드업: 1 / 0.15 ≈ 6.67x
```

디스크 I/O가 병목이 되어 아무리 코어가 많아도 6.67배가 한계.

## Amdahl의 법칙이 말하지 않는 것

### 1. 오버헤드 무시

실제로는 병렬화에 **오버헤드**가 있다:

- 스레드 생성/종료
- 동기화 (락, 배리어)
- 통신 (캐시 일관성, 메모리 대역폭)

```cpp
// 이론: 병렬 부분 시간 = P / N
// 실제: 병렬 부분 시간 = P / N + overhead(N)
```

오버헤드 때문에 코어가 많아지면 오히려 느려질 수 있다.

### 2. 메모리 대역폭 무시

모든 코어가 메모리에 접근하면:

```
┌──────┬──────┬──────┬──────┐
│Core 1│Core 2│Core 3│Core 4│
└──┬───┴──┬───┴──┬───┴──┬───┘
   │      │      │      │
   ▼      ▼      ▼      ▼
┌─────────────────────────────┐
│       Memory Bus (병목)      │
└─────────────────────────────┘
```

**메모리 바운드** 워크로드는 코어 수에 비례하지 않는다.

### 3. 문제 크기 고정 가정

Amdahl의 법칙은 **문제 크기가 고정**됐다고 가정한다.

하지만 실제로는 코어가 많으면 **더 큰 문제**를 풀 수 있다.

→ 이것이 **Gustafson의 법칙** (다음 글)

## 실전 의미

### 1. 순차 부분을 줄여라

```cpp
// Before: 순차 I/O
data = load_all_at_once();  // 순차
process(data);               // 병렬

// After: 병렬 I/O + 파이프라인
// 로딩과 처리를 겹친다
std::thread loader([&] {
    for (auto chunk : chunks) {
        queue.push(load(chunk));  // 병렬 로드
    }
});

#pragma omp parallel
{
    while (auto chunk = queue.pop()) {
        process(chunk);  // 병렬 처리
    }
}
```

### 2. 순차 부분을 측정하라

병렬화 전에 프로파일링:

```
$ perf record ./program
$ perf report

순차 부분 찾기:
- 락 경쟁
- I/O 대기
- 순차 알고리즘
```

### 3. 기대치를 현실적으로

| 순차 비율 | 최대 스피드업 |
|----------|-------------|
| 50% | 2x |
| 25% | 4x |
| 10% | 10x |
| 5% | 20x |
| 1% | 100x |

"90%를 병렬화했으니 16코어에서 16배 빨라진다"는 환상.

## 확장된 Amdahl의 법칙

### 오버헤드 포함

$$
\text{Speedup}(N) = \frac{1}{S + \frac{P}{N} + O(N)}
$$

O(N)은 병렬화 오버헤드. 보통 O(N) = α·N 형태.

### 효율성

$$
\text{Efficiency}(N) = \frac{\text{Speedup}(N)}{N}
$$

- Efficiency = 1.0: 완벽한 확장성
- Efficiency < 1.0: 손실 있음

| 코어 | 스피드업 | 효율성 |
|-----|---------|--------|
| 1 | 1.0x | 100% |
| 4 | 3.0x | 75% |
| 16 | 8.0x | 50% |
| 64 | 16.0x | 25% |

효율성이 떨어지면 비용 대비 효과가 줄어든다.

## 핵심 요약

| 개념 | 설명 |
|-----|------|
| Amdahl의 법칙 | Speedup = 1 / (S + P/N) |
| 한계 | 무한 코어도 1/S가 최대 |
| 순차 부분 | 전체 스피드업을 지배 |
| 실전 의미 | 순차 부분 줄이기가 핵심 |
| 한계점 | 오버헤드, 메모리 대역폭 무시 |
