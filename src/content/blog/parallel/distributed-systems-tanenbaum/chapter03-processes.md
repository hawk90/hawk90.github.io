## 스레드

분산 시스템에서 **스레드**는 단일 프로세스 내에서 독립적인 실행 흐름을 제공한다.

### 스레드 vs 프로세스

| 특성 | 프로세스 | 스레드 |
|------|---------|--------|
| 주소 공간 | 독립 | 공유 |
| 생성 비용 | 높음 | 낮음 |
| 컨텍스트 스위치 | 비쌈 | 저렴 |
| 통신 | IPC 필요 | 직접 메모리 접근 |

```
프로세스:
┌─────────────┐  ┌─────────────┐
│ 주소 공간 A  │  │ 주소 공간 B  │
│ ┌─────────┐ │  │ ┌─────────┐ │
│ │Thread 1 │ │  │ │Thread 1 │ │
│ └─────────┘ │  │ └─────────┘ │
└─────────────┘  └─────────────┘

멀티스레드 프로세스:
┌─────────────────────────────┐
│         공유 주소 공간        │
│ ┌───────┐ ┌───────┐ ┌─────┐ │
│ │Thread │ │Thread │ │ ... │ │
│ │   1   │ │   2   │ │     │ │
│ └───────┘ └───────┘ └─────┘ │
└─────────────────────────────┘
```

### 분산 시스템에서 스레드의 역할

**서버 측**:

```
요청 처리 모델:

1. 단일 스레드 (Single-threaded)
   요청 → [처리] → [처리] → [처리] (순차)
   문제: I/O 블로킹 시 전체 대기

2. 스레드 풀 (Thread Pool)
   요청 → Thread Pool → 병렬 처리
   장점: 동시 요청 처리, 스레드 재사용

3. 이벤트 기반 (Event-driven)
   요청 → Event Loop → 비동기 처리
   장점: 적은 스레드로 많은 연결
```

**클라이언트 측**:

```
- UI 스레드: 사용자 응답성 유지
- 네트워크 스레드: 서버 통신
- 백그라운드 스레드: 데이터 처리
```

### 스레드 구현

| 구현 방식 | 설명 | 장단점 |
|----------|------|--------|
| **사용자 수준** | 커널 모름 | 빠른 스위치, I/O 블로킹 문제 |
| **커널 수준** | OS가 관리 | 느린 스위치, 진정한 병렬 |
| **하이브리드** | M:N 매핑 | 유연, 복잡 |

## 가상화

**가상화**는 물리 자원을 논리 자원으로 추상화한다.

### 가상화의 역할

```
가상화가 중요한 이유:
1. 자원 격리 — 서로 영향 없음
2. 이식성 — 환경 독립적 배포
3. 활용도 — 물리 자원 공유
4. 복제/마이그레이션 — 상태 스냅샷
```

### 가상 머신 아키텍처

```
Type 1 (Bare-metal):
┌─────┐ ┌─────┐ ┌─────┐
│ VM1 │ │ VM2 │ │ VM3 │
└──┬──┘ └──┬──┘ └──┬──┘
   └───────┴───────┘
         ▼
┌─────────────────────┐
│    Hypervisor       │ ← 하드웨어 직접 제어
└─────────────────────┘
         ▼
┌─────────────────────┐
│     Hardware        │
└─────────────────────┘

Type 2 (Hosted):
┌─────┐ ┌─────┐
│ VM1 │ │ VM2 │
└──┬──┘ └──┬──┘
   └───┬───┘
       ▼
┌─────────────┐
│ Hypervisor  │
└──────┬──────┘
       ▼
┌─────────────┐
│  Host OS    │
└──────┬──────┘
       ▼
┌─────────────┐
│  Hardware   │
└─────────────┘
```

### 컨테이너

컨테이너는 OS 수준 가상화. VM보다 가볍다.

```
┌─────────┐ ┌─────────┐ ┌─────────┐
│Container│ │Container│ │Container│
│  App A  │ │  App B  │ │  App C  │
└────┬────┘ └────┬────┘ └────┬────┘
     └───────────┴───────────┘
              ▼
┌────────────────────────────────┐
│     Container Runtime          │
│     (Docker, containerd)       │
└────────────────────────────────┘
              ▼
┌────────────────────────────────┐
│         Host OS Kernel         │
└────────────────────────────────┘
```

| 특성 | VM | 컨테이너 |
|------|-----|---------|
| 격리 수준 | 강함 | 중간 |
| 오버헤드 | 높음 | 낮음 |
| 부팅 시간 | 분 | 초 |
| 이미지 크기 | GB | MB |

## 클라이언트

### 네트워크 사용자 인터페이스

클라이언트가 사용자에게 서비스를 보여주는 방식.

| 유형 | 설명 | 예 |
|------|------|-----|
| **Thin Client** | 표시만 담당 | X Window, VNC |
| **Fat Client** | 로직도 실행 | 네이티브 앱 |
| **웹 기반** | 브라우저에서 실행 | SPA, PWA |

```
Thin Client:
┌────────────┐         ┌────────────┐
│  Client    │◀───────▶│  Server    │
│ (Display)  │ 화면전송 │ (Processing)│
└────────────┘         └────────────┘

Fat Client:
┌────────────────┐     ┌────────────┐
│    Client      │◀───▶│   Server   │
│ (UI + Logic)   │데이터│  (Data)    │
└────────────────┘     └────────────┘
```

### 클라이언트 측 소프트웨어 배치

```
분산 투명성을 위한 클라이언트 역할:
- 접근 투명성: 데이터 형식 변환
- 장애 투명성: 재시도 로직
- 복제 투명성: 복제본 중 선택
- 위치 투명성: 서비스 디스커버리
```

## 서버

### 일반 설계 이슈

**반복 vs 동시 서버**:

```
반복 (Iterative):
while (true) {
    request = accept();
    handle(request);  // 블로킹
}

동시 (Concurrent):
while (true) {
    request = accept();
    spawn_thread(handle, request);  // 병렬
}
```

**상태 유지 vs 무상태**:

| 유형 | 장점 | 단점 |
|------|------|------|
| **Stateless** | 단순, 확장 용이 | 매번 상태 전송 |
| **Stateful** | 효율적 | 장애 복구 어려움 |

### 서버 클러스터

```
3-티어 클러스터:
                    ┌─────────────┐
                    │Load Balancer│
                    └──────┬──────┘
         ┌─────────────────┼─────────────────┐
         ▼                 ▼                 ▼
    ┌─────────┐       ┌─────────┐       ┌─────────┐
    │ Web 1   │       │ Web 2   │       │ Web 3   │
    └────┬────┘       └────┬────┘       └────┬────┘
         └─────────────────┼─────────────────┘
                           ▼
    ┌─────────┐       ┌─────────┐       ┌─────────┐
    │ App 1   │       │ App 2   │       │ App 3   │
    └────┬────┘       └────┬────┘       └────┬────┘
         └─────────────────┼─────────────────┘
                           ▼
                    ┌─────────────┐
                    │  Database   │
                    │  (Primary)  │
                    └─────────────┘
```

**요청 라우팅**:

```
1. Transport Layer (L4):
   - TCP/IP 연결 기반
   - 빠름, 단순

2. Application Layer (L7):
   - HTTP 헤더, 쿠키 기반
   - 유연, 세션 어피니티
```

## 코드 마이그레이션

코드나 실행 상태를 다른 노드로 이동.

### 이동성의 이유

```
이유:
1. 성능 — 데이터 근처에서 처리
2. 유연성 — 동적 서비스 배치
3. 부하 분산 — 과부하 노드에서 이동
```

### 마이그레이션 모델

| 모델 | 이동 대상 | 예 |
|------|----------|-----|
| **Weak Mobility** | 코드만 | JavaScript, Java Applet |
| **Strong Mobility** | 코드 + 실행 상태 | VM Live Migration |

```
Weak Mobility:
┌──────────┐     code      ┌──────────┐
│ Source   │ ─────────────▶│  Target  │
│          │               │ (restart)│
└──────────┘               └──────────┘

Strong Mobility:
┌──────────┐   code+state  ┌──────────┐
│ Source   │ ─────────────▶│  Target  │
│ (pause)  │               │ (resume) │
└──────────┘               └──────────┘
```

**VM 라이브 마이그레이션**:

```
단계:
1. Pre-copy: 메모리 페이지 복사 (실행 중)
2. Stop-and-copy: 잠시 멈추고 나머지 복사
3. Resume: 새 호스트에서 재개

목표: 다운타임 최소화 (수 ms ~ 수백 ms)
```

## 정리

- **스레드**는 서버의 동시 요청 처리에 필수
- **가상화**로 자원 격리, 이식성, 마이그레이션 가능
- **컨테이너**는 VM보다 가볍고 빠름
- **서버 클러스터**로 확장성과 가용성 확보
- **코드 마이그레이션**으로 유연한 서비스 배치

## 핵심 비교

| 주제 | 옵션 A | 옵션 B |
|------|--------|--------|
| 스레드 구현 | 사용자 수준 (빠름) | 커널 수준 (진정한 병렬) |
| 가상화 | VM (강한 격리) | 컨테이너 (가벼움) |
| 클라이언트 | Thin (단순) | Fat (성능) |
| 서버 | Stateless (확장) | Stateful (효율) |
| 마이그레이션 | Weak (단순) | Strong (투명) |
