## 클럭 동기화

분산 시스템에서 **시간**은 복잡한 문제. 글로벌 클럭이 없다.

### 물리적 클럭

각 컴퓨터는 자체 **하드웨어 클럭**을 가진다. 문제는 **드리프트**.

```
클럭 드리프트:
시간이 지나면서 각 클럭이 다르게 흐름

    실제 시간 ─────────────────────────▶

    Clock A   ━━━━━━━━━━━━━━━━━━━━━━▶  (빠름)
    Clock B   ────────────────────────▶  (정확)
    Clock C   ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─▶  (느림)
```

**드리프트 율 (ρ)**: 1 - dC/dt, 일반적으로 10⁻⁵ ~ 10⁻⁶.

### NTP (Network Time Protocol)

인터넷에서 가장 널리 쓰이는 **시간 동기화 프로토콜**.

```
NTP 계층 구조:
                 ┌─────────┐
    Stratum 0    │  GPS/   │  원자시계
                 │ Atomic  │
                 └────┬────┘
                      │
                 ┌────┴────┐
    Stratum 1    │ Primary │  직접 연결된 서버
                 │ Server  │
                 └────┬────┘
           ┌─────────┼─────────┐
           ▼         ▼         ▼
    Stratum 2    Server   Server   Server
                   │
                   ▼
    Stratum 3    Client
```

**왕복 시간 기반 보정**:

```
A → B: T1에 전송, T2에 수신
B → A: T3에 전송, T4에 수신

왕복 시간: δ = (T4 - T1) - (T3 - T2)
오프셋:   θ = ((T2 - T1) + (T3 - T4)) / 2
```

### Berkeley 알고리즘

NTP와 달리 **내부 동기화** — 외부 시간 소스 없이 노드들끼리 동기화.

```
1. 마스터가 모든 슬레이브에게 시간 질의
   Master ─▶ Slave1: 시간?
          ─▶ Slave2: 시간?
          ─▶ Slave3: 시간?

2. 슬레이브들이 자신의 시간 응답
   Slave1 ─▶ Master: 10:05:15
   Slave2 ─▶ Master: 10:05:17
   Slave3 ─▶ Master: 10:05:13

3. 마스터가 평균 계산 후 조정값 전송
   평균: 10:05:15
   Master ─▶ Slave1: +0초
          ─▶ Slave2: -2초
          ─▶ Slave3: +2초
```

**특징**: 마스터 장애 시 새 마스터 선출 필요.

## 논리적 클럭

물리적 시간보다 **이벤트 순서**가 중요한 경우가 많다.

### Lamport 타임스탬프

**happens-before** 관계 (→)를 캡처.

```
정의:
1. 같은 프로세스에서 a가 b 전에 발생 → a → b
2. a가 메시지 전송, b가 수신 → a → b
3. a → b, b → c → a → c (추이적)

동시 (||): a → b도 b → a도 아니면 동시
```

**알고리즘**:

```
각 프로세스 Pi:
- 로컬 카운터 Ci 유지
- 이벤트 발생 시: Ci++
- 메시지 전송 시: Ci++, 메시지에 Ci 포함
- 메시지 수신 시: Ci = max(Ci, 수신된 타임스탬프) + 1
```

```
예:
P1: (1)────────(2)─────────────────(5)───▶
         ╲              ╱
          ╲            ╱
P2: ───────(1)────(2)─────(3)──────────▶
                    ╲
                     ╲
P3: ────────────────(3)────(4)─────────▶
```

**한계**: a → b이면 L(a) < L(b). 하지만 역은 성립 안 함.

### 벡터 클럭

**인과 관계를 완전히 캡처**.

```
N개 프로세스: 각각 벡터 [C1, C2, ..., CN] 유지

규칙:
- 이벤트 발생 시: Vi[i]++
- 메시지 전송 시: Vi[i]++, 벡터 포함
- 메시지 수신 시: Vi[j] = max(Vi[j], Vm[j]) for all j, then Vi[i]++
```

```
예 (3 프로세스):
P1: [1,0,0]──[2,0,0]────────────────[2,2,2]──▶
                 ╲              ╱
                  ╲            ╱
P2: ──────[0,1,0]──[2,2,0]──[2,2,1]───────────▶
                        ╲
                         ╲
P3: ────────────────[0,0,1]──[2,2,2]──────────▶
```

**비교 규칙**:

| 조건 | 관계 |
|------|------|
| V(a) < V(b) (모든 원소 ≤, 적어도 하나 <) | a → b |
| V(a) > V(b) | b → a |
| 그 외 (비교 불가) | a \|\| b (동시) |

## 상호 배제

분산 환경에서 **임계 영역 접근 제어**.

### 중앙집중식 알고리즘

**코디네이터**가 락을 관리.

```
1. 요청: 프로세스가 코디네이터에 요청
2. 대기/허가: 코디네이터가 큐에 넣거나 즉시 허가
3. 해제: 완료 후 코디네이터에 알림

Client1 ──REQUEST──▶ ┌────────────┐
                     │Coordinator │
Client2 ──REQUEST──▶ │  (Queue)   │
                     └────────────┘
                          │
                     ◀──GRANT──
```

**장점**: 단순, 공정.
**단점**: 단일 장애 지점, 병목.

### 분산 알고리즘 (Ricart-Agrawala)

모든 노드에게 허가 요청.

```
요청:
- 타임스탬프와 함께 모든 노드에 REQUEST

응답 규칙:
- 임계 영역 안 아님 + 관심 없음 → OK
- 임계 영역 안 아님 + 관심 있음 → 타임스탬프 비교
  - 내 타임스탬프 < 수신 타임스탬프 → 지연
  - 내 타임스탬프 > 수신 타임스탬프 → OK
- 임계 영역 안 → 지연

진입:
- 모든 (N-1)개 OK 받으면 진입

해제:
- 지연시킨 모든 노드에 OK 전송
```

```
예:
P1 (ts=5) ──REQUEST──▶ P2 (ts=7)
          ◀──OK────── P2 (내 타임스탬프 > 5)

P2 (ts=7) ──REQUEST──▶ P1 (ts=5)
                       P1 (내 타임스탬프 < 7, 지연)
          ◀──OK────── (P1 해제 후)
```

**장점**: 단일 장애 지점 없음.
**단점**: 메시지 많음 (2(N-1)), 모든 노드 알아야 함.

### 토큰 링 알고리즘

**토큰을 가진 노드만 진입**.

```
논리적 링:
    ┌───▶ P1 ───┐
    │           ▼
    P4         P2
    ▲           │
    └─── P3 ◀───┘

토큰이 P2에 있으면 P2만 진입 가능
사용 안 하면 다음 노드로 전달
```

**장점**: 공정 (토큰 순환), 기아 없음.
**단점**: 토큰 분실 시 복구 필요, 노드 장애 처리.

| 알고리즘 | 메시지/진입 | 지연 | 문제점 |
|----------|-------------|------|--------|
| 중앙집중 | 3 | 2 | 코디네이터 장애 |
| Ricart-Agrawala | 2(N-1) | 2(N-1) | 모든 노드 필요 |
| 토큰 링 | 1~∞ | 0~N-1 | 토큰 분실 |

## 선거 알고리즘

**리더(코디네이터) 선출**. 장애 후 새 리더 필요할 때.

### Bully 알고리즘

**ID가 가장 높은 노드가 승리**.

```
가정: 프로세스에 고유 번호, 번호가 높을수록 우선

절차:
1. P가 코디네이터 장애 감지
2. P보다 번호 높은 모든 노드에 ELECTION 메시지
3. 응답 없으면 → P가 코디네이터, COORDINATOR 브로드캐스트
4. 응답 있으면 → 응답한 노드가 선거 진행, P는 대기
```

```
예 (P4가 장애 감지, P7이 코디네이터였음):

P4 ──ELECTION──▶ P5, P6, P7
P5 ◀──OK─────── P5 (응답)
P6 ◀──OK─────── P6 (응답)
(P7 무응답)

P5 ──ELECTION──▶ P6, P7
P6 ◀──OK─────── P6 (응답)

P6 ──ELECTION──▶ P7
(무응답)

P6 ──COORDINATOR──▶ 모든 노드
P6이 새 코디네이터
```

**이름 유래**: 숫자 높은 놈이 "불리"처럼 이김.

### Ring 알고리즘

노드가 링으로 연결. **선거 메시지가 링을 돈다**.

```
절차:
1. P가 코디네이터 장애 감지
2. ELECTION 메시지 (자신 ID 포함) 를 후속자에게 전송
3. 각 노드가 자신 ID 추가하며 전달
4. 메시지가 시작점으로 돌아오면 → 가장 높은 ID가 코디네이터
5. COORDINATOR 메시지로 결과 브로드캐스트
```

```
예 (P2 시작):
     P1 ─────▶ P2(시작)
     ▲           │
     │    ELECTION[2]
     │           ▼
     P5 ◀───── P3
     │    ELECTION[2,3]
     │           │
     ▼           ▼
     P4 ◀───── (P3→P4)
        ELECTION[2,3,4]

최종: [2,3,4,5,1] → P5가 코디네이터
```

## 위치 시스템

**물리적 위치 추적**. 모바일/IoT 환경에서 중요.

### GPS (Global Positioning System)

위성 기반 삼각측량.

```
4개 이상 위성에서 신호 수신:
- 각 위성: (x, y, z, t) 정보 송신
- 수신기: 신호 도달 시간으로 거리 계산
- 4개 방정식 → (x, y, z, Δt) 풀이

    Sat1 ●───────────┐
                     │ d1
    Sat2 ●──────┐    │
                │ d2 ▼
    Sat3 ●──┐   ▼  ┌───┐
            │ d3   │수신│
    Sat4 ●──┴──────│기 │
                   └───┘
```

### Wi-Fi 기반 위치

**신호 강도 (RSSI)** 또는 **핑거프린트** 기반.

```
삼각측량:
    AP1 ●─────────── d1 = 5m
                        ╲
    AP2 ●──── d2 = 3m ───● 위치
                        ╱
    AP3 ●─────────── d3 = 4m

핑거프린트:
1. 학습 단계: 각 위치에서 AP별 RSSI 기록
2. 위치 추정: 현재 RSSI와 가장 유사한 패턴 찾기
```

## 정리

- **물리적 클럭**: NTP (외부 동기화), Berkeley (내부 동기화)
- **논리적 클럭**: Lamport (순서), Vector (인과 관계)
- **상호 배제**: 중앙집중(단순), 분산(N-1 메시지), 토큰 링(공정)
- **선거**: Bully (높은 ID 승리), Ring (순환 선거)
- **위치**: GPS (실외), Wi-Fi (실내)

| 주제 | 기법 | 특징 | 약점 |
|------|------|------|------|
| 시간 동기화 | NTP | 외부 정확도 | 네트워크 지연 |
| | Berkeley | 상대 동기화 | 마스터 장애 |

| | Ring | 공정 | 느림 |
