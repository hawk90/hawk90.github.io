## 분산 시스템이란

> **분산 시스템**: 독립적인 컴퓨터들의 집합으로, 사용자에게 단일 일관된 시스템으로 보이는 것.

이 정의에서 두 가지 핵심이 있다.

1. **독립적인 컴퓨터들** — 하드웨어적으로 자율적인 노드
2. **단일 시스템으로 보임** — 소프트웨어가 협력하여 투명성 제공

```
┌─────────────────────────────────────────────────────┐
│                    사용자 관점                        │
│              "하나의 시스템처럼 보인다"                  │
└─────────────────────────────────────────────────────┘
                         │
         ┌───────────────┼───────────────┐
         ▼               ▼               ▼
    ┌─────────┐    ┌─────────┐    ┌─────────┐
    │ Node A  │    │ Node B  │    │ Node C  │
    │ (독립)   │    │ (독립)   │    │ (독립)   │
    └─────────┘    └─────────┘    └─────────┘
         │               │               │
         └───────────────┴───────────────┘
                    네트워크
```

### 특징

| 특징 | 설명 |
|------|------|
| **동시성** | 여러 노드가 동시에 작업 수행 |
| **글로벌 클럭 부재** | 완벽한 시간 동기화 불가능 |
| **독립적 장애** | 일부 노드만 실패 가능 |

## 설계 목표

분산 시스템은 네 가지 핵심 목표를 추구한다.

### 자원 공유 (Resource Sharing)

원격 자원에 쉽게 접근할 수 있어야 한다.

```
자원의 예:
- 프린터, 스토리지, 데이터
- 컴퓨팅 파워 (클라우드)
- 서비스 (웹 API)
```

**경제성**: 비싼 자원을 여러 사용자가 공유하면 비용 절감.

**협업**: 지리적으로 분산된 팀이 데이터와 서비스를 공유.

### 분산 투명성 (Distribution Transparency)

사용자와 애플리케이션에게 **분산을 숨긴다**.

| 투명성 유형 | 설명 | 예 |
|------------|------|-----|
| **접근 (Access)** | 데이터 표현과 접근 방식 차이를 숨김 | 빅엔디안/리틀엔디안 변환 |
| **위치 (Location)** | 자원의 물리적 위치를 숨김 | URL은 위치를 노출하지 않음 |
| **이전 (Migration)** | 자원이 이동해도 사용자는 모름 | VM 라이브 마이그레이션 |
| **재배치 (Relocation)** | 사용 중 이동해도 투명 | 모바일 사용자 핸드오프 |
| **복제 (Replication)** | 복제본 존재를 숨김 | CDN, 데이터베이스 복제 |
| **동시성 (Concurrency)** | 여러 사용자의 동시 접근을 숨김 | 락, 트랜잭션 |
| **장애 (Failure)** | 장애와 복구를 숨김 | 자동 장애 조치 |

**완전한 투명성은 불가능하다.** 네트워크 지연, 장애는 근본적으로 숨길 수 없다.

```
투명성의 트레이드오프:
- 완전 투명 → 성능 저하 (원격 호출 vs 로컬 호출)
- 장애 투명 → 복잡성 증가
- 위치 투명 → 법적/규제 문제 (데이터 거주)
```

### 개방성 (Openness)

**표준 규칙에 따라 서비스를 제공**하는 시스템.

```
개방성의 핵심:
1. 잘 정의된 인터페이스 (IDL, API 명세)
2. 이식성 (Portability) — 다른 시스템으로 쉽게 이동
3. 상호운용성 (Interoperability) — 다른 구현과 협력
4. 확장성 (Extensibility) — 새 컴포넌트 추가 용이
```

**정책과 메커니즘의 분리**가 중요하다.

```
메커니즘: HOW (캐싱 방법)
정책: WHAT/WHEN (무엇을 언제 캐시할지)

좋은 설계: 메커니즘은 제공, 정책은 사용자가 결정
```

### 확장성 (Scalability)

시스템이 성장할 때도 성능을 유지해야 한다.

**세 가지 확장성 차원**:

| 차원 | 설명 | 도전 |
|------|------|------|
| **크기 (Size)** | 사용자/자원 수 증가 | 중앙집중 병목 |
| **지리 (Geographic)** | 노드 간 거리 증가 | 통신 지연 |
| **관리 (Administrative)** | 여러 조직이 공유 | 신뢰, 정책 충돌 |

**확장성 기법**:

```
1. 비동기 통신 — 동기 대기 제거
2. 분산 — 단일 서버 → 여러 서버
3. 복제 — 데이터를 여러 곳에
4. 캐싱 — 자주 쓰는 데이터를 가까이
```

**확장성의 함정 — 일관성**:

```
복제/캐싱 → 여러 복사본 → 일관성 문제
강한 일관성 → 동기화 비용 → 확장성 저하
약한 일관성 → 확장성 ↑, 복잡성 ↑
```

## 분산 시스템의 유형

### 고성능 분산 컴퓨팅 (HPC)

**클러스터 컴퓨팅**: 동일한 하드웨어, 동일 OS, 고속 LAN.

```
┌─────────────────────────────────────┐
│         마스터 노드                   │
└───────────────┬─────────────────────┘
    ┌───────────┼───────────┐
    ▼           ▼           ▼
┌───────┐  ┌───────┐  ┌───────┐
│Worker │  │Worker │  │Worker │
│Node 1 │  │Node 2 │  │Node N │
└───────┘  └───────┘  └───────┘
```

**그리드 컴퓨팅**: 이기종 하드웨어, 여러 조직, 광역 네트워크.

**클라우드 컴퓨팅**: 가상화된 자원을 서비스로 제공.

| 서비스 모델 | 설명 | 예 |
|------------|------|-----|
| **IaaS** | 인프라 (VM, 스토리지) | AWS EC2, GCP Compute |
| **PaaS** | 플랫폼 (런타임, DB) | Heroku, GAE |
| **SaaS** | 애플리케이션 | Gmail, Salesforce |

### 분산 정보 시스템

**트랜잭션 처리 시스템 (TP 모니터)**:

```
ACID 속성:
- Atomicity: 전부 또는 전무
- Consistency: 일관된 상태 유지
- Isolation: 동시 트랜잭션 간섭 없음
- Durability: 커밋 후 영구 저장
```

**엔터프라이즈 애플리케이션 통합 (EAI)**:

```
미들웨어로 이기종 시스템 연결:
- RPC (Remote Procedure Call)
- MOM (Message-Oriented Middleware)
- ESB (Enterprise Service Bus)
```

### 퍼베이시브 시스템 (Pervasive Systems)

일상 환경에 스며든 작은 노드들.

| 유형 | 특징 | 예 |
|------|------|-----|
| **유비쿼터스** | 연속적 환경 | 스마트홈 |
| **모바일** | 위치 변화 | 스마트폰 앱 |
| **센서** | 대량 데이터 수집 | IoT, 스마트 시티 |

## 분산 시스템의 함정

Peter Deutsch의 **분산 컴퓨팅의 8가지 오류**:

| # | 오류 (잘못된 가정) | 현실 |
|---|------------------|------|
| 1 | 네트워크는 신뢰할 수 있다 | 패킷 손실, 연결 끊김 |
| 2 | 지연 시간은 0이다 | 수 ms ~ 수백 ms |
| 3 | 대역폭은 무한하다 | 병목, 혼잡 |
| 4 | 네트워크는 안전하다 | 도청, 변조, 공격 |
| 5 | 토폴로지는 변하지 않는다 | 노드 추가/제거, 장애 |
| 6 | 관리자는 한 명이다 | 여러 조직, 정책 충돌 |
| 7 | 전송 비용은 0이다 | 대역폭 비용, 인코딩 오버헤드 |
| 8 | 네트워크는 동질적이다 | 이기종 프로토콜, 장비 |

```
이 오류들을 가정하고 설계하면:
→ 프로덕션에서 예상치 못한 장애
→ 성능 문제
→ 보안 취약점
```

## 정리

- **분산 시스템**은 독립 노드들이 협력하여 단일 시스템처럼 보이는 것
- **설계 목표**: 자원 공유, 투명성, 개방성, 확장성
- **투명성은 트레이드오프** — 완전한 투명성은 비용이 크다
- **확장성의 세 차원**: 크기, 지리, 관리
- **8가지 오류**를 항상 염두에 두고 설계해야 한다
