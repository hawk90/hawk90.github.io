## 정확성이란

동시성 프로그램의 **정확성**: 순차 버전과 같은 결과를 생성.

```
정확성의 두 측면:

1. Safety (안전성):
   "나쁜 일이 일어나지 않는다"
   - 레이스 컨디션 없음
   - 데드락 없음
   - 데이터 손상 없음

2. Liveness (활동성):
   "좋은 일이 결국 일어난다"
   - 프로그램이 종료
   - 요청이 처리됨
   - 기아 없음
```

---

### 레이스 컨디션

**정의**: 여러 스레드가 공유 데이터에 접근하고, 최소 하나가 쓰기를 할 때, 결과가 실행 순서에 의존.

```
예: 계좌 이체

잔고: A = 1000, B = 1000

Thread 1: A → B 500 이체
Thread 2: B → A 300 이체

정상 순서:
T1: A = 1000 - 500 = 500
T1: B = 1000 + 500 = 1500
T2: B = 1500 - 300 = 1200
T2: A = 500 + 300 = 800
결과: A = 800, B = 1200 ✓

레이스 순서:
T1: read A (1000)
T2: read B (1000)
T2: read A (1000)
T1: read B (1000)
T1: write A = 500
T1: write B = 1500
T2: write B = 700   ← T2가 옛날 B 값 사용!
T2: write A = 1300  ← T2가 옛날 A 값 사용!
결과: A = 1300, B = 700 ✗ (돈이 생김!)
```

**해결**: 동기화 (뮤텍스, 락).

---

### 데드락

**정의**: 두 개 이상의 스레드가 서로가 가진 자원을 기다리며 영원히 블록.

```
4가지 조건 (모두 만족해야 발생):
1. 상호 배제 (Mutual Exclusion)
2. 점유 대기 (Hold and Wait)
3. 비선점 (No Preemption)
4. 순환 대기 (Circular Wait)
```

```
예:
Thread 1:            Thread 2:
lock(A)              lock(B)
lock(B) ← 대기       lock(A) ← 대기
...                  ...

순환 대기:
T1 → B → T2 → A → T1

해결:
- 락 순서 고정 (항상 A 먼저, 그 다음 B)
- 타임아웃
- 데드락 탐지 및 복구
```

---

### 라이브락

**정의**: 스레드들이 진행은 하지만 실제 작업은 안 하는 상태.

```
예: 복도에서 마주친 두 사람
A: 왼쪽으로 피함
B: 왼쪽으로 피함 (같은 방향)
A: 오른쪽으로 피함
B: 오른쪽으로 피함 (또 같은 방향)
... 무한 반복

코드 예:
while (conflicting) {
    release_lock();
    random_backoff();  // 항상 같은 타이밍
    acquire_lock();
}
```

**해결**: 랜덤 백오프, 우선순위.

## 성능 측정

병렬 프로그램의 성능을 어떻게 측정할까?

---

### Amdahl의 법칙

**핵심**: 순차 부분이 병렬화의 **한계를 결정**.

```
속도 향상 (Speedup) = T_순차 / T_병렬

S = 1 / (s + p/N)

s = 순차 부분 비율
p = 병렬 부분 비율 (s + p = 1)
N = 프로세서 수
```

```
예: 순차 부분 10% (s = 0.1)

N = 2:  S = 1 / (0.1 + 0.9/2)  = 1.82배
N = 4:  S = 1 / (0.1 + 0.9/4)  = 3.08배
N = 8:  S = 1 / (0.1 + 0.9/8)  = 4.71배
N = 16: S = 1 / (0.1 + 0.9/16) = 6.40배
N = ∞:  S = 1 / 0.1            = 10배 (최대)
```

```
그래프:

Speedup
   ▲
10 │                    ────────── 이론적 최대 (1/s)
   │                 ╱
 8 │               ╱
   │             ╱
 6 │           ╱
   │         ╱
 4 │       ╱
   │     ╱
 2 │   ╱
   │ ╱
   └───────────────────────────▶ 프로세서 수 (N)
       2   4   8   16  32  ∞
```

**교훈**: 10%의 순차 부분이 있으면 아무리 코어를 늘려도 10배가 한계.

---

### Gustafson의 법칙

**Amdahl과 다른 관점**: 문제 크기를 키우면?

```
Amdahl: 고정된 문제 크기, 프로세서 증가
Gustafson: 프로세서에 맞춰 문제 크기 증가

Gustafson의 공식:
S = N - s × (N - 1)
또는
S = s + p × N

s = 순차 시간 비율 (병렬 실행 시)
N = 프로세서 수
```

```
예: s = 0.1 (병렬 실행 시 순차 비율)

N = 2:  S = 2 - 0.1 × 1 = 1.9배
N = 4:  S = 4 - 0.1 × 3 = 3.7배
N = 8:  S = 8 - 0.1 × 7 = 7.3배
N = 16: S = 16 - 0.1 × 15 = 14.5배

→ 프로세서 수에 거의 선형!
```

**핵심 차이**:

| Amdahl | Gustafson |
|--------|-----------|
| 문제 크기 고정 | 문제 크기 확장 |
| 비관적 (한계 강조) | 낙관적 (확장성 강조) |
| 강한 확장성 | 약한 확장성 |

---

### 속도 향상 (Speedup)

```
정의:
S(N) = T_1 / T_N

T_1 = 1개 프로세서로 실행 시간
T_N = N개 프로세서로 실행 시간
```

**종류**:

| 종류 | 정의 | 의미 |
|------|------|------|
| **절대 속도향상** | T_best_seq / T_N | 최고 순차 대비 |
| **상대 속도향상** | T_1_parallel / T_N | 1스레드 병렬 버전 대비 |

```
이상적: S(N) = N (선형 속도향상)
현실적: S(N) < N (오버헤드 때문)
초선형: S(N) > N (드물게, 캐시 효과 등)
```

---

### 효율성 (Efficiency)

```
정의:
E(N) = S(N) / N = T_1 / (N × T_N)

의미: 각 프로세서가 얼마나 효율적으로 사용되는가
```

```
예:
S(4) = 3.2 → E(4) = 3.2 / 4 = 0.8 = 80%
S(8) = 5.6 → E(8) = 5.6 / 8 = 0.7 = 70%
S(16) = 9.6 → E(16) = 9.6 / 16 = 0.6 = 60%

→ 프로세서가 늘수록 효율 감소 (일반적)
```

**효율 감소 원인**:

```
1. 순차 부분 (Amdahl)
2. 동기화 오버헤드
3. 통신 오버헤드
4. 부하 불균형
5. 메모리 경쟁
```

---

## 확장성

**정의**: 프로세서 수 증가에 따른 성능 향상 정도.

### 강한 확장성 (Strong Scalability)

```
문제 크기 고정, 프로세서 증가

이상적: T_N = T_1 / N
현실: 순차 부분, 오버헤드로 제한
```

### 약한 확장성 (Weak Scalability)

```
프로세서당 문제 크기 고정 (전체 문제 크기 증가)

이상적: T_N = T_1 (일정)
의미: 2배 프로세서로 2배 문제를 같은 시간에
```

```
비교:

                강한 확장성              약한 확장성
문제 크기:      고정                    프로세서에 비례
목표:           실행 시간 감소          처리량 증가
적합한 경우:    고정 크기 문제          대규모 데이터
```

---

## 오버헤드 분석

병렬화의 숨은 비용.

```
총 실행 시간 = 계산 + 통신 + 동기화 + 부하 불균형

계산: 실제 작업
통신: 데이터 이동, 메시지
동기화: 락 획득/해제, 배리어
부하 불균형: 가장 느린 스레드 대기
```

```
예: 4 스레드, 불균형

Thread 0: ████████░░░░ 80ms + 20ms 대기
Thread 1: ██████████░░ 100ms
Thread 2: ████████████ 120ms ← 가장 느림
Thread 3: ███████░░░░░ 70ms + 50ms 대기

총 시간: 120ms (가장 느린 스레드)
이상적: 100ms (평균 × 1.25)

불균형 오버헤드: 20%
```

## 정리

- **정확성** = Safety + Liveness
- **레이스**: 공유 데이터 + 쓰기 + 순서 의존
- **데드락**: 4가지 조건 모두 만족 시 발생
- **Amdahl**: S = 1/(s + p/N), 순차 부분이 한계
- **Gustafson**: 문제 크기 확장 시 선형에 가까움
- **효율성**: E = S/N, 프로세서 활용도
