## 왜 동시성인가

**"공짜 점심은 끝났다."** — Herb Sutter의 유명한 선언.

```
클럭 속도의 역사:
1970s: MHz 시대
1990s: GHz 진입
2004: ~3.8 GHz 정점
2010s: 멀티코어로 방향 전환
2020s: 여전히 ~5 GHz 근처 (단일 코어)

┌──────────────────────────────────────────────┐
│  Clock Speed                                 │
│     ▲                                        │
│     │        ┌────── 한계 (발열, 전력)       │
│     │       ╱                                │
│     │      ╱                                 │
│     │     ╱                                  │
│     │    ╱                                   │
│     │───╱─────────────────────────────────▶  │
│           Time                               │
└──────────────────────────────────────────────┘
```

**더 빨리 가려면 더 많은 코어를 써야 한다.**

## 멀티코어의 부상

**단일 코어 한계**:

| 문제 | 설명 |
|------|------|
| **발열** | 클럭 ↑ → 전력 ↑ → 열 ↑ |
| **전력 벽** | P ∝ V² × f (주파수 비례) |
| **ILP 한계** | 명령어 수준 병렬성 포화 |

**멀티코어 해결책**:

```
단일 코어 4 GHz → 열 문제
vs
4개 코어 × 2 GHz → 같은 전력, 더 많은 처리량

┌─────────┐        ┌───┬───┬───┬───┐
│  CPU    │  vs    │ C1│ C2│ C3│ C4│
│ 4 GHz   │        │2G │2G │2G │2G │
└─────────┘        └───┴───┴───┴───┘
   열 한계              분산 처리
```

**현실**:

```
2024년 기준:
- 데스크톱: 8~24 코어 일반
- 서버: 64~128 코어
- 모바일: 8 코어 (빅.리틀)

소프트웨어가 코어를 활용하지 못하면?
→ 싱글 스레드 성능만 사용
→ 나머지 코어는 놀고 있음
```

## 동시성 vs 병렬성

자주 혼동되지만 다른 개념.

```
동시성 (Concurrency):
- 여러 작업이 "논리적으로" 동시에 진행
- 단일 코어에서도 가능 (인터리빙)
- 관점: 구조 (Structure)

병렬성 (Parallelism):
- 여러 작업이 "물리적으로" 동시에 실행
- 여러 코어/프로세서 필요
- 관점: 실행 (Execution)
```

```
동시성 (단일 코어):
┌─────────────────────────────────────────┐
│ Task A │ Task B │ Task A │ Task B │ ... │
└─────────────────────────────────────────┘
         ↑ 컨텍스트 스위칭

병렬성 (멀티 코어):
Core 1: ────Task A────────────────────────▶
Core 2: ────Task B────────────────────────▶
         ↑ 실제로 동시 실행
```

**비유**:

```
동시성: 한 바리스타가 여러 주문을 번갈아 처리
병렬성: 여러 바리스타가 각자 주문을 동시에 처리
```

## 성능 향상의 기회

**어디서 병렬화할 것인가?**

```
프로그램 분석:
┌─────────────────────────────────────┐
│  순차 부분 (20%)                    │
│  ┌───────────────────────────────┐  │
│  │ 병렬화 가능 부분 (80%)          │  │
│  │                               │  │
│  │   ┌───────┐  ┌───────┐       │  │
│  │   │루프 1 │  │루프 2 │ ...   │  │
│  │   └───────┘  └───────┘       │  │
│  └───────────────────────────────┘  │
└─────────────────────────────────────┘
```

**일반적인 병렬화 대상**:

| 대상 | 예 |
|------|-----|
| **데이터 병렬 루프** | 행렬 연산, 이미지 처리 |
| **독립 작업** | 파일 처리, 웹 요청 |
| **파이프라인** | 비디오 인코딩, 컴파일 |
| **분할 정복** | 정렬, 검색 |

**Amdahl의 법칙 미리보기**:

```
속도 향상 = 1 / (S + P/N)

S = 순차 부분 비율
P = 병렬 부분 비율 (S + P = 1)
N = 프로세서 수

예: S = 0.2, P = 0.8, N = 4
→ 1 / (0.2 + 0.8/4) = 1 / 0.4 = 2.5배

N = ∞ 일 때: 1 / 0.2 = 5배 (최대)
```

**교훈**: 순차 부분이 병목. 병렬화 비율이 중요.

## 동시성 프로그래밍의 도전

**왜 어려운가?**

```
1. 비결정성 (Nondeterminism):
   실행할 때마다 결과가 다를 수 있음

2. 레이스 컨디션 (Race Condition):
   공유 데이터 접근 순서에 따라 결과 달라짐

3. 데드락 (Deadlock):
   서로가 서로를 기다리며 영원히 멈춤

4. 디버깅 어려움:
   문제가 재현이 안 됨 (Heisenbug)
```

**예: 레이스 컨디션**

```
// 공유 변수
int counter = 0;

// 스레드 1
void increment() {
    for (int i = 0; i < 1000; i++) {
        counter++;  // 읽기-수정-쓰기
    }
}

// 스레드 2
void increment() {
    for (int i = 0; i < 1000; i++) {
        counter++;
    }
}

// 기대: counter = 2000
// 실제: counter < 2000 (높은 확률)

이유:
T1: 읽기 (counter = 5)
T2: 읽기 (counter = 5)  ← 같은 값
T1: 수정 (5 + 1 = 6)
T2: 수정 (5 + 1 = 6)    ← 같은 결과
T1: 쓰기 (counter = 6)
T2: 쓰기 (counter = 6)  ← 하나 잃음
```

## 정리

- **공짜 점심은 끝났다** — 클럭 속도 한계로 멀티코어 시대
- **동시성 ≠ 병렬성** — 구조 vs 실행
- **Amdahl의 법칙** — 순차 부분이 병목
- **도전** — 비결정성, 레이스, 데드락, 디버깅
- **이 시리즈** — 실용적 패턴 중심 접근
