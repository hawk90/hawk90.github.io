> **Working Effectively with Legacy Code** Chapter 6 요약

## 이 챕터의 메시지

기존 메서드 / 클래스를 테스트로 감쌀 시간은 없지만 — 변경은 해야 한다. 그렇다면 **새 코드만이라도 깨끗하게**.

```
원본 코드: 테스트 없음, 시간 없음
새 코드: 테스트 있음, 격리됨
```

원본은 그대로 두고 — 새 기능을 옆에 붙인다.

## Sprout Method — 새 메서드 싹 틔우기

기존 메서드에 코드를 추가해야 하는데 — 메서드 전체에 테스트를 쓸 시간이 없다.

```java
// 이전:
class Order {
    void postEntries(List<Entry> entries) {
        for (Entry e : entries) {
            // ... 기존 로직 ...
        }
    }
}

// 변경 요구: 중복 entry 제거.
// Sprout: 새 메서드로 분리, 테스트는 새 메서드에만.
class Order {
    void postEntries(List<Entry> entries) {
        List<Entry> deduped = uniqueEntries(entries);   // 새 호출
        for (Entry e : deduped) {
            // ... 기존 로직 그대로 ...
        }
    }

    // 새 메서드 — 순수 함수로 만들어 테스트 가능.
    List<Entry> uniqueEntries(List<Entry> entries) {
        // ...
    }
}
```

`uniqueEntries`에 대해 — 단위 테스트를 작성. 기존 `postEntries`는 손대지 않음.

**장점** — 새 코드는 테스트됨. 위험 격리.

**단점** — 원본의 부패는 그대로.

## Sprout Class — 새 클래스 싹 틔우기

새 메서드로 부족할 때 — 새 클래스를 만들고 호출.

```java
class Order {
    void postEntries(List<Entry> entries) {
        EntryProcessor processor = new EntryProcessor();   // 새 클래스
        List<Entry> processed = processor.process(entries);
        for (Entry e : processed) {
            // ... 기존 ...
        }
    }
}

class EntryProcessor {
    public List<Entry> process(List<Entry> entries) {
        // 깨끗한, 테스트된 코드
    }
}
```

원본 클래스에 의존성이 너무 많아 새 메서드조차 어렵다면 — 클래스 자체를 빼낸다.

## Wrap Method — 메서드 감싸기

기존 메서드의 호출 전/후에 동작을 추가해야 할 때.

```java
// 이전:
class Order {
    void pay() {
        // ... 결제 처리 ...
    }
}

// 변경 요구: 결제 후 영수증 전송.
// Wrap: 기존 메서드 이름 바꾸고 — 새 메서드가 감쌈.
class Order {
    void pay() {
        payProcess();
        sendReceipt();   // 새 동작
    }

    private void payProcess() {   // 원본의 이름만 변경
        // ... 기존 결제 처리 그대로 ...
    }
}
```

호출자는 `pay()`를 그대로 호출. 동작 확장됨.

## Wrap Class — 클래스 감싸기

비슷한 패턴을 클래스 수준에서 — **데코레이터 패턴**.

```java
class LoggingOrder extends Order {
    @Override
    public void pay() {
        log("paying");
        super.pay();
        log("paid");
    }
}
```

기존 `Order`는 변경하지 않음. 호출 측에서 `LoggingOrder`를 주입.

## 네 가지의 선택

```
변경이 메서드 안에서 끝남 → Sprout Method
새 클래스가 필요함 → Sprout Class
기존 동작의 전/후 추가 → Wrap Method
시스템 전반에 동작 추가 → Wrap Class
```

## 한계

- 원본은 — 계속 부패 중
- 단기적 우회. 장기적으로는 원본도 정리 필요
- 하지만 **지금 당장 안전하게 변경**이 우선

## 정리

- **Sprout Method** — 새 메서드 분리, 테스트
- **Sprout Class** — 새 클래스 분리, 테스트
- **Wrap Method** — 메서드를 감싸 동작 확장
- **Wrap Class** — 데코레이터로 시스템 확장
- 시간 없을 때 — **새 코드는 깨끗하게**
