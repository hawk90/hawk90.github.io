> **Working Effectively with Legacy Code** Chapter 13 요약

## 이 챕터의 메시지

코드가 — 무엇을 해야 하는지 모른다. 문서도 없고 명세도 없다.

> **Characterization Test** — 코드가 지금 *하고 있는* 동작을 그대로 받아 적는다.

## Characterization Test

```
의도 검증이 아니라:
"이 코드가 옳다"

현실 검증:
"이 코드가 지금 이렇게 동작한다"
```

옳은지 그른지 — 모른다. 다만 — **지금의 동작이 무엇인지** 고정.

## 작성법

```java
// 1단계 — 임의의 값으로 테스트.
@Test
void characterize() {
    Service s = new Service();
    int result = s.compute(42);
    assertEquals(/* 모름 */, result);   // ← 실패할 의도
}

// 2단계 — 실행. 실제 값이 나옴 (e.g. 123).

// 3단계 — 그 값을 — 그대로 받아 적음.
@Test
void characterize_compute_42_returns_123() {
    Service s = new Service();
    assertEquals(123, s.compute(42));
}
```

이 테스트가 — 통과. 의미는 **현재 동작 = 123**.

## 가치

- 변경 시 — 이 테스트가 깨지면 동작이 바뀐 것
- 그 변경이 의도된 것이면 → 테스트 업데이트
- 의도하지 않았으면 → 버그

**현 상태의 안전망**.

## 버그를 발견할 때

```java
// characterize 작성 중:
assertEquals(123, s.compute(42));   // 통과

// 그런데 — "이게 정말 옳은 결과?" 의문.
```

답이 — 명백히 잘못이면 (e.g. 음수 입력에 양수 결과) → **알려진 동작이 버그**.

```
이때 옵션:
1. 버그 그대로 — characterize (현 동작 고정)
2. 버그를 — 별도 이슈로 기록
3. 버그를 수정 — 단, 다른 의존이 없는 경우만
```

종종 — 버그가 **기능으로 의존되고 있을 수도**. 함부로 못 고침.

## 정리

- **Characterization** — 지금 동작을 받아쓴 테스트
- 옳고 그름 판단은 — 나중
- 변경 안전망의 역할
- 버그 발견 시 — 신중히 결정
