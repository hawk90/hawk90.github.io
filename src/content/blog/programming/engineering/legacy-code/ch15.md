> **Working Effectively with Legacy Code** Chapter 15 요약

## 이 챕터의 메시지

코드의 90%가 — 외부 API 호출. 통합 코드.

```java
void process() {
    AwsClient aws = new AwsClient(...);
    aws.connect();
    StripeClient stripe = new StripeClient(...);
    stripe.charge(...);
    EmailClient email = new EmailClient(...);
    email.send(...);
    // ... 모든 줄이 — 다른 시스템 호출
}
```

테스트할 — 비즈니스 로직 자체가 별로 없다.

## 두 가지 전략

### 1. Skin and Wrap the API

각 API를 — 우리 인터페이스로 감싼다 (ch14 패턴).

```java
interface Storage { void put(...); }
interface Payment { void charge(...); }
interface Notifier { void send(...); }

class Service {
    Service(Storage s, Payment p, Notifier n) { /* DI */ }
    void process() {
        storage.put(...);
        payment.charge(...);
        notifier.send(...);
    }
}
```

이제 `Service`의 흐름 — mock으로 테스트 가능.

### 2. Responsibility-Based Extraction

API 호출 안에 — 작은 책임들이 섞여 있음. 분리.

```java
// 이전:
void process() {
    // 데이터 변환 (우리 로직)
    Order order = transform(request);

    // API 호출
    awsClient.upload(order);

    // 응답 후처리 (우리 로직)
    Result result = parseResponse(awsClient.getLastResponse());
}

// 이후 — 책임 분리:
void process() {
    Order order = transformer.transform(request);   // 우리 로직 - 테스트 가능
    Response resp = uploader.upload(order);         // API - mock
    Result result = parser.parse(resp);             // 우리 로직 - 테스트 가능
}
```

API 호출 사이의 — **우리의 로직**을 분리. 그 부분만 테스트.

## 통합 테스트는 별도

API 호출 자체 — 통합 테스트로. 느리고 외부 의존이지만 필요.

```
Unit Test ─── Wrapper로 격리한 비즈니스 로직
Integration Test ─── 실제 API와의 연결
```

두 종류 — 다른 목적.

## 정리

- API 위주 코드 = 흔한 패턴
- **Skin and Wrap** — API 추상화
- **Responsibility Extraction** — 우리 로직 분리
- Integration test로 — 실제 연결 검증
