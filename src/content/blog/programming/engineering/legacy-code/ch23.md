> **Working Effectively with Legacy Code** Chapter 23 요약

## 이 챕터의 메시지

테스트가 — 부족. 변경이 안전한지 — 어떻게 확신?

> 비공식 기법들 — 자동 검증을 *보완*.

## Hyperaware Editing

지금 — 무엇을 하고 있는지 — **극도로 의식**.

```
키 누를 때마다:
- 무엇을 바꾸려 했는가?
- 실제 무엇을 바꾸고 있는가?
- 의도와 일치하는가?
```

자동 조작 / 멍 때리며 — 절대 금지.

## Single-Goal Editing

한 세션에 — **한 가지 목표**만.

```
Bad:
"이거 고치고, 저것도 보기 좋게 정리하고, 이 이름도 바꾸고..."

Good:
"이 함수를 추출한다. 다른 건 메모만."
```

여러 변경이 섞이면 — 어느 변경이 문제인지 — 추적 어려움.

## Preserve Signatures

리팩터링 시 — 시그니처를 — **보존**.

```java
// 이전:
int compute(int x, int y) { /* ... */ }

// Extract — 이름은 바꾸지만 시그니처 그대로:
private int computeInternal(int x, int y) { /* ... */ }
public int compute(int x, int y) {
    return computeInternal(x, y);
}
```

호출자 — 변경 없음. 영향 범위 — 최소.

## Lean on the Compiler

타입 시스템 — 검증자.

```java
// 메서드 시그니처 변경:
int compute(Order o)  →  int compute(Customer c)

// 컴파일러 — 모든 호출처 보고
// 일일이 찾아 헤맬 필요 없음
```

타입 변경 / 메서드 이름 변경 — 컴파일러가 모든 영향처를 알려줌. C / C++ / Java / C# 등 — 정적 타입 언어의 장점.

## Pair Programming

또 다른 — 검증 방법. 동료가 — 옆에서 본다.

```
"이 변경 — 정말 시그니처만 바꾼 거지?"
"네, 호출자는 안 건드림."
```

실수 — 즉시 발견.

## 정리

- 자동 테스트가 — 1순위
- 보완 기법들:
  - **Hyperaware** — 의식적 편집
  - **Single-Goal** — 한 가지 목표
  - **Preserve Signatures** — 시그니처 보존
  - **Lean on Compiler** — 컴파일러 활용
- Pair Programming — 사람의 검증
