> **Working Effectively with Legacy Code** Chapter 22 요약

## 이 챕터의 메시지

500줄 메서드 — 테스트 없이 바꿔야 한다.

```java
void doEverything() {
    // 줄 1 ~ 500
}
```

## 괴물의 종류

### Bulleted Method

```java
void process() {
    step1();
    step2();
    step3();
    // ...
    step20();
}
```

서로 — 비교적 독립. **차례 그대로 호출** 형태.

→ Extract Method 안전. 각 step이 — 자기 단위.

### Snarled Method

```java
void process() {
    int a = compute();
    while (a > 0) {
        if (cond) {
            for (...) {
                a = update(...);
            }
        }
    }
    // 흐름이 — 얽힘
}
```

논리가 — 얽힘. 분해가 어려움.

## 분해 전략

### Skeletonize

논리 구조의 — 뼈대만 남긴다.

```java
// 원본:
if (cond1) {
    // 100줄
    if (cond2) {
        // 200줄
    }
}

// Skeleton:
if (cond1) {
    handleCond1();      // 100줄 → 메서드 호출로 치환
    if (cond2) {
        handleCond2();
    }
}
```

조건 / 루프 구조는 — 그대로. 안의 내용은 — 메서드로.

### Find Sequences

연속된 코드 블록을 — 식별.

```java
// 50줄을 본다 — 어디서 어디까지가 한 단위?
// 변수의 사용 범위 / 빈 줄 / 주석으로 단위 추정
// 단위 단위로 — Extract Method
```

### IDE에 의존

Refactoring IDE의 Extract Method — 자동.

```
1. 추출할 코드 블록 선택
2. Refactor → Extract Method
3. 이름 지정
4. IDE가 — 매개변수 / 반환값 자동 처리
```

**의미 보존이 보장됨** (IDE의 약속).

## 변수 추적

```java
void monster() {
    int x = 1;
    // ... 100줄 ...
    int y = process(x);
    // ... 100줄 ...
    output(y);
}
```

각 변수의 — **사용 범위**를 추적. 사용 범위가 — Extract 단위 힌트.

## 정리

- 괴물 메서드 — 테스트 없이 분해 가능 (IDE 신뢰)
- **Skeletonize** — 구조 유지, 내용은 메서드로
- **Find Sequences** — 단위 식별
- Refactoring IDE에 의존
- 변수 사용 범위로 — 추출 단위 힌트
