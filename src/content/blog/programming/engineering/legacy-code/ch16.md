> **Working Effectively with Legacy Code** Chapter 16 요약

## 이 챕터의 메시지

코드를 — 이해 못 한다. 변경할 자신이 없다.

> **이해하지 못한 채 — 변경하지 말 것.**

이해를 돕는 — 몇 가지 기법.

## Notes / Sketching

종이 / 화이트보드에 — 그린다.

```
[OrderService]
    ├─ Customer (멤버)
    ├─ Items (List)
    └─ Total
        ↑
        calculate() ─ Tax 계산 호출
                       ↓
                   [TaxService] ← 외부 lib
```

코드를 — 시각적 모형으로. 의외로 명확해진다.

## Listing Markup

코드를 — **출력**해서 펜으로 표시.

```
1: void process() {                    [ ENTRY ]
2:     validate(request);              [ check ]
3:     if (request.isValid()) {        [ branch1 ]
4:         transform(request);
5:         save(request);
6:     } else {                        [ branch2 ]
7:         log("invalid");
8:     }
9: }                                   [ EXIT ]
```

브랜치 / 호출 / 책임 — 표시. 구조가 — 보인다.

## Scratch Refactoring

이해를 위해 — **버릴 작정으로 리팩터링**.

```
1. 새 브랜치
2. 거리낌없이 리팩터링 (테스트 없어도)
3. 코드의 의미가 — 드러남
4. 끝나면 — 브랜치 버림 (실제로는 머지 안 함)
5. 본 작업에서 — 알게 된 구조로 다시 정식 리팩터링
```

**투자한 시간은 — 이해의 비용**.

## Delete Unused Code

이해를 — 어렵게 하는 것은 **죽은 코드**.

```java
void process() {
    // 5년 전 코드 — 안 쓰임
    legacyA();
    legacyB();

    // 진짜 동작:
    realLogic();
}
```

쓰이지 않는 코드 — 삭제. 남은 코드의 의미가 — 명확.

```
git log으로 — 언제든 복구 가능
```

## 정리

- 이해 못 한 채 변경 — 금지
- **Sketching** — 시각적 모형
- **Listing Markup** — 종이 + 펜
- **Scratch Refactoring** — 학습 목적 리팩터링
- 죽은 코드 — 삭제
