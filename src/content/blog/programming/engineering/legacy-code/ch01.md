> **Working Effectively with Legacy Code** Chapter 1 요약

## 이 챕터의 메시지

Feathers의 정의가 책의 출발점이다.

> **Legacy code = 테스트가 없는 코드.**

오래된 코드가 아니다. 못 짠 코드도 아니다. **테스트가 없어서 변경이 두려운 코드** — 그게 legacy다.

## 네 가지 변경 이유

소프트웨어를 바꾸는 이유는 네 가지뿐이다.

1. **새 기능 추가**
2. **버그 수정**
3. **디자인 개선** (리팩터링)
4. **자원 사용 최적화** (성능)

네 가지 모두 같은 형태로 본다 — **기존 동작을 유지하면서 다른 동작을 추가하거나 다른 형태로**.

## 변경의 위험

기존 코드를 만질 때 묻는다.

- 어떤 변경을 할지
- 변경이 옳은지 어떻게 알지
- 다른 곳이 깨지지 않을지 어떻게 알지

테스트가 없으면 — **마지막 질문에 답할 방법이 없다**. 그래서 두려움이 생긴다.

## 두려움이 만드는 행동

두려움에 빠진 개발자는.

- 변경을 안 한다 (코드가 굳음)
- 한다면 최소한만 (의도 안 한 결과)
- 새 코드를 옆에 짠다 (duplication)
- 리팩터링을 안 한다 (점점 망가짐)

악순환. 시간이 갈수록 더 두려워진다.

## 책의 약속

이 책의 약속:

> **테스트 없는 코드를 — 테스트 있는 코드로 만드는 기술.**

그게 가능하면 두려움이 사라진다. 두려움이 사라지면 변경이 가능해진다.

## 정리

- Legacy = **테스트 없는 코드**
- 변경 이유 네 가지 — 기능 / 버그 / 디자인 / 성능
- 테스트 없음 → 두려움 → 악순환
- 책의 약속 — 테스트 없는 코드를 테스트 있게
