> **Working Effectively with Legacy Code** Chapter 24 요약

## 이 챕터의 메시지

코드베이스 — 백만 줄. 한 번도 — 정리된 적 없음. 압도된다.

> "다 정리한다"는 — 환상. 작게, 꾸준히.

## 큰 그림 vs 매일의 변경

```
큰 그림: "전체 아키텍처를 재설계해야 해"
현실: 매주 5개의 새 버그 / 새 기능 요구
```

전체 재설계 — 거의 불가능. 그 사이에도 — 코드는 더 더러워짐.

**해법** — 매일의 변경에 — 좋은 습관.

## Boy Scout Rule

> 캠핑장을 — 떠날 때보다 깨끗하게.

코드도 — 마찬가지.

```java
// 버그 수정을 위해 메서드를 본다 → 한 줄 추가하면서:
// - 변수 이름 하나를 — 더 좋게
// - 의미 없는 주석 — 제거
// - 한 메서드를 — 추출
```

**1% 개선**이 — 누적되면 — 시간이 지나 좋아진다.

## 작게 / 자주

```
× 한 달 동안 — 대규모 리팩터링 브랜치
○ 매일 — 작은 정리 커밋
```

큰 브랜치 — 충돌 폭탄. 머지 안 됨.

작은 변경 — 빠른 머지. 위험 적음.

## 보이는 곳부터

```
1. 자주 변경하는 영역 (Hot Spot)
   → 정리의 ROI 높음

2. 새 기능 추가 영역
   → 어차피 손대니까 — 정리

3. 자주 안 가는 영역
   → 손 안 댐 (필요할 때만)
```

**파레토** — 20%의 영역에 — 80%의 작업이 집중. 그 20%를 정리.

## 팀의 분위기

압도감은 — 개인의 문제가 아니라 — **팀의 문제**.

```
- 정리에 대한 — 인정
- 작은 개선의 — 공유
- "아무도 안 고치는 코드"의 영광 거부
```

코드 품질 — **팀 책임**.

## 정리

- 백만 줄 — 한 번에 못 정리
- **Boy Scout Rule** — 떠날 때 깨끗하게
- 작게 / 자주
- Hot Spot 우선
- 팀 문화로 — 분위기 조성
