> **Working Effectively with Legacy Code** Chapter 17 요약

## 이 챕터의 메시지

애플리케이션 — 구조가 없다. 모든 것이 모든 것과 연결.

> 구조는 — 발견되는 것. 추상화를 통해.

## Telling the Story

팀원에게 — "이 시스템이 뭘 하는지" 설명.

```
"음... Order가 들어오고...
 그게 Customer에 연결되고...
 Inventory를 확인하고...
 그리고 또..."
```

설명이 — 막힌다. 막히는 자리가 — **구조의 결함**.

설명을 — **부드럽게 할 수 있게** 구조를 재배열.

```
"Order를 받는다 → Validation → Processor → Delivery"

이 네 단계의 모듈 / 책임을 — 만든다.
```

## CRC Cards

**C**lass **R**esponsibility **C**ollaborator.

카드 한 장에 — 한 클래스. 클래스 별로:

```
[Order]
Responsibilities:
- 항목 추가/삭제
- 총액 계산
Collaborators:
- Customer
- Item
- Inventory
```

- 책임이 너무 많으면 → 분리
- Collaborator가 너무 많으면 → 응집도 낮음

종이로 — 빠른 모델링.

## Naming Components

이름이 — 구조를 만든다.

```
이전: Service, Manager, Helper, Util
이후: PaymentValidator, OrderTransformer, InventoryChecker
```

각 이름에 — 단일 책임. 책임이 흩어지면 — 이름이 어색해짐 → 분리 신호.

## 점진적 도입

전체를 — 한 번에 못 고친다.

```
1. 작은 책임을 — 추출
2. 다음 작은 책임을 — 추출
3. 반복
4. 점차 — 구조가 드러남
```

큰 그림은 — 처음엔 보이지 않음. 만들어가며 드러난다.

## 정리

- 구조 없는 시스템 = 흔한 Legacy
- **Telling the Story** — 막히는 자리가 결함
- **CRC** — 책임 모델링
- **Naming** — 이름이 구조 유도
- 점진적 도입
