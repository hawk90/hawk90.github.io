> **Working Effectively with Legacy Code** Chapter 21 요약

## 이 챕터의 메시지

하나의 비즈니스 규칙이 — 바뀌었다. 그런데 코드 — 7군데에 흩어져 있어서 — 7번 바꿔야 한다.

> **중복** = Legacy의 핵심 문제.

## 중복의 형태

### Type 1 — 글자 그대로 복사

```java
// PaymentA.java
int discount = (level >= 3) ? amount * 9 / 10 : amount;

// PaymentB.java
int discount = (level >= 3) ? amount * 9 / 10 : amount;   // 같음
```

**Extract Method**로 한 곳에 모음.

### Type 2 — 구조는 같고 변수만 다름

```java
// 메서드 A:
for (Item i : itemsA) { processA(i); }

// 메서드 B:
for (Order o : ordersB) { processB(o); }
```

루프 패턴 동일. 추상화 가능.

### Type 3 — 비슷한 로직

```java
class PaymentA {
    int compute() { /* 비슷 */ }
}
class PaymentB {
    int compute() { /* 비슷 */ }
}
```

**Pull Up Method** — 공통 슈퍼클래스로.

## 단계적 제거

```
1. 가장 명확한 중복 찾기 (Type 1)
2. Extract Method → 한 곳에
3. 다른 곳의 호출을 — 점진적으로 마이그레이션
4. 모두 한 곳을 호출하면 — 다른 곳들 삭제
5. 다음 중복으로
```

## 변경의 효과

```
이전: 비즈니스 규칙 변경 → 7곳 수정
이후: 비즈니스 규칙 변경 → 1곳 수정
```

**변경 비용** = 1/7. 더 중요한 건 — 누락 가능성이 사라진다.

## 정리

- 중복 = Legacy의 핵심 문제
- **Extract Method** — 단순 복사
- **Pull Up Method** — 비슷한 로직
- 점진적 제거 — 한 번에 못 함
- 변경 비용을 — 결정적으로 낮춤
