> **Working Effectively with Legacy Code** Chapter 20 요약

## 이 챕터의 메시지

이미 — 5000줄짜리 클래스. 더 키우면 안 된다.

```
새 기능 추가 → 또 이 클래스에?
```

새 기능을 — **새 클래스로**. 큰 클래스는 — 점진적으로 분해.

## SRP 위반의 신호

```
- 메서드가 100개 이상
- 멤버 변수가 50개 이상
- 다른 클래스를 — 만들고/사용/관리 모두
- import / using 문이 — 50줄 이상
```

**책임이 여러 개** = SRP 위반.

## 책임 식별

메서드 이름을 — 죽 본다.

```java
class GodService {
    void validateOrder()    // 검증 책임
    void saveOrder()        // 저장 책임
    void emailUser()        // 알림 책임
    void calculateTax()     // 계산 책임
    void logActivity()      // 로깅 책임
    // ...
}
```

그룹지을 수 있다 — 별도 클래스 후보.

```
[OrderValidator] - validate
[OrderRepository] - save
[Notifier] - email
[TaxCalculator] - calculate
[ActivityLogger] - log
```

## Extract Class

한 책임씩 — 빼낸다.

```java
// 1단계 — 메서드 그룹 식별
// 2단계 — 새 클래스 생성
class OrderValidator {
    boolean validate(Order o) { /* ... */ }
}

// 3단계 — GodService에서 위임
class GodService {
    OrderValidator validator = new OrderValidator();
    void validateOrder(Order o) {
        boolean ok = validator.validate(o);
        // ...
    }
    // 다른 책임은 — 차차 추출
}

// 4단계 — 호출자 점진적 마이그레이션
```

5000줄짜리 — 한 번에 못 깬다. **한 책임씩**.

## 신규 기능

새 기능은 — God Class에 더하지 않는다.

```
이전 패턴: GodService.newFeature() 추가
새 패턴: NewFeatureService 만들고 GodService에서 호출
```

God Class의 — 추가 부담을 막음.

## 정리

- 큰 클래스 = SRP 위반
- 메서드 이름으로 — 책임 식별
- **Extract Class** — 한 책임씩
- 새 기능은 — 새 클래스로
