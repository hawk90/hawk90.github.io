> **Working Effectively with Legacy Code** Chapter 5 요약

## 이 챕터의 메시지

좋은 도구가 — Legacy 작업을 가능하게 한다.

## 필수 도구들

### 1. 자동화된 Refactoring IDE

- **IntelliJ IDEA** (Java)
- **Visual Studio** + ReSharper (C#)
- **CLion** (C++)
- **VSCode** + 플러그인 (다양)

**Extract Method, Rename, Move** 같은 리팩터링이 자동. **테스트가 부족한 코드**에 매우 중요 — 손으로 리팩터링은 위험.

### 2. xUnit 프레임워크

- **JUnit** (Java)
- **NUnit** (.NET)
- **pytest** (Python)
- **GoogleTest** (C++)
- **RSpec** (Ruby)

테스트 작성의 표준 도구.

### 3. Mock 프레임워크

- **Mockito** (Java)
- **Moq** (.NET)
- **unittest.mock** (Python)
- **gMock** (C++)

복잡한 의존을 — mock으로 대체. Legacy 코드에서 거의 필수.

### 4. Code Coverage 도구

테스트가 얼마나 코드를 커버하는지.

- **JaCoCo** (Java)
- **coverage.py** (Python)
- **gcov** (C/C++)

100% 커버리지가 목표가 아니라 — **변경할 영역의 커버리지**가 충분한지 확인.

## "Safe Refactoring"의 조건

리팩터링 IDE의 자동 변환은 — **테스트 없이도 안전**해야 한다.

```
이름 변경, 메서드 추출 같은 변환은 — 의미 보존 보장
```

이런 변환을 활용하면 — 테스트 작성 전에도 어느 정도 정리 가능.

다만 — **자동이라도 검증은 필요**. IDE 버그 / 사용자 실수 가능.

## 정리

- **Refactoring IDE** — 자동 변환의 안전망
- **xUnit** — 테스트 표준
- **Mock 프레임워크** — 의존 격리
- **Coverage** — 변경 영역 확인
- 자동 변환은 신뢰할 만하지만 — 검증 필요
