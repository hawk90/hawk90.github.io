> **Working Effectively with Legacy Code** Chapter 10 요약

## 이 챕터의 메시지

클래스는 만들었는데 — 테스트하고 싶은 메서드가 호출 어려움.

```
- private이다 → 접근 불가
- 너무 큼 → 입력/출력 추적 어려움
- 부작용 많음 → 격리 어려움
```

## Private 메서드

private 메서드를 — 테스트하고 싶다.

### Subclass and Override

`private`을 `protected`로.

```java
// 이전:
class Service {
    private int calculate(int x) { return x * 2; }
}

// 이후:
class Service {
    protected int calculate(int x) { return x * 2; }
}

// 테스트:
class TestService extends Service {
    public int testCalculate(int x) {
        return calculate(x);
    }
}
```

또는 — 패키지 private(`default`)로. 같은 패키지의 테스트 클래스에서 호출 가능.

### Expose Static Method

private이 — 정적이고 상태 의존 없으면, public static으로 추출.

```java
// 이전:
class Order {
    private int discount(int total, int level) { /* ... */ }
}

// 이후:
class Order {
    int discount(int total, int level) {
        return DiscountUtil.calculate(total, level);
    }
}

class DiscountUtil {
    public static int calculate(int total, int level) { /* ... */ }
}
```

`DiscountUtil.calculate` — 독립 테스트 가능.

## 거대한 메서드

100줄짜리 메서드 — 무엇을 테스트하나.

### Extract Method 시도

```java
void process() {
    // step1 (30줄)
    // step2 (40줄)
    // step3 (30줄)
}
```

각 step을 — 메서드로 추출. 각 메서드에 테스트.

문제 — 추출 자체가 위험 (Refactoring IDE 도움). 일부 단계는 — Sensing Variable 필요.

### Sensing Variable

```java
void process() {
    int intermediate = step1();   // 중간 상태 노출
    setLastIntermediate(intermediate);   // ← 테스트 위해
    step2(intermediate);
}

// 테스트:
service.process();
assertEquals(42, service.getLastIntermediate());
```

추후 — Sensing 변수 제거.

## Pass-Through 메서드

테스트하고 싶은 코드가 — 깊이 묻혀있다.

```java
class A {
    void doIt() { b.doIt(); }
}
class B {
    void doIt() { c.realLogic(); }
}
class C {
    private void realLogic() { /* ← 여기 */ }
}
```

A를 통해서만 호출 가능하지만 — A의 의존성이 너무 많아 인스턴스화 불가.

**해결** — C를 직접 인스턴스화. 그리고 `realLogic`을 노출.

## 정리

- private → Subclass and Override / protected
- 정적 분리 가능 → Expose Static Method
- 거대 메서드 → Extract Method + Sensing Variable
- 깊은 호출 → 직접 인스턴스화
