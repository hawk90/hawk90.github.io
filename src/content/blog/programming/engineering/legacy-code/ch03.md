> **Working Effectively with Legacy Code** Chapter 3 요약

## 이 챕터의 메시지

코드를 테스트하려면 두 가지가 필요하다.

1. **Sensing** — 코드의 결과를 감지할 수 있는가?
2. **Separation** — 대상을 다른 것에서 분리할 수 있는가?

Legacy 코드의 어려움은 — 이 둘 중 하나 또는 둘 다 없다는 것.

## Sensing — 감지

```java
class Calculator {
    void compute() {
        // 결과가 어디로? 화면? 파일? 다른 객체?
        // 테스트에서 어떻게 검증?
    }
}
```

`compute()`의 결과를 — 어떻게 감지하나?

**해법**:
- 메서드가 값을 반환하게 (return type 변경)
- 사이드 이펙트를 — 검증 가능한 객체로 흘림
- Mock으로 — 인자로 받은 의존을 대체

## Separation — 분리

```java
class OrderProcessor {
    void process(Order o) {
        Database.save(o);            // 직접 DB 호출
        EmailService.notify(o);       // 직접 이메일 호출
        // 테스트에서 진짜 DB / 이메일 호출하기 싫음
    }
}
```

Database / EmailService를 — 어떻게 분리하나?

**해법**:
- 의존성 주입 (생성자, setter)
- 인터페이스 추출
- Mock 객체

## 두 문제는 자주 함께 온다

```java
class System {
    void run() {
        // 직접 호출 + 검증 어려움
        Logger.log("started");
        DB.update(...);
        Email.send(...);
    }
}
```

- Separation 안 됨 — Logger, DB, Email이 글로벌
- Sensing 안 됨 — 결과가 외부로 흘러나감

해법은 둘 다 — **의존성을 명시적으로 받기**.

## "Object Seam"

Feathers의 핵심 개념 — **seam** (이음새). 코드를 변경 없이 동작을 바꿀 수 있는 지점.

```java
class System {
    Logger logger;
    Database db;
    Email email;
    
    System(Logger l, Database d, Email e) { ... }  // seam
    
    void run() {
        logger.log("started");   // 같은 코드, 다른 객체 가능
        db.update(...);
        email.send(...);
    }
}
```

생성자가 seam — 테스트에서는 mock 주입, 운영에서는 진짜 객체. 같은 코드.

## 정리

- 테스트하려면 — **Sensing** + **Separation**
- Sensing — 결과 감지
- Separation — 대상 분리
- 해법 — **의존성 주입 + 인터페이스 + Mock**
- **Seam** — 코드 변경 없이 동작 변경 가능 지점
