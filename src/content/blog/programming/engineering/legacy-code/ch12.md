> **Working Effectively with Legacy Code** Chapter 12 요약

## 이 챕터의 메시지

한 영역에 여러 변경이 — 모이는 경우. 각각 따로 테스트하면 — 비효율.

> **Pinch Point** — 영향들이 모이는 좁은 지점.

## Pinch Point의 정의

```
변경1 ──→ 영향A ┐
변경2 ──→ 영향B ┼──→ Pinch Point ──→ 결과
변경3 ──→ 영향C ┘
```

여러 변경의 영향이 — 한 지점을 통과. 그 지점에 — **하나의 테스트**.

## 예시

```java
class Order {
    void addItem(Item item) { items.add(item); }       // 변경 1 예정
    void applyDiscount(int level) { /* ... */ }        // 변경 2 예정
    void applyPromotion(String code) { /* ... */ }     // 변경 3 예정

    int calculateTotal() {   // ← Pinch Point
        // items, discount, promotion이 모두 모이는 곳
    }
}
```

세 변경 모두 — `calculateTotal`의 결과에 영향.

**전략** — `calculateTotal`에 대해 테스트 작성. 변경 후에도 — 이 테스트가 검증.

## 찾는 법

1. 각 변경에 대해 — Effect Sketch
2. 영향들이 — 어디서 만나는가?
3. 그 지점 = Pinch Point
4. 그 지점에 — 집중 테스트

## 이점

- 적은 테스트로 — 넓은 변경 검증
- Pinch Point가 — 자연스러운 경계
- 더 작은 단위 테스트는 — 나중에 보강

## 한계

- 모든 변경이 Pinch Point를 만들지는 않음
- 너무 큰 메서드면 — Pinch Point도 큼
- 작은 단위로 분해 후 — 다시 찾기

## 정리

- **Pinch Point** — 영향이 모이는 좁은 지점
- 한 테스트로 여러 변경 검증
- Effect Sketch + Pinch Point = 효율적 테스트 전략

## 다음 장 예고

다음 장 — **어떤 테스트를 써야 하나**. 변경 자체가 보이지 않을 때.
