> **Working Effectively with Legacy Code** Chapter 8 요약

## 이 챕터의 메시지

새 기능을 추가하는 — 두 가지 방법.

```
1. TDD — 테스트 먼저, 코드 다음
2. Programming by Difference — 상속으로 차이만 추가
```

## TDD in Legacy

TDD는 — Legacy에서도 유효. 다만 적용이 어려움.

```
new class / new code → TDD 자연스럽게
existing code modification → seam 필요
```

**새 기능**이 새 클래스로 표현 가능하다면 — TDD 적용. 작은 단위로.

### TDD Cycle

```
1. Red — 실패하는 테스트 작성
2. Green — 최소 코드로 통과
3. Refactor — 정리
```

Legacy 환경에서 — 이 사이클이 격리됨. 기존 코드는 영향 없음.

```java
// 새 기능: 할인 계산
@Test
void discount_for_vip_customer() {
    Discount d = new DiscountCalculator();
    assertEquals(100, d.calculate(1000, CustomerType.VIP));
}

// 새 클래스 — 기존 시스템에 영향 없음
class DiscountCalculator { /* ... */ }
```

기존 `Order`에 통합은 — 나중에 (sprout method로).

## Programming by Difference

다형성 / 상속을 활용 — 기존 동작을 유지하면서 새 동작 추가.

```java
class StandardPayment {
    int total() { return amount; }
}

// 새 요구: VIP 고객은 10% 할인.
// Programming by Difference — 차이만 표현:
class VIPPayment extends StandardPayment {
    @Override
    int total() {
        return super.total() * 9 / 10;   // 차이만 표현
    }
}
```

기존 코드는 — 그대로. 새 클래스가 차이만 표현.

호출 측에서 적절한 타입을 선택.

### 주의 — 임시 해결책

Programming by Difference는 — **빠른 임시 해결책**.

```
장점: 기존 코드 변경 없음
단점: 상속 계층이 깊어지면 — 이해 어려움
```

Feathers의 권고 — 일단 working code로 만들고 — **나중에 리팩터링** (composition over inheritance).

## 두 가지의 조합

```
새 클래스가 가능 → TDD
기존 클래스의 변형 → Programming by Difference
나중에 정리 → Refactoring (Replace Inheritance with Delegation 등)
```

## 정리

- 새 기능 추가 = **TDD** (가능하면)
- 또는 **Programming by Difference** (상속으로 차이)
- 둘 다 기존 코드의 영향 최소화
- 상속은 임시 — 나중에 정리
