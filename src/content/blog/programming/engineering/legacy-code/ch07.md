> **Working Effectively with Legacy Code** Chapter 7 요약

## 이 챕터의 메시지

코드 한 줄 바꾸면 — 빌드 + 테스트가 30분. 변경 자체가 비현실적이다.

**원인** — 강한 의존성. 작은 변경이 시스템 전체를 다시 빌드.

## 시간이 길어지는 이유

```
의존성이 단단할수록:
- 한 클래스의 변경 → 많은 클래스 재컴파일
- 한 모듈의 변경 → 전체 통합 테스트 필요
- 한 메서드의 변경 → 큰 환경 셋업 필요
```

**짧은 피드백 루프** — 변경의 핵심.

## 의존성 깨기

물리적 / 컴파일 의존성을 깨야 — 작은 단위로 작업 가능.

### 인터페이스 추출

```java
// 이전:
class PaymentService {
    void pay() { /* ... */ }
}
class Order {
    PaymentService payment;   // 구체 클래스에 의존
}

// 이후 — 인터페이스 추출:
interface IPayment {
    void pay();
}
class PaymentService implements IPayment { /* ... */ }
class Order {
    IPayment payment;   // 인터페이스에 의존
}
```

이제 `PaymentService` 변경이 — `Order`의 재컴파일을 요구하지 않음 (인터페이스가 안정적이라면).

### 모듈 분리

큰 프로젝트 — 여러 jar / dll / module로 분리. 각 모듈 독립 빌드.

```
core.jar (안 바뀜)
business.jar (바뀜)
ui.jar (바뀜)
```

UI 작업 시 — core만 빌드 캐시. 빌드 시간 1/10.

## 테스트 격리

```java
// 이전 — 전체 환경 필요:
@Test
void test() {
    Application app = new Application();
    app.initialize();   // 30초
    // ... 테스트
}

// 이후 — 단위 테스트:
@Test
void test() {
    Order order = new Order(mockPayment);
    order.pay();
    // 0.01초
}
```

**Sensing / Separation**으로 의존성 제거 → 독립적 단위 테스트.

## Build Dependencies

C++에서 특히 심각.

```cpp
// 이전:
class Service {
    Database db;   // db.h 포함 필요 → 컴파일 의존성 폭증
};

// 이후 — Pimpl:
class Service {
    struct Impl;
    std::unique_ptr<Impl> pimpl;
};
// service.cpp에서만 db.h 포함
```

**Pimpl / Forward Declaration** — 헤더 의존성 깨기.

## 정리

- 빌드/테스트 시간 = 변경의 마찰
- 길어지는 이유 = **의존성**
- 인터페이스 추출, 모듈 분리, Pimpl
- 격리된 단위 테스트로 빠른 피드백
- 짧은 피드백 루프 = Legacy 작업의 토대
