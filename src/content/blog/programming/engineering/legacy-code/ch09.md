> **Working Effectively with Legacy Code** Chapter 9 요약

## 이 챕터의 메시지

테스트 하네스에 — 클래스를 인스턴스화. 이 단계에서 막히는 경우가 많다.

```
new MyClass(/* 50개의 파라미터 */);   // 어떻게 만드나?
```

## 흔한 문제

### 1. 의존성이 너무 많은 생성자

```java
class OrderService {
    OrderService(Database db, Logger log, EmailService email,
                 PaymentGateway gw, Cache cache, Config cfg) {
        // ...
    }
}
```

테스트에서 — 6개 객체 모두 만들어야 함. 각각이 또 의존성 있음.

**해결** — **Parameterize Constructor**.

```java
// 기존 생성자 유지:
OrderService() {
    this(new Database(), new Logger(), /* ... */);
}

// 테스트용 생성자 추가:
OrderService(Database db, Logger log, /* ... */) {
    this.db = db;
    this.log = log;
    // ...
}
```

테스트는 — mock 주입.

### 2. 숨겨진 의존성

```java
class OrderService {
    OrderService() {
        this.db = new Database();   // 생성자 안에서 — 외부 시스템 호출!
    }
}
```

`new OrderService()` 호출만으로 — 데이터베이스 연결 시도. 테스트 실패.

**해결** — **Extract and Override Factory Method**.

```java
class OrderService {
    OrderService() {
        this.db = createDatabase();
    }
    protected Database createDatabase() {   // ← 오버라이드 가능
        return new Database();
    }
}

// 테스트:
class TestOrderService extends OrderService {
    @Override
    protected Database createDatabase() {
        return new MockDatabase();   // ← 다른 객체 반환
    }
}
```

### 3. 부작용이 있는 생성자

```java
class Service {
    Service() {
        registerWithSystem();   // 글로벌 상태 변경
        startBackgroundThread();
    }
}
```

테스트마다 — 백그라운드 스레드. 격리 불가.

**해결** — **Supersede Instance Variable** 또는 **Subclass and Override**.

생성자의 부작용 — 별도 메서드로 분리.

```java
class Service {
    Service() {}   // 부작용 없는 빈 생성자
    void initialize() {   // 부작용은 별도
        registerWithSystem();
        startBackgroundThread();
    }
}
```

테스트 — 생성만 하고 initialize는 호출하지 않음.

## 일반 전략

1. **생성자를 단순화** — 의존성을 주입받게
2. **숨은 의존성을 노출** — factory method
3. **부작용을 분리** — initialize, start 메서드

## 정리

- 큰 생성자 → Parameterize Constructor
- 숨은 의존성 → Extract and Override Factory
- 부작용 생성자 → 별도 initialize 메서드
- 목표: **테스트 환경에서 인스턴스화 가능**
