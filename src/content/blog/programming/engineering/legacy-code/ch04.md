> **Working Effectively with Legacy Code** Chapter 4 요약

## 이 챕터의 메시지

Feathers의 핵심 개념을 정식화.

> **Seam** = 그 자리의 코드를 변경하지 않고 — 그 자리의 동작을 변경할 수 있는 지점.

테스트를 위한 모든 기술의 기반.

## Seam의 정의

```java
class Service {
    void run() {
        logger.log("running");   // ← 이 자리가 seam?
    }
}
```

- 이 줄을 변경하지 않고
- `logger.log`의 동작을 다른 것으로 바꿀 수 있는가?
- 가능 → seam

`logger`가 멤버 변수이고 생성자로 주입된다면 — **생성자가 seam**. 다른 logger 객체로 생성하면 동작이 바뀐다.

## 세 종류의 Seam

### 1. Preprocessing Seam

C/C++의 매크로 / `#define`.

```c
#define DATABASE_QUERY(q) actual_query(q)
// 테스트 시:
#define DATABASE_QUERY(q) mock_query(q)
```

컴파일 시 다른 코드로 치환. C 레거시 코드에서 자주.

**Enabling Point** — `#define` 자리. 그 한 줄로 동작이 바뀜.

### 2. Link Seam

링크 시점에 결정.

```c
// production: real_logger.o와 링크
// test: mock_logger.o와 링크
```

같은 함수 이름, 다른 구현체. 빌드 시스템이 선택.

**Enabling Point** — 링커 설정 / 빌드 스크립트.

### 3. Object Seam

객체 지향 언어의 다형성.

```java
class Service {
    Logger logger;
    Service(Logger l) { this.logger = l; }   // ← seam
}

// production: new Service(new FileLogger())
// test: new Service(new MockLogger())
```

**Enabling Point** — 객체 생성 시 (생성자 주입). 가장 일반적, 가장 깔끔.

## Seam의 가치

Legacy 코드에 — seam이 없으면 테스트 불가.

```
seam 없음 → 코드 변경 필요 → 위험
seam 있음 → 동작만 바꿈 → 안전
```

따라서 Legacy 작업의 첫 단계 — **seam을 찾거나 만든다**.

## Seam 만들기

기존 코드에 seam이 없으면 — 추가.

```java
// 이전 (seam 없음):
class Service {
    void run() {
        Logger.log("...");   // 글로벌 호출
    }
}

// 이후 (object seam 추가):
class Service {
    LoggerInterface logger;
    Service(LoggerInterface l) { this.logger = l; }   // seam!
    void run() {
        logger.log("...");
    }
}
```

생성자 추가 — 작은 변경, 큰 이득. 다음 장부터 다양한 기법을 다룬다.

## 정리

- **Seam** — 코드 변경 없이 동작 변경 가능 지점
- **Preprocessing** — 매크로 / #define
- **Link** — 빌드 시 다른 구현체
- **Object** — 다형성 / 의존성 주입
- 각 seam에 **Enabling Point** — 동작을 바꾸는 자리
- Legacy 작업의 핵심 — **seam 찾기 / 만들기**
