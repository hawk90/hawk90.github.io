> **Working Effectively with Legacy Code** Chapter 2 요약

## 이 챕터의 메시지

변경의 두 가지 방식.

> **Edit and Pray** (편집 후 기도) vs **Cover and Modify** (보호 후 수정).

Legacy 코드 작업의 정신은 — 후자.

## Edit and Pray

대부분의 개발자가 무의식적으로 하는 방식.

1. 코드를 변경한다
2. 잘 됐기를 빈다
3. 운영에서 깨지면 디버깅

문제 — **피드백이 늦다**. 깨졌다는 신호가 며칠 / 몇 주 / 몇 달 후.

## Cover and Modify

1. 변경할 영역에 **테스트 먼저** 추가 (cover)
2. 그 다음 변경 (modify)
3. 테스트가 깨지면 — 즉시 발견

피드백이 **초 단위**. 변경이 안전.

## 두 종류의 테스트

테스트의 목적 두 가지.

**1. Correctness 검증 (전통적)**

코드가 옳은지 확인.

**2. 안전망 (Feathers의 강조)**

변경 시 깨지는지 즉시 알려 줌.

Legacy 코드에서는 — 두 번째가 더 중요. **이미 동작하는 것을 깨뜨리지 않게** 보호.

## 테스트의 빠른 피드백

테스트가 1초 안에 실행되어야 — 매 변경 후 돌릴 수 있다.

- Unit test — ms 단위
- Integration test — 초 단위
- E2E test — 분 단위 (너무 느림)

Legacy 작업은 — **단위 테스트** 중심. 빠른 피드백이 핵심.

## 정리

- **Edit and Pray** = 피드백 없음 = 두려움
- **Cover and Modify** = 테스트 먼저 = 안전
- 테스트는 — 정확성 검증 + **안전망**
- **빠른 피드백** (단위 테스트)이 핵심
