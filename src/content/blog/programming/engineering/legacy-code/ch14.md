> **Working Effectively with Legacy Code** Chapter 14 요약

## 이 챕터의 메시지

외부 라이브러리 — final class, static method, sealed class. 직접 mock 불가.

```java
// 외부 lib의 final class:
final class ThirdPartyService {
    public String fetch() { /* ... */ }
}

// 직접 mock 불가 — Mockito가 final 못 mock (기본 설정)
```

## 해결 — Wrapper

라이브러리를 — **우리 인터페이스로 감싼다**.

```java
// 우리의 인터페이스:
interface Fetcher {
    String fetch();
}

// production:
class ThirdPartyFetcher implements Fetcher {
    private final ThirdPartyService service = new ThirdPartyService();
    @Override
    public String fetch() {
        return service.fetch();
    }
}

// test:
class MockFetcher implements Fetcher {
    @Override
    public String fetch() { return "mocked"; }
}
```

이제 우리 코드는 — `Fetcher`에 의존. 라이브러리 의존은 격리.

## 추가 이점

- 라이브러리 교체가 쉬움 (구현만 변경)
- 라이브러리 API 변경의 영향 — wrapper에만 국한
- 우리 도메인 언어로 — API 재명명 가능

```java
interface PaymentGateway {
    PaymentResult charge(int amount, Card card);   // 우리 도메인 어휘
}

// implementation
class StripeGateway implements PaymentGateway {
    // Stripe SDK의 복잡한 API → 우리 API로 단순화
}
```

## 정적 메서드

```java
// 라이브러리:
class Util {
    public static int compute(int x) { /* ... */ }
}

// 사용:
class Service {
    int run() {
        return Util.compute(42);   // 정적 — mock 어려움
    }
}
```

**해결** — Wrapper 인스턴스:

```java
interface UtilWrapper {
    int compute(int x);
}

class RealUtilWrapper implements UtilWrapper {
    @Override
    public int compute(int x) { return Util.compute(x); }
}

class Service {
    UtilWrapper util;
    Service(UtilWrapper u) { this.util = u; }
    int run() { return util.compute(42); }
}
```

## 정리

- 라이브러리 직접 의존 = 테스트 장애물
- **Wrapper로 격리** — 우리 인터페이스
- 정적도 동일 — wrapper 인스턴스
- 부수 효과: 도메인 어휘 재명명
