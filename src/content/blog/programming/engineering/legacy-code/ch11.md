> **Working Effectively with Legacy Code** Chapter 11 요약

## 이 챕터의 메시지

코드를 바꾸려고 한다. 바꾸는 자리에 — 테스트가 필요. 그런데 **무엇을 테스트하나**.

> **Effect Sketch** — 변경의 영향이 퍼지는 방향을 그림.

## Effect Sketch

```java
class Account {
    int balance;
    void deposit(int amount) {
        balance += amount;       // ← 변경 지점
    }
    int getBalance() { return balance; }
}
```

`deposit`을 바꾸려고 한다. 영향:

```
deposit() ──→ balance ──→ getBalance() ──→ 호출자들
```

`balance`가 영향을 받는 것 → `getBalance` 호출자 모두 — 잠재적 영향.

## 사용법

1. **변경 지점** 찾기 (e.g. `deposit`)
2. 이 지점에서 — 어떤 값이 바뀌는가? (`balance`)
3. 그 값을 — 누가 읽는가? (`getBalance`)
4. 그 메서드를 — 누가 호출하는가? (밖)
5. **테스트해야 할 메서드** — 영향 경로의 끝

```
deposit (변경)
    ↓
balance (멤버 변수)
    ↓
getBalance (읽음)
    ↓
TestPoint!   ← 여기 테스트
```

## 영향의 전파 규칙

- **값** → 그 값을 읽는 모든 곳
- **메서드** → 호출하는 모든 곳
- **수퍼클래스 변경** → 모든 서브클래스
- **인터페이스 변경** → 모든 구현
- **글로벌** → 시스템 전체

```
글로벌 변수 = 영향 추적 가장 어려움
```

## 단순화

큰 클래스 — 영향 그래프가 거대.

**Encapsulation**으로 — 영향 범위를 좁힌다.

```java
// 이전:
class Service {
    public int state;   // 누가 읽는지 모름
}

// 이후:
class Service {
    private int state;
    public int getState() { return state; }
}
```

`state` 읽는 곳 → `getState` 호출하는 곳만 추적.

## 정리

- **Effect Sketch** — 영향이 퍼지는 그림
- 변경 지점 → 영향받는 값 → 읽는 메서드 → 호출자
- 영향 경로의 끝에 — 테스트
- Encapsulation으로 — 영향 범위 축소
