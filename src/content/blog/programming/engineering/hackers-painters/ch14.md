> **Hackers and Painters** Essay 14 요약

## 핵심 주장

**꿈의 프로그래밍 언어**를 설계한다면 어떤 모습일까? Graham이 생각하는 이상적인 언어의 특성.

## 언어 설계의 목표

### 누구를 위한 것인가

언어의 대상:
- 훌륭한 해커들
- 초보자가 아님
- **최고의 프로그래머가 원하는 것**

### 왜 그런가

이유:
- 좋은 해커가 쓰면 다른 사람도 따라옴
- 언어의 성공 = 해커의 선택
- 전문가를 만족시켜야

## 꿈의 언어 특성

### 1. 간결함 (Succinctness)

좋은 언어:
- 적은 코드로 많은 일
- 보일러플레이트 없음
- **생각을 직접 표현**

### 2. 해커블함 (Hackability)

해킹하기 쉬움:
- 쉽게 수정 가능
- 실험하기 좋음
- **언어 자체를 바꿀 수 있음**

### 3. 일회용 프로그램 (Throwaway Programs)

빠른 스크립팅:
- 작은 일에도 쓸 수 있음
- 시작 비용 낮음
- 대화형 개발

### 4. 라이브러리 (Libraries)

풍부한 생태계:
- 좋은 표준 라이브러리
- 활발한 커뮤니티
- 바퀴 재발명 불필요

### 5. 효율성 (Efficiency)

충분히 빠름:
- 대부분의 경우 충분
- 필요하면 최적화 가능
- **개발 속도가 더 중요**

### 6. 시간 (Time)

성숙함:
- 버그가 잡힘
- 문서화됨
- 커뮤니티 형성됨

### 7. 재설계 (Redesign)

지속적 개선:
- 버전업
- 피드백 반영
- 진화

## 중요하지 않은 것

### 속도

순수 속도:
- 대부분 충분
- 병목은 일부분
- **조기 최적화는 악**

### 안전

극단적 안전:
- 자유를 희생
- 해커를 방해
- 적당히만

### 인기

초기 인기:
- 좋은 언어는 결국 퍼짐
- 마케팅보다 품질
- 해커들이 선택하면 됨

## 언어와 구현

### 언어 ≠ 구현

구분해야 함:
- 언어: 추상적 설계
- 구현: 구체적 실행
- 좋은 구현이 나중에 올 수 있음

### 성장 전략

언어 성장:
1. 작게 시작
2. 해커들 모음
3. 라이브러리 성장
4. 구현 개선
5. 대중화

## 현실과 이상

### 존재하는 언어 중

Graham의 평가:
- Python: 꽤 좋음
- Ruby: 좋음
- Lisp: 이상에 가까움 (구현이 문제)

### Arc 프로젝트

Graham의 시도:
- 자신의 꿈의 언어
- Lisp 방언
- 간결함에 초점

## 실용적 조언

### 언어 선택

프로그래머에게:
- 가장 강력한 언어
- 해킹하기 좋은 언어
- 자신이 좋아하는 언어

### 언어 설계

설계자에게:
- 해커를 타겟
- 간결함 최우선
- 유연성 중시

## 시사점

1. **간결함이 핵심**: 적은 코드 = 좋은 언어
2. **해커 우선**: 전문가를 만족시켜라
3. **유연성**: 수정 가능하고 확장 가능
4. **실용성**: 이론보다 실제 사용

## 인용

> "The dream language would be clean, simple, and expressive."

> "A programming language is for thinking about programs, not for expressing programs you've already thought of."

> "Succinctness is power."
