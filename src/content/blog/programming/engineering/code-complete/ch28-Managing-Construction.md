## 이 챕터의 메시지

좋은 construction에는 — **좋은 관리자**가 필요하다. 단지 일정과 자원을 챙기는 게 아니라, 좋은 환경과 의사결정을 만드는 일.

> 좋은 관리자 = **개발자가 잘 일할 수 있는 조건을 만든다**.

## 핵심 내용

- **일정 추정**은 어렵다 — 과거 데이터·기준선·범위 의식.
- **측정**이 관리의 토대 — 측정 없는 개선 X.
- **프로그래머 처우** — 개인 차이가 매우 크다.
- 좋은 관리자 = **기술 이해 + 사람 이해**.

## 일정 추정

McConnell이 인용 — 일정 추정 정확도는 **시간이 갈수록만** 향상.

| 시점 | 정확도 (range) |
| --- | --- |
| 프로젝트 시작 | 0.25x ~ 4x |
| 요구사항 후 | 0.5x ~ 2x |
| 디자인 후 | 0.67x ~ 1.5x |
| Construction 중 | 0.85x ~ 1.15x |

처음엔 — **4배 차이**가 정상. 시간이 가면서 좁아진다.

### 추정 기법

- **과거 데이터** — 비슷한 프로젝트의 실제 시간.
- **상향식** — 작업 단위별 더하기.
- **유추** — 비슷한 프로젝트와 비교.
- **전문가 합의** — 여러 사람의 평균.

단일 기법으로 부족. **여러 기법 조합**.

## 측정

> 측정 없이 — 개선도 없다.

측정할 가치 있는 것:

- **결함률** (LOC당, 시간당).
- **생산성** (LOC/일, 기능/일).
- **순환 복잡도** — 함수별.
- **테스트 커버리지**.
- **빌드 시간**.
- **PR 머지 시간**.

### 측정의 함정

- **숫자가 목표가 되면 — 게임화**된다. (Goodhart의 법칙)
- **LOC 자체는 가치 X** — 코드 양이 아니라 가치.
- **개인 비교에 사용 금지** — 팀 전체로.

## 프로그래머 처우

McConnell이 인용 — **프로그래머 개인 차이가 10~20배**. 같은 일에 한 사람은 1일, 다른 사람은 10일.

이게 의미하는 바:

- **좋은 사람을 채용**이 가장 큰 레버리지.
- 사람을 더 추가해도 — 의사소통 비용으로 인해 효과가 작다 (Brooks의 법칙).
- **개인의 차이를 인정**하고 — 적재적소.

## Brooks의 법칙

> "**늦은 프로젝트에 사람을 더하면 — 더 늦어진다**."

이유:

- 신입 교육 시간.
- 의사소통 경로 증가.
- 일이 분할 불가능한 부분.

해결책 — **사람보다 일을 줄이거나 일정을 늘려라**.

## 좋은 관리자의 특성

McConnell이 정리한 좋은 관리자.

- **기술 이해** — 프로그래머의 일을 안다.
- **신뢰** — 마이크로매니지먼트 X.
- **장애물 제거** — 환경·도구·정책 개선.
- **명확한 우선순위** — 무엇이 중요한지.
- **인정** — 좋은 일을 알아본다.
- **솔직** — 좋은 소식과 나쁜 소식 모두.

## 관리자 vs 기술 리더

큰 조직에선 — 두 역할이 분리될 수 있다.

- **관리자**: 사람·일정·자원.
- **기술 리더**: 디자인·아키텍처·코드 품질.

두 역할이 협력 — 한 사람이 둘을 다 하기 어렵다.

## 정리

- **일정 추정** — 시간이 갈수록 정확도 향상.
- **측정**이 개선의 토대 — 단 게임화 주의.
- 프로그래머 **개인 차이 10~20배** — 채용이 핵심.
- **Brooks의 법칙** — 사람을 더해도 빨라지지 않는다.
- 좋은 관리자 = **기술 + 사람** 이해.
