## 이 챕터의 메시지

루틴(함수, 메서드)은 — **construction의 가장 자주 만나는 단위**다. 매일 새 함수를 짜고, 기존 함수를 본다. 좋은 루틴 한 줄이 만들어 내는 차이가 — 시스템 전체의 가독성과 변경 가능성을 결정한다.

McConnell의 가이드는 단호하다.

> **좋은 루틴 = 응집도 한 가지 + 의도가 드러나는 이름 + 적절한 길이 + 적은 매개변수**.

## 핵심 내용

- 루틴이 존재하는 **정당한 사유**가 있어야 한다.
- **기능적 응집도**(한 가지 일)가 가장 좋은 응집도.
- 좋은 이름: **동사 + 객체**, 책임을 정확히 묘사.
- **적절한 길이** — 한 화면, 50~100줄 정도까지.
- **매개변수는 적게** — 7개 이하, 이상적으로 3개 이하.

## 루틴은 왜 만드나

루틴을 만드는 정당한 사유들.

1. **복잡성을 줄인다** — 큰 본문을 작은 명명된 단위로 나눈다.
2. **중간 추상을 도입** — 같은 추상 수준에서 흐름이 보인다.
3. **코드 중복 제거** — DRY.
4. **흐름을 숨긴다** — 디테일을 함수 안에 가두고 호출자는 의도만 본다.
5. **임의 코드 시퀀스를 명명** — 의미를 부여.
6. **신뢰성 향상** — 한 자리에서 검증된 코드를 여러 곳에서 활용.
7. **인자화** — 비슷한 동작의 변형.
8. **포팅성** — 환경 의존성을 한 자리에.

이 중 어느 하나도 안 해당하면 — **그 루틴은 만들 이유가 없다**.

## 응집도 — 한 가지 일

응집도의 종류 (좋은 순서대로):

| 응집도 | 의미 | 평가 |
| --- | --- | --- |
| **Functional** | 한 가지 일 | ✅ 가장 좋음 |
| **Sequential** | 한 단계가 다음 단계 입력 | OK |
| **Communicational** | 같은 데이터에 대한 여러 일 | 분할 검토 |
| **Temporal** | 같은 시점에 일어나는 일들 | 보통 |
| **Procedural** | 한 순서로 일어나는 일들 | 약함 |
| **Logical** | switch로 분기되는 여러 일 | 약함 — 분할 |
| **Coincidental** | 임의로 묶인 일들 | ❌ — 분할 |

가장 좋은 응집도는 **functional** — 함수가 한 가지 일만 한다.

## 좋은 루틴 이름

루틴 이름은 — **함수가 하는 일을 정확히 묘사**해야 한다.

### 좋은 이름의 속성

- **동사 + 객체** — `printDocument`, `calculateTotal`.
- **함수가 하는 일을 모두** — 부작용이 있다면 이름에 포함.
- **불특정 동사 X** — `handle`, `process` 같은 모호한 단어 피하기.
- **이름의 길이는 함수의 책임 폭에 비례** — 단순하면 짧게, 복잡하면 길게.
- **반환값을 기준으로** — boolean 반환은 `isXxx`, `hasXxx`.

### 나쁜 이름

```
handleData()     // 무엇을?
process()        // 어떻게?
doStuff()        // 무슨 stuff?
mainLogic()      // 어떤 로직?
```

이런 이름은 **함수가 너무 많은 일을 한다**는 신호.

## 루틴의 길이

McConnell의 권장 길이는 **현대 통념보다 길다**.

- 짧은 함수가 좋다 — 일반적 통념.
- 그러나 강제로 너무 짧게 만들면 — 인위적 분할로 가독성이 떨어진다.
- McConnell의 연구 인용: **65~150줄의 함수가 결함률이 가장 낮다** (특정 조건에서).

다만 — 이건 옛 데이터(2004년)다. 현대 코딩 컨벤션은 — 더 짧게 (10~30줄)를 권장한다.

> 핵심: **한 가지 일**을 한다면 길이는 부차적이다. 한 일이 길면 길고, 짧으면 짧다.

## 매개변수

- **이상적**: 0~3개.
- **7개 이하** — 사람이 한 번에 머리에 담을 한계.
- **8개 이상** — 거의 항상 잘못된 설계.

### 매개변수 줄이는 방법

- **객체로 묶기** — 의미 있는 그룹은 클래스/struct로.
- **함수를 작은 단위로 쪼개기** — 한 함수가 여러 데이터를 알면 — 여러 함수.
- **상태를 객체에 두기** — 매번 전달하지 않고 멤버로.

## 매개변수 사용 규칙

- **입력 → 변경 → 출력** 순서로 매개변수 배치.
- 너무 깊이 들어가는 객체는 피한다 — Law of Demeter.
- **사용 안 하는 매개변수**는 즉시 제거.
- **출력 매개변수**는 피하고 반환값으로.

## 정리

- 루틴은 **정당한 사유**가 있어야 만든다.
- **functional cohesion**이 최선 — 한 가지 일.
- 이름은 **함수가 하는 일을 정확히** 묘사.
- 한 가지 일이라면 **길이는 부차적**.
- 매개변수는 **0~3개 이상적, 7개 이하**.
