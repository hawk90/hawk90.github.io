## 이 챕터의 메시지

디버깅은 — **과학적 방법**이다. 추측하고 시도하는 무작위 시행착오가 아니라, 가설을 세우고 검증하는 절차.

> 좋은 디버깅 = **재현 → 가설 → 검증 → 수정 → 회귀 방지**.

## 핵심 내용

- 디버깅은 **과학적**. 추측 대신 가설·검증.
- 첫 단계 — **확실히 재현**.
- 발견한 결함을 **반드시 기록**.
- 도구를 잘 활용 — 디버거, 로깅, 어셔션.
- 수정 후 — **회귀 테스트** 추가.

## 과학적 디버깅 절차

1. **재현 가능한 케이스** 만들기.
2. **결함 위치** 좁히기 (이진 탐색).
3. **가설 세우기** — 왜 이 결함이 발생하는가?
4. **가설 검증** — 코드 변경 또는 로그.
5. **수정** — 가설이 맞으면.
6. **회귀 테스트 추가** — 다시 발생 안 하게.

## 재현이 첫 단계

> 재현할 수 없으면 — 고칠 수 없다.

재현하기 어려운 결함:

- 동시성 (race).
- 시간 의존 (특정 시각, 시간대).
- 환경 의존 (특정 OS·하드웨어).
- 데이터 의존 (특정 입력만).

이런 경우 — **재현 조건을 좁히는 일** 자체가 디버깅의 큰 부분.

## 가설 세우기

> 추측이 아니라 **데이터에 기반한 가설**.

```
관찰: 사용자 X에게서만 결함 발생
가설 1: X의 데이터 형식이 다름 → 검증: 데이터 비교
가설 2: X의 권한이 다름 → 검증: 권한 확인
가설 3: X의 환경(브라우저)이 다름 → 검증: 브라우저 정보
```

각 가설을 — 실험으로 검증. 거짓이면 제외.

## 이진 탐색으로 위치 좁히기

긴 코드에서 결함 위치를 찾을 때.

```c
// 1번 자리에 print → 통과
// 4번 자리에 print → 실패
// 2~3번 사이가 문제
// 2번에 print → 통과 → 3번이 문제
```

큰 코드를 — 한 단계씩 좁혀 간다.

## 흔한 함정

### 1) 추측 기반 수정

```c
// "음 이 변수가 문제일 것 같다"
x = 0;    // 수정?

// 진짜 원인은 다른 자리 → 며칠 후 다시 깨짐
```

가설을 검증 안 하고 수정하면 — 진짜 원인이 잡히지 않는다.

### 2) 증상만 고치기

```c
// 결과: 5가 6으로 표시됨
// 수정: result -= 1   ← 증상만 고침

// 진짜 원인: 어딘가에서 잘못 계산
```

증상이 아니라 **근본 원인**을 찾는다.

### 3) 비슷한 코드 양산

```c
// 한 자리에서 결함 발견 → 수정
// 다른 자리에 같은 결함이 있는 줄 모르고 — 며칠 후 또 발생
```

발견된 결함의 **유사 패턴**을 코드 전체에서 검색.

### 4) "디버거 없이도 찾을 수 있다" 자만

print 디버깅도 좋지만 — **디버거가 더 빠를 때**가 많다. 도구 활용.

## 디버깅 도구

- **디버거** (gdb, lldb, IDE 통합) — 단계 실행, 브레이크포인트, 변수 검사.
- **로깅** — 큰 그림 추적, 분산 시스템.
- **assertion** — 가정 검증, 결함을 조기 검출.
- **메모리 도구** — Valgrind, AddressSanitizer.
- **프로파일러** — 성능 디버깅.

## 수정 후 — 회귀 방지

결함을 고친 후 — **테스트를 추가**한다. 같은 결함이 다시 발생하지 않도록.

```python
def test_user_can_login_with_apostrophe_in_email():
    """버그 #12345 — '가 포함된 이메일에서 로그인 실패"""
    user = User(email="o'brien@example.com")
    assert user.login(password="...") == True
```

테스트 이름에 **버그 번호 + 의도**를 적으면 — 미래의 다른 개발자가 이해.

## 정리

- 디버깅은 **과학적** — 가설·검증.
- 첫 단계는 **재현**.
- 증상 아닌 **근본 원인**.
- **이진 탐색**으로 위치 좁히기.
- 수정 후 — **회귀 테스트 추가**.
