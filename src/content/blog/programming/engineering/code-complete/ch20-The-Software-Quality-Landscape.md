## 이 챕터의 메시지

"좋은 소프트웨어"는 한 단어로 정의 못 한다. **여러 차원**이 있고, 일부는 서로 충돌한다.

> 품질의 차원을 알면 — 트레이드오프를 의식적으로 결정할 수 있다.

## 핵심 내용

- 품질의 **외부**(사용자 인식)와 **내부**(개발자 인식) 차원.
- 차원들이 서로 **트레이드오프**.
- 결함 검출 기법은 효과가 다르다 — **여러 기법 조합**이 핵심.

## 외부 품질 (사용자가 본다)

- **정확성** — 명세대로 동작.
- **사용성** — 학습·사용 쉬움.
- **효율** — 시간/메모리/배터리.
- **신뢰성** — 정해진 조건에서 실패 없음.
- **무결성** — 보안, 데이터 보호.
- **적응성** — 다른 환경 동작.
- **정확함**(accuracy) — 계산 결과의 신뢰성.
- **견고성** — 잘못된 입력에 대처.

## 내부 품질 (개발자가 본다)

- **유지보수성** — 변경·수정 쉬움.
- **유연성** — 새 요구에 적응.
- **포팅성** — 다른 환경 이식.
- **재사용성** — 다른 곳에서 활용 가능.
- **가독성** — 코드 이해 쉬움.
- **테스트 가능성** — 검증 가능.
- **이해 가능성** — 시스템 전체 파악 쉬움.

## 트레이드오프

| 한쪽 | vs | 다른쪽 |
| --- | --- | --- |
| 정확성 | ↔ | 견고성 |
| 효율 | ↔ | 가독성 |
| 정확성 | ↔ | 유연성 |
| 보안 | ↔ | 사용성 |

예: 의료 시스템은 **정확성** > 견고성. 게임은 **견고성** > 정확성.

도메인에 맞게 — **우선순위**를 정한다.

## 결함 검출 효과

McConnell이 인용하는 연구. 결함 검출 기법별 평균 효과.

| 기법 | 평균 검출률 |
| --- | --- |
| 비공식 디자인 리뷰 | 25~40% |
| 공식 디자인 리뷰 | 45~65% |
| 비공식 코드 리뷰 | 25~40% |
| 공식 코드 리뷰(인스펙션) | 45~70% |
| 단위 테스트 | 30~35% |
| 통합 테스트 | 35~40% |
| 시스템 테스트 | 30~35% |
| 베타 테스트(저강도) | 25~40% |
| 베타 테스트(고강도) | 60~85% |

가장 큰 메시지 — **단일 기법으로는 50% 이상 못 잡는다**. 여러 기법을 조합해야 한다.

McConnell의 권고: **조합 사용 시 95%+** 결함 검출이 가능. 한 기법만 쓰면 절반은 놓친다.

## 결함의 비용

| 단계 | 비용 (상대) |
| --- | --- |
| 요구사항 | 1x |
| 설계 | 3x |
| Construction | 5~10x |
| 시스템 테스트 | 10~25x |
| 운영 | 50~200x |

조기 발견 = 비용 절감. 그래서 — **여러 기법을 조기에**.

## 정리

- 품질의 **외부·내부 차원**이 있고 — 서로 트레이드오프.
- 도메인에 맞는 **우선순위** 결정.
- 결함 검출은 — **여러 기법 조합**으로 95%+.
- 단일 기법은 50%도 못 잡는다.
- 조기 발견 = **비용 기하급수 절감**.
