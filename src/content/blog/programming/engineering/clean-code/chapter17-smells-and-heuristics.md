## 이 챕터의 메시지

책의 마지막 챕터는 **전체의 압축**이다. Martin이 자기 경력에서 모은 코드 냄새(code smells)와 휴리스틱을 **카테고리별 체크리스트**로 정리한다.

> 이 챕터는 매일 코드 리뷰 옆에 둘 만한 **참고 자료**다.

각 항목은 짧다. 책 전체의 다른 챕터를 한 줄로 압축한 것이다. 처음 읽을 땐 너무 많아 보일 수 있지만 — 시간이 지나면 자기 코드에서 자동으로 떠오르는 신호가 된다.

## 핵심 내용

- 카테고리: 주석, 환경, 함수, 일반, Java, 이름, 테스트.
- 각 항목은 **냄새**(문제 신호) 또는 **휴리스틱**(권장 패턴).
- 매 코드 리뷰에서 이 목록을 머리에 두고 본다.
- 이 챕터는 책 전체의 **압축된 안내**.

## 주석 (Comments)

| 냄새 | 의미 |
| --- | --- |
| C1: 부적절한 정보 | 변경 이력, 작성자 정보 등은 git에 있다 |
| C2: 사용되지 않는 주석 | 옛 주석은 거의 항상 거짓말 — 삭제 |
| C3: 중복 주석 | 코드가 이미 말하는 걸 또 적지 마라 |
| C4: 잘못된 주석 | 코드와 일치 안 함 — 거짓말 |
| C5: 주석 처리된 코드 | 지워라. git이 보관한다 |

## 환경 (Environment)

| 휴리스틱 | 의미 |
| --- | --- |
| E1: 빌드 한 단계 | 한 명령으로 빌드되어야 한다 |
| E2: 테스트 한 단계 | 한 명령으로 모든 테스트 통과 확인 가능해야 |

복잡한 빌드는 개발자가 빌드를 안 돌리게 만든다.

## 함수 (Functions)

| 냄새 | 의미 |
| --- | --- |
| F1: 너무 많은 인자 | 0~3개가 이상적. 4+는 거의 항상 잘못 |
| F2: 출력 인자 | 매개변수를 변경하는 함수 X. 객체 자신을 변경하거나 새 값을 반환 |
| F3: 플래그 인자 | bool 인자는 두 함수로 나누는 신호 |
| F4: 죽은 함수 | 호출되지 않으면 지워라 |

## 일반 (General)

가장 많은 항목이 있는 카테고리다.

| 냄새 | 의미 |
| --- | --- |
| G1: 한 언어 안에 여러 언어 | HTML/CSS/SQL/Regex 등 섞임 — 최소화 |
| G2: 명백한 동작이 안 됨 | 가장 자연스러운 사용이 동작해야 — 놀라움 금지 |
| G3: 경계 조건 미처리 | off-by-one, 빈 리스트, null 등 — 명시적으로 |
| G4: 안전장치 무시 | 컴파일 경고 끄기, 테스트 끄기 등 — 폭탄을 두는 행위 |
| G5: 중복 | 모든 악의 뿌리. 항목별 형태가 있다: |
| - 명백한 중복 | 똑같은 코드가 둘 이상의 자리 |
| - switch/if 중복 | 같은 분기가 여러 자리에 — 다형성으로 |
| - 알고리즘 중복 | 같은 패턴이 미세하게 다른 형태로 — Template Method 등 |
| G6: 잘못된 추상 수준 | 한 함수/클래스에 여러 수준이 섞임 |
| G7: 추상화의 base가 derived를 안다 | 의존 방향 반대 — base는 derived를 몰라야 |
| G8: 너무 많은 정보 | 인터페이스의 메서드 수 최소화 — 정보 은닉 |
| G9: 죽은 코드 | 사용되지 않는 코드 — 지워라 |
| G10: 수직 분리 | 변수는 사용 가까이, 헬퍼는 호출자 아래 |
| G11: 일관성 부족 | 같은 일에 같은 단어 |
| G12: 잡음 | 의미 없는 코드, 빈 기본 생성자 등 |
| G13: 인공적 결합 | 한 가지 일의 디테일이 무관한 곳에 — 분리 |
| G14: Feature Envy | 한 메서드가 자기 클래스보다 다른 클래스의 데이터를 더 만짐 — 이동 |
| G15: Selector 인자 | bool/enum 인자로 두 동작을 분기 — 두 함수로 |
| G16: 모호한 의도 | 코드가 무엇을 하는지 즉시 안 보임 |
| G17: 잘못된 위치 | 함수가 자기 책임이 아닌 클래스에 있음 |
| G18: 부적절한 static | 다형성 가능성을 막음 |
| G19: 설명 변수 사용 | 복잡한 표현식은 이름 있는 변수로 분해 |
| G20: 함수 이름이 동사 | `addToList(item)`이 `list.add(item)`보다 자연스러우면 메서드로 |
| G21: 알고리즘 이해 | 단순히 동작하는 게 아니라 — 명확히 이해 |
| G22: 논리 결합 명확히 | `&&`, `||`의 의미를 변수로 추출 |
| G23: 부정 조건 회피 | `if (!condition)`보다 `if (condition)` |
| G24: 함수는 한 가지 일 | 한 함수, 한 책임 |
| G25: 시간 결합 명시 | 호출 순서가 중요하면 — 매개변수로 강제 |
| G26: 일관성 | 한 결정을 했다면 코드 전체에 적용 |
| G27: 구조 위에 컨벤션 | 강한 컨벤션보다 강한 타입/구조가 낫다 |
| G28: 조건 캡슐화 | 복잡한 조건은 의도 있는 메서드로 |
| G29: 부정 조건 피하기 | (G23과 같은 메시지) |
| G30: 함수가 한 가지 일 | (G24와 같음) |
| G31: 숨겨진 시간 결합 | 호출 순서가 중요한데 시그니처에 안 드러남 |
| G32: 임의의 결정 회피 | "왜 이 값인가"에 답이 없으면 — 의미 있는 이름이나 상수로 |
| G33: 경계 조건 캡슐화 | 위험한 경계 검사 한 자리에 |
| G34: 함수의 추상 수준 한 단계 아래 | 함수 본문은 함수보다 한 수준 아래여야 |
| G35: 구성 가능 데이터는 높은 수준에 | 설정 값은 main 가까이에, 비즈니스 로직에 묻지 마라 |
| G36: 전이적 탐색 회피 | Law of Demeter — `a.getB().getC().doSomething()` X |

## Java (J)

| 휴리스틱 | 의미 |
| --- | --- |
| J1: 긴 import 목록 회피 | wildcard `*` 사용도 OK |
| J2: 상수 상속 X | 상수는 상속이 아니라 정적 import |
| J3: 상수 vs Enum | 일련의 값엔 enum |

## 이름 (Names)

| 휴리스틱 | 의미 |
| --- | --- |
| N1: 묘사적 이름 | 노력해서 짓고, 자주 바꾸기 |
| N2: 추상 수준에 맞는 이름 | 추상 클래스: 추상적 이름; 구체 클래스: 구체적 이름 |
| N3: 표준 명명 활용 | `Decorator`, `Visitor`, `toString` 등 |
| N4: 명확 vs 짧음 | 좁은 스코프엔 짧게, 넓은 스코프엔 길게 |
| N5: 부작용 표현 | 함수 이름이 부작용을 드러내야 |
| N6: 부정 조건 회피 | 가능한 한 긍정 형태 |
| N7: 모호한 이름 회피 | `info`, `data`, `temp` 등 |

## 테스트 (Tests)

| 휴리스틱 | 의미 |
| --- | --- |
| T1: 충분한 테스트 | 모든 경계 조건, 모든 정상 경로 |
| T2: 커버리지 도구 사용 | 빠뜨린 자리 발견 |
| T3: 사소한 테스트 건너뛰지 마라 | 작은 테스트가 의도를 잘 보여 준다 |
| T4: 무시된 테스트는 모호함의 의문 | `@Ignore`된 테스트는 문제의 신호 |
| T5: 경계 조건 테스트 | 정상값 외 경계가 가장 자주 깨진다 |
| T6: 버그 주변에 더 테스트 | 한 버그 근처에 다른 버그 있을 확률 ↑ |
| T7: 실패 패턴 정보 풍부 | 실패 패턴에 디버깅 단서 |
| T8: 테스트 커버리지 패턴 | 커버되지 않은 영역 분석 |
| T9: 테스트는 빨라야 | 느린 테스트는 안 돌게 된다 |

## 이 챕터를 어떻게 쓰나

이 목록은 **외울 게 아니다**. 다음과 같이 활용한다.

1. **코드 리뷰 시 옆에 두고 본다** — 자기 코드와 동료 코드 모두.
2. **시간이 지나면서 자동으로 떠오른다** — 처음엔 의식적, 나중엔 자동.
3. **새 항목을 추가한다** — 개인 경험에서 발견한 패턴.

가장 좋은 활용은 — **자기만의 체크리스트**로 발전시키는 것이다. Martin의 항목을 출발점으로 하고, 자기 팀·도메인의 패턴을 추가한다.

## 책 전체의 종합

이 챕터로 책이 끝난다. 책 전체의 메시지를 한 문단으로 압축하면 다음과 같다.

> **클린 코드는 태도다.** 매일 매 커밋, 매 함수, 매 이름의 작은 선택들이 누적되어 — 시간이 지나도 변경 가능한, 다음 사람이 읽을 수 있는, 자부심을 가질 수 있는 시스템을 만든다.
>
> 그 태도는 한 사람의 능력이 아니라 **습관**이다. 습관은 매일 연습되어 자란다. 보이스카우트 규칙을 매일 따르고, 테스트를 안전망으로 두고, 거장들의 정의를 자기 기준으로 삼는다.

## 정리

- 카테고리별 코드 냄새와 휴리스틱의 **압축된 체크리스트**.
- 외우는 게 아니라 — 매 리뷰에서 옆에 두고 본다.
- 시간이 지나면 자동으로 떠오르는 신호가 된다.
- 자기만의 체크리스트로 발전시킨다.
- 책 전체의 메시지 — **클린 코드는 태도이고 습관이다**.

이로써 Clean Code 시리즈가 끝났다. 다음 단계는 코드를 실제로 짜는 일 — 그리고 이 책의 원칙들을 매일 매 커밋에 적용하는 것이다.
