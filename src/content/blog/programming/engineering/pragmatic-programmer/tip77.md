## 이 팁의 메시지

> **Tip 77: Work with a User to Think Like a User.** It's the best way to gain insight into how the system will really be used.

시스템이 실제로 어떻게 사용될지 통찰을 얻는 가장 좋은 방법이다.

## 개발자의 맹점

개발자는 사용자가 아니다. 우리의 가정이 틀릴 수 있다.

```text
개발자 생각: "키보드 단축키가 효율적이야"
사용자 현실: 마우스를 더 많이 쓴다

개발자 생각: "설정은 JSON 파일로 편집하면 되지"
사용자 현실: JSON이 뭔지 모른다

개발자 생각: "에러 메시지가 명확하잖아"
사용자 현실: "NullPointerException이 뭐예요?"
```

## 사용자 관찰

실제 사용자가 일하는 것을 본다.

```text
관찰 방법:
1. 사용자 옆에 앉는다
2. 평소대로 일하게 한다
3. 질문하지 않고 지켜본다
4. 막히는 부분, 반복하는 작업을 기록한다
5. 나중에 질문한다
```

말하는 것과 행동하는 것이 다를 수 있다.

## 하루를 함께

개발자가 사용자의 일을 해본다.

```text
예: 고객 지원 시스템 개발

Day 1: 고객 지원팀과 함께 앉기
- 같은 도구 사용
- 같은 전화 받기
- 같은 이메일 처리

발견:
- "고객 검색이 느려서 통화 중에 당황함"
- "이전 문의 내역을 찾으려면 3번 클릭해야 함"
- "복사-붙여넣기를 하루에 100번 함"
```

직접 겪으면 문제를 진짜로 이해한다.

## 페르소나

구체적인 가상 사용자를 만든다.

```text
페르소나: 김영희 (회계사, 45세)
- 회사에서 10년째 근무
- 엑셀에 익숙하지만 새 소프트웨어는 부담
- 하루에 200건의 전표 처리
- 실수하면 월말에 고생함
- 점심시간에 할 일을 끝내고 싶음
```

"사용자"가 아니라 "김영희"를 위해 만든다.

## 사용자 스토리

기능이 아니라 목표로 생각한다.

```text
기능 중심:
"주문 목록을 필터링하는 기능"

사용자 스토리:
"고객 지원 담당자로서,
 지난 주의 미처리 주문을 빠르게 찾아서
 고객에게 상황을 안내하고 싶다"
```

왜 필요한지가 드러난다.

## 맥락 이해

사용자의 환경을 이해한다.

| 질문 | 영향 |
|------|------|
| 어디서 쓰는가? | 모바일 필요성, 화면 크기 |
| 언제 쓰는가? | 바쁜 시간, 집중 가능 시간 |
| 얼마나 자주? | 학습 곡선 허용 범위 |
| 다른 도구는? | 통합 필요성 |
| 실수하면? | 에러 복구 중요도 |

## 지름길

사용자가 만든 지름길을 관찰한다.

```text
발견: 엑셀에 복사해서 계산하고 다시 붙여넣음
→ 소프트웨어 내에서 계산 기능이 필요

발견: 같은 값을 매번 입력함
→ 기본값이나 템플릿 기능 필요

발견: 화면 캡처해서 이메일로 공유함
→ 공유 기능이 필요
```

지름길은 빠진 기능을 알려준다.

## 사용자 참여 방법

| 방법 | 장점 | 단점 |
|------|------|------|
| 인터뷰 | 깊은 이해 | 시간 소요 |
| 설문 | 많은 응답 | 피상적 |
| 관찰 | 실제 행동 | 관찰 효과 |
| 베타 테스트 | 실제 사용 | 늦은 피드백 |
| 온사이트 | 맥락 이해 | 비용 |

여러 방법을 조합한다.

## 공감

사용자의 좌절을 직접 느낀다.

```text
"왜 이렇게 느리지?"
"이게 왜 안 되는 거야?"
"여기 어디 있는 거야?"
```

이 좌절을 직접 경험하면 우선순위가 달라진다.

## 정리

- 개발자와 사용자의 관점은 다르다.
- 사용자를 관찰하고 함께 일한다.
- 페르소나로 구체적인 사용자를 상상한다.
- 기능이 아닌 목표로 생각한다.
- 사용자의 맥락과 지름길을 이해한다.
- 공감으로 우선순위를 정한다.
