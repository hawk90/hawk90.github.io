## 이 팁의 메시지

> **Tip 80: Don't Think Outside the Box—Find the Box.** When faced with an impossible problem, identify the real constraints. Ask yourself: "Does it have to be done this way?"

불가능한 문제에 직면하면 진짜 제약을 찾아라. "이렇게 해야만 하는가?"라고 자문한다.

## "상자 밖" 사고의 문제

"상자 밖에서 생각하라"는 조언을 자주 듣는다. 그러나 상자가 뭔지 모르면 밖으로 나갈 수 없다.

```text
문제: "이 리스트를 정렬해야 하는데 O(n)으로 해야 해요"

상자를 모를 때:
"O(n) 정렬은 불가능해요" (포기)

상자를 찾을 때:
"잠깐, 정말 정렬해야 하나요? 최댓값만 찾으면 O(n)이에요"
```

제약을 명확히 하면 해결책이 보인다.

## 진짜 제약 vs 가짜 제약

| 진짜 제약 | 가짜 제약 |
|----------|----------|
| 물리 법칙 | "항상 이렇게 했으니까" |
| 법적 규정 | "이게 표준이니까" |
| 확정된 예산 | "더 나은 방법이 없을 거야" |
| 기술적 한계 (현재) | "그건 안 될 거예요" (검증 안 함) |

## 제약 발견 질문

문제가 막혔을 때 물어본다.

```text
1. "이것을 꼭 해야 하나?" (목표 자체를 의심)
2. "왜 이렇게 해야 하나?" (방법을 의심)
3. "누가 이렇게 하라고 했나?" (요구사항 출처 확인)
4. "제약이 바뀌면 어떻게 되나?" (제약 변경 가능성)
5. "다른 방법은 정말 없나?" (대안 탐색)
```

## 예: 불가능한 기한

```text
요청: "이 기능을 2주 안에 완성해 주세요"
반응: "불가능해요"

상자 찾기:
- "전체 기능"이 필요한가? → 핵심 부분만 2주에 가능
- "2주"가 절대적인가? → 데모용은 2주, 완성은 4주
- "완성"의 정의는? → MVP면 2주 가능
```

제약을 재정의하면 해결책이 나온다.

## 예: 성능 문제

```text
문제: "페이지 로딩이 10초 걸려요"
요구: "1초로 줄여주세요"
반응: "데이터가 너무 많아서 불가능해요"

상자 찾기:
- 모든 데이터를 한 번에 로드해야 하나? → 페이지네이션
- 서버에서 처리해야 하나? → 클라이언트 캐싱
- 실시간이어야 하나? → 미리 계산해두기
- 10초 전체가 문제인가? → 첫 화면 1초, 나머지 백그라운드
```

## 가정 나열

문제에 대한 모든 가정을 적어본다.

```text
문제: "사용자 인증 시스템 구축"

가정 목록:
- [ ] 직접 구축해야 한다 → OAuth 사용 가능?
- [ ] 비밀번호가 필요하다 → 패스키, 소셜 로그인?
- [ ] 이메일이 필요하다 → 전화번호 인증?
- [ ] 실시간 검증이 필요하다 → 토큰 기반?
- [ ] 모든 페이지에서 검증 → 공개 페이지 분리?
```

각 가정에 "정말?"이라고 묻는다.

## 제약 풀기

때로는 제약 자체를 바꿀 수 있다.

```text
제약: "기존 API와 호환되어야 한다"
→ "v2 API로 전환 기간을 두면?"

제약: "모든 브라우저 지원"
→ "IE 사용자가 0.1%면 제외 가능?"

제약: "실시간 동기화"
→ "1분 지연이 허용되면?"
```

제약을 바꾸는 게 더 나은 해결책일 수 있다.

## 다른 관점

문제를 다른 각도에서 본다.

```text
직접적 해결: "어떻게 빠르게 만들까?"
우회적 해결: "느려도 괜찮게 만들 수 있나?"
             → 로딩 중 다른 콘텐츠 표시
             → 진행률 표시로 체감 시간 감소

직접적 해결: "버그를 어떻게 고치지?"
우회적 해결: "이 기능이 꼭 필요한가?"
             → 기능 제거가 더 나을 수도
```

## 상자를 그려본다

제약을 시각화한다.

```text
┌─────────────────────────────────┐
│         예산: 1000만원           │
│  ┌─────────────────────────┐   │
│  │   기한: 3개월             │   │
│  │  ┌───────────────────┐  │   │
│  │  │ 팀원: 2명          │  │   │
│  │  │ ┌───────────────┐ │  │   │
│  │  │ │ 기술: Python  │ │  │   │
│  │  │ └───────────────┘ │  │   │
│  │  └───────────────────┘  │   │
│  └─────────────────────────┘   │
└─────────────────────────────────┘

어떤 상자를 넓힐 수 있나?
어떤 상자가 진짜인가?
```

## 정리

- "상자 밖"으로 나가려면 먼저 상자를 찾는다.
- 진짜 제약과 가짜 제약을 구분한다.
- "왜 이렇게 해야 하나?"라고 묻는다.
- 모든 가정을 나열하고 의심한다.
- 제약 자체를 바꿀 수 있는지 확인한다.
- 다른 관점에서 문제를 본다.
