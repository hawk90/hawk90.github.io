## 이 팁의 메시지

> **Tip 86: Do What Works, Not What's Fashionable.** Don't adopt development practices because they seem new and cool.

새롭고 멋져 보인다고 개발 관행을 채택하지 마라.

## 유행의 함정

기술 업계는 유행에 민감하다.

```text
흔한 유행:
"요즘 마이크로서비스가 대세야"
"쿠버네티스 안 쓰면 뒤처지는 거야"
"NoSQL이 미래야"
"서버리스가 답이야"
"함수형 프로그래밍해야 해"
```

유행을 따르는 것과 문제를 푸는 것은 다르다.

## 유행 vs 적합

| 유행 기반 결정 | 적합성 기반 결정 |
|----------------|------------------|
| "다들 쓰니까" | "우리 문제에 맞으니까" |
| "새로우니까" | "효과가 검증됐으니까" |
| "이력서에 좋으니까" | "팀이 잘 다룰 수 있으니까" |
| "블로그에서 봤으니까" | "실험해보니 맞으니까" |

## 질문으로 검증한다

```text
새 기술/방법론 도입 전:

1. 어떤 문제를 푸는가?
   - 우리에게 그 문제가 있는가?
   - 지금 방법으로 못 푸는가?

2. 비용은 무엇인가?
   - 학습 곡선은?
   - 기존 시스템 변경 비용은?
   - 운영 복잡도 증가는?

3. 실제 효과는?
   - 우리와 비슷한 규모의 성공 사례가 있는가?
   - 실패 사례도 찾아봤는가?

4. 대안은?
   - 기존 도구로 해결 가능한가?
   - 더 간단한 방법은 없는가?
```

## 사례: 마이크로서비스

```text
마이크로서비스가 필요한 경우:
- 팀이 100명 이상
- 서로 다른 배포 주기 필요
- 서로 다른 기술 스택 필요
- 특정 부분만 스케일아웃 필요

마이크로서비스가 과한 경우:
- 팀이 10명 미만
- 단일 배포로 충분
- 복잡한 트랜잭션 필요
- 운영 인력 부족
```

넷플릭스가 마이크로서비스를 쓴다고 5명 팀에 필요하진 않다.

## 사례: NoSQL

```text
NoSQL이 맞는 경우:
- 스키마가 자주 변한다
- 수평 확장이 필수다
- 관계형 조인이 거의 없다
- 읽기/쓰기 패턴이 단순하다

PostgreSQL이 나은 경우:
- 복잡한 관계 데이터
- ACID 트랜잭션 필요
- 익숙한 도구
- 작은 규모
```

## 실험으로 검증한다

```text
도입 프로세스:
1. 작은 파일럿으로 시작
2. 실제 문제에 적용
3. 결과 측정
4. 비용/이점 분석
5. 팀 피드백 수집
6. 확대 또는 철수 결정
```

유행이 아니라 데이터로 결정한다.

## 기존 도구의 가치

```text
"지루한" 기술의 장점:
- 검증된 안정성
- 풍부한 문서와 커뮤니티
- 팀이 이미 알고 있음
- 문제 해결 사례가 많음
- 장기 지원 가능성
```

새 기술은 흥미롭지만, 오래된 기술은 신뢰할 수 있다.

## 이력서 주도 개발

```text
위험 신호:
- "이 프로젝트로 새 기술 배우고 싶어"
- "이력서에 쿠버네티스 넣고 싶어"
- "최신 프레임워크 경험이 필요해"

올바른 태도:
- "이 문제에 가장 적합한 게 뭘까?"
- "팀이 가장 효과적으로 쓸 수 있는 건?"
- "장기적으로 유지보수 가능한 건?"
```

## 정리

- 유행이 아니라 효과를 따른다.
- 새 기술 도입 전에 질문으로 검증한다.
- 우리 문제에 맞는지 확인한다.
- 작은 실험으로 검증한다.
- 기존 도구의 가치를 인정한다.
- 이력서가 아니라 문제 해결을 위해 선택한다.
