## 이 팁의 메시지

> **Tip 82: Agile Is Not a Noun; Agile Is How You Do Things.** Agile is not a noun—it's how you do things.

애자일은 명사가 아니다. 애자일은 하는 방식이다.

## 애자일의 오해

"우리는 애자일을 합니다"라고 말하는 팀이 많다. 그러나 애자일은 제품이나 프레임워크가 아니다.

```text
잘못된 이해:
"우리는 SAFe를 도입했으니 애자일입니다"
"스크럼 마스터가 있으니 애자일입니다"
"스프린트를 하니 애자일입니다"
"칸반 보드가 있으니 애자일입니다"
```

도구와 의식이 애자일을 만들지 않는다.

## 애자일 선언문의 가치

```text
애자일 소프트웨어 개발 선언 (2001):

- 프로세스와 도구보다 "개인과 상호작용"
- 포괄적인 문서보다 "작동하는 소프트웨어"
- 계약 협상보다 "고객과의 협력"
- 계획을 따르기보다 "변화에 대응"
```

왼쪽도 가치 있지만, 오른쪽에 더 높은 가치를 둔다.

## 진짜 애자일 vs 가짜 애자일

| 진짜 애자일 | 가짜 애자일 |
|-------------|-------------|
| 지속적으로 개선한다 | 프로세스를 고정한다 |
| 피드백에 반응한다 | 계획대로만 한다 |
| 팀이 결정한다 | 위에서 지시한다 |
| 작동하는 소프트웨어 우선 | 문서와 회의 우선 |
| 필요하면 바꾼다 | "원래 이렇게 한다" |

## 애자일하게 일한다는 것

```text
애자일한 팀의 특징:
1. 짧은 주기로 가치를 전달한다
2. 피드백을 받고 방향을 조정한다
3. 막힌 곳을 빠르게 해결한다
4. 팀 내 의사소통이 활발하다
5. 회고하고 개선한다
6. 변화를 두려워하지 않는다
```

## 프레임워크의 함정

```text
흔한 함정:
- 스크럼을 도입했지만 회고가 형식적이다
- 스프린트가 있지만 항상 일정에 쫓긴다
- 스탠드업이 있지만 보고 회의가 됐다
- 칸반 보드가 있지만 WIP 제한이 없다
- "애자일 코치"가 있지만 팀이 자율적이지 않다
```

## 애자일의 본질

```text
본질적인 질문:
- 피드백 루프가 짧은가?
- 팀이 자율적으로 결정하는가?
- 지속적으로 개선하고 있는가?
- 고객에게 가치를 자주 전달하는가?
- 변화에 유연하게 대응하는가?
```

이 질문에 "예"라면 애자일하다. 프레임워크와 무관하다.

## 반(反) 애자일 신호

```text
경고 신호:
- "프로세스가 그렇게 정해졌으니"
- "스프린트 중간에는 바꿀 수 없다"
- "그건 스크럼 가이드에 없다"
- "애자일 전환 프로젝트는 2년 계획"
- 변화에 대한 두려움
```

## 정리

- 애자일은 명사가 아니라 형용사다.
- 프레임워크나 도구가 애자일을 만들지 않는다.
- 애자일 선언문의 가치를 따르는 것이 핵심이다.
- 지속적 개선과 피드백 대응이 본질이다.
- 의식과 도구에 집착하면 가짜 애자일이 된다.
- 팀이 자율적으로 일하고 변화에 대응하면 애자일하다.
