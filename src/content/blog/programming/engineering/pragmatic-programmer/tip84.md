## 이 팁의 메시지

> **Tip 84: Schedule It to Make It Happen.** If you don't schedule it, it's not going to happen.

일정에 넣지 않으면 일어나지 않는다.

## 급한 일 vs 중요한 일

급한 일이 중요한 일을 밀어낸다.

```text
급하고 중요한 일:     → 즉시 한다
급하지만 중요하지 않은: → 위임하거나 거절
급하지 않지만 중요한:  → 일정에 넣는다 ←
급하지도 중요하지도:   → 하지 않는다
```

일정에 넣지 않은 "급하지 않지만 중요한 일"은 영원히 미뤄진다.

## 일정에 넣어야 할 것들

**기술 부채 상환:**

- 리팩토링 시간
- 테스트 커버리지 향상
- 문서 정리
- 의존성 업데이트
- 레거시 코드 개선

**학습과 성장:**

- 새 기술 탐색
- 책 읽기
- 사이드 프로젝트
- 컨퍼런스/밋업

**팀 활동:**

- 코드 리뷰
- 지식 공유 세션
- 회고
- 1:1 미팅

## 구체적인 방법

```text
잘못된 방식:
"시간 나면 리팩토링해야지"
"언젠가 테스트 커버리지 올려야지"
"조만간 문서 정리해야지"

올바른 방식:
"매주 금요일 오후 2시간은 기술 부채 상환"
"스프린트당 20% 용량은 개선 작업"
"매일 퇴근 전 30분은 테스트 작성"
```

## 시간 박스

정해진 시간 동안만 집중한다.

```text
예: 기술 부채 금요일

14:00-16:00 기술 부채 시간
- 이번 주 발견한 코드 냄새 정리
- 가장 급한 것 하나 선택
- 2시간 동안 집중 작업
- 완벽하지 않아도 끝낸다
- 다음 주에 계속
```

## 팀 차원의 일정

```text
주간 일정 예:

월: 스프린트 계획
화-목: 개발
금 오전: 코드 리뷰 집중
금 오후: 기술 부채 / 학습
격주 금요일: 회고
```

## 보호해야 할 시간

**보호된 시간:**

- 회의 없는 집중 시간
- 학습 시간
- 개선 시간
- 1:1 미팅 시간

**지켜야 할 규칙:**

- 급한 일에도 양보하지 않는다
- 일정에 명시한다
- 팀이 합의한다
- 예외는 정말 예외일 때만

## 개인 차원의 일정

**개인 일정 예:**

**매일:**

- 08:00-09:00: 학습 (책, 온라인 강의)
- 09:00-12:00: 집중 개발 (회의 금지)
- 14:00-17:00: 협업, 회의
- 17:00-17:30: 테스트 작성

**매주:**

- 월요일 아침: 주간 계획
- 금요일 오후: 주간 회고 + 개선

## 일정이 안 지켜질 때

**원인 분석:**

- 일정 자체가 비현실적인가?
- 급한 일이 너무 많은가?
- 팀 합의가 없는가?
- 우선순위가 불명확한가?

**조정:**

- 작게 시작한다 (1시간부터)
- 팀과 협의한다
- 보호 장치를 둔다
- 성과를 측정하고 공유한다

## 정리

- 일정에 넣지 않으면 실행되지 않는다.
- 급하지 않지만 중요한 일을 명시적으로 일정에 넣는다.
- 기술 부채, 학습, 개선 시간을 확보한다.
- 시간 박스로 집중한다.
- 팀 차원에서 합의하고 보호한다.
- 작게 시작하고 점진적으로 늘린다.
