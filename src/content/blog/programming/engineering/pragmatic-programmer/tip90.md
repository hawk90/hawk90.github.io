## 이 팁의 메시지

> **Tip 90: Coding Ain't Done 'Til All the Tests Run.** Just because it seems to work doesn't mean it does.

작동하는 것처럼 보인다고 작동하는 게 아니다.

## "끝났다"의 정의

| 잘못된 정의 | 올바른 정의 |
|-------------|-------------|
| "코드 작성 끝났어요" | 코드 작성 완료 |
| "내 컴퓨터에서는 돼요" | 테스트 작성 완료 |
| "일단 돌아가요" | 모든 테스트 통과 |
| "테스트는 나중에 할게요" | 코드 리뷰 완료 / 문서 업데이트 완료 |

## 테스트가 증명하는 것

| 테스트 통과 | 테스트 없음 |
|-------------|-------------|
| 기능이 요구사항대로 동작 | 작동할지도 모름 |
| 기존 기능이 깨지지 않음 | 기존 기능이 괜찮을지도 |
| 엣지 케이스 처리됨 | 엣지 케이스는 모름 |
| 에러 핸들링 검증됨 | 에러가 나면 어떻게 될지 모름 |

## 완료의 체크리스트

기능 완료를 위한 영역별 체크 항목이다.

**코드:**

- 기능 구현 완료
- 코드 리뷰 통과
- 린터/포매터 통과

**테스트:**

- 단위 테스트 작성
- 통합 테스트 작성 (필요시)
- 모든 테스트 통과
- 기존 테스트 깨지지 않음

**문서:**

- API 문서 업데이트
- README 업데이트 (필요시)
- 변경 사항 기록

## CI가 보장한다

CI 파이프라인은 다음 순서로 동작한다.

1. 코드 푸시
2. 빌드 실행
3. 모든 테스트 실행
4. 결과 확인 — 통과면 머지 가능, 실패면 머지 불가

"테스트 나중에 고칠게"로 머지할 수 없다.

## 회귀 방지

회귀(regression)는 *기존에 되던 게 안 되는 것*이다.

- 버그 수정 시 테스트 추가
- 기존 기능에 대한 테스트 유지
- 모든 테스트가 항상 실행

이 사이클이 *버그 발생 → 테스트 추가 → 같은 버그 재발 방지*를 보장한다.

## 테스트 부채

테스트 없이 진행하면 다음 순서로 손해가 누적된다.

1. 단기적으로 빨라 보임
2. 버그 발생
3. 수동 테스트 시간 증가
4. 리팩토링 두려움
5. 코드 품질 하락
6. 결국 더 느려짐

## "나중에 테스트"의 함정

자주 듣는 말:

- "일단 기능 먼저, 테스트는 나중에"
- "시간이 없어서 테스트는 다음에"
- "동작하는 거 확인했으니 테스트는 생략"

현실:

- "나중"은 오지 않는다
- 시간이 없으면 테스트도 없다
- 확인했다고 버그가 없는 게 아니다

## 테스트 우선의 이점

| 테스트 없이 | 테스트와 함께 |
|-------------|---------------|
| 빠른 시작 | 약간 느린 시작 |
| 나중에 버그 | 버그 조기 발견 |
| 수동 확인 필요 | 자동 확인 |
| 리팩토링 두려움 | 자신감 있는 리팩토링 |
| 기술 부채 누적 | 지속 가능한 속도 |

## 문화로 정착

팀 규칙에 명시한다.

- PR 머지 조건에 테스트 포함
- CI 실패하면 머지 금지
- 테스트 커버리지 기준 설정
- 코드 리뷰 시 테스트 확인
- "완료" 정의에 테스트 포함

## 정리

- 테스트가 모두 통과해야 코딩이 끝난 것이다.
- "일단 돌아가니까"는 완료가 아니다.
- CI가 테스트 통과를 강제한다.
- 테스트 없이 진행하면 나중에 더 느려진다.
- "나중에 테스트"는 거의 항상 "테스트 없음"이다.
- 팀 문화로 정착시킨다.
