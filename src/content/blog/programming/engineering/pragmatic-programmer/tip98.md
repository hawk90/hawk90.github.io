## 이 팁의 메시지

> **Tip 98: Don't Enable Scoundrels.** Because you risk becoming one, too.

나쁜 사람을 돕지 마라. 당신도 그렇게 될 위험이 있기 때문이다.

## 기술의 양면성

같은 기술이 좋은 일에도, 나쁜 일에도 쓰이다.

| 기술 | 좋은 용도 | 나쁜 용도 |
|------|-----------|-----------|
| 얼굴 인식 | 실종자 찾기 | 시민 감시 |
| 위치 추적 | 배달 추적 | 스토킹 |
| 자동화 | 반복 업무 제거 | 대량 스팸, 봇 공격 |

## 고려해야 할 질문

기능을 개발할 때 미리 던지는 질문이다.

- 누가 이것을 악용할 수 있는가?
- 어떻게 악용할 수 있는가?
- 악용을 막을 방법이 있는가?
- 피해자는 누가 될 수 있는가?

## 실제 사례

| 행위 | 결과 |
|------|------|
| 데이터 브로커에 위치 데이터 판매 | 스토커가 피해자 추적에 사용 |
| 약한 인증 시스템 | 계정 탈취, 신분 도용 |
| 콘텐츠 추천 알고리즘 | 극단적 콘텐츠로 유도, 과격화 |
| 안면 인식 데이터베이스 | 권위주의 정권의 반체제 인사 감시 |

## 누가 나쁜 사람인가

**분명한 경우** — 사기꾼, 스토커, 독재 정권, 테러리스트.

**애매한 경우:**

- 법적으로 합법이지만 윤리적으로 문제
- 좋은 의도지만 나쁜 결과
- 지금은 괜찮지만 나중에 문제

## 거부의 기준

**거부해야 할 때:**

- 명백히 해로운 목적
- 사용자를 속이는 기능
- 프라이버시 침해
- 차별을 자동화
- 민주주의 훼손

**판단이 어려울 때:**

- 동료와 논의
- 윤리 위원회 참조
- 외부 전문가 상담
- 공개 토론

## 시스템 설계로 방지

악용을 어렵게 만드는 설계 요소이다.

- 레이트 리미팅 (대량 작업 제한)
- 인증 강화
- 감사 로그
- 데이터 최소 수집
- 기능 제한 (범위 한정)

## 내부 고발

조직이 해를 끼칠 때 단계별 대응 순서이다.

1. 내부에서 문제 제기
2. 더 높은 레벨로 에스컬레이션
3. 법적 보호 하에 외부 신고
4. 떠나기

"회사가 시켰다"는 변명이 아니다.

## 장기적 관점

| 단기 이익 | 장기 결과 |
|-----------|-----------|
| "이 계약이 돈이 되니까" | 평판 손상 |
| "일자리가 필요하니까" | 법적 책임 / 양심의 가책 / 사회적 해악 |

## 정리

- 우리가 만드는 것이 어떻게 쓰일지 고려한다.
- 악용 가능성을 미리 생각한다.
- 해로운 프로젝트는 거부할 수 있어야 한다.
- 시스템 설계로 악용을 어렵게 한다.
- "시켰다"는 변명이 아니다.
- 단기 이익보다 장기 결과를 본다.
