## 이 팁의 메시지

> **Tip 96: Sign Your Work.** Craftsmen of an earlier age were proud to sign their work. You should be, too.

과거의 장인들은 자신의 작품에 서명하는 것을 자랑스러워했다. 당신도 그래야 한다.

## 장인의 서명

중세 석공은 자신이 깎은 돌에 마크를 남겼다. 그 마크는 품질의 보증이었다.

```text
서명의 의미:
- "이것은 내가 만들었다"
- "내 이름을 걸고 품질을 보증한다"
- "문제가 있으면 나를 찾아라"
- "이 일에 자부심을 느낀다"
```

## 코드에 서명한다는 것

```text
물리적 서명이 아니라:
- git commit에 이름이 남는다
- 코드 리뷰에 이름이 남는다
- 버그 리포트에 이름이 연결된다

자부심 있는 서명:
- 내 이름이 붙어도 부끄럽지 않은 코드
- 다른 사람이 봐도 자랑스러운 코드
- 미래의 내가 봐도 괜찮은 코드
```

## 익명성의 문제

```text
익명 코드의 유혹:
"어차피 내 이름 안 남으니까"
"아무도 모르니까 대충 해도"
"나중에 누가 고치겠지"

결과:
- 품질 저하
- 책임 회피
- 기술 부채 누적
```

## 소유권과 책임

| 익명 | 서명 |
|------|------|
| "누가 이렇게 했어?" | "내가 했고, 책임진다" |
| 책임 회피 | 책임 수용 |
| 최소한의 노력 | 최선의 노력 |
| 대충 넘어감 | 끝까지 마무리 |

## 서명할 수 있는 코드

```text
서명할 수 있으려면:
- 읽기 쉬운 코드
- 테스트가 있는 코드
- 문서화된 코드
- 설계가 깔끔한 코드
- 보안이 고려된 코드
```

## 팀 서명

```text
개인 서명:
"이 함수는 내가 작성했다"

팀 서명:
"이 시스템은 우리 팀이 만들었다"

팀 서명의 조건:
- 코드 리뷰로 품질 보장
- 공동 소유권
- 집단 책임
- 일관된 표준
```

## 나쁜 코드에서 벗어나기

```text
상속받은 나쁜 코드:
- 내 서명이 아닌 코드
- 하지만 이제 내가 관리

할 수 있는 것:
1. 손대는 부분부터 개선
2. 테스트 추가
3. 점진적 리팩토링
4. 보이스카우트 규칙 적용
```

## 자부심의 균형

```text
지나친 자부심:
- "내 코드는 완벽해" (교만)
- "건드리지 마" (방어적)
- "내 방식이 최고야" (독단)

건강한 자부심:
- "최선을 다했다"
- "피드백 환영한다"
- "더 나아질 수 있다"
- "책임진다"
```

## 레거시

```text
내가 떠난 후에도:
- 코드가 남는다
- 아키텍처가 남는다
- 문화가 남는다
- 평판이 남는다

어떤 레거시를 남기고 싶은가?
```

## 정리

- 장인은 자신의 작품에 서명한다.
- 이름을 걸고 품질을 보증한다.
- 익명성은 품질 저하를 유혹한다.
- 내 이름이 붙어도 부끄럽지 않게 한다.
- 팀 서명은 공동 책임이다.
- 건강한 자부심과 책임감을 갖는다.
