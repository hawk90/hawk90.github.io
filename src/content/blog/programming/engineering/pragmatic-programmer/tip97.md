## 이 팁의 메시지

> **Tip 97: First, Do No Harm.** Failure to do no harm is a failure of imagination.

해를 끼치지 않는 것에 실패하는 것은 상상력의 실패다.

## 히포크라테스 선서

의사의 첫 번째 원칙은 "해를 끼치지 마라"다. 개발자에게도 적용된다.

```text
우리 코드가 할 수 있는 것:
- 개인 정보를 노출한다
- 사용자를 중독시킨다
- 차별을 자동화한다
- 환경을 해친다
- 민주주의를 위협한다
```

## 의도하지 않은 결과

```text
의도: 사용자 참여 증가
수단: 무한 스크롤, 알림, 게임화
결과: 스마트폰 중독, 정신 건강 악화

의도: 효율적인 채용
수단: AI 이력서 필터링
결과: 기존 편향 확대, 차별 자동화

의도: 맞춤형 콘텐츠
수단: 추천 알고리즘
결과: 필터 버블, 극단화
```

## 질문해야 할 것

```text
기능 개발 전:
- 이것이 오용될 수 있는가?
- 취약한 사용자에게 해가 되는가?
- 의도하지 않은 결과가 있을 수 있는가?
- 최악의 경우는 무엇인가?

데이터 수집 전:
- 꼭 필요한 데이터인가?
- 안전하게 보관할 수 있는가?
- 사용자가 알고 있는가?
- 동의를 받았는가?
```

## 다크 패턴

```text
사용자를 속이는 디자인:
- 숨겨진 비용
- 구독 취소 어렵게
- 수치심 유발 문구 ("정말 포기하시겠어요?")
- 혼란스러운 선택지
- 강제 지속
```

이런 것을 만들라고 하면 거부할 수 있어야 한다.

## 프라이버시

```text
수집하지 않는 것이 최선:
- 필요 없는 데이터는 수집하지 않는다
- 수집한 데이터는 보호한다
- 사용 목적을 명확히 한다
- 삭제 요청을 존중한다

"나중에 쓸 수 있으니까" → 위험
```

## 접근성

```text
해를 끼치지 않는 것은:
- 장애인도 사용할 수 있게 하는 것
- 느린 네트워크도 고려하는 것
- 오래된 기기도 지원하는 것
- 기술에 익숙하지 않은 사람도 배려하는 것
```

배제는 해를 끼치는 것이다.

## 환경

```text
소프트웨어의 환경 영향:
- 서버 전력 소비
- 불필요한 데이터 전송
- 비효율적인 코드
- 짧은 하드웨어 수명

더 적은 자원으로 같은 일을 할 수 있다면 그렇게 한다.
```

## "아니오"라고 말하기

```text
해가 되는 것을 요청받았을 때:
1. 우려를 표현한다
2. 대안을 제시한다
3. 에스컬레이션한다
4. 필요하면 거부한다

"명령을 따랐을 뿐"은 변명이 아니다.
```

## 정리

- 소프트웨어는 해를 끼칠 수 있다.
- 의도하지 않은 결과를 미리 상상한다.
- 다크 패턴을 만들지 않는다.
- 프라이버시를 존중한다.
- 모든 사람이 접근할 수 있게 한다.
- 해가 되는 요청은 거부할 수 있어야 한다.
