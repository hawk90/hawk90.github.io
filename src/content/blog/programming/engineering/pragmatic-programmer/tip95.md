## 이 팁의 메시지

> **Tip 95: Delight Users, Don't Just Deliver Code.** Your goal is to delight users, not just to deliver code.

당신의 목표는 사용자를 기쁘게 하는 것이지, 코드를 전달하는 게 아니다.

## 코드 전달 vs 가치 전달

| 코드 전달 사고 | 가치 전달 사고 |
|----------------|----------------|
| "요청한 기능 구현했습니다" | "이제 고객 응대가 30% 빨라집니다" |
| "명세대로 동작합니다" | "월말 보고서 작성 시간이 1시간에서 10분으로" |
| "버그 수정했습니다" | "환불 처리가 자동화되어 CS팀 업무가 줄었습니다" |

## 사용자가 원하는 것

사용자가 말하는 것과 *진짜 원하는 것*은 다릅니다.

- 사용자가 말하는 것 — "주문 검색 기능이 필요해요"
- 사용자가 진짜 원하는 것 — "고객 전화가 오면 빨리 주문을 찾고 싶어요"

더 나은 해결책:

- 전화번호 자동 인식 → 주문 자동 표시
- 최근 주문 바로 표시
- 검색 없이도 찾을 수 있게

## 기쁨의 요소

| 수준 | 설명 | 예 |
|------|------|-----|
| 기본 | 동작함 | 버튼 누르면 실행됨 |
| 기대 | 예상대로 동작 | 빠르고 정확함 |
| 기쁨 | 예상을 넘어섬 | 알아서 제안해줌 |

## 기쁨 주는 작은 것들

예상을 넘는 경험을 만드는 디테일입니다.

- 저장하지 않아도 자동 저장
- 오타를 자동 수정
- 자주 쓰는 항목 먼저 표시
- 다음에 할 일 제안
- 진행 상황을 실시간 표시
- 에러 메시지에 해결 방법 포함

## 사용자 관점으로 생각

| 개발자 관점 | 사용자 관점 |
|-------------|-------------|
| "API 응답 시간 200ms" | "클릭하면 바로 나와요" |
| "99.9% 가동률" | "항상 쓸 수 있어요" |
| "테스트 커버리지 80%" | "에러가 거의 없어요" |

## 비즈니스 목표 이해

기능을 만들기 전 던지는 질문입니다.

- 이 기능으로 누가 이득을 보는가?
- 얼마나 이득을 보는가?
- 이 기능이 없으면 어떻게 되는가?
- 더 큰 이득을 줄 수 있는 방법은?

## 성공 측정

| 코드 지표 (덜 중요) | 비즈니스 지표 (더 중요) |
|---------------------|-------------------------|
| 코드 줄 수 | 사용자 만족도 |
| 커밋 수 | 업무 처리 시간 단축 |
| 기능 수 | 에러율 감소 / 재방문율 / 매출 증가 |

## 피드백 수집

다양한 채널에서 *지속적으로* 모읍니다.

- 직접 인터뷰
- 사용 관찰
- 설문 조사
- 지원 티켓 분석
- 사용 패턴 분석
- A/B 테스트

## 기대 관리

과잉 약속보다 *과잉 전달*이 낫습니다.

- 나쁜 예 — "다음 주까지 완성할게요" → 2주 걸림 → 실망
- 좋은 예 — "2주 정도 걸릴 것 같아요" → 10일 만에 완성 → 기쁨

## 정리

- 코드가 아니라 사용자 기쁨이 목표다.
- 사용자가 말하는 것과 원하는 것을 구분한다.
- 예상을 넘는 경험이 기쁨을 준다.
- 비즈니스 지표로 성공을 측정한다.
- 사용자 피드백을 지속적으로 수집한다.
- 과잉 약속보다 과잉 전달한다.
