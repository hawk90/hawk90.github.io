## 이 팁의 메시지

> **Tip 81: Don't Go into the Code Alone.** Programming can be difficult and demanding. Take a friend.

프로그래밍은 어렵고 힘들다. 친구를 데려가라.

## 혼자 코딩의 위험

혼자 코딩하면 맹점이 생긴다.

**혼자일 때:**

- 같은 실수를 반복한다
- 터널 시야에 빠진다
- 더 나은 방법을 놓친다
- 설계 결정을 합리화한다
- 막히면 몇 시간씩 헤맨다

두 사람의 눈이 한 사람보다 낫다.

## 짝 프로그래밍

두 사람이 한 컴퓨터로 함께 코딩한다.

| 역할 | 하는 일 |
|------|---------|
| 드라이버 | 키보드를 치고 코드를 작성 |
| 내비게이터 | 전체를 보고 방향을 제시 |

**짝 프로그래밍 흐름:**


**1. 문제를 함께 이해한다**


**2. 드라이버가 코드를 작성한다**


**3. 내비게이터가 질문하고 제안한다**


**4. 20~30분마다 역할을 바꾼다**


**5. 둘 다 코드를 소유한다**

## 몹 프로그래밍

팀 전체가 한 화면으로 함께 코딩한다.

**몹 프로그래밍:**

- 3~6명이 한 화면을 본다
- 한 명이 타이핑, 나머지가 방향 제시
- 5~10분마다 타이피스트 교체
- 모든 결정이 팀 합의로

복잡한 문제, 중요한 설계에 효과적이다.

## 코드 리뷰

비동기적으로 함께 코드를 본다.

**코드 리뷰 체크리스트:**

- 요구사항을 만족하는가?
- 읽기 쉬운가?
- 테스트가 충분한가?
- 에러 처리가 적절한가?
- 보안 문제가 없는가?
- 더 간단한 방법이 있는가?

## 함께할 때의 이점

| 혼자 | 함께 |
|------|------|
| 맹점 발생 | 맹점 발견 |
| 지식 고립 | 지식 공유 |
| 막히면 혼자 고민 | 막히면 바로 해결 |
| 내 스타일만 | 다양한 관점 |
| 코드 설명 없음 | 설명하며 이해 깊어짐 |

## 러버덕 디버깅

짝이 없을 때 고무 오리에게 설명한다.

**러버덕 디버깅:**


**1. 문제를 오리에게 설명한다**


**2. 한 줄씩 코드를 설명한다**


**3. "이 줄은 왜 필요하지?"**


**4. 설명하다 보면 문제를 발견한다**

설명하는 행위 자체가 생각을 정리한다.

## 원격 협업

물리적으로 같은 공간이 아니어도 된다.

**원격 짝 프로그래밍 도구:**

- VS Code Live Share
- JetBrains Code With Me
- Tuple
- 화면 공유 + 음성 통화

## 언제 함께하면 좋은가

| 상황 | 작업 유형 |
|------|----------|
| 혼자 | 단순 반복 작업, 탐색적 프로토타입 |
| 함께 | 복잡한 버그, 설계 결정, 새 기술 학습, 중요한 코드, 온보딩, 지식 전파 |

## 정리

- 혼자 코딩하면 맹점이 생긴다.
- 짝 프로그래밍으로 두 눈을 활용한다.
- 몹 프로그래밍으로 팀 전체의 지혜를 모은다.
- 코드 리뷰로 비동기적으로 협업한다.
- 설명하는 행위 자체가 이해를 깊게 한다.
- 복잡한 문제, 중요한 코드는 함께 작업한다.
