> *Test-Driven Development by Example* Chapter 11 요약

## 이 챕터의 메시지

10장에서 `Money`가 구체 클래스가 되면서 `Dollar`와 `Franc`이 빈 껍데기처럼 남았다. 이 장은 그 두 클래스를 완전히 제거한다. 6장에서 신중하게 도입했던 상속 계층이 결국 사라진다. Beck은 이 사실에 대해 통렬한 결론을 내린다. *Premature inheritance가 모든 악의 근원이다.*

## 제거 후의 코드

남는 것은 `Money` 한 클래스뿐이다.

```java
class Money {
    private int amount;
    private String currency;

    Money(int amount, String currency) {
        this.amount = amount;
        this.currency = currency;
    }

    static Money dollar(int amount) {
        return new Money(amount, "USD");
    }

    static Money franc(int amount) {
        return new Money(amount, "CHF");
    }

    Money times(int multiplier) {
        return new Money(amount * multiplier, currency);
    }

    String currency() {
        return currency;
    }

    public boolean equals(Object obj) {
        Money money = (Money) obj;
        return amount == money.amount
            && currency.equals(money.currency);
    }
}
```

`Dollar`와 `Franc`이라는 단어는 Factory 메서드 이름에만 남았다. 모든 테스트가 통과한다.

## 외부 인터페이스의 일관성

사용자 코드는 6장부터 지금까지 무엇이 바뀌었을까?

```java
// 6장
Money five = new Dollar(5);
Money ten = five.times(2);

// 11장
Money five = Money.dollar(5);
Money ten = five.times(2);
```

`new Dollar(5)`가 `Money.dollar(5)`로 바뀐 것 하나뿐이다. 그 작은 외부 변경 뒤에서 상속 계층 전체가 만들어졌다가 사라졌다. 8장의 Factory Method 도입이 이 큰 재편을 안전하게 만들었다.

## "Root of All Evil"의 의미

제목은 1 Timothy 6:10의 *"돈에 대한 사랑이 모든 악의 근원이다"*를 패러디한 것이다. Beck이 가리키는 *root of all evil*은 **premature inheritance**다.

처음에 `Dollar`와 `Franc`을 별도 클래스로 만든 결정은 합리적으로 보였다. 두 가지 통화가 다른 동작을 할 수도 있다는 가정이 자연스러웠다. 그러나 실제로는 차이가 통화 문자열 하나뿐이었다.

| 시점 | 디자인 | 정당한가? |
|------|--------|----------|
| ch05 도입 시 | `Franc`을 `Dollar` 복사로 만듦 | 중복은 명확, 통합은 추후 |
| ch06 부모 추출 | `Money` 추상 부모 | 그 시점에선 합리적 |
| ch10 구체화 | `Money`가 직접 인스턴스화 | 자식 동작이 없음을 발견 |
| ch11 제거 | `Dollar`/`Franc` 사라짐 | 진짜 답이 드러남 |

처음부터 *진짜 답*을 알 수 있었을까? 아마 아니다. 디자인은 코드와 함께 자라난다. 그러나 한 가지 원칙은 살아남는다. *상속을 너무 일찍 도입하지 마라.*

## 상속의 비용

상속은 객체 지향의 가장 강력한 도구지만 가장 비싼 도구이기도 하다.

- 자식이 부모의 모든 구현 상세에 결합한다
- 자식 추가가 부모의 약속을 바꾼다 (Liskov 위반의 유혹)
- 다중 상속 / interface mixin / composition으로 풀 수 있는 문제를 상속으로 풀면 나중에 후회한다

Joshua Bloch의 *Effective Java* Item 18은 *Prefer composition over inheritance*라고 단언한다. *Refactoring*의 **Replace Inheritance with Delegation**도 같은 정신이다. 상속은 진짜 필요할 때만 도입하고, 의심스러우면 컴포지션이다.

## 모든 테스트 통과

11장의 변경은 큰 디자인 변경이지만 모든 테스트가 통과한다. 그 사실은 다음을 보장한다.

- 외부 동작이 그대로다
- 사용자 코드를 깨지 않는다
- 다음 단계로 안전하게 갈 수 있다

이게 *"테스트가 진화적 디자인을 가능하게 한다"*의 구체적 의미다.

## 정리

- `Dollar`/`Franc` 클래스를 완전히 제거 — `Money` 한 클래스로 통합
- **Premature inheritance**가 결국 비용으로 돌아왔다
- 그러나 진화 과정이 잘못이라 단정할 수는 없다 — 답은 코드와 함께 자란다
- 모든 테스트가 통과 — 외부 인터페이스는 안정
