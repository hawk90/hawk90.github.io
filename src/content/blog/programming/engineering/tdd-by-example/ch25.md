## 패턴 1 — Isolated Test

> 테스트는 서로 독립이어야 한다.

한 테스트의 결과가 다른 테스트의 결과에 영향을 주면 안 된다. 순서를 바꿔도, 일부만 돌려도 결과가 같아야 한다.

```python
# 나쁜 예 — 공유 상태
class TestThing(TestCase):
    counter = 0      # 클래스 변수, 모든 테스트가 공유

    def test_increment(self):
        TestThing.counter += 1
        assert TestThing.counter == 1   # 첫 번째 실행에서만 통과

# 좋은 예 — setUp으로 매번 초기화
class TestThing(TestCase):
    def setUp(self):
        self.counter = 0

    def test_increment(self):
        self.counter += 1
        assert self.counter == 1
```

19장의 `setUp`이 이 패턴의 토대다. 매 테스트가 깨끗한 fixture에서 출발한다.

## 패턴 2 — Test List

> 떠오르는 테스트와 리팩터링, 가정을 리스트에 적는다.

1장부터 책 전체에서 등장한 To-Do List다. 작업 중에 떠오르는 모든 생각을 즉시 적고, 현재 작업에만 집중한다.

**TODO:**

- [ ] $5 × 2 = $10
- [ ] Currency 비교
- [ ] equals(null) 처리
- [ ] 환율 적용
- [ ] hashCode

리스트는 *작업의 큐*가 아니라 *집중의 도구*다. 다음 한 가지만 보고, 나머지는 보관한다. 한 가지가 끝나면 리스트로 돌아간다.

## 패턴 3 — Test First

> 테스트를 먼저 짜라. 그 다음 통과시키는 코드.

TDD의 본질이다. 코드를 짜고 그 코드를 위한 테스트를 짜는 것이 아니라, 그 반대다.

이점은 세 가지다.

| 이점 | 의미 |
|------|------|
| 사용자 시각이 먼저 | 테스트는 코드의 첫 사용자 — API 결정이 자연스러워짐 |
| 안 짜도 되는 코드를 안 짠다 | 테스트가 요구하지 않으면 만들 필요 X |
| 테스트가 디자인 도구가 된다 | 테스트하기 어려운 코드는 디자인이 나쁘다는 신호 |

세 번째 항목이 깊다. GOOS(Freeman·Pryce)는 *"Listen to your tests"*라는 슬로건으로 이 사실을 강조한다. 테스트가 짜기 어려우면 디자인을 바꾼다.

## 패턴 4 — Assert First

> assert를 먼저 짜라.

테스트 작성의 순서를 뒤집는다. 보통은 setup → action → assert 순서로 짜지만, *역방향*으로 짜면 의도가 더 명확해진다.

```python
# 보통의 순서 (앞에서 뒤로)
def test_x():
    setup()
    actual = action()
    assertEquals(expected, actual)

# Assert First (뒤에서 앞으로)
def test_x():
    assertEquals(expected, actual)      # 1) 무엇을 검증?
    actual = something_returning_value()  # 2) 어디서?
    setup_something()                     # 3) 어떻게 준비?
```

첫 단계에서 *무엇을 검증할지*를 명확히 한다. 그게 정해지면 그것을 만들기 위해 무엇이 필요한지가 자연스럽게 나온다. *결과에서 출발해 입력으로*라는 사고 흐름이다.

## 패턴 5 — Test Data

> 테스트 데이터는 의도를 드러내야 한다.

데이터가 무엇을 의미하는지 코드 자체에서 보여야 한다.

```python
# 모호한 데이터 — 왜 10인가?
assertEquals(10, money.amount * 2)

# 명확한 데이터 — 의도가 드러남
assertEquals(20, Money.dollar(10).times(2).amount)
```

테스트는 코드의 *문서*이기도 하다. 6개월 뒤 누가 테스트를 봐도 *왜 그 데이터인지*가 보여야 한다. 매직 넘버를 피하고 의미 있는 이름을 쓴다.

## 패턴 6 — Evident Data

> 데이터의 출처가 분명해야 한다.

Test Data의 한 단계 깊은 버전이다. 데이터가 어디서 왔는지 코드 안에 표시한다.

```python
# 출처 불명 — 왜 7392인가?
assertEquals(7392, calculate_value())

# 출처 명시
days_per_year = 365
weeks_per_year = 52
result = days_per_year + weeks_per_year * 134  # 명시적 유래
assertEquals(result, calculate_value())
```

두 번째 버전은 *계산 과정 자체가 테스트의 일부*다. 미래의 독자가 *어떻게 7392가 나왔는지*를 코드에서 읽을 수 있다.

이게 *Refactoring*의 **Replace Magic Number with Symbolic Constant**와 같은 정신이다.

## 정리

- **Isolated Test** — 매 테스트가 독립, setUp으로 매번 초기화
- **Test List** — 떠오르는 생각의 보관소, 집중을 유지
- **Test First** — 테스트가 먼저, 디자인 도구가 된다
- **Assert First** — assert에서 출발해 setup으로
- **Test Data** — 의도가 드러나는 데이터
- **Evident Data** — 출처가 보이는 데이터
