> **Test-Driven Development by Example** Chapter 25 요약

## 이 챕터의 메시지

Part III 시작 — **TDD의 핵심 패턴들**. 1-24장에서 등장한 모든 패턴을 체계적으로.

## 패턴 1 — Isolated Test

> **테스트는 서로 독립적이어야 한다.**

한 테스트의 결과가 — 다른 테스트에 영향 X.

```python
test1: 객체 A 사용, 상태 변경
test2: 같은 객체 A 사용 → 의도 안 한 상태?
```

해법 — **매 테스트마다 새 객체**. setUp으로 매번 새로.

## 패턴 2 — Test List

> **떠오르는 테스트 / 리팩터링 / 가정을 — 리스트에 적는다.**

```
TODO:
[ ] $5 × 2 = $10
[ ] Currency 비교
[ ] equals null 처리
[ ] 환율 적용
```

생각이 떠올라도 — 지금 작업만 집중. 리스트가 — 다음 단계를 보존.

## 패턴 3 — Test First

> **테스트를 먼저 짜라. 그 다음 통과시키는 코드.**

TDD의 본질. 코드 짜고 — 테스트 짜는 게 아니라 — 그 반대.

장점:
- 코드의 사용자 시각이 먼저
- 안 짜도 되는 코드를 안 짜게 됨
- 테스트가 — 디자인 도구가 됨

## 패턴 4 — Assert First

> **assert를 먼저 짜라.**

테스트 작성 순서를 뒤집음.

```python
# 보통:
def test_x():
    setup()
    action()
    assertEquals(expected, actual)   # 마지막에 assert

# Assert First:
def test_x():
    assertEquals(expected, actual)   # 먼저 — 무엇을 검증?
    actual = ...                       # 어디서 오는가?
    setup = ...                        # 어떻게 준비?
```

뒤에서부터 앞으로 작성. 의도가 더 명확.

## 패턴 5 — Test Data

> **테스트 데이터는 명확해야 한다.**

```python
# 모호:
assertEquals(10, money.amount * 2)

# 명확:
assertEquals(20, Money.dollar(10).times(2).amount)
```

데이터가 — 의도를 드러내야.

## 패턴 6 — Evident Data

> **데이터의 출처가 분명해야 한다.**

```python
# 출처 불명:
assertEquals(7392, calculate())

# 출처 명시:
days_per_year = 365
assertEquals(days_per_year + 27, calculate())   # 왜 +27인지 코드에
```

매직 넘버를 — 의미 있는 표현으로.

## 정리

- **Isolated Test** — 매 테스트 독립
- **Test List** — 생각 보관, 집중 유지
- **Test First** — 테스트가 먼저, 그게 디자인 도구
- **Assert First** — assert부터 거꾸로 작성
- **Test Data** — 명확
- **Evident Data** — 출처 명시
