> *Test-Driven Development by Example* Chapter 14 요약

## 이 챕터의 메시지

13장에서 `Bank.reduce`가 `Sum`만 처리할 수 있었다. `Money` 단독 식을 넘기면 캐스트 실패다. 이 장은 `reduce`를 `Expression` 인터페이스의 메서드로 옮긴다. 각 구현 클래스가 자신을 어떻게 축소할지 알게 된다. 다형성이 `Bank`에서 `if`/`instanceof` 분기를 없앤다.

이 변경은 디자인의 정신을 한 단계 끌어올린다. *식이 자신을 축소할 줄 안다.* `Bank`는 단순한 위임자가 된다.

## reduce를 Expression으로

`Expression` 인터페이스에 `reduce`를 둔다.

```java
interface Expression {
    Money reduce(String to);
}

class Money implements Expression {
    public Money reduce(String to) {
        return this;   // 자기 자신
    }

    Expression plus(Money addend) {
        return new Sum(this, addend);
    }
}

class Sum implements Expression {
    Money augend;
    Money addend;

    Sum(Money augend, Money addend) {
        this.augend = augend;
        this.addend = addend;
    }

    public Money reduce(String to) {
        int amount = augend.amount + addend.amount;
        return new Money(amount, to);
    }
}
```

`Money.reduce`는 자기를 그대로 반환한다(아직 환율 없음). `Sum.reduce`는 augend와 addend의 amount를 더해서 새 `Money`를 만든다.

## Bank의 단순화

`Bank.reduce`는 이제 분기 없이 위임만 한다.

```java
class Bank {
    Money reduce(Expression source, String to) {
        return source.reduce(to);
    }
}
```

`Bank`는 `Expression`이 무엇인지 모른다. `Money`인지 `Sum`인지 신경 쓰지 않는다. 그저 `reduce`를 호출할 뿐이다. 다형성이 자기 일을 한다.

## Replace Conditional with Polymorphism

이 리팩터링은 *Refactoring* 카탈로그의 **Replace Conditional with Polymorphism**과 같은 정신이다. 이전 코드에서 `Bank`는 입력 타입에 따라 분기할 수밖에 없었다.

```java
// 만약 polymorphism 없이 확장한다면
class Bank {
    Money reduce(Expression source, String to) {
        if (source instanceof Sum) {
            // Sum 처리
        } else if (source instanceof Money) {
            // Money 처리
        } else if (source instanceof Product) {  // 미래에 추가?
            // Product 처리
        }
    }
}
```

새 `Expression` 종류가 추가될 때마다 `Bank`의 분기가 늘어난다. *Open/Closed Principle* 위반이다. 다형성으로 분기를 옮기면 새 종류를 추가해도 `Bank`는 그대로 둔다.

## Bank의 진짜 책임은 무엇인가

`Bank`가 단순한 위임자라면 왜 필요할까? 지금은 의미가 약해 보이지만, 15장에서 환율 표가 도입되면 `Bank`의 진짜 책임이 드러난다. *Bank는 환율 정보의 보관소다.* `reduce`는 환율을 적용하기 위해 `Bank`가 필요하다.

```java
class Bank {
    Money reduce(Expression source, String to) {
        return source.reduce(this, to);   // 15장에서 변경
    }
}
```

이미 그 변경의 방향이 보인다. `reduce`에 `Bank` 자신을 넘겨주면, `Money`와 `Sum`이 환율을 조회할 수 있다.

## 디자인의 진화

이 사이클에서 발견되는 정신은 다음과 같다.

```text
1. 처음 — Bank가 Sum을 직접 캐스트
2. 한 걸음 — reduce를 Expression의 메서드로 옮김
3. 다음 단계 — 환율 정보를 Bank가 제공 (15장)
```

각 사이클이 책임을 더 자연스러운 곳으로 옮긴다. 객체 지향 디자인의 정신을 그대로 따른다.

## 정리

- `reduce`를 `Expression`의 메서드로 옮긴다 — **Replace Conditional with Polymorphism**
- `Money.reduce`는 자신을 그대로, `Sum.reduce`는 합산
- `Bank`는 단순한 위임자가 된다
- 새 `Expression` 종류 추가가 `Bank` 변경 없이 가능 — Open/Closed
- 다음 장에서 환율이 추가되면 `Bank`의 진짜 책임이 드러난다
