> *Test-Driven Development by Example* Chapter 23 요약

## 이 챕터의 메시지

지금까지 테스트를 한 번에 하나씩 돌렸다. 실제 프로젝트는 수십에서 수천 개의 테스트를 한 번에 실행해야 한다. 이를 위한 도구가 `TestSuite`다. 여러 `TestCase`를 묶어서 한 번의 `run(result)` 호출로 모두 실행한다.

이 작은 추가가 자연스럽게 **Composite** 패턴을 만든다. `TestCase`와 `TestSuite`가 같은 `run(result)` 인터페이스를 따르므로, `TestSuite`가 다른 `TestSuite`를 포함하는 트리 구조까지 가능해진다.

## TestSuite 구현

가장 단순한 형태다. 테스트 리스트와 `add` 메서드, 그리고 모두를 돌리는 `run`이다.

```python
class TestSuite:
    def __init__(self):
        self.tests = []

    def add(self, test):
        self.tests.append(test)

    def run(self, result):
        for test in self.tests:
            test.run(result)
```

`tests`에 들어가는 것은 `TestCase`일 수도, 다른 `TestSuite`일 수도 있다. 같은 `run(result)` 메서드만 있으면 된다.

## 사용 예

```python
suite = TestSuite()
suite.add(TestCaseTest("testTemplateMethod"))
suite.add(TestCaseTest("testResult"))
suite.add(TestCaseTest("testFailedResult"))
suite.add(TestCaseTest("testSuite"))

result = TestResult()
suite.run(result)
print(result.summary())
# "4 run, 0 failed"
```

네 테스트를 한 번에 돌리고 누적된 result로 summary를 출력한다.

## Composite 패턴의 등장

`TestCase`와 `TestSuite`는 같은 인터페이스를 따른다.

```text
TestCase.run(result)   — 한 테스트 실행
TestSuite.run(result)  — 묶인 테스트들 실행
```

호출자는 둘을 구별할 필요가 없다. `something.run(result)`만 호출한다. 이게 GoF의 **Composite** 패턴이다. *부분과 전체를 같은 인터페이스로 다룬다.*

12-16장에서 본 `Money`/`Sum`/`Expression`과 같은 구조다.

| 영역 | Composite의 leaf | Composite의 branch |
|------|------------------|---------------------|
| Money 예제 | `Money` | `Sum` |
| xUnit | `TestCase` | `TestSuite` |
| GUI | `Button` 같은 위젯 | `Panel` 같은 컨테이너 |
| 파일시스템 | `File` | `Directory` |

같은 정신이 다른 옷을 입었다.

## 트리 구조

`TestSuite`가 다른 `TestSuite`도 포함할 수 있으므로 트리가 가능하다.

```python
core_suite = TestSuite()
core_suite.add(TestCore("testA"))
core_suite.add(TestCore("testB"))

ui_suite = TestSuite()
ui_suite.add(TestUI("testRender"))
ui_suite.add(TestUI("testClick"))

all_tests = TestSuite()
all_tests.add(core_suite)
all_tests.add(ui_suite)

all_tests.run(result)   # 모두 실행
```

큰 프로젝트의 테스트 구조가 이런 트리다. JUnit의 `@RunWith(Suite.class)`, pytest의 마커 시스템, NUnit의 카테고리 등이 모두 이 트리를 만드는 다양한 방법이다.

## 자동 검출의 길

현대 xUnit은 보통 `TestCase`를 손으로 `add`하지 않는다. 자동 검출이 가능하다.

```python
# pytest의 경우
# - test_*.py 파일을 찾아서
# - Test* 클래스를 찾아서
# - test_* 메서드를 찾아서
# - 자동으로 suite에 추가
```

Beck의 책은 자동 검출까지 가지 않는다. 29장에서 **All Tests** 패턴으로 가볍게 언급한다. 핵심 메커니즘 — `TestSuite`라는 추상 — 은 23장에서 이미 만들어졌다. 나머지는 그 위의 자동화일 뿐이다.

## 부트스트랩의 마지막 한 조각

이제 자기 자신을 자기로 검증하는 부트스트랩의 마지막 조각이 완성된다.

```python
class TestCaseTest(TestCase):
    def testSuite(self):
        suite = TestSuite()
        suite.add(WasRun("testMethod"))
        suite.add(WasRun("testBrokenMethod"))
        result = TestResult()
        suite.run(result)
        assert "2 run, 1 failed" == result.summary()
```

`TestCase`가 `TestSuite`까지 자기로 검증한다. 컴파일러가 자기 자신을 컴파일하는 것과 정확히 같은 정신이다.
