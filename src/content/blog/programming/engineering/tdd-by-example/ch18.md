> *Test-Driven Development by Example* Chapter 18 요약

## 이 챕터의 메시지

Part II가 시작된다. 새 도전은 *xUnit 테스트 프레임워크 자체를 TDD로 짜는 것*이다. 여기서 흥미로운 닭과 달걀 문제가 등장한다. *테스트 프레임워크 없이 테스트 프레임워크의 동작을 어떻게 검증할까?*

Beck의 답은 단순하다. 가장 원시적인 첫 버전을 손으로 검증하고, 그 버전이 자기 자신을 검증할 수 있을 정도로 자랄 때까지 작은 단계를 누적한다. 부트스트랩(bootstrapping)의 정신이다. 컴파일러가 자기 자신을 컴파일하는 것과 같다.

언어는 Python을 사용한다. Beck이 책을 쓸 때 Python을 골랐는데, 동적 타입의 간결함이 TDD 사이클을 짧게 보여 주기에 좋기 때문이다.

## 가장 단순한 TestCase

`TestCase`의 첫 버전은 메서드 이름을 받고 그것을 호출하는 것이 전부다.

```python
class TestCase:
    def __init__(self, name):
        self.name = name

    def run(self):
        method = getattr(self, self.name)
        method()
```

`getattr`은 객체에서 속성을 동적으로 가져온다. 이름이 `"testMethod"`면 `self.testMethod`를 가져와 호출한다. 이게 xUnit의 핵심 메커니즘이다. *테스트 메서드를 이름으로 동적 호출*한다.

## 첫 테스트 — WasRun

`TestCase`를 상속한 `WasRun`이 테스트 대상이자 테스트 자체다.

```python
class WasRun(TestCase):
    def __init__(self, name):
        TestCase.__init__(self, name)
        self.wasRun = None

    def testMethod(self):
        self.wasRun = 1
```

`testMethod`가 호출되면 `wasRun`이 `1`로 설정된다. 호출되지 않았다면 `None`이다.

## print로 첫 검증

진짜 `assertEquals`가 아직 없다. 검증은 `print`로 한다.

```python
test = WasRun("testMethod")
print(test.wasRun)   # None — 아직 안 돌림
test.run()
print(test.wasRun)   # 1 — 돌렸음
```

두 줄의 출력이 `None` 다음 `1`이면 `run`이 `testMethod`를 호출한다는 사실이 검증된다. 사람의 눈이 assertion 역할이다.

원시적이지만 의도된 출발점이다. 부트스트랩의 첫 발걸음은 항상 손으로 검증한다.

## 부트스트랩의 정신

부트스트랩의 정신은 컴퓨터 과학 전반에 흐른다.

- **컴파일러 자기 호스팅** — C 컴파일러를 C로 짠다. 처음 한 번은 기존 컴파일러(어셈블리 또는 다른 언어)로 컴파일한다. 그 뒤로는 자기로 자기를 컴파일한다.
- **운영체제 부팅** — BIOS가 1단계 로더를 메모리에 올린다. 1단계가 2단계를 올린다. 2단계가 커널을 올린다. 작은 것이 큰 것을 차례로 띄운다.
- **xUnit** — 손으로 검증한 작은 TestCase가 자라서 자기 자신을 검증한다.

각각 *원시적 외부 검증 → 점진적 자기 검증*의 길을 따른다.

## 다음 단계의 방향

이 첫 버전에는 분명한 한계가 있다.

| 한계 | 해결 장 |
|------|--------|
| `setUp`이 없음 | ch19 |
| `tearDown`이 없음 | ch20 |
| 결과를 카운트하지 않음 | ch21 |
| 예외 처리가 없음 | ch22 |

각 장이 한 한계를 풀어 나간다. 6개 사이클이 끝나면 진짜 xUnit이 된다.
