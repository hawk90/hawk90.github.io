> *Test-Driven Development by Example* Chapter 20 요약

## 이 챕터의 메시지

19장의 `setUp`이 테스트 전 준비를 담당했다면, `tearDown`은 테스트 후 정리를 담당한다. 두 메서드가 짝을 이루어 테스트 격리를 완성한다.

`setUp`만으로 충분하지 않은가? 메모리 객체라면 그렇다. 그러나 외부 자원(파일, DB 연결, 소켓, 임시 디렉터리)을 다루기 시작하면 정리가 필수다. 정리하지 않으면 자원 누수가 누적되고, 다른 테스트나 외부 환경에 영향을 준다.

## tearDown 추가

`run`이 testMethod 후 `tearDown`을 호출하도록 바꾼다.

```python
class TestCase:
    def __init__(self, name):
        self.name = name

    def setUp(self):
        pass

    def tearDown(self):
        pass

    def run(self):
        self.setUp()
        method = getattr(self, self.name)
        method()
        self.tearDown()    # 새로 추가
```

자식 클래스는 필요하면 `tearDown`을 오버라이드한다. 22장에서는 testMethod가 예외를 던져도 `tearDown`이 보장되도록 try/finally가 더해진다.

## 언제 tearDown이 필요한가

전형적인 사용 사례는 다음과 같다.

| 자원 | tearDown에서 |
|------|--------------|
| DB 연결 | 트랜잭션 롤백, 연결 닫기 |
| 임시 파일/디렉터리 | 삭제 |
| 네트워크 소켓 | 닫기 |
| 외부 프로세스 | 종료 |
| 전역 상태 변경 | 원복 |
| Mock의 expectation 검증 | 호출 패턴 확인 |

특히 마지막 항목은 GOOS의 mockist 스타일에서 중요하다. mock이 기대한 메시지를 받았는지를 tearDown에 가까운 시점에 검증한다.

## 예제 — 임시 파일 테스트

파일 입출력 테스트가 정리되지 않으면 다음 테스트가 깨지거나 시스템이 어질러진다.

```python
class TestFileWriter(TestCase):
    def setUp(self):
        self.path = "/tmp/test_output.txt"
        self.writer = FileWriter(self.path)

    def tearDown(self):
        import os
        if os.path.exists(self.path):
            os.remove(self.path)

    def test_writes_content(self):
        self.writer.write("hello")
        with open(self.path) as f:
            assert f.read() == "hello"
```

setUp이 path를 준비하고, tearDown이 결과 파일을 지운다. 같은 테스트를 100번 돌려도 시스템 상태가 같다.

## 격리의 두 축

setUp과 tearDown은 테스트 격리의 두 축이다.

```text
setUp    — 매 테스트가 같은 시작 상태에서 출발
tearDown — 매 테스트가 같은 종료 상태에 도달
```

둘이 모두 있어야 *어느 순서로 돌려도 결과가 같다*는 약속이 성립한다. xUnit의 표준 약속이다.

## 모든 xUnit 후예의 공통

| 프레임워크 | setUp 짝 | tearDown 짝 |
|-----------|----------|-------------|
| JUnit | `@Before` / `@BeforeEach` | `@After` / `@AfterEach` |
| NUnit | `[SetUp]` | `[TearDown]` |
| pytest | fixture (yield 앞) | fixture (yield 뒤) |
| RSpec | `before(:each)` | `after(:each)` |
| Mocha | `beforeEach` | `afterEach` |

pytest는 fixture 안의 `yield` 전후로 setUp/tearDown을 표현하는 우아한 방식을 가진다.

```python
@pytest.fixture
def writer(tmp_path):
    path = tmp_path / "out.txt"
    w = FileWriter(path)
    yield w           # 테스트 실행
    if path.exists():
        path.unlink()
```

같은 개념의 다른 표현이다.

## 예외 발생 시의 처리

지금까지의 구현은 testMethod가 예외를 던지면 `tearDown`이 호출되지 않는다. 자원이 누수된다. 22장에서 try/finally로 이 문제를 푼다.

```python
# 22장의 미리보기
def run(self, result):
    self.setUp()
    try:
        method = getattr(self, self.name)
        method()
    except:
        result.testFailed()
    self.tearDown()    # try/finally로 보장
```

자원 보장은 단순해 보이지만 실제로는 신중하게 다뤄야 하는 영역이다. setUp 자체가 실패하면 tearDown은 어떻게? testMethod가 성공하고 tearDown이 실패하면? 22장에서 다룬다.
