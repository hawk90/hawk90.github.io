> *Test-Driven Development by Example* Chapter 8 요약

## 이 챕터의 메시지

7장의 끝에서 `Dollar`와 `Franc`의 차이가 통화 문자열 하나로 줄어들었다. 이 장은 사용자가 그 두 클래스를 알 필요 자체가 없도록 만든다. **Factory Method** 패턴이 등장하면서 `Money`라는 한 가지 추상이 외부 인터페이스가 된다.

이 변경의 즉각적인 효과는 두 가지다. 첫째, 클라이언트 코드가 단순해진다. 둘째, 내부 표현을 자유롭게 바꿀 수 있게 된다. 곧 보겠지만, 책의 후반부에서 `Dollar`/`Franc` 클래스 자체가 사라지는 것도 사용자가 그 존재를 모르기 때문에 가능하다.

## Factory Method 추가

`Money`에 정적 메서드를 두고 그것이 객체 생성을 담당한다.

```java
class Money {
    static Money dollar(int amount) {
        return new Dollar(amount, "USD");
    }

    static Money franc(int amount) {
        return new Franc(amount, "CHF");
    }
}
```

`Dollar`/`Franc` 생성자는 이제 통화를 인자로 받는다.

```java
class Dollar extends Money {
    Dollar(int amount, String currency) {
        this.amount = amount;
        this.currency = currency;
    }
}
```

테스트도 새 인터페이스를 따른다.

```java
public void testMultiplication() {
    Money five = Money.dollar(5);
    assertEquals(Money.dollar(10), five.times(2));
}
```

`new Dollar(...)`가 사라졌다. 사용자가 보는 것은 `Money`뿐이다.

## 반환 타입을 Money로

`times`도 구체 타입 대신 `Money`를 반환하도록 바꾼다.

```java
class Dollar extends Money {
    Money times(int multiplier) {
        return new Dollar(amount * multiplier, currency);
    }
}

class Franc extends Money {
    Money times(int multiplier) {
        return new Franc(amount * multiplier, currency);
    }
}
```

여기서 자바의 공변 반환 타입(covariant return type) 덕에 자식이 더 구체적인 타입을 반환할 수도 있지만, 일관된 외부 인터페이스를 위해 모두 `Money`로 통일한다.

## 외부 인터페이스 vs 내부 구현

이 변경의 의미는 두 영역의 분리에 있다.

| 영역 | 보이는 것 |
|------|----------|
| 외부 (사용자 코드) | `Money` 한 종류, 생성은 `Money.dollar()`/`Money.franc()` |
| 내부 (구현) | `Dollar`, `Franc` 두 서브클래스 |

외부 인터페이스를 안정시키면 내부를 자유롭게 바꿀 수 있다. 10장에서 `Money`가 구체 클래스가 되고 11장에서 `Dollar`/`Franc`이 사라지는 변경이 사용자 코드를 깨지 않는 이유가 바로 여기에 있다.

## Factory Method 패턴의 정신

Beck이 사용한 것은 GoF의 *Factory Method*의 가장 단순한 형태다. 생성 책임을 한 곳에 모으면 다음 이점이 생긴다.

- 생성 시점에 추가 로직(검증, 캐싱, 등록)을 끼워 넣기 쉽다
- 반환 타입을 추상으로 만들어 내부 구체 클래스를 숨길 수 있다
- 새 변형을 추가할 때 클라이언트가 바뀌지 않는다

예를 들어 통화 코드 검증을 더하고 싶다면 `Money.dollar` 한 곳을 수정하면 된다. `new Dollar(...)`가 코드 곳곳에 흩어져 있었다면 모두 찾아야 했을 일이다.

## 정리

- **Factory Method** `Money.dollar()`/`Money.franc()`으로 객체 생성을 통합한다
- 사용자 코드는 `Dollar`/`Franc`을 모른다 — 외부 인터페이스는 `Money` 하나
- 외부 안정 + 내부 자유라는 분리가 후속 변경의 토대가 된다
- 반환 타입도 `Money`로 통일해 인터페이스를 좁힌다
