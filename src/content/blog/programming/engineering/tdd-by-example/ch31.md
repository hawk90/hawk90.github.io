> *Test-Driven Development by Example* Chapter 31 요약

## 이 챕터의 메시지

TDD 사이클의 세 번째 단계가 Refactor다. Beck은 이 장에서 자기 사이클에서 자주 등장하는 리팩터링 몇 가지를 정리한다. Fowler의 *Refactoring* 카탈로그와 같은 정신이다 — *외부 동작을 유지하면서 내부 구조를 개선*한다. 두 책이 짝의 관계다.

특히 이 장의 패턴들은 *TDD 맥락에 특화*된 면이 있다. *테스트가 매 단계 안전망 역할을 한다*는 전제 위에 서 있다.

## 자주 쓰는 리팩터링

### Reconcile Differences

> 두 코드가 거의 같으면 — 단계적으로 동일하게 만들어 합쳐라.

5-11장의 `Dollar`/`Franc` 통합이 이 패턴의 큰 사례다.

```python
# 단계 1: 두 함수
def foo_a(x):
    return x + 1

def foo_b(x):
    return x + 2

# 단계 2: 차이를 매개변수로 분리
def foo_a(x):
    delta = 1
    return x + delta

def foo_b(x):
    delta = 2
    return x + delta

# 단계 3: 이제 함수 본체가 동일 — 통합
def foo(x, delta):
    return x + delta
```

작은 단계로 두 코드의 차이를 *외부 매개변수로 밀어내면* 함수 본체가 같아진다. 그때 통합한다.

### Isolate Change

> 변경할 부분을 — 먼저 분리한다.

큰 코드의 일부만 바꿔야 할 때, 그 부분을 *먼저 함수로 추출*하고, 그 다음 그 함수만 바꾼다.

```python
# 변경 전 — 200줄짜리 함수
def process(data):
    # 100줄
    # 변경할 부분 (50줄)
    # 50줄

# 단계 1: Extract Function
def process(data):
    do_part_a(data)
    do_part_b(data)    # 추출된 변경 대상
    do_part_c(data)

# 단계 2: do_part_b만 변경
def do_part_b(data):
    # 새 구현
```

변경이 *국소화*된다. 영향 범위를 좁히면 잘못이 적어진다. *Refactoring*의 **Extract Function**과 같은 정신이다.

### Migrate Data

> 데이터 구조 변경 — 점진적으로.

운영 중인 시스템이나 분산 시스템에서 데이터 구조를 바꿀 때, 한 번에 바꾸지 않고 *점진적으로 옮긴다*.

**1. 새 구조 추가 (기존 구조는 그대로)**


**2. 새 구조에도 쓰기 (두 곳에 쓰기)**


**3. 새 구조에서 읽기 (읽기를 옮김)**


**4. 기존 구조의 쓰기 제거**


**5. 기존 구조 자체 제거**

각 단계가 *돌리는 동안에도 동작하는 상태*다. 다운타임 없이 마이그레이션 가능. 분산 데이터베이스의 schema migration이 정확히 이 패턴이다.

### Extract Method / Inline Method

Fowler의 카탈로그와 동일하다. TDD에서 매우 자주 등장한다.

```python
# Extract — 의미 있는 단위로 묶기
total = sum(item.price * item.quantity for item in cart)
# ↓
def cart_total(cart):
    return sum(item.price * item.quantity for item in cart)
total = cart_total(cart)

# Inline — 추상화가 과한 경우 풀기
def get_user_name(user):
    return user.name
name = get_user_name(user)
# ↓
name = user.name
```

언제 추출하고 언제 인라인할지의 감각은 경험으로 자란다. *Refactoring* 6장 참조.

## TDD와 Fowler의 정신

Beck의 *TDD*와 Fowler의 *Refactoring*은 같은 정신을 다른 각도에서 본다.

| 면 | Beck (TDD) | Fowler (Refactoring) |
|----|-----------|----------------------|
| 출발 | 새 기능 추가 | 기존 코드 개선 |
| 사이클 | Red → Green → Refactor | 작은 안전한 변경 |
| 안전망 | 테스트 (Beck이 짠) | 테스트 (이미 있는 것) |
| 결과 | 테스트가 자라난 코드 | 깔끔해진 기존 코드 |

두 책의 핵심은 같다. *작은 단계, 매 단계 안전, 누적된 안전이 큰 변경을 가능하게 한다*. Beck의 사이클에 Refactor 단계가 들어 있는 이유, 그리고 Fowler의 카탈로그가 *테스트의 존재를 전제하는* 이유가 같다.

## 작은 단계의 누적이 큰 변경을 막는다

이 장의 깊은 메시지는 다음과 같다.

> 매번 작은 리팩터링이 — 큰 리팩터링의 필요를 없앤다.

매 사이클의 Refactor 단계에서 코드를 조금씩 정리한다. 그 누적이 *Big Ball of Mud*가 되는 것을 막는다. 한 번도 정리하지 않은 코드는 어느 순간 *전체 재작성*이라는 큰 비용으로 돌아온다.

이 사고는 32장의 *Mastering TDD*에서 *균형*이라는 단어로 다시 등장한다.

## 정리

- **Reconcile Differences** — 두 비슷한 코드를 단계적으로 동일하게
- **Isolate Change** — 변경할 부분을 먼저 분리
- **Migrate Data** — 데이터 구조 변경을 점진적으로
- **Extract / Inline Method** — Fowler 카탈로그와 동일
- Beck과 Fowler의 두 책은 짝의 정신 — 작은 단계, 매 단계 안전
- 매번의 작은 리팩터링이 큰 리팩터링의 필요를 없앤다
