> *Test-Driven Development by Example* Chapter 9 요약

## 이 챕터의 메시지

8장에서 Factory Method를 도입하면서 외부 인터페이스가 `Money` 하나로 통일됐다. 내부에서는 `Dollar`와 `Franc`이 여전히 별도 클래스다. 그러나 자세히 보면 두 `times` 구현이 통화 문자열 전달을 빼면 거의 동일하다.

이 장은 그 `times`를 `Money` 부모로 끌어올리는 짧은 리팩터링이다. 한 번 더 **Pull Up Method** 패턴이다.

## 현재 상태

자식 클래스 두 곳에 거의 같은 코드가 있다.

```java
class Dollar extends Money {
    Money times(int multiplier) {
        return new Dollar(amount * multiplier, currency);
    }
}

class Franc extends Money {
    Money times(int multiplier) {
        return new Franc(amount * multiplier, currency);
    }
}
```

차이는 `new Dollar(...)`와 `new Franc(...)`뿐이다. 그러나 `currency`가 이미 데이터로 있으니, 어느 구체 클래스가 만들어지든 결과는 같은 `Money`다. 그렇다면 어느 한 쪽으로 통일해도 되지 않을까?

## 부모로 옮긴 times

`Money`로 끌어올리되 일단 `Dollar`를 만들도록 통일한다.

```java
class Money {
    Money times(int multiplier) {
        return new Dollar(amount * multiplier, currency);
    }
}
```

`Dollar`와 `Franc`의 `times`는 제거한다. 부모의 구현이 그대로 상속된다.

이 변경은 일견 이상해 보인다. `Franc`에서 호출해도 `new Dollar`가 만들어진다. 그러나 `equals`는 `currency` 문자열로 비교하므로, 통화가 같으면 같은 객체로 취급된다. 테스트는 모두 통과한다.

```java
Money five = Money.franc(5);
Money fifteen = five.times(3);   // 내부적으로 Dollar이지만 currency="CHF"
assertEquals(Money.franc(15), fifteen);   // 통과
```

여기서 *클래스 정체성과 도메인 정체성의 분리*가 작동한다. 7장에서 도입한 통화 데이터의 효과다.

## 더 단순한 길

`Dollar`와 `Franc` 가운데 어느 쪽을 만들든 동작이 같다면, 부모인 `Money` 자체를 직접 만들면 되지 않을까? 이 질문이 10장에서 본격적으로 다뤄진다. 거기서 `Money`가 추상 클래스에서 구체 클래스로 바뀐다.

## 테스트가 안전망

이번 리팩터링은 한 줄짜리 변경 같지만 의미상 큰 변화다. 만약 어떤 코드가 `times`의 반환 객체에 대해 `instanceof Franc`을 검사한다면 깨진다. 그런 코드가 없음을 보장하는 것은 테스트다.

```text
1. times를 Money로 옮긴다 → 모든 테스트 실행 → 통과
2. 자식의 times를 제거한다 → 다시 실행 → 통과
3. 다음 단계로
```

각 단계가 테스트 통과를 확인하는 작은 보폭이다. 한 단계라도 빨간 막대가 뜨면 즉시 되돌리고 원인을 찾는다.

## 정리

- `times`를 `Money` 부모로 **Pull Up Method**
- `Dollar` 생성으로 통일해도 `currency` 비교 덕에 동작이 같다
- 클래스 정체성이 도메인 정체성과 분리됐다
- 테스트가 매 단계의 안전망이다 — 한 단계씩, 모두 통과 확인
