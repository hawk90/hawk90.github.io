> *Test-Driven Development by Example* Chapter 27 요약

## 이 챕터의 메시지

이 장은 *테스트 자체를 어떻게 디자인할 것인가*의 패턴들이다. 26장이 *언제 어떤 테스트를 짤지*였다면, 27장은 *어떻게 짤지*에 관한 도구상자다.

특히 Mock Object 관련 패턴들이 중요하다. 의존성이 무거운 객체를 다루는 테스트의 표준 도구들이다. Khorikov의 *Unit Testing* 5장이 이 영역을 현대적으로 정리한다.

## 패턴 1 — Child Test

> 너무 큰 테스트가 실패하면 — 더 작은 테스트로 쪼개라.

큰 테스트가 실패했을 때 *어디서* 실패했는지 알기 어려운 경우가 있다.

```python
def test_full_workflow():
    # 100줄짜리 시나리오
    user = create_user()
    profile = build_profile(user)
    order = place_order(profile, items)
    invoice = generate_invoice(order)
    email = send_confirmation(invoice)
    # ... 어디서 실패?
```

해법은 *큰 시나리오를 작은 component 테스트로 쪼개는 것*이다.

```python
def test_create_user_returns_valid_id():
    user = create_user()
    assert user.id is not None

def test_build_profile_with_user():
    user = mock_user()
    profile = build_profile(user)
    assert profile.user_id == user.id

# ... 각 단계마다 작은 테스트
```

각 작은 테스트가 통과하면 큰 시나리오로 돌아간다. 큰 테스트가 여전히 실패하면 *integration* 문제다.

## 패턴 2 — Mock Object

> 실제 객체가 비싸거나 사용하기 어렵다면 — Mock으로 대체한다.

DB, 네트워크, 시간, 무거운 외부 라이브러리. 이런 의존성을 실제로 호출하면 테스트가 느려지고 환경에 의존한다. Mock이 같은 인터페이스로 *예측 가능한 응답*을 준다.

```python
class MockDatabase:
    def __init__(self):
        self.saved = []

    def save(self, record):
        self.saved.append(record)

    def find(self, id):
        for r in self.saved:
            if r.id == id:
                return r
        return None

def test_save_user():
    mock = MockDatabase()
    service = UserService(mock)
    service.create_user("alice")
    assert "alice" == mock.find(0).name
```

GOOS(Freeman·Pryce)가 *Mockist TDD*의 정점이다. Khorikov는 *London vs Detroit* 스타일로 두 학파를 정리한다.

| 학파 | Mock 사용 |
|------|----------|
| London (Mockist) | 모든 외부 의존을 mock |
| Detroit (Classicist) | 가능하면 진짜, 외부 자원만 mock |

Beck은 둘 사이의 중간 입장이다. *유용할 때만* mock을 쓴다. 과용은 *깨지기 쉬운 테스트*를 만든다.

## 패턴 3 — Self Shunt

> 테스트 자체가 — collaborator의 역할을 한다.

테스트가 검증 대상의 협력자를 mock으로 만드는 대신, 테스트 클래스 자체가 그 협력자가 된다.

```python
class TestWidget(TestCase):
    def setUp(self):
        self.received = None

    def callback(self, value):
        self.received = value     # 테스트가 콜백 받음

    def test_fires_callback(self):
        widget = Widget(on_fire=self.callback)
        widget.fire("hello")
        assert "hello" == self.received
```

Mock 클래스를 별도로 만들지 않는다. 테스트 클래스가 callback을 받는 객체 역할을 한다.

장점은 두 가지다.

| 장점 | 의미 |
|------|------|
| 추가 클래스 불필요 | 단순함 |
| 검증 데이터에 직접 접근 | `self.received` 즉시 사용 |

단점도 있다. 테스트 클래스가 여러 역할을 겸하므로 *복잡한 콜백 시나리오*에는 부적합하다.

## 패턴 4 — Log String

> 순서가 중요한 호출을 — 문자열에 기록해 검증한다.

여러 메서드가 *특정 순서로* 호출되어야 한다는 검증.

```python
class LogString:
    def __init__(self):
        self.log = ""

    def call(self, what):
        self.log += what + " "

def test_call_order():
    log = LogString()
    obj = Thing(logger=log)
    obj.do_a()
    obj.do_b()
    obj.do_c()
    assert "a b c " == log.log    # 순서 검증
```

문자열이 *호출 순서의 기록*이 된다. assertion이 한 줄로 끝난다. 18-23장의 xUnit 예제에서도 비슷한 정신이 등장한다.

## 패턴 5 — Crash Test Dummy

> 에러 처리 코드를 테스트하려면 — 일부러 예외 던지는 Mock.

평소에 깨지지 않는 자원을 *깨진 척* 만든다.

```python
class CrashDummy:
    def operation(self):
        raise IOError("simulated disk failure")

def test_error_handling():
    obj = MakeThing(disk=CrashDummy())
    try:
        obj.save_to_disk()
        assert False, "예외가 안 던져짐"
    except IOError:
        pass    # 의도된 예외 — 적절히 처리됨
```

자동차 충돌 테스트의 *crash test dummy*에서 빌린 이름이다. 진짜 운전자를 다치게 하지 않고 충돌을 시험한다. 진짜 디스크 장애를 일으키지 않고 에러 처리를 검증한다.

이 패턴은 *resilience testing*의 기초다. Chaos Engineering(Netflix)이 같은 정신을 분산 시스템 수준으로 확장한 것이다.

## 모의(Mock) 객체 관련 정리

이 다섯 패턴 가운데 Mock Object, Self Shunt, Log String, Crash Test Dummy 네 가지가 모두 *테스트 대역(test double)* 패턴이다. Gerard Meszaros의 *xUnit Test Patterns*가 더 세밀하게 분류한다.

| 종류 | 역할 |
|------|------|
| Dummy | 자리만 채움, 호출되지 않음 |
| Stub | 미리 정한 응답 |
| Spy | 호출을 기록 |
| Mock | 기대(expectation) 검증 포함 |
| Fake | 단순화된 실제 구현 (예: in-memory DB) |

Beck의 책은 이 분류 이전이라 *Mock*이라는 단어를 더 넓게 쓴다.

## 정리

- **Child Test** — 큰 테스트를 작은 테스트로 쪼개
- **Mock Object** — 외부 의존성을 가짜로 대체
- **Self Shunt** — 테스트 자체가 collaborator
- **Log String** — 호출 순서를 문자열로 검증
- **Crash Test Dummy** — 일부러 깨지는 mock으로 에러 처리 검증
