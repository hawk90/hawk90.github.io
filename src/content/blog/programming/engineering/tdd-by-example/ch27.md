> **Test-Driven Development by Example** Chapter 27 요약

## 이 챕터의 메시지

테스트 자체를 어떻게 디자인할 것인가의 패턴.

## 패턴 — Child Test

> **너무 큰 테스트가 실패하면 — 더 작은 테스트로 쪼개라.**

```
대형 테스트 실패 → 어디서 실패? 알기 어려움
```

해법: 작은 component 테스트로 쪼개. 각각 통과시키고 — 다시 큰 테스트.

## 패턴 — Mock Object

> **실제 객체가 비싸거나 — 사용 어렵다면 — Mock으로 대체.**

```python
class MockDatabase:
    def save(self, record):
        self.saved = record  # 실제 DB 호출 X

def test_save():
    mock = MockDatabase()
    service = Service(mock)
    service.create_user("alice")
    assertEquals("alice", mock.saved.name)
```

DB / 네트워크 / 외부 API는 — 종종 mock.

## 패턴 — Self Shunt

> **테스트 자체가 — collaborator의 역할.**

```python
class TestThing(TestCase):
    def __init__(self, name):
        super().__init__(name)
    
    def callback(self, value):
        self.received = value
    
    def test_x():
        widget = Widget(callback=self.callback)
        widget.fire()
        assertEquals("expected", self.received)
```

테스트 클래스 자체가 — callback 받는 객체. Mock 별도 객체 없이.

## 패턴 — Log String

> **순서가 중요한 호출을 — 문자열에 기록해 검증.**

```python
class LogString:
    def __init__(self):
        self.log = ""
    
    def call(self, what):
        self.log += what + " "

def test_order():
    log = LogString()
    obj = MakeThing(logger=log)
    obj.do_a()
    obj.do_b()
    assertEquals("a b ", log.log)   # 순서 검증
```

호출 순서를 — 문자열로 확인.

## 패턴 — Crash Test Dummy

> **에러 처리 코드를 테스트하려면 — 일부러 에러 던지는 Mock.**

```python
class CrashDummy:
    def operation(self):
        raise Exception("simulated failure")

def test_error_handling():
    obj = MakeThing(collaborator=CrashDummy())
    try:
        obj.do_something()
        fail()
    except:
        pass  # 예외 처리됨
```

실제로 깨지지 않는 환경에서 — 깨진 척.

## 정리

- **Child Test** — 큰 테스트를 작게 쪼갬
- **Mock Object** — 외부 의존 대체
- **Self Shunt** — 테스트 자체가 collaborator
- **Log String** — 호출 순서 검증
- **Crash Test Dummy** — 에러 시뮬레이션
