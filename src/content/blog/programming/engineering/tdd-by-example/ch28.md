> **Test-Driven Development by Example** Chapter 28 요약

## 이 챕터의 메시지

테스트를 통과시키는 — 세 가지 방법.

## 패턴 — Fake It (Till You Make It)

> **가능한 한 빨리 통과시킨다 — 하드코딩이라도.**

```java
// 첫 시도
public int times(int multiplier) {
    return 10;   // 하드코딩
}
```

명백히 잘못. 그러나 — 초록 막대를 얻고 — 다음 테스트로 일반화.

작은 한 걸음. 빠른 피드백.

## 패턴 — Triangulate

> **두 데이터 포인트로 일반화 강제.**

```java
assertEquals(10, money.times(2).amount);   // 첫 테스트
assertEquals(15, money.times(3).amount);    // 두 번째 — Triangulation
```

이제 — `return 10`은 통과 안 됨. 진짜 구현 필요.

```java
public int times(int multiplier) {
    return amount * multiplier;
}
```

## 패턴 — Obvious Implementation

> **답이 명백하면 — 바로 짜라.**

```python
def add(a, b):
    return a + b   # Fake it / Triangulate 필요 X
```

명백한 경우 — 단계별 게임 필요 없음.

**기준** — 5분 안에 정확히 짤 수 있다는 자신감.

## 세 방법의 선택

| 상황 | 추천 |
|---|---|
| 답이 어떨지 모름 | Fake It |
| 일반화 어려움 | Triangulate |
| 명백 | Obvious Implementation |

대부분의 경우 — Obvious Implementation. 어려운 경우 — Fake It → Triangulate.

## 패턴 — One to Many

> **여러 객체를 다루기 전에 — 하나만 다루는 코드부터.**

```python
# 1단계
def sum_one(item):
    return item.value

# 2단계 — 일반화
def sum_many(items):
    return sum(item.value for item in items)
```

하나의 케이스부터 — 여러 개로 확장.

## 정리

- **Fake It** — 하드코딩으로 빠른 초록 막대
- **Triangulate** — 두 데이터 포인트로 일반화
- **Obvious Implementation** — 명백하면 바로
- **One to Many** — 하나 → 여러
