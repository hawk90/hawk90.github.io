> *Test-Driven Development by Example* Chapter 28 요약

## 이 챕터의 메시지

빨간 막대를 어떻게 초록으로 만들 것인가? Beck은 세 가지 정형화된 방법을 제시한다. **Fake It**, **Triangulate**, **Obvious Implementation**. 어느 것을 선택할지는 상황에 따라 다르다. 그리고 보너스 패턴 **One to Many**가 추가된다.

이 세 방법은 1-3장에서 이미 사례로 등장했다. 이 장은 그것을 패턴으로 정리한다.

## 패턴 1 — Fake It (Till You Make It)

> 가능한 한 빨리 통과시킨다. 하드코딩이라도 OK.

답이 어떨지 확실하지 않을 때, 일단 *가짜* 답으로 초록을 만든다.

```java
// 1장의 사례 — Dollar.times
class Dollar {
    int amount = 10;    // 하드코딩
    public Dollar(int amount) {}
    public void times(int multiplier) {}
}
```

분명히 잘못된 코드다. 그러나 빨간 막대가 사라졌다. 다음 사이클에서 두 번째 테스트나 리팩터링으로 진짜 구현으로 옮긴다.

장점은 *심리적 안정*이다. 빨간 막대를 보면서 큰 고민을 하는 것보다, 초록 상태에서 다음 단계를 생각하는 것이 마음이 가볍다.

## 패턴 2 — Triangulate

> 두 데이터 포인트로 일반화를 강제한다.

Fake It의 다음 단계가 자주 Triangulate다. 두 번째 테스트가 *진짜 구현*을 강제한다.

```java
// 3장의 사례 — equals
assertTrue(new Dollar(5).equals(new Dollar(5)));   // 첫 번째
assertFalse(new Dollar(5).equals(new Dollar(6)));  // Triangulate

// 이제 return true로는 통과 못함
public boolean equals(Object obj) {
    Dollar dollar = (Dollar) obj;
    return amount == dollar.amount;     // 진짜 구현 강제됨
}
```

기하학 비유로 *두 점이 직선을 결정한다*. 한 점은 무수한 직선이 통과한다. 두 점은 직선이 유일하다.

## 패턴 3 — Obvious Implementation

> 답이 명백하면 — 바로 짠다.

Fake It과 Triangulate가 너무 의례적이라 답답할 수 있다. *답이 분명한 경우*에는 단계 게임 없이 바로 진짜 구현을 짠다.

```python
def add(a, b):
    return a + b      # Fake It 필요 없음, Triangulate도 필요 없음
```

기준은 단순하다. *5분 안에 정확히 짤 수 있다는 자신감*. 그게 있으면 Obvious. 의심스러우면 Fake It으로 후퇴.

자신감이 잘못된 경우 — 즉 *Obvious인 줄 알았는데 빨간 막대가 안 사라지는 경우* — 그게 바로 *Fake It으로 후퇴할 신호*다. 정직한 후퇴.

## 세 방법의 선택

| 상황 | 추천 |
|------|------|
| 답이 어떨지 모름 | Fake It |
| 일반화 방향이 불확실 | Triangulate |
| 답이 5분 안에 명백 | Obvious Implementation |
| Obvious가 실패 | Fake It으로 후퇴 |

실무에서는 대부분이 Obvious Implementation이다. 어려운 부분에서만 Fake It → Triangulate의 사이클이 등장한다. 31장의 *Mastering TDD*에서 *어려운 곳은 작게, 쉬운 곳은 크게*라는 메시지가 같은 결론이다.

## 패턴 4 — One to Many

> 여러 객체를 다루기 전에 — 하나만 다루는 코드부터.

집합 처리가 필요할 때, 곧바로 list/loop로 가지 않고 *원소 하나 처리*부터 짠다.

```python
# 1단계 — 하나 처리
def sum_one(item):
    return item.value

# 2단계 — 일반화 (One to Many)
def sum_many(items):
    return sum(item.value for item in items)
```

처음부터 `sum_many`를 짜면 두 가지 결정을 동시에 해야 한다. *원소 처리*와 *반복 구조*다. One to Many는 둘을 분리한다.

이 정신은 함수형 프로그래밍의 *map / fold* 분리와 통한다. 원소 변환은 map이, 반복은 fold가 담당한다. 책임의 분리가 코드를 단순하게 만든다.

## 정리

- **Fake It** — 하드코딩으로 빠른 초록
- **Triangulate** — 두 데이터 포인트로 일반화 강제
- **Obvious Implementation** — 명백하면 바로 짠다
- **One to Many** — 하나의 케이스 먼저, 그 뒤 일반화
- 대부분은 Obvious. 어려운 곳에서만 Fake It → Triangulate
