> *Test-Driven Development by Example* Chapter 21 요약

## 이 챕터의 메시지

지금까지의 xUnit은 `print`로 결과를 확인했다. 사람의 눈이 통과/실패를 판단했다. 그러나 테스트가 100개가 되면 그 방식은 한계에 부딪힌다. 어느 테스트가 통과했는가? 어느 게 실패했는가? 자동화된 카운터가 필요하다.

이 장은 `TestResult`를 도입한다. 각 테스트가 자기 결과를 result에 등록하고, 마지막에 한 줄짜리 summary가 출력된다. JUnit의 빨간/초록 막대의 원형이다.

## TestResult 클래스

가장 단순한 결과 카운터다.

```python
class TestResult:
    def __init__(self):
        self.runCount = 0
        self.errorCount = 0

    def testStarted(self):
        self.runCount += 1

    def testFailed(self):
        self.errorCount += 1

    def summary(self):
        return f"{self.runCount} run, {self.errorCount} failed"
```

세 메서드뿐이다. `testStarted`는 카운트 증가, `testFailed`는 에러 증가, `summary`는 결과 문자열 반환.

## TestCase.run에 result 전달

`run`이 `TestResult`를 받도록 시그니처를 바꾼다.

```python
class TestCase:
    def run(self, result):
        result.testStarted()
        self.setUp()
        try:
            method = getattr(self, self.name)
            method()
        except:
            result.testFailed()
        self.tearDown()
```

`testStarted`는 무조건 호출(돌렸으니 카운트). `testFailed`는 예외가 발생했을 때만. try/except로 잡는다.

## 사용 예

```python
test = WasRun("testMethod")
result = TestResult()
test.run(result)
print(result.summary())
# 출력: "1 run, 0 failed"
```

이제 테스트의 결과가 객체에 누적된다. 여러 테스트를 같은 result로 돌리면 누적 카운트가 나온다.

```python
result = TestResult()
for name in ["testA", "testB", "testC"]:
    TestSomething(name).run(result)
print(result.summary())
# 출력: "3 run, 1 failed" (예시)
```

23장의 `TestSuite`가 이 반복을 추상화한다.

## 왜 result를 외부에서 주입하는가

`run`이 result를 *생성*하지 않고 *주입받는* 이유는 두 가지다.

| 이유 | 의미 |
|------|------|
| 누적 가능 | 여러 테스트가 같은 result에 카운트 누적 |
| 격리 가능 | 테스트마다 다른 result도 가능 (필요 시) |
| 테스트 가능 | TestResult 자체를 다른 result로 테스트 |

이 패턴은 *의존성 주입(Dependency Injection)*의 미니어처다. 객체가 자기에게 필요한 협력자를 외부에서 받는다.

## TestResult 자체를 테스트하기

자기 자신을 자기로 테스트할 수 있는가? 거의 그렇다.

```python
class TestCaseTest(TestCase):
    def testResult(self):
        result = TestResult()
        WasRun("testMethod").run(result)
        assert "1 run, 0 failed" == result.summary()
```

이게 부트스트랩이 완성되는 순간이다. `TestCase`가 자기 동작을 검증하는 테스트를 자기 자신의 형태로 짠다. 24장에서 다시 회고한다.

## 실패의 표시

지금의 구현은 `print(result.summary())`로 마지막에 출력한다. 진짜 xUnit이라면 IDE가 빨간/초록 막대를 표시한다.

| 시각화 | 의미 |
|--------|------|
| 초록 막대 (green bar) | 모든 테스트 통과 |
| 빨간 막대 (red bar) | 하나 이상 실패 |
| 노란 막대 (어떤 도구) | 예외 등 비통과/비실패 |

TDD에서 *빨간 막대* / *초록 막대*라는 비유가 이 시각화에서 왔다. 색이 즉각적인 피드백을 준다.

## To-Do List 갱신

```text
[X] TestResult — 통과/실패 카운트
[ ] Exception 처리의 신중함 (setUp 실패 등)
[ ] TestSuite — 여러 테스트 묶음
```

## 정리

- `TestResult` 도입 — 통과/실패 카운트와 summary
- `run(result)`로 result를 주입받음 → 누적 가능, 격리 가능
- 자기 자신을 자기로 검증하는 부트스트랩이 완성에 가까워짐
- 빨간/초록 막대 비유의 토대 — 즉각적 시각 피드백
