> *Test-Driven Development by Example* Chapter 6 요약

## 이 챕터의 메시지

5장에서 `Franc`을 도입하면서 `Dollar`와의 중복이 명확해졌다. 이 장은 공통 부모 `Money` 클래스를 만들고 `equals`를 위로 끌어 올린다. Fowler 카탈로그의 **Pull Up Method**가 처음으로 등장한다.

또한 이 통합 과정에서 새 결함이 드러난다. `Dollar.equals(Franc)`이 무엇을 반환해야 하는가? 통화가 다르면 동등하지 않아야 한다. 임시 해법으로 `getClass()` 비교를 도입하고, 더 나은 해법은 다음 장으로 미룬다.

## Money 부모 클래스

`amount` 필드와 `equals`를 부모로 옮긴다.

```java
class Money {
    protected int amount;

    public boolean equals(Object obj) {
        Money money = (Money) obj;
        return amount == money.amount;
    }
}

class Dollar extends Money {
    Dollar(int amount) {
        this.amount = amount;
    }

    Dollar times(int multiplier) {
        return new Dollar(amount * multiplier);
    }
}

class Franc extends Money {
    Franc(int amount) {
        this.amount = amount;
    }

    Franc times(int multiplier) {
        return new Franc(amount * multiplier);
    }
}
```

`equals`는 이제 한 곳에만 존재한다. `times`는 아직 자식에 남아 있는데, 반환 타입이 다르기 때문이다. 이 통합은 8장의 Factory Method 도입 이후에 가능해진다.

## 새 결함이 드러난다

부모로 끌어올린 `equals`는 `amount`만 비교한다. 그러면 다음 시나리오가 잘못된 결과를 낸다.

```java
new Dollar(5).equals(new Franc(5))   // ?
```

도메인적으로 둘은 다른 통화이므로 같지 않아야 한다. 그러나 현재 코드는 `amount`만 보므로 `true`를 반환한다. 버그다.

테스트로 그 사실을 드러낸다.

```java
public void testEquality() {
    assertTrue(new Dollar(5).equals(new Dollar(5)));
    assertFalse(new Dollar(5).equals(new Dollar(6)));
    assertTrue(new Franc(5).equals(new Franc(5)));
    assertFalse(new Franc(5).equals(new Franc(6)));
    assertFalse(new Dollar(5).equals(new Franc(5)));   // 새 케이스
}
```

마지막 줄이 빨간 막대를 만든다.

## 임시 해법 — class 비교

가장 빠른 초록은 자바의 런타임 클래스를 비교하는 것이다.

```java
public boolean equals(Object obj) {
    Money money = (Money) obj;
    return amount == money.amount
        && getClass().equals(money.getClass());
}
```

`Dollar`와 `Franc`은 서로 다른 클래스이므로 `getClass()`가 다르고, 따라서 결과는 `false`다. 테스트가 통과한다.

## 임시 해법의 약점

이 해법은 동작하지만 디자인 관점에서 문제가 있다.

| 약점 | 의미 |
|------|------|
| 도메인 개념이 아닌 언어 디테일 | `getClass()`는 자바의 reflection, 비즈니스 개념이 아니다 |
| 새 통화마다 새 클래스 필요 | 통화 100개면 클래스 100개 |
| 동등성과 클래스 정체성을 묶음 | 같은 USD인데 표현 클래스가 다르면 다르다고 판단 |

진짜 도메인 개념은 *통화*다. `Dollar`와 `Franc`이라는 클래스 이름은 그 통화를 표현하는 한 가지 방법일 뿐이다. 7장에서 통화를 데이터로 만들면서 이 임시 해법을 제거한다.

## Pull Up Method라는 리팩터링

Beck이 보여 준 이 통합은 Fowler의 *Refactoring* 카탈로그에서 **Pull Up Method**라는 이름을 가진다. 자식 클래스들에 같은 메서드가 있으면 부모로 끌어 올린다.

**1. 자식 A, B의 메서드가 같은지 확인**


**2. 부모로 메서드 복사**


**3. 자식 A에서 제거 → 테스트 통과 확인**


**4. 자식 B에서 제거 → 테스트 통과 확인**

각 단계마다 모든 테스트가 통과해야 다음 단계로 간다. 이 일정한 보폭이 TDD의 안전망이다.

## 정리

- 공통 `Money` 클래스를 도입하고 `equals`를 부모로 **Pull Up**한다
- 통합 과정에서 `Dollar.equals(Franc)`이 잘못된 결과를 내는 새 버그가 드러난다
- 임시 해법으로 `getClass()`를 비교한다
- `getClass()` 비교는 도메인 개념이 아니라 언어 디테일 — 7장에서 통화 데이터로 대체
