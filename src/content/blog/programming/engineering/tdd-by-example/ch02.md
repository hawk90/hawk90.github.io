> *Test-Driven Development by Example* Chapter 2 요약

## 이 챕터의 메시지

1장의 마지막에서 발견한 사이드 이펙트 문제를 해결한다. 같은 `Dollar` 객체로 `times`를 두 번 호출하면 결과가 누적되는 버그였다. 이 장에서는 `Dollar`를 불변 값 객체로 바꾸어 문제 자체가 발생하지 않게 만든다.

값 객체는 한 번 생성되면 내부 상태가 바뀌지 않는다. 자연수, 문자열, 날짜처럼 "값 자체가 정체성"인 객체의 표준 모델이다.

## 새 테스트로 문제를 드러낸다

먼저 사이드 이펙트가 사라졌음을 검증할 테스트를 짠다. 같은 `five`로 두 번 곱해도 원본이 바뀌지 않아야 한다.

```java
public void testMultiplication() {
    Dollar five = new Dollar(5);
    Dollar product = five.times(2);
    assertEquals(10, product.amount);
    product = five.times(3);
    assertEquals(15, product.amount);   // five는 여전히 5
}
```

두 번째 `times(3)` 호출이 결정적이다. 원본 `five.amount`가 그대로 5라면 결과는 15가 된다. 만약 1장의 누적되는 구현을 그대로 쓰면 결과는 30이 되어 테스트가 실패한다.

## 구현 — 새 객체 반환

`times`가 자기 자신을 수정하는 대신 새 `Dollar`를 만들어 반환한다.

```java
class Dollar {
    int amount;

    Dollar(int amount) {
        this.amount = amount;
    }

    Dollar times(int multiplier) {
        return new Dollar(amount * multiplier);
    }
}
```

호출자는 반환값을 받아야 결과를 쓸 수 있다. 원본은 결코 변하지 않는다. 자바의 `String`이 `concat`을 그렇게 처리하는 것과 같은 방식이다.

## Value Object 패턴의 첫 등장

이 작은 변경이 책 전체에서 반복될 큰 패턴의 시작이다. Beck은 25장에서 값 객체를 한 패턴으로 정리하는데, 핵심 규칙은 다음 세 가지다.

| 규칙 | 의미 |
|------|------|
| 객체의 값이 정체성 | `Dollar(5)`라는 두 객체는 둘 다 "5 달러"로 동일하다 |
| 한 번 생성되면 불변 | 내부 상태를 바꾸는 메서드가 없다 |
| 동등성은 값의 동등성 | `equals`는 필드 값을 비교한다 |

자바의 `String`, `Integer`, `BigDecimal`, 그리고 Java 8 이후의 `Optional`, `LocalDate`가 모두 값 객체다. 도메인 모델에서도 돈, 좌표, IP 주소 같은 단순 값은 거의 항상 값 객체로 표현한다.

## 왜 불변이 자연스러운가

가변 값 객체가 일으키는 문제는 책 전체에서 반복된다. 가변이면 다음 함정이 생긴다.

```java
// 가변 Money가 있다고 가정
Money price = ...;
order.setBasePrice(price);
price.times(2);            // 누가 영향을 받나?
order.applyDiscount(0.1);  // price도 같이 바뀌나?
```

호출자는 자기 객체가 다른 어딘가에 의해 변경됐는지 알 수 없다. 불변이면 이 모든 의심이 사라진다. 한 번 만들어진 5달러는 영원히 5달러다.

병렬·동시성에서도 불변 객체는 동기화 없이 공유할 수 있다. Khorikov가 *Unit Testing* 6장에서 함수형 스타일 테스트를 권하는 이유 가운데 하나가 바로 이것이다.

## To-Do List 갱신

```text
[ ] $5 + 10 CHF = $10 (환율 2:1)
[X] $5 × 2 = $10
[ ] amount를 private으로
[X] Dollar의 사이드 이펙트 제거
[ ] Money rounding
[ ] equals()                ← 새로 추가
[ ] hashCode()              ← 새로 추가
```

다음 단계는 `equals`다. 값 객체는 동등성을 값으로 비교해야 의미가 있다. 그게 없으면 `new Dollar(5)`와 `new Dollar(5)`가 다른 객체로 취급된다.

## 정리

- `Dollar`를 불변 **값 객체**로 바꾼다
- `times`는 자기 자신을 수정하지 않고 새 `Dollar`를 반환한다
- 값 객체의 세 규칙: 값이 정체성, 불변, 동등성은 값으로
- 가변 객체가 일으키는 모든 의심이 불변으로 사라진다
- 다음 항목 `equals`가 To-Do에 추가됐다
