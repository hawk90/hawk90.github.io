> *Test-Driven Development by Example* Chapter 7 요약

## 이 챕터의 메시지

6장에서 임시 해법으로 도입한 `getClass()` 비교는 도메인 개념이 아닌 언어의 reflection에 의존한다. 이 장은 통화를 도메인 데이터로 옮겨 그 의존을 제거한다. 작은 변경이지만 디자인의 방향이 크게 바뀐다.

핵심 원칙은 다음 한 줄이다. *비즈니스 개념을 코드에 명시적으로 표현하라.* DDD가 *유비쿼터스 언어*라는 이름으로 이 원칙을 정식화한 것과 같은 결론이다.

## Currency를 데이터로

`Money`에 `currency` 필드를 추가한다. `Dollar`와 `Franc`은 생성 시 자신의 통화를 지정한다.

```java
class Money {
    protected int amount;
    protected String currency;

    String currency() {
        return currency;
    }

    public boolean equals(Object obj) {
        Money money = (Money) obj;
        return amount == money.amount
            && currency.equals(money.currency);
    }
}

class Dollar extends Money {
    Dollar(int amount) {
        this.amount = amount;
        this.currency = "USD";
    }

    Dollar times(int multiplier) {
        return new Dollar(amount * multiplier);
    }
}

class Franc extends Money {
    Franc(int amount) {
        this.amount = amount;
        this.currency = "CHF";
    }

    Franc times(int multiplier) {
        return new Franc(amount * multiplier);
    }
}
```

`equals`는 이제 `getClass()` 대신 `currency` 문자열을 비교한다.

## 왜 이게 더 좋은가

`getClass()`에서 `currency` 데이터로의 전환은 다음 이점을 가져온다.

| 측면 | getClass() 비교 | currency 데이터 비교 |
|------|-----------------|---------------------|
| 사용자 이해 | 자바 reflection 지식 필요 | "통화"라는 비즈니스 어휘 |
| 새 통화 추가 | 새 클래스 정의 | 문자열 한 줄 |
| 환율 정보 | 클래스에 묶여 어색 | `currency`를 키로 환율 테이블 가능 |
| 직렬화 | 클래스 이름 매핑 필요 | `currency` 문자열 그대로 |

특히 마지막 두 항목이 결정적이다. 환율 표는 통화 쌍을 키로 하는 자료 구조다. 통화가 클래스가 아닌 문자열이면 표가 자연스럽게 만들어진다. 15장의 `Bank.addRate`가 그 형태다.

## 도메인 모델링의 첫걸음

이 변경은 디자인을 한 단계 끌어올린다. 이전에는 "Dollar는 클래스다"라는 정적 사실이었다면, 이제는 "Dollar는 통화가 USD인 Money다"라는 동적 정의가 된다.

DDD(Eric Evans)는 이 사고를 다음과 같이 정식화한다.

> 도메인의 핵심 개념은 코드의 명시적 요소가 되어야 한다.

통화가 클래스에 *숨어* 있던 상태에서 데이터로 *드러난* 상태로 옮긴 셈이다. 이후 장에서 `Dollar`/`Franc` 클래스 자체가 사라지는 것도 이 변경 덕분이다.

## 다음 결합 — Dollar와 Franc 차이가 사라진다

`currency`가 데이터로 옮기고 나면 `Dollar`와 `Franc`은 거의 같다.

```java
class Dollar extends Money {
    Dollar(int amount) {
        this.amount = amount;
        this.currency = "USD";   // 이 줄만 다름
    }

    Dollar times(int multiplier) {
        return new Dollar(amount * multiplier);
    }
}
```

차이는 생성자의 통화 문자열뿐이다. 이러면 클래스 자체가 사라질 수 있지 않을까? 8장에서 Factory Method로 그 시도가 시작된다.

## 정리

- `Currency`를 데이터로 옮겨 `getClass()` 비교의 임시 해법을 제거한다
- 도메인 개념(통화)이 코드의 명시적 요소가 된다 — DDD의 정신
- 환율 표 같은 후속 디자인이 자연스러워진다
- `Dollar`/`Franc`의 차이가 한 문자열로 줄어든다 — 통합의 길이 보인다
