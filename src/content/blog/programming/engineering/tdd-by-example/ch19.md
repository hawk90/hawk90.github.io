> *Test-Driven Development by Example* Chapter 19 요약

## 이 챕터의 메시지

18장의 TestCase는 메서드를 그냥 호출했다. 그러나 진짜 테스트는 검증 대상 객체와 그 주변 상태(fixture)가 필요하다. 여러 테스트가 비슷한 fixture를 공유하면 코드가 반복되고, 한 테스트의 상태 변경이 다음 테스트에 영향을 미친다.

해법은 `setUp` 메서드다. 각 테스트가 호출되기 직전에 자동으로 실행되어 fixture를 깨끗한 초기 상태로 준비한다. JUnit, pytest, NUnit, RSpec, Mocha — 모든 xUnit 후예들이 같은 패턴을 따른다.

## setUp 도입

`TestCase.run`이 `setUp`을 먼저 호출하도록 바꾼다.

```python
class TestCase:
    def __init__(self, name):
        self.name = name

    def setUp(self):
        pass    # 기본은 아무것도 안 함

    def run(self):
        self.setUp()                           # 매 테스트 전
        method = getattr(self, self.name)
        method()
```

자식 클래스는 필요하면 `setUp`을 오버라이드한다.

```python
class WasRun(TestCase):
    def setUp(self):
        self.wasRun = None
        self.wasSetUp = 1

    def testMethod(self):
        self.wasRun = 1
```

`setUp`이 호출됐는지 검증하기 위해 `wasSetUp` 플래그를 둔다. 테스트 메서드 안에서 그 플래그를 확인하면 setUp이 작동했다는 증거가 된다.

## 검증

```python
test = WasRun("testMethod")
test.run()
assert test.wasSetUp == 1   # setUp이 돌았다
assert test.wasRun == 1     # testMethod도 돌았다
```

물론 이 시점에는 아직 진짜 `assert`가 없으므로 `print`로 확인한다.

## 왜 setUp이 필요한가

setUp이 없을 때 두 테스트가 같은 객체를 공유한다고 가정하자.

```python
# setUp 없이
class TestThing(TestCase):
    def __init__(self, name):
        super().__init__(name)
        self.thing = Thing()       # 한 번만 만들어짐
        self.thing.add("a")
        self.thing.add("b")

    def test_size(self):
        assert self.thing.size() == 2

    def test_clear(self):
        self.thing.clear()
        assert self.thing.size() == 0
```

`test_clear`가 먼저 실행되면 `thing`이 비워진다. `test_size`가 다음에 돌면 `size`가 `0`이라 실패한다. 테스트 순서가 결과를 바꾼다. 재현 불가능한 버그가 된다.

setUp으로 매 테스트마다 새 `Thing`을 만들면 이 문제가 사라진다.

```python
class TestThing(TestCase):
    def setUp(self):
        self.thing = Thing()
        self.thing.add("a")
        self.thing.add("b")

    def test_size(self):
        assert self.thing.size() == 2

    def test_clear(self):
        self.thing.clear()
        assert self.thing.size() == 0
```

이게 25장의 **Isolated Test** 패턴이다. *테스트는 서로 독립이어야 한다.*

## fixture라는 이름

`setUp`이 준비하는 객체와 상태를 *fixture*라 부른다. 사진관에서 사진 배경과 조명이 미리 설치된 상태를 말하는 단어에서 왔다. 매 사진(테스트)이 같은 배경에서 찍힌다.

xUnit Patterns(Gerard Meszaros)는 fixture를 더 정교하게 분류한다. *Fresh Fixture*는 매번 새로 만든다(테스트당). *Shared Fixture*는 여러 테스트가 공유한다(고비용일 때만). Beck의 기본은 Fresh Fixture다.

## setUp의 비용

setUp이 매 테스트마다 돈다는 사실은 비용이다. 무거운 객체(DB 연결, 큰 객체 그래프)면 테스트가 느려진다.

| 전략 | 장점 | 단점 |
|------|------|------|
| Fresh Fixture (Beck 기본) | 완전 격리, 안전 | 느릴 수 있음 |
| Shared Fixture (BeforeAll) | 빠름 | 순서 의존, 누수 위험 |
| Mock으로 무거운 부분 대체 | 빠름 + 격리 | Mock 정확성 의존 |

JUnit의 `@BeforeAll`, pytest의 `session` 스코프 fixture는 Shared Fixture를 위한 장치다. 그러나 기본은 Fresh다.

## 모든 xUnit 후예의 공통

setUp 패턴은 그 후 모든 테스트 프레임워크의 표준이 됐다.

```text
JUnit  — @Before / @BeforeEach
NUnit  — [SetUp]
pytest — fixture (autouse)
RSpec  — before(:each)
Mocha  — beforeEach()
```

이름과 문법은 달라도 정신은 같다. 매 테스트 전에 fixture를 준비한다.

## 정리

- `setUp` 메서드 도입 — 매 테스트 직전 자동 호출
- 매 테스트가 깨끗한 fixture에서 시작 → 테스트 격리
- 25장의 **Isolated Test** 패턴의 토대
- 모든 xUnit 후예가 같은 패턴(이름만 다름)
- Fresh Fixture가 기본, Shared는 비용 때문에만
