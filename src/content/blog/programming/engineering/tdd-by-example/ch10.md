> *Test-Driven Development by Example* Chapter 10 요약

## 이 챕터의 메시지

9장에서 `times`를 `Money`로 끌어올린 결과, `Money`가 더 이상 추상일 이유가 없어졌다. 자식 클래스가 추가하는 동작이 없기 때문이다. 이 장은 `Money`를 구체 클래스로 만들어 다음 단계인 `Dollar`/`Franc` 제거의 길을 연다.

## 추상에서 구체로

`Money`에 생성자와 필드 접근을 두고 직접 인스턴스화 가능하게 한다.

```java
class Money {
    protected int amount;
    protected String currency;

    Money(int amount, String currency) {
        this.amount = amount;
        this.currency = currency;
    }

    Money times(int multiplier) {
        return new Money(amount * multiplier, currency);
    }

    public boolean equals(Object obj) {
        Money money = (Money) obj;
        return amount == money.amount
            && currency.equals(money.currency);
    }
}
```

이전에는 `times`가 `new Dollar(...)`를 만들었다. 이제는 `new Money(...)`를 만든다. 어차피 `currency`로 통화를 구별하므로 결과의 동작은 동일하다.

## Factory도 단순화

`Money.dollar`와 `Money.franc`이 `Money` 자체를 만들도록 바꾼다.

```java
class Money {
    static Money dollar(int amount) {
        return new Money(amount, "USD");
    }

    static Money franc(int amount) {
        return new Money(amount, "CHF");
    }
}
```

`Dollar`와 `Franc` 서브클래스가 더는 필요 없다. 다음 장에서 그것들을 제거할 준비가 끝났다.

## 사용자 코드는 무엇을 보는가

외부 코드 입장에서 보면 아무것도 바뀌지 않았다.

```java
Money five = Money.dollar(5);
Money ten = five.times(2);
assertEquals(Money.dollar(10), ten);
```

8장의 Factory Method 도입이 외부 인터페이스를 안정시킨 덕분이다. 외부가 안정돼 있으면 내부의 큰 재편이 사용자 코드를 깨지 않는다. 이게 캡슐화의 실용적 가치다.

## 점진적 변경의 안전성

여기서 일어난 단계를 정리하면 다음과 같다.

```text
ch06: Money 추상 부모 추출, equals를 Pull Up
ch07: currency를 데이터로
ch08: Factory Method 도입
ch09: times를 Pull Up
ch10: Money를 concrete로
ch11: Dollar/Franc 제거 (예정)
```

각 장이 한 가지 변경만 한다. 매 장 끝에 모든 테스트가 통과한다. 6장에서 11장까지 디자인이 크게 바뀌는데도 매 단계가 안전하다.

이게 Beck이 17장의 회고에서 말하는 *"TDD는 진화적 디자인의 도구다"*의 의미다. 처음 옳다고 생각한 디자인(`Dollar` + `Franc` 별도 클래스)이 결국 다른 모습(`Money` 한 클래스)으로 진화하는 동안 테스트가 안전망 역할을 했다.
