> *Test-Driven Development by Example* Chapter 22 요약

## 이 챕터의 메시지

21장의 `run`은 testMethod의 예외만 잡았다. 그러나 실제로는 세 곳에서 예외가 날 수 있다. setUp, testMethod, tearDown이다. 각각이 다른 의미를 가진다.

이 장은 그 세 가지 실패를 분리해 처리한다. 핵심 결정은 *setUp이 실패하면 testMethod를 실행하지 않는다*는 것이다. fixture가 없으므로 테스트가 의미 없기 때문이다.

## setUp 실패 분리

setUp을 별도 try로 감싸고, 실패 시 즉시 반환한다.

```python
class TestCase:
    def run(self, result):
        result.testStarted()
        try:
            self.setUp()
        except:
            result.testFailed()
            return    # testMethod 실행하지 않음

        try:
            method = getattr(self, self.name)
            method()
        except:
            result.testFailed()

        self.tearDown()
```

setUp이 던지면 result에 실패를 기록하고 거기서 끝낸다. testMethod도 tearDown도 호출하지 않는다.

## 왜 setUp 실패에 testMethod를 안 돌리나

fixture가 준비되지 않은 상태로 testMethod를 돌리면 무엇이 일어날까?

```python
class TestThing(TestCase):
    def setUp(self):
        self.thing = make_thing()    # 여기서 예외

    def test_size(self):
        assert self.thing.size() == 0    # self.thing이 없으므로 AttributeError
```

setUp 실패 → testMethod도 자동으로 실패. 그러나 진짜 실패 원인은 setUp이고, testMethod의 실패는 부수적이다. 두 번 실패로 카운트되면 통계가 왜곡된다.

setUp이 실패하면 거기서 멈추는 것이 정직하다.

## 세 가지 실패의 의미

각 단계의 실패는 다른 정보다.

| 실패 위치 | 의미 | 디버깅 단서 |
|-----------|------|------------|
| setUp | fixture를 만들 수 없음 | 의존성, 환경, 설정 |
| testMethod | 검증 대상이 잘못 동작 | 진짜 버그 |
| tearDown | 정리 실패 | 자원 누수, 외부 상태 |

좋은 xUnit은 실패 메시지에 위치를 표시한다. *"setUp failed"*는 *"test failed"*와 다른 의미다.

## tearDown 처리의 미묘함

위의 구현은 tearDown 실패를 잡지 않는다. testMethod가 통과해도 tearDown이 예외를 던지면 그 예외가 그대로 전파된다. result에는 통과로 기록된 채로.

더 견고한 구현은 tearDown도 try로 감싼다.

```python
def run(self, result):
    result.testStarted()
    try:
        self.setUp()
    except:
        result.testFailed()
        return

    try:
        method = getattr(self, self.name)
        method()
    except:
        result.testFailed()

    try:
        self.tearDown()
    except:
        result.testFailed()   # 또는 별도 카운트
```

진짜 xUnit 프레임워크들은 더 정교하다. JUnit은 `@After`에서 던진 예외를 별도로 보고한다. AssertJ나 pytest는 *test error* vs *test failure*를 구분한다. Beck의 책은 그 정도까지는 가지 않는다 — Part II의 목적은 패턴의 본질을 보여 주는 것이다.

## try/finally의 자바 표현

자바라면 같은 정신이 `try`/`finally`로 표현된다.

```java
public void run(TestResult result) {
    result.testStarted();
    try {
        try {
            setUp();
        } catch (Throwable t) {
            result.testFailed();
            return;
        }
        try {
            invokeTestMethod();
        } catch (Throwable t) {
            result.testFailed();
        }
    } finally {
        try {
            tearDown();
        } catch (Throwable t) {
            result.testFailed();
        }
    }
}
```

`finally`가 tearDown 호출을 보장한다. testMethod가 어떤 예외를 던져도 tearDown이 돌아간다. 20장에서 미뤘던 *예외 시 tearDown 보장*이 여기서 완성된다.

## Exception 처리의 일반 원칙

이 장의 결정들은 일반화될 수 있다.

**1. 예외를 잡되, 잡은 사실을 기록한다 (silent catch 금지)**


**2. 의미가 다른 예외는 다른 곳에서 잡는다**


**3. 자원 정리는 finally 또는 with/RAII로 보장한다**


**4. 예외 메시지에 위치 정보를 포함한다**

이 원칙들은 *Effective Java* 9장, *Clean Code* 7장과 같은 결론에 닿는다.

## 정리

- setUp 실패 시 testMethod를 실행하지 않는다 — fixture 없으면 의미 없음
- 세 단계의 실패가 각각 다른 의미를 가진다
- tearDown은 항상 호출되도록 try/finally
- 진짜 xUnit은 test error vs test failure를 구분 — Beck의 책은 핵심만
- 같은 원칙이 일반 예외 처리에도 적용
