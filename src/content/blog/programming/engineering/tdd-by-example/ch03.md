> *Test-Driven Development by Example* Chapter 3 요약

## 이 챕터의 메시지

2장에서 `Dollar`를 값 객체로 바꿨다. 이제 값 객체의 두 번째 의무인 동등성 비교를 구현한다. 그 과정에서 Beck의 핵심 패턴인 **Triangulation**이 처음 등장한다. 한 데이터 포인트로는 강제할 수 없는 일반화를 두 번째 테스트가 끌어내는 기법이다.

## 첫 테스트

가장 단순한 시나리오부터 시작한다. 같은 값을 가진 두 `Dollar`가 동등해야 한다.

```java
public void testEquality() {
    assertTrue(new Dollar(5).equals(new Dollar(5)));
}
```

가장 빠른 초록을 위한 구현은 명백하다.

```java
public boolean equals(Object obj) {
    return true;
}
```

분명히 잘못된 코드다. 그러나 일단 빨간 막대는 사라졌다. 1장에서 본 **Fake It** 패턴의 또 다른 모습이다.

## Triangulation으로 일반화 강제

지금 상태에서 멈추면 `equals`는 영원히 `true`만 반환한다. 진짜 구현으로 옮기는 가장 좋은 방법은 두 번째 테스트를 추가하는 것이다.

```java
public void testEquality() {
    assertTrue(new Dollar(5).equals(new Dollar(5)));
    assertFalse(new Dollar(5).equals(new Dollar(6)));   // 추가
}
```

이제 `return true`로는 통과할 수 없다. 진짜 비교를 해야 한다.

```java
public boolean equals(Object obj) {
    Dollar dollar = (Dollar) obj;
    return amount == dollar.amount;
}
```

두 테스트가 모두 통과한다.

## Triangulation의 기하학적 직관

Beck이 이 기법을 *Triangulation*이라 부르는 이유는 측량 비유에서 나온다. 한 점에서는 무수히 많은 직선이 통과하기 때문에 방향을 결정할 수 없다. 두 점이 있으면 통과하는 직선은 단 하나로 결정된다.

```text
한 데이터 포인트  → 한 점을 통과하는 직선은 무한
두 데이터 포인트  → 두 점을 통과하는 직선은 유일
```

테스트도 마찬가지다. 한 케이스만으로는 어떤 일반화도 정당화되지 않는다. 비슷한 두 번째 케이스가 추가되어야 진짜 패턴이 드러난다.

## Fake It vs Triangulation vs Obvious

Beck은 28장에서 초록 막대를 만드는 세 가지 방법을 정리한다. `equals` 사례는 그 가운데 두 가지가 차례로 등장한다.

| 방법 | 사용 시점 | `equals` 사례 |
|------|----------|---------------|
| Fake It | 가장 빠른 초록을 원할 때 | `return true` |
| Triangulation | 일반화 방향이 불확실할 때 | 두 번째 테스트 추가 |
| Obvious Implementation | 답이 5분 안에 보일 때 | `amount == dollar.amount` 한 줄 |

`equals` 정도는 사실 *Obvious Implementation*으로 한 번에 짤 수도 있다. 그러나 Beck이 이 단계를 일부러 거쳐 보이는 이유는 어려운 영역에서 같은 기법이 어떻게 작동하는지를 익히게 하기 위함이다.

## 코드의 약점

방금 만든 `equals`에는 두 가지 결함이 남아 있다.

```java
public boolean equals(Object obj) {
    Dollar dollar = (Dollar) obj;
    return amount == dollar.amount;
}
```

첫째, `obj`가 `null`이면 `NullPointerException`이 난다. 둘째, `Dollar`가 아닌 객체가 들어오면 `ClassCastException`이 난다. 자바의 표준 관례는 다음 형태다.

```java
public boolean equals(Object obj) {
    if (!(obj instanceof Dollar)) return false;
    Dollar dollar = (Dollar) obj;
    return amount == dollar.amount;
}
```

Beck은 이 정도의 견고함을 지금 단계에서 강제하지 않는다. 대신 To-Do에 적어 둔다. 그게 *한 번에 한 가지*의 정신이다.

## To-Do List 갱신

```text
[ ] $5 + 10 CHF = $10 (환율 2:1)
[X] $5 × 2 = $10
[ ] amount를 private으로
[X] Dollar의 사이드 이펙트 제거
[ ] Money rounding
[X] equals()
[ ] hashCode()
[ ] equals(null)             ← 새로 추가
[ ] equals(다른 타입의 객체)  ← 새로 추가
```

다음 장은 캡슐화로 넘어간다. `equals`가 생겼으니 외부에서 `amount`에 직접 접근할 이유가 사라졌다.

## 정리

- `equals`를 구현하면서 **Triangulation** 패턴을 처음 본다
- 한 테스트로는 강제할 수 없는 일반화를 두 번째 테스트가 끌어낸다
- Fake It → Triangulate → Refactor의 순서가 한 사이클 안에 모두 들어간다
- 모든 약점을 한 번에 해결하지 않고 To-Do에 적어 둔다 (한 번에 한 가지)
