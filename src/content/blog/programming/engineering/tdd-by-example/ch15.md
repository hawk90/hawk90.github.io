> *Test-Driven Development by Example* Chapter 15 요약

## 이 챕터의 메시지

15장은 책의 처음 목표였던 다중 통화 덧셈을 완성한다. `Bank`가 환율 표를 가지고, `reduce`가 그 표를 참조해 통화 변환을 수행한다. 1장에서 시작한 한 줄짜리 곱셈에서 14장에 걸친 모든 준비가 이 한 테스트로 결실을 맺는다.

## 목표 테스트

원래 책 시작 페이지의 요구사항이다.

```java
public void testMixedAddition() {
    Expression fiveBucks = Money.dollar(5);
    Expression tenFrancs = Money.franc(10);
    Bank bank = new Bank();
    bank.addRate("CHF", "USD", 2);
    Money result = bank.reduce(fiveBucks.plus(tenFrancs), "USD");
    assertEquals(Money.dollar(10), result);
}
```

CHF 2단위가 USD 1단위와 같다는 환율을 등록하고, `$5 + 10 CHF`을 USD로 축소한다. 답은 `$5 + ($10 / 2 USD환산) = $10`이다.

## Bank의 환율 표

`Bank`가 통화 쌍을 키로 환율을 보관한다.

```java
class Bank {
    Map<Pair, Integer> rates = new HashMap<>();

    void addRate(String from, String to, int rate) {
        rates.put(new Pair(from, to), rate);
    }

    int rate(String from, String to) {
        if (from.equals(to)) return 1;
        return rates.get(new Pair(from, to));
    }
}
```

`Pair`는 `(from, to)` 두 문자열을 묶은 값 객체다. `equals`/`hashCode`를 구현해 Map의 키로 쓸 수 있다.

```java
class Pair {
    private String from;
    private String to;

    Pair(String from, String to) {
        this.from = from;
        this.to = to;
    }

    public boolean equals(Object obj) {
        Pair pair = (Pair) obj;
        return from.equals(pair.from) && to.equals(pair.to);
    }

    public int hashCode() {
        return 0;   // 단순 구현, 충분
    }
}
```

`hashCode`가 `0`인 것은 의도다. 모든 Pair가 같은 버킷으로 가지만 `equals`가 충돌을 해결한다. 성능은 떨어지지만 정확하다. 더 나은 hashCode는 To-Do로 미룬다.

## Money.reduce에 환율 적용

`Money.reduce`도 이제 `Bank`를 받는다. 자기 통화와 목표 통화가 다르면 환율로 나눈다.

```java
class Money implements Expression {
    public Money reduce(Bank bank, String to) {
        int rate = bank.rate(currency, to);
        return new Money(amount / rate, to);
    }
}
```

`Sum.reduce`도 마찬가지로 `Bank`를 받아 각 피연산자에 위임한다.

```java
class Sum implements Expression {
    public Money reduce(Bank bank, String to) {
        int amount = augend.reduce(bank, to).amount
                   + addend.reduce(bank, to).amount;
        return new Money(amount, to);
    }
}
```

`Bank.reduce`도 시그니처가 바뀐다.

```java
class Bank {
    Money reduce(Expression source, String to) {
        return source.reduce(this, to);
    }
}
```

## 메인 목표 달성

이제 테스트가 통과한다. 책의 처음 두 페이지에서 약속한 목표가 14장의 사이클을 거쳐 달성됐다.

```text
$5 + 10 CHF, rate(CHF→USD) = 2
↓ Sum.reduce
$5.reduce + (10 CHF).reduce
↓
5 USD + (10 / 2 USD) = 5 + 5 = 10 USD
```

각 단계가 명확하고 검증되어 있다. 그게 TDD의 약속이다.

## 환율의 정수 나눗셈 한계

`amount / rate`이라는 정수 나눗셈이 약점을 드러낼 수 있다. `11 CHF`을 USD로 환산하면 `11 / 2 = 5`가 되어 소수점이 잘린다. 실제 금융 코드라면 이 부분은 `BigDecimal`이나 통화 단위(센트 단위 정수)로 정밀하게 처리해야 한다.

Beck은 책에서 이 단순화를 의도적으로 받아들인다. 핵심은 *TDD의 흐름*을 보여 주는 것이지 금융 라이브러리를 짜는 것이 아니다. To-Do의 *Money rounding* 항목이 이 한계를 가리킨다.
