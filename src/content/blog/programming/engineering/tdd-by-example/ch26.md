> **Test-Driven Development by Example** Chapter 26 요약

## 이 챕터의 메시지

테스트가 실패하는 시점 — 어떤 테스트를 짤지의 패턴.

## 패턴 — One Step Test

> **현재 위치에서 가장 작은 다음 한 걸음**을 찾아라.

너무 큰 한 걸음은 — 실패. 너무 작은 한 걸음은 — 진척 없음. 적당한 한 걸음.

```
지금: $5 + $5 = $10 동작
다음 너무 큼: $5 + 10 CHF = $10
다음 너무 작음: $6 + $6 = $12 (실질적 진척 X)
다음 적당: 같은 통화 덧셈 메서드 시그니처 결정
```

## 패턴 — Starter Test

> **첫 테스트는 가장 단순한 것.**

새 시스템 / 모듈을 시작할 때 — 가장 단순한 시나리오.

```
Money — "$5 × 2 = $10"
xUnit — "테스트 메서드가 실행됐는지"
```

복잡한 케이스부터 시작 → 진척 안 됨. 단순부터.

## 패턴 — Explanation Test

> **이미 동작하는 코드에 — 설명용 테스트 추가.**

레거시 코드 / 외부 라이브러리를 — 테스트로 문서화.

```python
def test_explanation():
    # 이 함수가 X 입력에 Y 출력함을 보여 줌
    assertEquals(Y, function(X))
```

## 패턴 — Learning Test

> **외부 API를 — 테스트로 학습.**

새 라이브러리 / API를 — 직접 호출하기 전에 테스트로.

```python
def test_library_behavior():
    result = thirdParty.doSomething(input)
    # 진짜 동작하는지 확인
    assertEquals(expected, result)
```

문서가 거짓말일 수 있음. 테스트로 — 실제 동작 검증.

## 패턴 — Another Test

> **새 아이디어가 떠오르면 — Test List에 추가, 현재 작업 계속.**

집중 유지. 산만함 회피.

## 패턴 — Regression Test

> **버그 발견 시 — 그 버그를 잡는 테스트를 먼저.**

```
1. 버그 발견
2. 그 버그를 재현하는 테스트 작성 → 실패
3. 버그 수정
4. 테스트 통과
5. 미래에 — 같은 버그가 재발하면 즉시 발견
```

매 버그가 — 영원한 테스트.

## 정리

- **One Step Test** — 적당한 한 걸음
- **Starter Test** — 가장 단순부터
- **Explanation Test** — 코드 / 라이브러리 문서화
- **Learning Test** — 외부 API 학습
- **Another Test** — Test List 활용, 집중 유지
- **Regression Test** — 버그가 영원한 테스트
