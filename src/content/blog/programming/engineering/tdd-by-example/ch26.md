> *Test-Driven Development by Example* Chapter 26 요약

## 이 챕터의 메시지

테스트가 실패하는 시점에 — *어떤 테스트를 짤지*에 대한 패턴들이다. TDD의 사이클에서 가장 어려운 결정 가운데 하나가 *다음 테스트를 무엇으로?*이다. 이 장은 그 결정을 돕는 여섯 가지 발견적 도구다.

## 패턴 1 — One Step Test

> 현재 위치에서 가장 작은 의미 있는 다음 한 걸음을 찾아라.

너무 큰 걸음은 통과시키기 어렵다. 너무 작은 걸음은 진척이 없다. *적당히 작은* 한 걸음을 고른다.

```text
지금  : $5 + $5 = $10 동작
너무 큼: $5 + 10 CHF = $10 (환율, Expression, reduce 전부 필요)
너무 작음: $6 + $6 = $12 (같은 코드, 의미 없음)
적당 : $5 + 10 CHF의 결과 타입 결정 (Expression 도입)
```

12장의 결정이 정확히 이 패턴이다. 한 번에 *Expression 추상 도입*만 한다. 환율은 다음 사이클로 미룬다.

이 결정에는 경험이 필요하다. 익숙한 영역은 큰 걸음, 낯선 영역은 작은 걸음. 32장에서 같은 메시지가 다시 나온다.

## 패턴 2 — Starter Test

> 첫 테스트는 가장 단순한 것.

새 시스템이나 모듈을 시작할 때 *가장 단순한 시나리오*를 첫 테스트로 한다.

```text
Money 예제   — "$5 × 2 = $10"
xUnit 예제   — "testMethod가 실행됐는지"
HTTP 라이브러리 — "200 OK를 받는다"
파서          — "빈 입력은 빈 결과"
```

복잡한 시나리오부터 시작하면 진척이 막힌다. 모든 것이 한꺼번에 필요하기 때문이다. 단순한 시나리오는 작은 코드만으로도 통과시킬 수 있다. 그 통과가 다음 단계의 발판이 된다.

## 패턴 3 — Explanation Test

> 이미 동작하는 코드에 — 설명용 테스트를 추가한다.

테스트는 코드의 *실행 가능한 문서*다. 새로 짜는 코드뿐 아니라 *기존 코드의 동작을 설명*하는 데도 쓴다.

```python
def test_explanation():
    """
    parse_date()는 "2024-01-15" 형식을 (2024, 1, 15) 튜플로 반환한다.
    """
    assert (2024, 1, 15) == parse_date("2024-01-15")
```

레거시 코드를 다룰 때 특히 유용하다. 코드의 의도를 모른 채 만지면 위험하다. 먼저 *현재 동작을 테스트로 기록*하면 안전한 변경이 가능해진다.

이 정신은 Michael Feathers의 *Working Effectively with Legacy Code*의 *Characterization Test*와 같다.

## 패턴 4 — Learning Test

> 외부 API를 — 테스트로 학습한다.

새 라이브러리, 새 API를 쓸 때 *진짜 사용 코드에 바로 쓰는 대신* 학습용 테스트로 동작을 검증한다.

```python
def test_library_behavior():
    """
    이 라이브러리의 some_function()이
    빈 리스트에 대해 어떤 동작을 하는가?
    """
    result = third_party.some_function([])
    # 문서엔 None이라 했지만 진짜로?
    assert result is None
```

장점은 두 가지다.

| 장점 | 의미 |
|------|------|
| 문서의 거짓말을 잡는다 | 문서와 실제 동작이 다른 경우가 있음 |
| 미래 회귀 감지 | 라이브러리 업그레이드 시 동작 변화를 즉시 발견 |

특히 두 번째가 결정적이다. Learning Test가 있으면 의존성의 silent breaking change를 즉시 잡는다.

## 패턴 5 — Another Test

> 새 아이디어가 떠오르면 — Test List에 추가, 현재 작업 계속.

작업 중에 또 다른 테스트나 리팩터링이 떠오르는 것은 자연스럽다. 그러나 그것을 *지금* 한다면 현재 사이클이 길어진다. 길어지면 깨질 위험이 커진다.

해법은 명확하다.

```text
1. 떠오른 아이디어 → Test List에 적는다
2. 현재 작업으로 돌아온다
3. 현재 사이클이 끝나면 → 리스트에서 다음 한 가지 선택
```

집중력의 보존이다. 25장의 Test List 패턴의 운용 규칙이다.

## 패턴 6 — Regression Test

> 버그 발견 시 — 그 버그를 잡는 테스트를 먼저 짜라.

버그가 발견되면 즉시 코드를 고치고 싶어진다. TDD는 그 충동을 한 걸음 늦춘다. *먼저 버그를 재현하는 테스트를 짠다.*

```text
1. 버그 발견 — 어떤 입력에서 잘못된 동작
2. 그 입력으로 테스트를 짠다 → 실패 (빨간 막대)
3. 버그를 수정한다 → 통과 (초록 막대)
4. 이 테스트가 영구히 남는다
5. 미래에 같은 버그가 재발하면 즉시 빨간 막대로 알림
```

매 버그가 *영원한 테스트*가 된다. 같은 버그가 두 번 일어나지 않는다는 약속이다.

이 정신은 모든 회사의 버그 추적 시스템과 결합한다. *"버그를 닫기 전에 테스트를 추가했는가?"*가 코드 리뷰의 기본 체크리스트가 된다.

## 정리

- **One Step Test** — 적당히 작은 다음 한 걸음
- **Starter Test** — 가장 단순한 첫 시나리오
- **Explanation Test** — 기존 코드의 동작 설명
- **Learning Test** — 외부 API의 동작 학습 + 회귀 감지
- **Another Test** — 떠오른 아이디어는 리스트에, 현재 작업에 집중
- **Regression Test** — 버그가 영원한 테스트
