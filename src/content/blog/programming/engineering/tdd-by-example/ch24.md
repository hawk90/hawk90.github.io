> *Test-Driven Development by Example* Chapter 24 요약

## 이 챕터의 메시지

Part II가 끝난다. 18장부터 23장까지 6개 사이클로 xUnit의 핵심을 모두 만들었다. 이 장은 그 작은 프레임워크의 우아함을 회고한다. 핵심 통찰은 **self-testing**의 정신이다. *내가 만든 도구가 자기 자신을 검증한다.*

또한 Beck의 이 작은 디자인이 어떻게 산업 표준이 됐는지를 돌아본다. 한 사람의 단순한 모델이 거의 모든 프로그래밍 언어로 퍼졌다.

## Self-Testing의 우아함

부트스트랩의 정신을 도식으로 보면 다음과 같다.

**1. TestCase 클래스를 만든다 (print로 손 검증)**


**2. 그 TestCase를 사용해 — TestCase 자체의 동작을 테스트**


**3. 새 기능을 추가할 때마다 — 그 기능을 자기로 검증**


**4. 결국 "내가 만든 도구가 자기 자신을 완전히 검증한다"**

이 정신은 컴퓨터 과학 전반에 흐른다.

| 사례 | self-testing/hosting |
|------|---------------------|
| C 컴파일러를 C로 짠다 | 자기 자신을 컴파일 |
| LISP 인터프리터를 LISP로 짠다 | 자기 자신을 평가 |
| Smalltalk 시스템이 Smalltalk로 구성된다 | 자기를 자기로 표현 |
| xUnit이 xUnit으로 테스트된다 | 자기를 자기로 검증 |

도구가 자기에 대한 충분한 표현력을 가졌다는 증거다.

## xUnit이 산업에 미친 영향

Beck의 작은 책 한 권이 산업 전체를 바꾼 사례는 흔하지 않다. xUnit의 후예들은 거의 모든 언어와 플랫폼에 존재한다.

| 언어 | xUnit 후예 |
|------|-----------|
| Java | JUnit, TestNG |
| .NET | NUnit, xUnit.net, MSTest |
| Python | unittest, pytest |
| Ruby | Test::Unit, RSpec |
| JavaScript / TypeScript | Mocha, Jest, Vitest |
| C++ | GoogleTest, Catch2, doctest |
| Go | testing 표준 라이브러리 |
| Rust | 내장 #[test] |
| Swift | XCTest |
| Kotlin | KotlinTest, Kotest |

이름과 문법은 다르지만 정신은 같다. `TestCase`, `setUp`/`tearDown`, `assertion`, `TestResult`, `TestSuite`라는 다섯 추상이 모두 등장한다.

## TestCase.run의 Template Method

`TestCase.run`은 GoF의 **Template Method** 패턴의 교과서 사례다. 부모가 알고리즘의 골격(run의 흐름)을 정하고, 자식이 빈 슬롯(setUp/testMethod/tearDown)을 채운다.

```python
def run(self, result):
    self.setUp()      # 자식이 오버라이드 가능
    try:
        method = getattr(self, self.name)
        method()       # 자식이 정의한 testMethod
    except:
        result.testFailed()
    self.tearDown()    # 자식이 오버라이드 가능
```

부모는 *언제 무엇을 호출할지*를 안다. 자식은 *무엇을 할지*만 채운다. 이 패턴은 30장에서 다시 다뤄진다.
