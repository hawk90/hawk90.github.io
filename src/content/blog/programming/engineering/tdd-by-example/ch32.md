> **Test-Driven Development by Example** Chapter 32 요약

## 이 챕터의 메시지

책의 마지막. TDD의 한계와 미래.

> **TDD는 도구이지 종교가 아니다.**

## TDD가 잘 맞는 곳

- **순수 로직** — 비즈니스 규칙, 알고리즘
- **잘 정의된 인터페이스** — 입력 → 출력
- **장기 유지보수** — 테스트가 자산
- **팀 작업** — 회귀 방지

## TDD가 어려운 곳

- **UI** — 시각적 검증 필요
- **레거시 코드** — 테스트 없는 코드에 테스트 추가 (Feathers의 "Legacy Code")
- **외부 자원** — DB, 네트워크 (Mock 필요)
- **동시성** — 비결정적 실행

이런 곳에서도 — 부분 TDD 가능. 그러나 100% TDD는 비현실적.

## 자주 묻는 질문

**Q: 얼마나 많은 테스트?**

> 자기 두려움이 사라질 때까지.

너무 적으면 — 변경이 두렵다. 너무 많으면 — 유지 비용 큼. 균형.

**Q: 큰 단계로 갈 수 있나?**

> 어려운 곳에서는 작게. 쉬운 곳에서는 크게.

같은 작가도 — 익숙한 곳은 크게, 새 영역은 작게.

**Q: 리팩터링을 너무 자주?**

> 작은 단계가 — 큰 변경의 예방.

매번 작은 리팩터링 = 큰 리팩터링이 필요 없음.

## TDD의 한계 — Beck의 솔직함

```
"TDD가 모든 문제를 풀지 않는다."
```

- 시스템 설계 — TDD가 알려 주지 않음
- 도메인 이해 — TDD가 만들어 주지 않음
- 사람 / 팀 / 비즈니스 — TDD 영역 아님

**좋은 디자인의 도구 하나일 뿐**. 다른 도구도 필요.

## 미래

Beck이 2003년에 쓴 책. 20년이 지난 지금:

- TDD가 산업 표준이 됐다 (일부 영역)
- xUnit이 모든 언어에 (JUnit, pytest, RSpec, ...)
- CI/CD가 보편화 — 매 commit이 모든 테스트 실행
- TDD vs BDD vs ATDD 논쟁

다음 20년 — AI, formal verification, property-based testing이 더 큰 역할.

## 시리즈 마무리

32장의 여정.

- **Part I** — Money 예제, TDD 리듬
- **Part II** — xUnit 예제, self-bootstrapping
- **Part III** — 모든 패턴 정리

핵심 메시지를 한 줄로:

> **테스트가 안 짠다면 — 코드를 짤 자격이 없다.**

(과장이지만 — 정신은 그렇다.)

## 정리

- TDD는 **도구이지 종교가 아니다**
- 잘 맞는 곳 — 로직, 인터페이스, 장기, 팀
- 어려운 곳 — UI, 레거시, 외부 자원, 동시성
- 균형 — 너무 적지도, 너무 많지도
- 다른 도구도 필요 — 디자인 / 도메인 / 사람
