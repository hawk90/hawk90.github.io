> *Test-Driven Development by Example* Chapter 32 요약

## 이 챕터의 메시지

책의 마지막 장이다. 32개 사이클의 사례 워크스루를 끝낸 뒤, Beck은 한 걸음 물러서서 TDD의 한계와 미래를 본다. 핵심 메시지는 단순하면서도 종종 잊힌다.

> TDD는 도구이지 종교가 아니다.

도구는 적절한 곳에 쓴다. 모든 망치질이 정당하지 않듯, 모든 코드가 TDD로 짤 필요는 없다. 어디서 빛나고 어디서 흐릿한지를 분별하는 것이 *Mastering*의 의미다.

## TDD가 잘 맞는 곳

| 영역 | 이유 |
|------|------|
| 순수 로직 (알고리즘, 비즈니스 규칙) | 입력과 출력이 명확, 부작용 없음 |
| 잘 정의된 인터페이스 | 계약이 분명하면 테스트가 자연스러움 |
| 장기 유지보수 코드 | 테스트가 미래의 자산 |
| 팀 작업 | 회귀 방지, 변경의 신뢰 |

Money 예제는 모든 면에서 적합했다. 순수 로직, 명확한 인터페이스, 다른 사람도 이해할 수 있는 코드. TDD가 빛을 발한다.

## TDD가 어려운 곳

| 영역 | 어려움 |
|------|--------|
| UI | 시각적 검증 필요, 자동화 비용 큼 |
| 레거시 코드 | 테스트 없는 코드에 테스트 추가가 닭과 달걀 |
| 외부 자원 (DB, 네트워크, 시간) | 환경 의존, mock의 비용 |
| 동시성 | 비결정적, 재현 어려움 |
| GUI 애니메이션, 게임 물리 | 시각/체감이 검증의 핵심 |

이런 영역에서도 *부분 TDD*는 가능하다. 비즈니스 로직을 분리해 그 부분만 TDD로, UI나 외부 의존은 다른 검증 방법으로. 100% TDD는 비현실적이다.

레거시 코드의 경우 Michael Feathers의 *Working Effectively with Legacy Code*가 별도의 전문서로 발전했다. *Characterization Test*와 *Seam* 개념이 그 책의 핵심이다.

## 자주 묻는 질문 — Beck의 답

### 얼마나 많은 테스트가 충분한가?

> 자기 두려움이 사라질 때까지.

너무 적으면 변경이 두렵다. 너무 많으면 유지 비용이 크다. *균형*이 답이지만 정답은 영역마다 다르다.

코드 커버리지를 100%로 만드는 것은 자체 목표가 아니다. *변경할 때 두렵지 않은가?*가 진짜 기준이다.

### 큰 단계로 갈 수 있나?

> 어려운 곳에서는 작게. 쉬운 곳에서는 크게.

같은 사람도 익숙한 영역은 한 사이클을 크게, 낯선 영역은 작게 잡는다. 보폭은 *상황 판단*이다.

신참은 모든 곳에서 작게 시작하는 것이 안전하다. 경험이 자라면 어디서 큰 보폭이 정당화되는지 알게 된다.

### 리팩터링이 너무 자주 아닌가?

> 작은 단계가 — 큰 변경의 예방이다.

매 사이클마다 작은 리팩터링을 한다. 그 누적이 *Big Ball of Mud*를 방지한다. 한 번도 정리하지 않으면 어느 시점에 *전체 재작성*이라는 큰 비용이 돌아온다.

매번의 작은 비용이 미래의 큰 비용을 막는다.

## TDD의 한계 — Beck의 솔직함

Beck은 이 장에서 솔직하다.

> TDD가 모든 문제를 풀지 않는다.

| 영역 | TDD가 못하는 것 |
|------|----------------|
| 시스템 디자인 | 큰 그림은 다른 도구 필요 (DDD, C4 모델 등) |
| 도메인 이해 | 코드가 도메인을 *만들지는* 않음 |
| 사람/팀/비즈니스 | 사회적 문제는 TDD 영역 밖 |
| 성능 최적화 | TDD는 동작 검증, 성능은 별도 |
| 사용성/UX | 측정 가능한 요구가 아님 |

좋은 디자인의 *한 도구*일 뿐이다. 다른 도구도 필요하다. 도메인 모델링, 시스템 디자인, UX 연구, 성능 프로파일링.

## TDD의 20년 — 2003년에서 2026년까지

Beck이 책을 쓴 것은 2003년이다. 23년이 지난 지금, TDD의 풍경은 어떻게 변했나?

| 변화 | 의미 |
|------|------|
| TDD가 산업 표준이 됨 (일부 영역) | 알고리즘 라이브러리, 핵심 비즈니스 로직 |
| xUnit이 모든 언어에 | 24장에서 본 후예들의 폭발 |
| CI/CD가 보편화 | 매 commit이 모든 테스트 실행 |
| TDD vs BDD vs ATDD 논쟁 | 각각의 strength 분화 |
| Property-based testing | QuickCheck → Hypothesis, fast-check |
| Mutation testing | 테스트의 품질을 테스트 |
| AI-assisted testing | 테스트 생성, 회귀 감지 |

다음 20년은 어떨까? Beck이 책에서 직접 다루지는 않지만 흐름은 짐작 가능하다.

- AI가 *테스트 작성을 보조* (이미 시작)
- *Formal verification*과 TDD의 결합
- Property-based / fuzz testing이 *기본 도구*로
- *Live programming* — IDE가 매 키 입력에 테스트 실행

도구는 진화하지만 *작은 단계, 매 단계 검증, 두려움이 사라질 때까지*라는 정신은 그대로다.

## 시리즈 마무리 — 세 부분의 여정

32장의 여정을 한 표로 정리한다.

| Part | 장 | 다룬 것 |
|------|-----|--------|
| Part I | ch01-17 | Money 예제 — TDD의 리듬과 디자인 진화 |
| Part II | ch18-24 | xUnit 예제 — 자기 자신을 자기로 검증 |
| Part III | ch25-32 | 패턴 카탈로그 — 모든 패턴을 정리 |

핵심 메시지를 한 줄로 압축한다면 다음이다.

> 테스트가 안 짠다면 — 코드를 짤 자격이 없다.

과장이지만, 정신은 그렇다. 검증 없이 만들어진 코드는 *돌아갈 가능성*에 의존한다. TDD는 그 가능성을 *증거*로 바꾼다.

## 정리

- TDD는 **도구이지 종교가 아니다**
- 잘 맞는 곳 — 로직, 인터페이스, 장기, 팀
- 어려운 곳 — UI, 레거시, 외부 자원, 동시성
- 자기 두려움이 사라질 때까지 — 너무 적지도, 너무 많지도
- 어려운 곳은 작게, 쉬운 곳은 크게
- 매번의 작은 리팩터링이 큰 리팩터링의 필요를 없앤다
- TDD가 모든 문제를 풀지 않는다 — 디자인/도메인/사람은 다른 도구

## 시리즈 완결

32장의 여정을 끝까지 함께해 주셔서 감사하다. TDD는 평생의 기술이다. 처음 시작은 어색하고 느리다. 익숙해지면 *두 번째 본성*이 된다.

Beck의 책 표지에 적힌 한 줄을 인용한다.

> I am not a great programmer; I am just a good programmer with great habits.
>
> — Kent Beck

좋은 습관의 누적이 좋은 프로그래머를 만든다. TDD는 그 습관의 가장 강력한 형태 가운데 하나다.
