> **Test-Driven Development by Example** Chapter 30 요약

## 이 챕터의 메시지

TDD를 따라가다 보면 — 자연스럽게 GoF 패턴이 등장.

> **패턴은 추구하지 마라. 그러나 자연스럽게 나오면 — 알아채라.**

## TDD에서 자주 등장하는 패턴

### Command

```
함수 실행 시점 / 위치를 분리하고 싶을 때 — Command 객체.
```

TDD에서 — 비동기 / 콜백 / 큐 작업에 자연스럽게.

### Value Object

```
객체의 정체성 = 값. 불변.
```

Money 예제의 핵심. TDD에서 가장 자주 등장.

### Null Object

```
"없음"을 — null이 아닌 객체로 표현.
```

```python
class NullCustomer:
    def name(self): return "Anonymous"
    def discount(self): return 0
```

코드의 if-not-null 체크가 사라짐.

### Template Method

```
부모가 알고리즘의 골격 — 자식이 구체 단계.
```

TestCase의 `run` 메서드가 — 이 패턴.

```python
def run(self, result):
    self.setUp()
    try:
        self.testMethod()   # 자식이 정의
    except:
        result.failed()
    self.tearDown()
```

### Pluggable Object

```
필드로 보유한 객체 — 실행 시 변경 가능.
```

Strategy 패턴의 다른 이름. TDD에서 종종 — Mock 주입의 토대.

### Composite

```
한 객체와 객체 묶음이 — 같은 인터페이스.
```

TestSuite / TestCase가 — 이 패턴 (23장).

## TDD와 패턴의 관계

```
TDD를 하면 → 패턴이 자연스럽게 나온다
        ↓
패턴 카탈로그를 읽으면 → TDD의 결과를 더 빠르게
```

순서가 거꾸로 일 수도. 패턴을 알고 — TDD에서 빨리 알아챔.

## 정리

- TDD가 자연스럽게 만드는 패턴 — Value Object / Command / Null Object / Template Method / Pluggable Object / Composite
- 패턴 추구가 아니라 — 자연스러운 등장
- 패턴 지식이 — TDD를 더 빠르게
