> *Test-Driven Development by Example* Chapter 30 요약

## 이 챕터의 메시지

TDD를 따라가다 보면 GoF 디자인 패턴이 *자연스럽게* 등장한다. 의도적으로 패턴을 추구하는 게 아니다. 작은 사이클로 디자인을 진화시키다 보면 같은 구조가 반복적으로 떠오른다.

Beck의 한 줄은 다음과 같다.

> 패턴을 추구하지 마라. 그러나 자연스럽게 나오면 알아채라.

이 장은 Money/xUnit 예제에서 만난 패턴들과 그 일반적 사용을 정리한다.

## TDD에서 자주 보는 패턴들

여섯 패턴이 두 예제에서 모두 등장했다.

### Value Object

> 객체의 정체성 = 값. 불변.

`Money`가 정확히 이 패턴이다. 2장에서 처음 등장하고 책 전체의 핵심이 된다.

```java
class Money {
    private int amount;
    private String currency;
    // 불변, equals는 값으로
}
```

도메인 모델에서 가장 흔한 패턴이다. 돈, 좌표, IP 주소, 시간 등 *값이 정체성인 모든 것*이 후보다. TDD에서 자주 등장하는 이유는 *불변이라 테스트가 쉽기* 때문이다.

### Command

> 실행할 동작을 객체로 표현한다.

함수 호출을 객체로 캡슐화한다. 실행 시점/위치를 분리하고 싶을 때 자연스럽게 등장한다.

```python
class SaveCommand:
    def __init__(self, doc, path):
        self.doc = doc
        self.path = path

    def execute(self):
        with open(self.path, "w") as f:
            f.write(self.doc.content)

# 나중에 실행
command = SaveCommand(doc, "/tmp/out.txt")
queue.add(command)
# ...
for c in queue:
    c.execute()
```

비동기, undo/redo, 콜백, 작업 큐 — Command가 등장하는 흔한 영역이다.

### Null Object

> "없음"을 — null이 아닌 객체로 표현한다.

null 체크가 코드 곳곳에 흩어지는 것을 막는다. *동작 없는 객체*가 null의 자리에 들어간다.

```python
class NullCustomer:
    def name(self): return "Anonymous"
    def discount(self): return 0
    def credit_limit(self): return 0

# 사용 코드
customer = repository.find(id) or NullCustomer()
# 이제 if customer is None: 분기가 필요 없음
greeting = f"Hello {customer.name()}"
```

코드의 *if-not-null 체크가 사라진다*. 다형성이 분기를 대신한다.

### Template Method

> 부모가 알고리즘 골격을 정하고, 자식이 빈 슬롯을 채운다.

`TestCase.run`이 교과서적 사례다. 24장에서 이미 분석했다.

```python
class TestCase:
    def run(self, result):
        self.setUp()                       # 자식이 채울 빈 슬롯
        try:
            method = getattr(self, self.name)
            method()                       # 자식이 정의한 testMethod
        except:
            result.testFailed()
        self.tearDown()                    # 자식이 채울 빈 슬롯
```

xUnit, GUI 프레임워크의 lifecycle, 데이터 처리 파이프라인 등에서 흔하다.

### Pluggable Object

> 필드로 보유한 객체 — 실행 시 변경 가능.

Strategy 패턴의 다른 이름이다. 객체의 동작 한 부분을 외부에서 주입받는다.

```python
class Sorter:
    def __init__(self, comparator):
        self.comparator = comparator

    def sort(self, items):
        return sorted(items, key=self.comparator)

# 다른 비교 전략 주입
ascending = Sorter(comparator=lambda x: x.value)
descending = Sorter(comparator=lambda x: -x.value)
```

TDD에서 자주 등장하는 이유는 *Mock 주입의 토대*이기 때문이다. 테스트에서 진짜 객체 대신 mock을 주입할 수 있다.

### Composite

> 한 객체와 객체 묶음이 — 같은 인터페이스.

`Money` ↔ `Sum`, `TestCase` ↔ `TestSuite` 두 사례에서 모두 등장했다. *부분과 전체를 같은 인터페이스로 다룬다*.

```text
Expression
├── Money (leaf)
└── Sum   (composite)
```

```text
Test
├── TestCase   (leaf)
└── TestSuite  (composite)
```

GUI 위젯, 파일 시스템, 컴파일러 AST 등 *재귀 구조가 자연스러운 모든 곳*에서 등장한다.

## TDD와 패턴의 관계

패턴 지식이 TDD에 어떻게 작용하는가? 양방향 관계다.

```text
TDD를 하면        → 패턴이 자연스럽게 떠오른다
패턴 카탈로그 학습 → TDD에서 그 패턴을 더 빨리 알아챈다
```

패턴을 *추구하지는 않지만 알아채면 좋다*. 알아채지 못해도 다음 사이클에서 다시 떠오른다. 한 번에 모를 뿐이다.

이 정신은 Christopher Alexander의 *패턴 언어*에서 시작된다. 패턴은 문제와 해법의 짝이다. 비슷한 문제에 같은 형태의 해법이 반복적으로 좋은 결과를 낸다는 관찰이다.

## 일찍 도입하지 않는다

Beck의 또 다른 메시지는 *premature optimization*과 비슷하다. 패턴을 일찍 도입하면 후회한다.

```text
1. 단순한 코드로 시작
2. 사이클을 돌면서 중복/복잡성 등장
3. 그 시점에 적절한 패턴이 보인다
4. 그때 도입
```

11장의 *Premature inheritance가 모든 악의 근원*이 같은 정신이다. 일찍 도입한 상속 계층이 결국 제거됐다.

## 정리

- TDD가 자연스럽게 만드는 패턴 — Value Object, Command, Null Object, Template Method, Pluggable Object, Composite
- *패턴을 추구하지 마라, 자연스럽게 나오면 알아채라*
- 패턴 지식은 TDD를 더 빠르게 만든다 — 양방향 관계
- 일찍 도입하지 않는다 — 사이클이 패턴의 필요를 드러낼 때까지
