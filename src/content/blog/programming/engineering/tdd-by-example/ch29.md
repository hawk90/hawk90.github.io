> *Test-Driven Development by Example* Chapter 29 요약

## 이 챕터의 메시지

Part II의 xUnit 예제에서 자연스럽게 등장한 패턴들을 정리한다. 모두 18-23장에서 사례로 본 것들이지만, 여기서는 이름과 형태를 명시한다. Gerard Meszaros의 *xUnit Test Patterns*(2007) 책이 이 카탈로그를 훨씬 자세히 다루지만, 그 원형이 모두 이 장에 있다.

## 패턴 1 — Assertion

> Boolean 검증.

테스트의 가장 작은 단위다. 모든 테스트가 어떤 형태의 assertion으로 끝난다.

```python
assert_true(value)
assert_false(value)
assert_equals(expected, actual)
assert_null(value)
assert_not_null(value)
assert_same(expected, actual)       # 같은 객체 (identity)
assert_raises(ExceptionType, callable)
```

각 assertion은 두 가지를 한다. *조건 검사*와 *실패 시 진단 메시지*. 좋은 assertion은 실패할 때 *왜* 실패했는지를 명확히 보여 준다.

| 메시지 | 평가 |
|--------|------|
| `expected 10, but was 7` | 좋은 예 — 값 명시 |
| `expected non-null, but was null` | 좋은 예 — null 여부 명시 |
| `assertion failed` | 나쁜 예 — *무엇이 잘못됐는지 모름* |

JUnit, pytest, NUnit 모두 풍부한 assertion 라이브러리(AssertJ, Hamcrest, FluentAssertions)와 결합해 표현력을 높인다.

## 패턴 2 — Fixture

> 테스트의 setUp으로 준비된 객체들.

19장에서 본 setUp이 만드는 객체 그래프 전체를 fixture라 부른다.

```python
class TestOrder(TestCase):
    def setUp(self):
        # 다음 모두가 fixture
        self.user = User("alice")
        self.product = Product("book", 10)
        self.order = Order(self.user)
        self.order.add(self.product)

    def test_total(self):
        assert 10 == self.order.total()

    def test_user(self):
        assert "alice" == self.order.user.name
```

여러 테스트가 비슷한 시작 상태를 공유할 때 fixture가 *그 상태를 한 곳에 표현*한다.

## 패턴 3 — External Fixture

> 테스트 외부 자원 — DB, 파일, 네트워크.

메모리 객체가 아닌 *외부 자원*을 fixture로 다룰 때의 패턴이다. setUp과 tearDown이 자원의 lifecycle을 관리한다.

```python
class TestRepository(TestCase):
    def setUp(self):
        self.db = open_test_database()
        self.repo = UserRepository(self.db)

    def tearDown(self):
        self.db.close()
        clean_test_data()

    def test_save_and_find(self):
        self.repo.save(User("alice"))
        assert "alice" == self.repo.find_by_name("alice").name
```

이 패턴은 통합 테스트(integration test)의 표준 형태다. Khorikov의 *Unit Testing* 8-9장이 통합 테스트의 fixture 관리를 자세히 다룬다.

특히 트랜잭션 롤백 패턴이 자주 쓰인다. setUp에서 트랜잭션을 열고, tearDown에서 롤백한다. 테스트의 어떤 DB 변경도 영구화되지 않는다.

## 패턴 4 — Test Method

> 테스트 메서드 한 개 = 한 시나리오.

한 테스트가 하나의 시나리오만 검증한다. 여러 시나리오면 여러 메서드로 분리한다.

```python
# 나쁜 예 — 한 메서드에 여러 시나리오
def test_user(self):
    assert "alice" == User("alice").name
    assert User("alice").is_active is True
    assert User("alice").orders == []
    # 첫 번째가 실패하면 나머지가 안 돌고, 무엇이 잘못인지 불명확

# 좋은 예 — 시나리오 분리
def test_user_has_name(self):
    assert "alice" == User("alice").name

def test_user_is_active_by_default(self):
    assert User("alice").is_active is True

def test_new_user_has_no_orders(self):
    assert [] == User("alice").orders
```

이름 자체가 *무엇을 검증하는지*를 보여 준다. 실패 시 어느 시나리오인지 즉시 보인다.

이 원칙은 *Refactoring*의 **Extract Function**과 같은 정신이다. 한 함수가 한 가지 일.

## 패턴 5 — Exception Test

> 예외가 던져지는 것을 검증한다.

특정 입력에서 예외가 *나야 한다*는 검증이다.

```python
# 전통 스타일
def test_invalid_input(self):
    try:
        process(invalid_data)
        self.fail("예외가 던져지지 않음")
    except ValueError:
        pass     # 의도된 예외

# 현대 스타일 — assert_raises
def test_invalid_input(self):
    self.assert_raises(ValueError, lambda: process(invalid_data))

# pytest 스타일
def test_invalid_input():
    with pytest.raises(ValueError):
        process(invalid_data)
```

세 스타일 모두 *예외 메시지를 추가로 검증*하는 옵션을 가진다.

```python
with pytest.raises(ValueError, match="invalid format"):
    process(invalid_data)
```

이러면 *어떤* `ValueError`인지까지 검증한다.

## 패턴 6 — All Tests

> 모든 테스트를 한 번에 실행한다.

23장의 `TestSuite`로 가능해진 것이지만, 현대 xUnit은 자동 검출을 결합한다.

```python
# 수동 — Beck의 모델
def all_tests():
    suite = TestSuite()
    suite.add(TestSomething.test_a)
    suite.add(TestSomething.test_b)
    return suite

# pytest 자동 검출
# test_*.py 파일을 찾는다
# 그 안의 Test* 클래스를 찾는다
# 그 안의 test_* 메서드를 찾는다
# 자동으로 suite에 추가
```

자동 검출은 새 테스트를 추가할 때 등록 코드를 짤 필요가 없게 한다. 단점은 *naming convention에 종속*된다는 것. 큰 프로젝트에서는 어느 쪽이든 비용이 있다.

## 패턴 간의 관계

이 여섯 패턴이 xUnit 한 사이클의 전 과정을 덮는다.

**1. setUp — Fixture (또는 External Fixture)를 준비**


**2. Test Method — 한 시나리오 실행**


**3. Assertion — 검증 (또는 Exception Test)**


**4. tearDown — External Fixture 정리**


**5. All Tests — 위 사이클이 모든 테스트에 반복**

이 다섯 단계가 JUnit, pytest, NUnit 모두의 표준이다.

## 정리

- **Assertion** — 검증의 핵심 단위, 좋은 실패 메시지 필수
- **Fixture** — setUp이 만드는 객체 그래프
- **External Fixture** — DB/파일/네트워크, lifecycle 관리
- **Test Method** — 한 메서드 = 한 시나리오
- **Exception Test** — 예외가 나야 함을 검증
- **All Tests** — 통합 실행, 자동 검출 또는 수동 등록
- 여섯 패턴이 한 xUnit 사이클의 전 과정을 덮는다
