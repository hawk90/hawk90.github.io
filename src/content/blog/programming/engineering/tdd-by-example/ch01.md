> **Test-Driven Development by Example** Chapter 1 요약

## 이 챕터의 메시지

Kent Beck의 TDD 책은 — **워크스루**. Refactoring 1장처럼 한 실제 예제를 끝까지 따라간다.

> **TDD의 리듬을 코드로 보여 준다.**

규칙이 아니라 — 흐름.

## 시작 — Money 예제

요구사항: 다중 통화 보고서.

```
Instrument  Shares  Price    Total
IBM         1000    25 USD   25000 USD
GE          400     100 USD  40000 USD
                              -----
                              65000 USD
```

다중 통화면 — 환율 적용.

```
IBM   1000  25 USD     25000 USD
Novartis  400  150 CHF  60000 CHF
                        (환율 적용 후)
                        125000 USD
```

복잡. 그러나 TDD는 — **작은 한 조각**부터 시작.

## 첫 테스트 — "$5 × 2 = $10"

가장 단순한 시작.

```java
public void testMultiplication() {
    Dollar five = new Dollar(5);
    five.times(2);
    assertEquals(10, five.amount);
}
```

이 코드 — **실패한다** (Dollar 클래스 없음). 의도된 실패.

## TDD의 리듬 — Red / Green / Refactor

```
1. RED — 실패하는 테스트
2. GREEN — 가능한 한 빨리 통과
3. REFACTOR — 중복 제거 / 정리
```

이 사이클을 — 끝없이 반복.

## Green 만들기

가장 단순한 통과 방법.

```java
class Dollar {
    int amount = 10;  // 하드코딩!
    public Dollar(int amount) {}
    public void times(int multiplier) {}
}
```

**"가짜 구현(fake it)"** — 일단 통과시킨다. 옳지 않아도 OK.

## Refactor

정직한 코드로.

```java
class Dollar {
    int amount;
    public Dollar(int amount) {
        this.amount = amount;
    }
    public void times(int multiplier) {
        amount *= multiplier;
    }
}
```

테스트 — 여전히 통과. 일반화 완료.

## 이 작은 단계의 가치

```
단계 1: 빨간 막대 (테스트 실패)
단계 2: 가짜 구현으로 초록
단계 3: 일반화 (refactor)
```

**한 사이클이 분 단위**. 매 단계마다 — 테스트 통과 확인.

## 왜 이렇게 작게

Beck의 통찰:

> **작은 단계는 학습용 과장이 아니다. 어려운 부분에서는 진짜로 작게 가야 한다.**

쉬운 부분은 — 큰 단계 OK. 어려운 부분은 — 작게.

## 사이드 이펙트 문제

```java
Dollar five = new Dollar(5);
five.times(2);   // five.amount = 10
five.times(3);   // five.amount = 30 (!)
```

`times`가 객체를 변경. 같은 객체로 두 번 호출 → 누적.

문제. **value object은 불변**이 자연스럽다.

## To-Do List

Beck의 핵심 도구 — **To-Do List**.

```
TODO:
[ ] $5 + 10 CHF = $10 (환율 2:1)
[ ] $5 × 2 = $10   ← DONE
[ ] make "amount" private
[ ] Dollar의 사이드 이펙트?
[ ] Money rounding?
```

생각이 떠오를 때마다 — 리스트에. 그러나 **지금 작업만** 집중.

## 이 챕터의 메시지

```
1. 가장 작은 한 테스트
2. 가장 단순한 통과
3. 리팩터링
4. To-Do list로 다음 단계
```

이 리듬이 — 책 전체.

## 정리

- TDD = **Red / Green / Refactor** 사이클
- 가장 작은 한 테스트부터
- 일단 통과 → 그 다음 정직하게
- **To-Do list**가 생각을 보관
- 작은 단계는 학습용 과장이 아닌 — 진짜 도구
