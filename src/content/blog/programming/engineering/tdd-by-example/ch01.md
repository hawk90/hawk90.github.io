> *Test-Driven Development by Example* Chapter 1 요약

## 이 챕터의 메시지

Kent Beck의 TDD 책은 정의나 규칙을 늘어놓지 않는다. 대신 한 가지 실제 예제를 처음부터 끝까지 따라가는 **워크스루**의 형식을 택했다. Fowler의 *Refactoring* 1장이 비디오 가게 예제를 끝까지 끌고 가는 것과 같은 정신이다.

저자가 보여 주고자 하는 것은 규칙이 아니라 흐름이다. 빨간 막대를 보고, 가능한 한 빨리 초록으로 만들고, 정직한 코드로 정리하는 사이클이 작은 단위로 끝없이 반복되는 모습을 코드로 따라가게 한다.

## 출발점 — Money 예제

다중 통화 보고서를 만들어야 한다고 가정한다. 모든 종목이 USD라면 단순하다.

```text
Instrument  Shares  Price    Total
IBM         1000    25 USD   25000 USD
GE          400     100 USD  40000 USD
                              -----
                              65000 USD
```

여기에 CHF 같은 다른 통화가 끼면 환율을 적용해 합산해야 한다.

```text
IBM       1000  25 USD     25000 USD
Novartis  400   150 CHF    60000 CHF
                            (환율 적용 후)
                            125000 USD
```

문제는 크고 복잡해 보인다. 그러나 TDD는 큰 문제 전체를 한 번에 풀려고 들지 않는다. 가장 작은 한 조각부터 시작한다.

## 첫 테스트 — `$5 × 2 = $10`

전체 시스템 가운데 가장 단순한 조각을 고른다. 한 통화의 곱셈이다.

```java
public void testMultiplication() {
    Dollar five = new Dollar(5);
    five.times(2);
    assertEquals(10, five.amount);
}
```

`Dollar` 클래스조차 아직 없으므로 이 테스트는 컴파일에서부터 실패한다. 이것이 의도된 첫 빨간 막대다. 무엇을 만들지 테스트가 먼저 결정한다.

## TDD의 리듬 — Red, Green, Refactor

이 책의 모든 장이 같은 사이클을 반복한다.

**1. RED      — 실패하는 테스트를 짠다**


**2. GREEN    — 어떤 방법으로든 가장 빨리 통과시킨다**


**3. REFACTOR — 중복을 제거하고 정직한 코드로 만든다**

한 사이클이 보통 몇 분 이내에 끝난다. 매 단계마다 테스트가 통과하는지를 눈으로 확인하면서 진행한다.

## 가장 단순한 Green

테스트를 통과시키는 가장 단순한 방법은 답을 하드코딩하는 것이다.

```java
class Dollar {
    int amount = 10;  // 하드코딩
    public Dollar(int amount) {}
    public void times(int multiplier) {}
}
```

옳은 구현은 아니다. 그러나 빨간 막대는 사라지고 초록이 됐다. Beck은 이 단계를 **Fake It (Till You Make It)** 패턴이라 부른다. 일단 통과시키고 다음 단계에서 진짜를 만든다.

## Refactor

이제 정직한 구현으로 옮긴다.

```java
class Dollar {
    int amount;
    public Dollar(int amount) {
        this.amount = amount;
    }
    public void times(int multiplier) {
        amount *= multiplier;
    }
}
```

테스트는 여전히 통과한다. 하드코딩이 사라지면서 일반화가 완료됐다. 한 사이클이 끝났다.

## 작은 단계의 가치

이 한 사이클을 정리하면 다음과 같은 흐름이다.

```text
단계 1: 빨간 막대 (Dollar 없음)
단계 2: 가짜 구현으로 초록 (amount=10 하드코딩)
단계 3: 일반화 (amount *= multiplier)
```

`$5 × 2`라는 사소한 한 줄을 위해 세 단계를 거친다. 비효율로 보이지만 이 작은 단위가 TDD의 핵심이다. 매 단계마다 테스트가 통과한다는 확신이 쌓이기 때문에, 다음 단계로 넘어가는 결정이 두렵지 않다.

## 왜 이렇게 작게 가는가

Beck의 통찰은 다음 한 줄로 요약된다.

> 작은 단계는 학습용 과장이 아니다. 어려운 부분에서는 진짜로 작게 가야 한다.

쉬운 영역에서는 큰 보폭으로 가도 된다. 그러나 도메인이 낯설거나, 의존성이 복잡하거나, 디자인이 불확실한 영역에서는 한 사이클을 정말로 작게 잡아야 한다. 이 사실은 31장의 *Mastering TDD*에서 다시 강조된다.

## 사이드 이펙트라는 새 문제

방금 만든 `Dollar.times`는 자기 자신을 변경한다. 같은 객체로 두 번 호출하면 결과가 누적된다.

```java
Dollar five = new Dollar(5);
five.times(2);   // five.amount = 10
five.times(3);   // five.amount = 30 (의도와 다름)
```

값 객체(value object)의 자연스러운 의미는 불변이다. 이 문제는 다음 장에서 해결한다. 지금 당장 손대지 않는 이유는 TDD의 또 다른 원칙 때문이다. 한 번에 한 가지 문제만 본다.

## To-Do List라는 도구

새로 떠오른 문제는 코드에서 손대지 않는 대신 To-Do List에 적는다. Beck이 책 전체에서 가장 자주 사용하는 도구다.

**TODO:**

- [ ] $5 + 10 CHF = $10 (환율 2:1)
- [x] $5 × 2 = $10
- [ ] amount를 private으로
- [ ] Dollar의 사이드 이펙트 제거
- [ ] Money rounding

생각이 떠오를 때마다 리스트에 적되, 지금 작업에만 집중한다. 다음 장은 이 리스트의 위에서 두 번째 항목을 가져온다.

## 이 한 사이클이 책 전체

이 작은 예제가 보여 준 흐름은 책 전체에 그대로 반복된다.

**1. 가장 작은 한 테스트**


**2. 가장 단순한 통과**


**3. 리팩터링으로 정리**


**4. To-Do List에서 다음 항목**

같은 패턴이 32장 동안 다른 옷을 입고 등장한다.

## 정리

- TDD는 **Red → Green → Refactor** 사이클의 반복이다
- 한 사이클은 분 단위로 짧게, 매 단계마다 테스트 통과를 확인한다
- 가장 빠른 초록 막대를 위해 하드코딩(**Fake It**)도 허용한다
- 정직한 코드는 그 다음 리팩터링에서 만든다
- 떠오르는 문제는 **To-Do List**에 적고 현재 작업에만 집중한다
- 작은 단계는 학습용 과장이 아니라 어려운 부분에서 진짜로 필요한 도구다
