> *Test-Driven Development by Example* Chapter 4 요약

## 이 챕터의 메시지

3장에서 `equals`가 생겼다. 이제 외부 코드가 `Dollar`의 내부 `amount`에 직접 접근할 필요가 사라졌다. 비교는 동작(`equals`)으로 한다. 이 장은 그 사실을 따라 `amount`를 `private`으로 옮기고, 테스트 또한 데이터가 아닌 동작에 의존하도록 다시 쓴다.

캡슐화는 객체 지향의 기본 원칙이지만, TDD에서는 더 실용적인 의미가 있다. 테스트가 내부 데이터에 결합하면 리팩터링이 어려워진다. 테스트가 동작에만 의존하면 내부 구조를 자유롭게 바꿀 수 있다.

## 테스트를 동작 기반으로 바꾼다

이전 곱셈 테스트는 `product.amount`를 직접 읽었다.

```java
// 이전 버전
public void testMultiplication() {
    Dollar product = new Dollar(5).times(2);
    assertEquals(10, product.amount);   // amount 직접 접근
}
```

이제 `equals`가 있으므로 비교를 그리로 옮긴다.

```java
// 동작 기반 버전
public void testMultiplication() {
    Dollar product = new Dollar(5).times(2);
    assertTrue(new Dollar(10).equals(product));
}
```

두 코드가 의미상 같다고 생각될 수 있지만 차이는 크다. 새 버전은 `Dollar`의 내부 표현이 어떻든 신경 쓰지 않는다. 동등성이라는 동작만 있으면 통과한다.

## amount를 private으로

테스트가 `amount`를 더는 보지 않으므로 안전하게 캡슐화할 수 있다.

```java
class Dollar {
    private int amount;

    Dollar(int amount) {
        this.amount = amount;
    }

    Dollar times(int multiplier) {
        return new Dollar(amount * multiplier);
    }

    public boolean equals(Object obj) {
        Dollar dollar = (Dollar) obj;
        return amount == dollar.amount;   // 같은 클래스이므로 OK
    }
}
```

같은 클래스 내부에서는 `private` 필드에 접근할 수 있으므로 `equals`는 그대로 동작한다.

## 데이터 대신 객체의 의미를 다룬다

작은 변경이지만 사고의 전환이 들어 있다.

```text
이전: 외부가 데이터(amount)에 직접 접근
이후: 외부가 동작(equals)으로 객체를 비교
```

테스트가 객체의 *의미*를 다루기 시작한다. 같은 5달러인가? 같은 통화인가? 객체가 자신을 설명하는 메서드를 통해 답한다. 내부 표현은 객체의 사적인 영역이다.

GOOS(Freeman·Pryce)가 *"Mock 객체 테스트는 메시지가 흐르는 모양을 본다"*라고 말하는 것과 같은 정신이다. 데이터가 아니라 행동을 본다.

## Privacy 패턴의 일반 원칙

Beck은 26장에서 이 사고를 한 패턴으로 정리한다. 핵심은 **가능한 한 좁은 가시성**이다.

| 가시성 | 의미 | 사용 시점 |
|--------|------|----------|
| `private` | 같은 클래스만 | 기본값 |
| 패키지 (default) | 같은 패키지만 | 같은 모듈의 협력 클래스 |
| `protected` | 자식 클래스까지 | 상속 계층의 확장점 |
| `public` | 모두 | 외부 API |

기본은 `private`이고, 더 넓혀야 할 분명한 이유가 있을 때만 한 단계 올린다. Bertrand Meyer의 *"클래스의 모든 필드는 비밀이어야 한다"*는 *Object-Oriented Software Construction*의 원칙도 같은 결론이다.

## 작은 리팩터링의 안전망

이 변경은 단 두 줄이지만 만약 테스트가 없었다면 두려운 일이다.

```text
1. 테스트에서 amount 직접 접근을 제거
2. amount를 private으로
```

각 단계마다 모든 테스트가 통과해야 다음 단계로 간다. 한 번에 두 일을 하지 않는다. 이게 Beck이 32장에서 *"매번 작은 리팩터링이 큰 리팩터링의 필요를 없앤다"*고 말하는 이유다.

## 정리

- `equals`가 생기면 테스트는 내부 데이터 대신 동작으로 비교한다
- 그 결과 `amount`를 안전하게 `private`으로 옮길 수 있다
- 캡슐화는 객체 지향 원칙이자 리팩터링을 가능하게 하는 **TDD의 도구**다
- 기본 가시성은 `private`, 넓히는 데는 분명한 이유가 필요하다
- 작은 변경 한 번에 한 가지씩, 매 단계 테스트를 통과시킨다
