> **초안** — 정리 진행 중

## 개요

`inline` 키워드는 컴파일러에게 **요청**일 뿐 보장이 아닙니다. 인라이닝의 장점(호출 비용 제거, 최적화 기회)과 단점(코드 부피, 디버깅, 코드 변경 시 재컴파일)을 이해해야.

## inline의 두 형태

```cpp
// 명시적 inline
inline void f() { /* ... */ }

// 암묵적 inline — 클래스 정의 안의 함수
class Widget {
public:
    void f() { /* ... */ }    // 암묵적 inline
};
```

## inline이 거부되는 경우

- **함수가 너무 복잡** — 컴파일러가 비용 대비 손해라 판단
- **재귀 함수**
- **virtual 함수** — 동적 디스패치라 보통 인라인 불가
- **함수 포인터로 호출** — 인라인 적용 안 됨

## 단점 — 코드 부피

```cpp
inline void big() { /* 100줄 코드 */ }

// 이 함수가 100군데 호출되면 → 100 × 100줄 코드 부피
```

작은 함수만 인라인이 진짜 이득. 큰 함수는 코드 부피만 늘고 속도는 비슷.

## 디버깅과 라이브러리

- 인라인된 함수는 **디버거에서 break 못 검** (호출 자체가 사라짐)
- 라이브러리 헤더의 inline 함수를 변경하면 **사용자가 모두 재컴파일** 필요
- 일반 함수 변경은 **리링크만**으로 충분

## 권장 전략

- **80/20 원칙**: 대부분 함수는 inline 안 해도 됨. 진짜 hot path만
- **간단한 getter/setter**는 자연스럽게 인라인 (한 줄짜리)
- **inline 키워드 남발 금지** — 컴파일러가 대부분 알아서 결정 (특히 LTO와 함께)

## C++17 inline variable

C++17부터 `inline`은 변수에도 적용 가능 — 헤더에 정의된 변수의 ODR 문제 해결.

```cpp
// header.h
inline int globalCounter = 0;   // 여러 TU에 포함되어도 한 정의
```

## 핵심 정리

1. `inline`은 요청, 보장 X
2. 작은 함수만 인라인 — 큰 함수는 코드 부피 손해
3. 라이브러리 인터페이스에선 신중히 (재컴파일 영향)
4. 컴파일러/LTO가 대부분 잘 결정 — 무지성 inline 키워드 피하기
