## 한 줄 요약

> **"구조 7개 + 행위 7개"** — UML 2.5.1의 14가지 다이어그램은 같은 시스템을 다른 카메라 앵글로 보는 것이다.

## 어떤 문제를 푸는가

UML 다이어그램이 너무 많다고 느낀다면, **두 그룹**으로 나누면 단숨에 정리됩니다.

- **구조** — 시간에 따라 안 변하는 것 (클래스, 컴포넌트, 노드, ...)
- **행위** — 시간에 따라 변하는 것 (상호작용, 상태, 활동)

## 구조 다이어그램 (7종)

| 다이어그램 | 무엇을 보여주나 |
| --- | --- |
| **클래스** | 클래스·인터페이스·관계 |
| **객체** | 특정 시점의 인스턴스 스냅샷 |
| **패키지** | 패키지 간 의존 |
| **컴포넌트** | 배치 가능한 단위와 그 인터페이스 |
| **컴포지트 구조** | 클래스 내부의 부품·연결 (UML 2.0 신규) |
| **배포** | 노드(하드웨어)와 그 위의 아티팩트 |
| **프로파일** | 도메인 확장(스테레오타입·태그값·제약) (UML 2.5 first-class) |

> 💡 UML 2.0(2005)이 13종이었다가, **UML 2.5**(2015)에서 **프로파일 다이어그램**이 first-class로 승격되며 14종이 되었습니다. "아티팩트 다이어그램"은 별도 종류가 *아닙니다* — 아티팩트는 배포 다이어그램에 그려집니다.

## 행위 다이어그램 (7종)

행위 다이어그램은 다시 **"흐름·상태·시나리오" 3종**과 **상호작용 4종**으로 갈립니다.

| 다이어그램 | 무엇을 보여주나 |
| --- | --- |
| **유스케이스** | 시스템과 외부 행위자의 상호작용 |
| **활동** | 워크플로/알고리즘 흐름 |
| **상태 머신** | 객체의 라이프사이클 |
| **시퀀스** *(상호작용)* | 시간 순으로 본 객체 간 메시지 |
| **커뮤니케이션** *(상호작용)* | 메시지 + 객체 관계 (UML 1.x의 "협력 다이어그램" 개명) |
| **상호작용 개요** *(상호작용)* | 활동 + 시퀀스 결합 |
| **타이밍** *(상호작용)* | 시간 축 기준 상태 변화 |

> 💡 시퀀스·커뮤니케이션·상호작용 개요·타이밍 4종을 합쳐 **상호작용 다이어그램(Interaction Diagrams)** 으로 부릅니다. UML 2.5.1 메타모델에서 모두 `Interaction`의 시각적 변형.

## 4+1 뷰에 매핑

Kruchten의 4+1 뷰에 각 다이어그램을 매핑하면 어떤 시각에 쓸지가 분명해집니다.

| 뷰 | 주요 다이어그램 |
| --- | --- |
| 논리 (Logical) | 클래스, 객체, 상태 머신 |
| 프로세스 (Process) | 활동, 시퀀스, 커뮤니케이션, 타이밍 |
| 구현 (Implementation) | 컴포넌트, 패키지, 컴포지트 구조 |
| 배포 (Deployment) | 배포 (아티팩트는 그 위에) |
| 유스케이스 (+1) | 유스케이스 |
| 전 영역 확장 | 프로파일 |

## 어디서부터 시작해야 하나

새 시스템을 모델링한다면 다음 순서를 권장합니다.

1. **유스케이스** — 무엇을 할 시스템인가?
2. **활동** — 각 유스케이스의 큰 흐름은?
3. **클래스** — 어떤 도메인 객체가 필요한가?
4. **시퀀스** — 주요 시나리오에서 객체들이 어떻게 협력하나?
5. **상태 머신** — 라이프사이클이 복잡한 객체는?
6. **컴포넌트·배포** — 어떻게 패키징·배치하나?

> 💡 모든 다이어그램을 다 그릴 필요는 없습니다. 보통 5종(클래스·시퀀스·유스케이스·상태·활동) 정도면 99% 프로젝트에 충분.

## 자주 하는 실수

> ⚠️ 다이어그램 종류부터 고르고 그리기

먼저 **무엇을 보여주고 싶은지** 정한 다음 다이어그램을 고릅니다. 다이어그램이 목적이 되면 안 됩니다.

> ⚠️ 한 다이어그램에 모든 걸

다이어그램 하나에 너무 많은 요소가 들어가면 **읽히지 않습니다**. 7±2 원칙 — 한 다이어그램에 7개 안팎의 핵심 요소.

> ⚠️ 구조와 행위를 한 다이어그램에 섞기

같은 다이어그램 안에서 클래스 정의와 메시지 흐름을 섞으면 의미가 흐려집니다. 다이어그램 종류는 한 가지로.

## 정리

- UML 2.5.1의 14종 다이어그램은 **구조 7 + 행위 7**로 갈린다.
- 행위 7 중 **상호작용 4종**(시퀀스 · 커뮤니케이션 · 상호작용 개요 · 타이밍)이 한 묶음.
- 2.5에서 **프로파일 다이어그램**이 first-class로 승격, "협력" → "**커뮤니케이션**" 개명.
- **시스템을 보는 다른 시각** 이상도 이하도 아니다 — 다 그릴 필요 없다.
- 4+1 뷰에 매핑하면 **어떤 다이어그램을 언제 그릴지** 명확해진다.

다음 편부터(item08~) **개별 다이어그램**을 하나씩 깊이 다룹니다. 클래스 다이어그램이 시작.
