## 한 줄 요약

> **"어휘 + 규칙 + 메커니즘"** — UML은 단어를 모은 게 아니라 단어·문법·관용구가 다 있는 그림 언어다.

## 어떤 문제를 푸는가

처음 UML을 보면 다이어그램 종류만 14개라 길을 잃기 쉽습니다. "각각 다 따로 외워야 하나?" 그렇지 않습니다.

UML은 사실 **세 덩어리**로 구성됩니다. 이 셋의 조합으로 14개 다이어그램이 모두 만들어집니다.

1. **Building Blocks** — 어휘 (사물·관계·다이어그램)
2. **Rules** — 문법 (이름·범위·가시성·무결성·실행)
3. **Common Mechanisms** — 관용구 (명세·꾸밈·공통 분류·확장)

## 한눈에 보는 구조

![UML 구조](/images/blog/uml/diagrams/item02-uml-overview.svg)

세 덩어리(Building Blocks, Rules, Common Mechanisms)가 UML의 뿌리입니다. 14개 다이어그램은 모두 이 셋의 조합으로 만들어집니다.

## Building Blocks — 어휘

### 사물 (Things)

UML이 다루는 대상은 네 부류입니다.

| 분류 | 예시 |
| --- | --- |
| 구조적 (Structural) | 클래스, 인터페이스, 컴포넌트, 노드, 협력 |
| 행위적 (Behavioral) | 상호작용, 상태 머신, 활동 |
| 그룹핑 (Grouping) | 패키지 |
| 주석 (Annotational) | 노트 |

"명사 = 구조적", "동사 = 행위적"이라고 외워두면 거의 맞습니다.

### 관계 (Relationships)

사물 사이의 연결은 네 가지로 충분합니다.

- **Dependency** (의존) — 점선 화살표. "A가 변하면 B도 영향받는다."
- **Association** (연관) — 실선. "A는 B를 안다."
- **Generalization** (일반화) — 빈 삼각형. "A는 B의 일종이다."
- **Realization** (실체화) — 점선 + 빈 삼각형. "A는 B를 구현한다."

### 다이어그램 (Diagrams)

**UML 2.5.1**(OMG, 2017)은 14개 다이어그램을 정의합니다.

**구조 다이어그램 (7개)**
- 클래스, 객체, 패키지, 컴포넌트, 컴포지트 구조, 배포, **프로파일**

**행위 다이어그램 (7개)**
- 유스케이스, 활동, 상태 머신, **상호작용 4종**(시퀀스 · 통신 · 상호작용 개요 · 타이밍)

> 💡 UML 2.0(2005)이 13개였고, 2.5(2015)에서 **프로파일 다이어그램**이 first-class로 승격되어 14개가 되었습니다. 또 "**컬래보레이션 다이어그램**"은 UML 2.0에서 "**커뮤니케이션 다이어그램**"으로 개명되었으니 옛 문헌과 용어 혼동에 주의.

## Rules — 문법

UML 모델은 다음에 대해 규칙을 가집니다.

| 규칙 | 의미 |
| --- | --- |
| 이름 (Names) | 사물·관계·다이어그램은 이름을 가진다 |
| 범위 (Scope) | 이름이 어디서 유효한지 |
| 가시성 (Visibility) | 누가 그 이름을 볼 수 있는지 (+, -, #, ~) |
| 무결성 (Integrity) | 사물끼리 일관되게 연결되어야 한다 |
| 실행 (Execution) | 동적 모델이 어떻게 진행되는지 |

이 규칙은 모델이 "**그림이 아닌 형식 언어**"가 되게 합니다 — 도구가 검증할 수 있게.

## Common Mechanisms — 관용구

UML이 30년 가까이 쓰일 수 있는 이유는 다음 네 가지 메커니즘 덕분입니다.

### 1. 명세 (Specifications)

모든 그림 요소 뒤에는 **텍스트 명세**가 있습니다. 다이어그램은 명세의 시각적 투영(projection)일 뿐. 같은 명세에서 여러 다이어그램을 그릴 수 있고, 다이어그램만으로는 채워지지 않는 디테일이 명세에 들어있습니다.

### 2. 꾸밈 (Adornments)

직사각형으로 그린 클래스 옆에 `+name`, `-age` 같은 가시성 표시, 다중도(`1..*`), 역할 이름 등을 붙여 정보를 더합니다.

### 3. 공통 분류 (Common Divisions)

- **클래스 vs 객체** — 추상 vs 인스턴스
- **인터페이스 vs 구현** — 약속 vs 실현

UML은 모든 그림 요소에 이 두 분류를 일관되게 적용합니다.

### 4. 확장 메커니즘 (Extensibility)

UML은 자체로 충분하지 않은 도메인을 위해 세 가지 확장을 제공합니다.

- **스테레오타입** (`<<entity>>`, `<<controller>>`) — 새로운 어휘
- **태그값** (`{author="hawk"}`) — 새로운 속성
- **제약** (`{ordered}`, `{xor}`) — 새로운 의미 규칙

이 셋으로 UML을 도메인 전용 언어로 확장할 수 있습니다 — **UML Profile**.

## 4+1 뷰 — 보는 면

Kruchten의 4+1 뷰는 시스템을 다섯 시각으로 봅니다.

| 뷰 | 무엇을 보는가 | 주요 다이어그램 |
| --- | --- | --- |
| 논리 | 기능 구조 | 클래스, 객체 |
| 프로세스 | 동시성·성능 | 활동, 시퀀스, 통신 |
| 구현 | 코드 모듈 | 컴포넌트 |
| 배포 | 물리 토폴로지 | 배포 |
| 유스케이스 (+1) | 사용자 시나리오 | 유스케이스 |

UML 다이어그램 14종은 이 다섯 시각 어딘가에 속합니다. **어떤 뷰를 보려는지 먼저 정하고** 다이어그램을 고르세요.

## 자주 하는 실수

> ⚠️ "다이어그램 14개를 다 그려야 한다"

아닙니다. 보통 프로젝트는 클래스 + 시퀀스 + 유스케이스 3종이면 충분합니다. **필요한 시각만** 그리세요.

> ⚠️ "그림이 곧 모델이다"

그림은 모델의 **투영**입니다. 같은 모델에서 여러 그림이 나오고, 그림에 안 보이는 명세가 항상 더 있습니다.

> ⚠️ "스테레오타입을 남발해도 된다"

스테레오타입은 강력하지만 남발하면 표준 어휘를 잃고 팀 전용 방언이 됩니다.

## 정리

- UML은 **사물 + 관계 + 다이어그램**의 어휘를 가지고 있다.
- **이름·범위·가시성·무결성·실행** 다섯 가지 규칙으로 문법이 잡힌다.
- **명세·꾸밈·공통 분류·확장**의 네 메커니즘이 14개 다이어그램의 일관성을 만든다.
- 시스템은 **여러 뷰**에서 그려져야 한다 — 4+1이 대표적인 분류.

다음 편은 "Hello, World!" — 가장 단순한 UML 모델로 위 어휘를 실제로 써봅니다.
