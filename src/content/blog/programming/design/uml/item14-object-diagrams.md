## 한 줄 요약

> **"특정 시점의 객체 그래프"** — 클래스 다이어그램의 인스턴스 버전. 한 순간의 메모리 상태를 그린다.

## 어떤 문제를 푸는가

클래스 다이어그램으로 "Library는 Book을 1..* 가진다"라고 그렸습니다. 그런데 **실행 시점**엔 Library 인스턴스 하나가 Book 인스턴스 몇 개를 가지는지, Member가 어느 Book을 빌렸는지가 더 궁금합니다.

객체 다이어그램은 이 **순간의 사진**을 찍습니다.

## 언제 그리나

### 1. 알고리즘 시연

자료구조 변화를 단계별로 보여줄 때 — 6편 LinkedList 삭제, 트리 회전 등.

### 2. 시나리오 설명

"주문이 결제 완료 상태일 때 시스템엔 이런 객체들이 이렇게 연결돼 있다."

### 3. 테스트 케이스 시각화

"이 테스트의 fixture 객체 그래프는 다음과 같다."

### 4. 버그 재현

"이 시점에 객체 A와 B가 양쪽 다 객체 C를 가리켜 무한 루프가 발생했다."

## 객체 다이어그램 vs 클래스 다이어그램

| 항목 | 클래스 다이어그램 | 객체 다이어그램 |
| --- | --- | --- |
| 표기 | 굵은 글씨, 다중도 | 밑줄, 다중도 없음 |
| 시간 | 무시간 | 특정 시점 |
| 정보 | 정의 (속성·타입) | 값 (속성 값) |
| 변화 | 시스템 전 생애 | 한 순간 |

## 좋은 객체 다이어그램의 조건

### 시점을 명시

예 — *"3편 주문 완료 직후"*.

객체 다이어그램은 **순간**을 그립니다. 어느 순간인지 제목에 적으세요.

### 관련 객체만

시스템 전체 객체를 다 그리면 의미가 없습니다. **이 시나리오에 관계된 객체**만.

### 클래스 다이어그램과 일관성

객체 다이어그램의 모든 link는 클래스 다이어그램의 association을 만족해야 합니다. 만약 만족하지 않으면 — 클래스 다이어그램 또는 객체 다이어그램 둘 중 하나가 틀린 것.

## 활용 — 시퀀스와의 차이

객체 다이어그램과 시퀀스 다이어그램은 자주 헷갈립니다.

| | 객체 | 시퀀스 |
| --- | --- | --- |
| 보여주는 것 | 그 순간의 **상태** | 시간에 따른 **메시지** |
| 시간축 | 없음 | 있음 (위→아래) |
| 사용 시점 | 한 순간 | 흐름 전체 |

객체 다이어그램이 **사진**이라면, 시퀀스 다이어그램은 **동영상**입니다.

## 자주 하는 실수

> ⚠️ 시간 축 추가

객체 다이어그램에 화살표를 그리며 "이 다음에 이게 호출됨"이라고 적는 건 시퀀스 다이어그램의 일. 객체 다이어그램은 **한 순간**.

> ⚠️ 다중도 표시

클래스 다이어그램과 헷갈려 `1..*` 같은 다중도를 적으면 안 됩니다 — 객체는 이미 셀 수 있는 인스턴스니까.

> ⚠️ 너무 많은 속성

가독성을 위해 **핵심 속성만**. 모든 필드를 다 적을 필요 없음.

## 패턴 — Step-by-step 시연

알고리즘 시각화에 가장 효과적인 패턴:

```
Step 1:  head → [3] → [1] → [4] → null
Step 2:  head → [1] → [3] → [4] → null   (swap 3 and 1)
Step 3:  head → [1] → [3] → [4] → null   (sorted)
```

각 step마다 객체 다이어그램 한 장. 텍스트 설명은 짧게.

## 정리

- 객체 다이어그램은 **특정 시점의 객체 그래프**.
- 알고리즘·시나리오·테스트·버그 재현에 강력.
- 시점·관련 객체에만 집중, 클래스 다이어그램과 일관성 유지.
- 시퀀스 다이어그램(동영상)과 달리 **한 순간(사진)**.
