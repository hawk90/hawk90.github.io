## 한 줄 요약

> **"도메인 명사들의 지도"** — 가장 흔하고 가장 오래 가는 UML 다이어그램. 시스템의 정적 골격을 한 장에 본다.

## 어떤 문제를 푸는가

코드만 봐서는 도메인의 큰 그림이 안 보입니다. 30개 클래스가 어떻게 얽혀 있는지, 어디가 핵심이고 어디가 부속인지 — 코드는 한 줄씩 펼쳐줄 뿐 **지도**가 아닙니다.

클래스 다이어그램이 그 지도입니다.

## 한눈에 보는 예시 — 도서관

![Library class diagram](/images/blog/uml/diagrams/item08-class-diagram-example.svg)

세 클래스, 한 추상 클래스, 세 관계.

- `Library` ◆ `Book` (composition, 1..*) — 도서관이 책을 소유
- `Member` ── `Book` (association, 0..3) — 회원이 책을 빌림
- `Book` ─▷ `LibraryItem` (generalization) — 책은 도서관 자료의 일종

이 한 장에서 도서관 시스템의 **핵심 도메인 모델**이 보입니다.

## 클래스 다이어그램 4단계

### 1단계: 어휘 (Vocabulary)

핵심 도메인 명사를 골라 클래스로 만듭니다. 책임만 적고 속성·연산은 비웁니다.

```
[Library]  [Book]  [Member]  [Loan]
```

도메인 전문가와 합의하는 단계.

### 2단계: 단순 협력 (Simple Collaborations)

관계를 그리기 시작합니다. 다중도·역할 이름도 붙입니다.

```
Library  ◆ — 1..*  Book
Member   ─ borrows 0..3 ─ Book
```

### 3단계: 논리 스키마 (Logical Schema)

DB 테이블이나 데이터 구조에 매핑할 수 있는 수준으로 채웁니다. 속성·타입·키.

```
Book
  + isbn : String {unique}
  + title : String
  + year : int
```

### 4단계: 구현 골격 (Implementation Skeleton)

코드와 1:1 매핑되는 수준. 가시성·연산 시그너처·예외까지.

```
+ borrow(b : Book) : Loan throws AlreadyBorrowed
```

같은 시스템도 이 네 단계 중 **어디서 멈출지** 정해 그립니다.

## 클래스 다이어그램의 좋은 습관

### 핵심 클래스를 가운데에

다이어그램 가운데에 **가장 중요한 클래스**를 두고 주변에 협력자를 배치합니다. 시선이 자연스럽게 흐릅니다.

### 비슷한 추상화 수준

한 다이어그램 안의 클래스들은 **추상화 수준**이 비슷해야 합니다. `Customer`와 `StringUtils`가 같이 있으면 부자연스럽습니다.

### 패키지로 분리

클래스가 10개를 넘으면 의미 그룹으로 나눠 **패키지 박스**로 묶거나, 다이어그램을 쪼개세요.

### 다이어그램 이름

다이어그램에 이름을 붙입니다 — `"주문 도메인 모델 v2"`, `"권한 시스템 클래스 다이어그램"`. 6개월 뒤의 자신에게 보내는 메시지.

## 자주 하는 실수

> ⚠️ 모든 클래스를 한 다이어그램에

대규모 시스템에선 클래스가 수백 개. 한 다이어그램에 다 그리면 **읽히지 않는 벽지**가 됩니다. **시각마다 다이어그램을 쪼개세요**.

> ⚠️ getter/setter 다 적기

`getId()`, `setName()` 같은 자명한 메서드는 빼고 **도메인 연산**만 적습니다.

> ⚠️ 화살표를 너무 많이

연관·의존·일반화·실체화를 다 그리면 화살표가 거미줄이 됩니다. **이 다이어그램의 메시지**가 뭔지 정하고 메시지에 기여하는 관계만 남기세요.

## 정리

- 클래스 다이어그램은 **시스템 도메인의 지도** — 가장 흔하게 그려진다.
- 어휘 → 협력 → 스키마 → 구현 4단계로 **추상화 수준을 골라** 그린다.
- 핵심 클래스를 가운데에, 비슷한 추상화 수준끼리, 너무 많으면 패키지로 분리.
- 모든 클래스가 아니라 **이 다이어그램의 메시지**에 기여하는 것만 그린다.
