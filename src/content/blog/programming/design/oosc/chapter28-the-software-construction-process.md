## 한 줄 요약

> 소프트웨어 개발은 **클러스터 단위**로 진행하고, **반복적으로 정제**한다. 폭포수의 경직성과 무계획의 혼란 사이에서 균형을 찾는다.

## 프로세스의 필요성

### 왜 프로세스가 필요한가

| 프로세스 없이 | 과도한 프로세스 | 적절한 프로세스 |
|-------------|---------------|---------------|
| 언제 끝나는지 모름 | 문서만 양산 | 가이드라인 제공 |
| 누가 무엇을 하는지 불명확 | 개발 속도 저하 | 위험 조기 발견 |
| 품질 일관성 없음 | 창의성 억압 | 진행 상황 가시화 |
| 위험 관리 불가 | 형식에 매몰 | 팀 협업 촉진 |

### 전통적 모델의 문제

**폭포수 (Waterfall)**: 요구사항 → 분석 → 설계 → 구현 → 테스트

| 문제점 | 설명 |
|--------|------|
| 요구사항 변경 대응 불가 | "요구사항은 항상 변한다" |
| 후반 통합의 위험 | "모든 것이 한꺼번에 합쳐질 때 폭발" |
| 고객 피드백 지연 | "보여줄 게 있을 때는 이미 늦음" |
| 추정의 불확실성 | "처음에 전체를 알 수 없다" |

## 클러스터 모델

### 클러스터란

| 클러스터 (Cluster) 특징 |
|----------------------|
| 관련된 클래스들의 그룹 |
| 응집력 있는 기능 단위 |
| 독립적으로 개발 가능 |
| 명확한 인터페이스 |

| 클러스터 예 | 포함 클래스 |
|-----------|-----------|
| BANKING_CORE | ACCOUNT, CUSTOMER, TRANSACTION, MONEY |
| REPORTING | REPORT, STATEMENT, FORMATTER |

### 클러스터 기반 개발

| 클러스터 기반 개발 원칙 |
|---------------------|
| 시스템을 클러스터로 분해 |
| 각 클러스터를 독립적으로 개발 |
| 클러스터 내에서 반복적 정제 |
| 클러스터 간 점진적 통합 |

| 장점 |
|------|
| 병렬 개발 가능 |
| 위험 분산 |
| 조기 피드백 |
| 관리 가능한 단위 |

### 클러스터 의존성

**의존성 관리**: BANKING_CORE ← REPORTING, UI, PERSISTENCE

| 규칙 |
|------|
| 순환 의존성 피함 |
| 안정된 클러스터에 의존 |
| 인터페이스 먼저 정의 |
| 모의 객체로 독립 개발 |

## 반복적 개발

### 반복 (Iteration)

| 한 반복 주기 | 활동 |
|------------|------|
| 계획 | 이번 반복의 범위 결정 |
| 개발 | 분석 → 설계 → 구현 |
| 테스트 | 단위 + 통합 테스트 |
| 검토 | 결과 평가, 다음 계획 |

| 특징 |
|------|
| 짧은 주기 (2-4주) |
| 실행 가능한 결과물 |
| 고객 피드백 수집 |
| 계획 조정 |

### 점진적 정제

| 초기 반복 | 이후 반복 |
|---------|---------|
| 핵심 클래스 스켈레톤 | deferred를 effected로 |
| 주요 피처만 구현 | 예외 처리 추가 |
| 많은 것이 deferred | 성능 최적화 |
| — | 에지 케이스 처리 |

**"먼저 동작하게, 그 다음 올바르게, 그 다음 빠르게"**

### 각 반복의 산출물

```eiffel
-- 반복 1: 핵심 구조
deferred class ACCOUNT
feature
    withdraw (amount: MONEY)
        deferred
        end
end

-- 반복 2: 기본 구현
class BASIC_ACCOUNT
inherit
    ACCOUNT

feature
    withdraw (amount: MONEY)
        do
            balance := balance - amount
        end
end

-- 반복 3: 계약 추가
class BASIC_ACCOUNT
inherit
    ACCOUNT

feature
    withdraw (amount: MONEY)
        require
            sufficient: balance >= amount
        do
            balance := balance - amount
        ensure
            decreased: balance = old balance - amount
        end
end

-- 반복 4: 예외 처리, 로깅 등
```

## 개발 단계

### 일반화 (Generalization)

| 일반화 단계 |
|-----------|
| 구체적 문제 해결 |
| 패턴 발견 |
| 일반화된 솔루션 추출 |

| 예 | 특수 → 일반 |
|----|----------|

| 클러스터 통합 단계 |
|-----------------|
| 인터페이스 확인 |
| 통합 테스트 작성 |
| 점진적 통합 |
| 문제 발견 시 되돌림 |

| 규칙 |
|------|
| 작은 단위로 자주 통합 |

### 검증 (Verification)

| 검증 활동 | 내용 |
|---------|------|

| 타입 검증 | 컴파일 타임 타입 체크, 제네릭 타입 일관성 |

| 성능 검증 | 벤치마크, 프로파일링 |

## 문서화

### 코드 중심 문서화

**Meyer의 관점**: "코드가 최고의 문서"

| 이유 |
|------|
| 코드는 항상 최신 |
| 계약이 명세 역할 |
| Short Form이 문서 역할 |

| 보충 문서 | 역할 |
|---------|------|
| 아키텍처 개요 | 큰 그림 |
| 설계 결정 근거 | 왜? |
| 사용 예시 | 어떻게? |

### Short Form 활용

```eiffel
-- 자동 생성되는 문서
class interface ACCOUNT
    -- 계좌 추상화

feature -- 접근
    holder: CUSTOMER
        -- 계좌 소유자

    balance: MONEY
        -- 현재 잔고

feature -- 연산
    withdraw (amount: MONEY)
        -- `amount`만큼 인출
        require
            sufficient: balance >= amount
        ensure
            withdrawn: balance = old balance - amount

invariant
    positive_or_credit: balance >= -credit_limit
end
```

### 설계 문서

| 클러스터 문서 항목 | 내용 |
|-----------------|------|
| 목적 | 이 클러스터가 해결하는 문제 |
| 책임 | 주요 클래스와 역할 |
| 의존성 | 다른 클러스터와의 관계 |
| 인터페이스 | 공개 클래스와 피처 |
| 제약 | 성능, 보안 요구사항 |

## 자주 하는 실수

소프트웨어 구축 프로세스에서 흔히 빠지는 함정이다.

| 실수 | 증상 | 해결 |
|------|------|------|
| **프로세스 없이 개발** | 언제 끝나는지 모름, 품질 일관성 없음 → 위험 관리 불가 | 클러스터 모델 도입. 반복 주기 설정, 진행 가시화 |
| **과도한 프로세스** | 문서만 양산, 개발 속도 저하 → 창의성 억압, 형식에 매몰 | 프로세스 경량화. 코드 중심 문서화, Short Form 활용 |
| **클러스터 경계 불명확** | 순환 의존성, 통합 시 폭발 → 병렬 개발 불가 | 클러스터 인터페이스 먼저 정의. 의존성 방향 명확히 |
| **빅뱅 통합** | 모든 것이 한꺼번에 합쳐질 때 문제 폭발 → 디버깅 악몽 | 점진적 통합. 작은 단위로 자주 통합, CI 활용 |
| **위험 관리 부재** | 후반에 기술/일정 위험 발현 → 프로젝트 실패 | 조기 프로토타입. 스파이크 솔루션, 핵심 경로 관리 |
| **추정 과신** | 초기 추정 그대로 진행 → 일정 지연, 범위 폭주 | 버퍼 포함 추정. 반복마다 재추정, 범위 조정 준비 |
| **문서와 코드 불일치** | 별도 문서가 코드와 동기화 안 됨 → 문서 신뢰도 하락 | 코드가 문서. 계약이 명세, Short Form 자동 생성 |

## 정리

- **클러스터 모델**: 관련 클래스를 그룹으로 묶어 개발
- **반복적 개발**: 짧은 주기로 점진적 정제
- **위험 관리**: 기술/요구사항/일정 위험 조기 대응
- **팀 조직**: 클러스터별 팀, 명확한 책임
- **문서화**: 코드 중심, Short Form 활용
- **품질**: 계약 기반, 다층 테스트
