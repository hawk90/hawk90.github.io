## 한 줄 요약

> 소프트웨어 품질은 외적 요소(사용자가 보는 것)와 내적 요소(개발자가 보는 것)로 나뉘며, 객체지향은 이 둘을 동시에 달성하는 방법론이다.

## 왜 품질인가

소프트웨어 공학의 목표는 **품질 높은 소프트웨어**를 만드는 것이다. 그러나 "품질"이란 무엇인가? 이 질문에 답하지 않고는 어떤 방법론이 좋은지 판단할 수 없다.

Meyer는 품질을 두 범주로 나눈다:

| 범주 | 누가 보는가 | 예 |
|------|------------|-----|
| **외적 요소** | 사용자, 구매자 | 정확성, 견고성, 사용 용이성 |
| **내적 요소** | 개발자, 유지보수자 | 모듈성, 가독성, 문서화 |

핵심 통찰: **내적 요소는 외적 요소를 달성하기 위한 수단**이다. 사용자는 코드가 모듈화되어 있는지 신경 쓰지 않는다. 그러나 모듈화 없이는 확장성과 재사용성을 달성할 수 없다.

## 외적 품질 요소

### 정확성 (Correctness)

> 소프트웨어가 명세대로 동작하는 능력

정확성은 **가장 중요한** 품질 요소다. 명세와 다르게 동작하는 소프트웨어는 그 외의 모든 요소가 완벽해도 쓸모없다.

```text
정확성 = 명세 충족 여부

명세: "두 정수의 합을 반환한다"
구현: add(2, 3) → 5  ✓ 정확
구현: add(2, 3) → 6  ✗ 부정확
```

정확성은 **조건부**다. 명세가 없으면 정확성도 정의할 수 없다. 따라서 정확성을 논하려면 먼저 명세가 있어야 한다 — 이것이 나중에 **Design by Contract**로 이어진다.

### 견고성 (Robustness)

> 비정상적 조건에서도 합리적으로 동작하는 능력

정확성이 "명세된 경우"를 다룬다면, 견고성은 "명세되지 않은 경우"를 다룬다.

```text
명세: "양의 정수 두 개의 합을 반환한다"

정확성: add(2, 3) → 5
견고성: add(-1, 3) → ?  (명세에 없는 입력)
```

견고한 시스템은 예상치 못한 입력에도 **crash하지 않고** 합리적인 방식으로 대응한다 — 에러 메시지, 기본값 반환, graceful degradation 등.

### 확장성 (Extendibility)

> 명세 변경에 적응하는 용이성

소프트웨어는 변한다. 요구사항이 바뀌고, 시장이 바뀌고, 사용자가 새로운 기능을 원한다. 확장성이 낮은 소프트웨어는 작은 변경에도 전체를 다시 작성해야 한다.

확장성을 달성하는 두 가지 원칙:

1. **설계 단순성** — 단순한 아키텍처일수록 변경하기 쉽다
2. **분산화** — 변경 영향이 시스템 전체가 아닌 일부에만 미친다

![확장성 비교](/images/blog/oosc/diagrams/ch01-extensibility.svg)

### 재사용성 (Reusability)

> 소프트웨어 요소가 여러 응용에서 사용될 수 있는 능력

재사용성은 소프트웨어 산업의 오랜 꿈이다. 하드웨어 산업은 표준화된 부품(저항, 커패시터, 칩)을 조합해 제품을 만든다. 소프트웨어도 그래야 하지 않겠는가?

그러나 현실은 다르다. 대부분의 소프트웨어는 처음부터 다시 작성된다. 왜?

- **일반화 비용**: 특정 문제 해결 코드를 일반화하려면 추가 노력 필요
- **발견 비용**: 재사용 가능한 코드가 있는지 찾기 어려움
- **이해 비용**: 남의 코드를 이해하는 비용
- **적응 비용**: 내 문제에 맞게 수정하는 비용

객체지향은 이 문제들을 해결하려 한다 — 클래스, 상속, 다형성을 통해.

### 호환성 (Compatibility)

> 다른 소프트웨어 요소와 쉽게 결합되는 능력

소프트웨어는 고립되어 존재하지 않는다. 파일 시스템, 네트워크, 데이터베이스, 다른 응용 프로그램과 상호작용해야 한다.

호환성의 핵심은 **표준 인터페이스**:

- 표준 파일 형식 (JSON, XML, CSV)
- 표준 프로토콜 (HTTP, TCP/IP)
- 표준 데이터 구조 (배열, 맵, 트리)

비호환 시스템은 독자 형식 때문에 서로 통신할 수 없다. 표준 형식(JSON, XML, CSV)을 쓰면 상호운용이 가능해진다.

### 효율성 (Efficiency)

> 하드웨어 자원(프로세서, 메모리, 네트워크)을 적게 요구하는 정도

효율성은 중요하지만 **과대평가되는 경향**이 있다. Meyer의 관찰:

1. 대부분의 비효율은 **알고리즘 선택**에서 온다 (O(n²) vs O(n log n))
2. 나머지는 **아키텍처 결정**에서 온다
3. 미시적 최적화(변수 하나 줄이기)는 거의 의미 없다

| 최적화 유형 | 효과 |
|------------|------|
| 알고리즘 변경 | 매우 큼 |
| 아키텍처 변경 | 상당함 |
| 미시적 최적화 | 미미함 |

따라서 "빠른 코드"보다 "좋은 설계"가 먼저다. 좋은 설계를 갖추면 나중에 최적화할 여지가 남는다.

### 이식성 (Portability)

> 다양한 하드웨어·소프트웨어 환경으로 옮기기 쉬운 정도

이식성은 **플랫폼 의존성을 최소화**할 때 높아진다.

![이식성 비교](/images/blog/oosc/diagrams/ch01-portability.svg)

### 사용 용이성 (Ease of Use)

> 다양한 숙련도의 사용자가 쉽게 배우고 사용할 수 있는 정도

좋은 UI/UX는 사용자 숙련도에 **적응**한다:

- 초보자: 간단한 기본 동작
- 중급자: 단축키, 고급 옵션
- 전문가: 스크립팅, 커스터마이징

### 기능성 (Functionality)

> 시스템이 제공하는 서비스의 범위

기능성은 양날의 검이다:

- **너무 적으면**: 사용자 요구를 충족하지 못함
- **너무 많으면**: 복잡성 증가, 다른 품질 요소 저하 (feature creep)

Meyer는 **기능성과 다른 요소 사이의 균형**을 강조한다. 모든 기능을 추가하는 것이 능사가 아니다.

### 적시성 (Timeliness)

> 사용자가 필요로 할 때 출시되는 정도

늦은 소프트웨어는 완벽해도 쓸모없을 수 있다. 시장 기회는 지나가고, 경쟁자가 먼저 출시하고, 사용자는 다른 해결책을 찾는다.

## 내적 품질 요소

외적 요소는 사용자가 직접 경험한다. 내적 요소는 개발자만 본다. 그러나 내적 요소 없이는 외적 요소를 달성할 수 없다.

| 내적 요소 | 기여하는 외적 요소 |
|----------|-------------------|
| 모듈성 | 확장성, 재사용성 |
| 가독성 | 확장성, 정확성 |
| 문서화 | 재사용성, 확장성 |
| 테스트 용이성 | 정확성, 견고성 |

## 품질 요소 간의 상충

모든 요소를 최대화할 수는 없다. 상충 관계가 존재한다:

| 상충 | 설명 |
|------|------|
| 효율성 vs 이식성 | 플랫폼 최적화는 이식성을 낮춤 |
| 효율성 vs 재사용성 | 일반적 코드는 특수 최적화보다 느림 |
| 기능성 vs 사용 용이성 | 기능이 많으면 배우기 어려움 |
| 적시성 vs 정확성 | 빨리 출시하면 버그 증가 |

좋은 소프트웨어 공학은 이 상충을 **인식하고 관리**하는 것이다.

## 객체지향과 품질

Meyer의 핵심 주장: **객체지향은 여러 품질 요소를 동시에 달성하는 방법론**이다.

| 품질 요소 | OO의 기여 |
|----------|----------|
| 정확성 | Design by Contract |
| 견고성 | 예외 처리, 타입 시스템 |
| 확장성 | 상속, 다형성 |
| 재사용성 | 클래스 라이브러리, 제네릭 |
| 호환성 | 표준 인터페이스, 타입 체계 |

이 책의 나머지 부분은 이 주장을 **구체적으로 증명**하는 데 할애된다.

## 자주 하는 실수

품질 추구 시 흔히 빠지는 함정이다.

| 실수 | 증상 | 해결 |
|------|------|------|
| **정확성 무시** | "일단 빠르게 만들고 나중에 고치자" → 명세와 다른 동작이 누적 | 정확성이 최우선. 틀린 소프트웨어는 나머지가 완벽해도 쓸모없다 |
| **효율성 과대평가** | 변수 하나 줄이는 미시적 최적화에 집중 | 알고리즘과 아키텍처가 효율의 대부분. 미시적 최적화는 마지막에 |
| **기능성 과잉** | 모든 요청을 추가 → feature creep → 복잡성 폭발 | 기능과 다른 품질 사이 균형. "이 기능이 정말 필요한가?" 질문 |
| **내적 요소 무시** | "사용자가 코드를 보지 않으니까" → 모듈성·가독성 방치 | 내적 요소 없이는 외적 요소(확장성, 재사용성) 달성 불가 |
| **상충 관계 무시** | 모든 품질 요소를 100% 달성하려 함 | 상충 인식, 프로젝트에 따라 우선순위 결정 |
| **적시성 경시** | 완벽해질 때까지 출시 미룸 | 늦은 완벽한 소프트웨어보다 적시의 충분한 소프트웨어 |

## 정리

- **외적 품질 요소**: 사용자가 직접 경험 — 정확성, 견고성, 확장성, 재사용성, 호환성, 효율성, 이식성, 사용 용이성, 기능성, 적시성
- **내적 품질 요소**: 개발자만 봄 — 모듈성, 가독성, 문서화
- **내적 → 외적**: 내적 요소는 외적 요소를 달성하는 수단
- **상충 관계**: 모든 요소를 최대화할 수 없음, 균형 필요
- **정확성이 최우선**: 나머지가 아무리 좋아도 틀리면 쓸모없음
- **객체지향의 약속**: 여러 품질 요소를 동시에 달성
