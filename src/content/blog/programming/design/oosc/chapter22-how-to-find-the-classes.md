## 한 줄 요약

> 클래스 발견의 핵심은 **ADT**(추상 데이터 타입)를 찾는 것이다. 명사 추출, 책임 할당, 도메인 분석을 조합해서 사용한다.

## 클래스 발견의 어려움

OO 설계에서 가장 어려운 질문: **"어떤 클래스를 만들어야 하는가?"**

| 초보자의 어려움 | 설명 |
|---------------|------|
| 과잉 분해 | 너무 많은 클래스 |
| 과소 분해 | 너무 적은 클래스 |
| 잘못된 추상화 | 엉뚱한 클래스 |
| 책임 분배 오류 | 책임이 잘못된 곳에 |

### 클래스가 아닌 것

| 잘못된 클래스 | 이유 | 예 |
|-------------|------|-----|
| 단일 동작 | function으로 충분 | `class CalculateSquareRoot` |
| 물리적 표현 | 구현 세부사항 | `class DatabaseRecord` |
| 변하는 값 | 값은 객체가 아님 | `class CurrentTemperature` |
| 시스템 일부분 | 모듈과 혼동 | `class InputOutputSubsystem` |

## Meyer의 접근법: ADT 찾기

### ADT 관점

**좋은 클래스 = 잘 정의된 ADT**

| ADT의 조건 | 설명 |
|-----------|------|
| 명확한 의미론적 개념 | 추상화가 무엇을 나타내는지 분명 |
| 관련된 연산들의 집합 | 해당 개념에 적용 가능한 연산 |
| 불변식으로 특징지어짐 | 항상 참인 속성 |
| 독립적으로 재사용 가능 | 다른 맥락에서도 사용 가능 |

| ADT 예시 | 개념 | 연산 | 불변식 |
|---------|------|------|--------|
| STACK | 수학적 스택 | push, pop, top | 후입선출 |
| ACCOUNT | 비즈니스 계좌 | deposit, withdraw, balance | 잔고 ≥ 0 |

### ADT 후보 찾기

| 검증 질문 | 목적 |
|----------|------|
| 이 개념에 대해 어떤 연산이 가능한가? | 연산 집합 파악 |
| 이 개념의 불변 속성은 무엇인가? | 불변식 정의 |
| 이 개념은 다른 곳에서 재사용될 수 있는가? | 재사용성 평가 |
| 이 개념은 다른 개념과 어떻게 다른가? | 경계 명확화 |

| 유용한 ADT의 특징 | 설명 |
|-----------------|------|
| 적절한 추상화 수준 | 너무 높지도 낮지도 않음 |
| 연산의 완전성 | 필요한 것 모두 포함 |
| 연산의 일관성 | 같은 수준의 추상화 |

## 명사 추출법

### 기본 방법

| 단계 | 활동 |
|------|------|
| 1 | 요구사항 문서에서 명사 추출 |
| 2 | 후보 목록 작성 |
| 3 | 필터링과 정제 |
| 4 | 관계 분석 |

### 예시: 도서관 시스템

**요구사항**: "도서관은 책을 보유한다. 회원은 책을 대출할 수 있다. 대출 기간은 2주이며, 연체 시 벌금이 부과된다. 사서는 새 책을 등록하고 회원을 관리한다."

**명사 추출**: 도서관, 책, 회원, 대출, 기간, 벌금, 사서

### 필터링

| 필터링 기준 | 예시 | 결정 |
|------------|------|------|
| 중복 제거 | "도서"와 "책" | 하나로 통합 |
| 속성 vs 클래스 | "기간", "벌금" | LOAN의 속성? 별도 클래스? |
| 시스템 외부 | "사서" | 액터? USER의 역할? |
| 구현 세부사항 | "데이터베이스" | 클래스 아님 |

### 결과 클래스 목록

| 분류 | 클래스 | 설명 |
|------|--------|------|
| 핵심 | LIBRARY | 도서관 전체 |
| 핵심 | BOOK (또는 COPY) | 책 (물리적 책) |
| 핵심 | MEMBER | 회원 |
| 핵심 | LOAN | 대출 기록 |
| 지원 | DATE | 날짜 |
| 지원 | FINE | 벌금 (필요하다면) |
| 역할 | LIBRARIAN | 사서 (USER의 하위 타입?) |

## 명사 추출의 한계

| 문제점 | 설명 |
|--------|------|
| 요구사항 품질에 의존 | 모호한 문서 → 모호한 클래스 |
| 언어적 우연에 의존 | 같은 개념/다른 명사, 다른 개념/같은 명사 |
| 동적 개념 누락 | 프로세스, 상태 변화 |
| 추상화 누락 | 문서에 없는 추상 클래스 |

**결론**: 명사 추출만으로는 부족하다. 다른 기법과 조합해야 한다.

## 책임 주도 설계 (RDD)

### 책임이란

| 책임 유형 | 내용 |
|----------|------|
| **아는 것 (Knowing)** | 자신의 데이터, 관련 객체, 파생 값 |
| **하는 것 (Doing)** | 계산 수행, 다른 객체에 요청, 활동 제어 |

### CRC 카드

**CRC = Class, Responsibility, Collaborator**

| LOAN 클래스 | |
|------------|---|
| **책임** | 대출 정보 유지, 연체 여부 판단, 벌금 계산 |
| **협력자** | BOOK, MEMBER, DATE |

### 책임 할당 원칙

**GRASP 원칙 (Larman)**:

| 원칙 | 의미 | 예 |
|------|------|-----|
| Information Expert | 정보를 가진 클래스에 책임 할당 | 벌금 계산은 LOAN에 |
| Creator | 관련 정보가 있는 클래스가 생성 | LIBRARY가 LOAN 생성 |
| Low Coupling | 결합도 최소화 | LOAN을 통해 BOOK↔MEMBER 연결 |
| High Cohesion | 응집도 최대화 | LOAN에 대출 관련 책임 집중 |
| Controller | 시스템 이벤트 처리 담당 | LIBRARY_CONTROLLER가 대출/반납 |

## 도메인 분석

### 도메인 모델

| 도메인 분석의 핵심 | 설명 |
|-------------------|------|
| 개념 파악 | 문제 영역의 주요 개념 |
| 실세계 모델링 | 현실을 반영 |
| 전문가 협력 | 도메인 전문가와 함께 |

| 구분 | 도메인 모델 | 설계 모델 |
|------|------------|----------|
| 관점 | 문제 영역의 개념 | 소프트웨어 클래스 |
| 단계 | 분석 | 설계 |
| 연결 | 도메인 개념 → 소프트웨어 클래스 (항상 1:1은 아님) |

### 도메인 패턴

| 패턴 | 상위 개념 | 하위 개념 |
|------|----------|----------|
| Party (당사자) | PERSON, ORGANIZATION | MEMBER, LIBRARIAN |
| Product (제품) | ITEM, COPY | BOOK, DVD |
| Transaction (거래) | LOAN, RESERVATION | PURCHASE |
| Role (역할) | MEMBER_ROLE, STAFF_ROLE | 한 사람이 여러 역할 |

## 기존 라이브러리 활용

### 재사용 우선

**Meyer의 원칙**: "새 클래스를 만들기 전에 기존 클래스를 찾아라"

| 기존 소스 | 예 |
|----------|-----|
| 언어 표준 라이브러리 | 컬렉션, 문자열, 날짜 |
| 도메인 라이브러리 | 금융, 의료, 제조 |
| 프레임워크 | UI, 웹, 데이터 접근 |
| 이전 프로젝트 | 내부 라이브러리 |

### 상속 vs 조합

| 방법 | 설명 | 예 |
|------|------|-----|
| 상속 | is-a 관계 | `class SPECIAL_STACK inherit STACK [G]` |
| 조합 | has-a 관계 | `due_date: DATE` |
| 위임 | 포함 객체에 요청 전달 | `books.extend(b)` |

## 잘못된 클래스의 징후

### 클래스가 너무 크다

| 징후 | 해결 |
|------|------|
| 피처가 50개 이상 | 책임별로 분리 |
| 여러 책임이 섞임 | 추상 클래스 추출 |
| 이름이 모호 ("Manager", "Handler") | 조합으로 재구성 |

### 클래스가 너무 작다

| 징후 | 해결 |
|------|------|
| 피처가 2-3개뿐 | 관련 클래스와 병합 |
| 단일 속성 래퍼 | 속성으로 전환 |
| 다른 클래스와 항상 함께 사용 | 값 객체로 재정의 |

### 책임이 분산됨

| 징후 | 해결 |
|------|------|
| 하나의 변경이 여러 클래스 수정 필요 | 책임 재할당 |
| 관련 데이터가 여러 클래스에 분산 | 클래스 병합 |
| Feature Envy (다른 클래스 데이터 자주 사용) | 중재자 클래스 도입 |

## 실전 접근법

### 반복적 정제

| 단계 | 활동 |
|------|------|
| 1차 시도 | 명사 추출로 후보 목록 |
| 2차 정제 | 책임 분석으로 조정 |
| 3차 정제 | 구현하면서 발견 |

**핵심**: 완벽한 클래스 목록을 처음부터 알 수 없다. 반복적으로 정제한다.

### 체크리스트

| 검증 질문 |
|----------|
| 이 클래스의 목적을 한 문장으로 설명할 수 있는가? |
| 이 클래스의 불변식은 무엇인가? |
| 이 클래스는 다른 곳에서 재사용될 수 있는가? |
| 이 클래스의 책임은 응집력 있는가? |
| 이 클래스의 이름은 적절한가? |
| 비슷한 기존 클래스가 있는가? |

### Meyer의 조언

**"클래스는 서비스 제공자다."**

| 좋은 클래스 특성 | 설명 |
|----------------|------|
| 클라이언트 관점 | 클라이언트가 필요한 서비스 제공, 구현 세부사항 숨김 |
| 추상화 수준 일관 | 모든 피처가 같은 추상화 수준 |
| 완전하면서 최소 | 필요한 것은 모두, 불필요한 것은 제외 |
| 독립적 | 최소 결합, 다른 클래스 변경에 영향 최소 |

## 정리

- **ADT 관점**: 클래스 = 연산과 불변식을 가진 추상화
- **명사 추출**: 요구사항에서 명사 추출, 필터링
- **책임 주도**: 아는 것/하는 것, CRC 카드
- **도메인 분석**: 문제 영역의 개념 모델링
- **재사용 우선**: 새로 만들기 전에 기존 것 찾기
- **반복적 정제**: 완벽한 목록은 처음부터 없음
