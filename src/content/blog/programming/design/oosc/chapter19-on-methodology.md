## 한 줄 요약

> **방법론**은 가이드라인이지 규칙이 아니다. 좋은 방법론은 **원칙**을 제공하고, **판단**은 개발자에게 맡긴다.

## 방법론이란 무엇인가

소프트웨어 방법론(methodology)은 소프트웨어 개발 과정을 안내하는 체계적 접근법이다.

| 구성 요소 | 설명 |
|----------|------|
| **원칙** | 근본적인 지침 |
| **기법** | 구체적인 방법 |
| **도구** | 지원 소프트웨어 |
| **표기법** | 표현 방식 |
| **프로세스** | 단계별 절차 |

### 방법론의 역할

| 방법론이 해야 할 것 | 방법론이 하지 못하는 것 |
|-------------------|---------------------|
| 문제 해결의 방향 제시 | 창의적 사고 대체 |
| 팀 간 의사소통 표준화 | 모든 상황에 적용되는 해답 |
| 경험의 축적과 전달 | 판단력과 경험 대신 |
| 품질 향상을 위한 체크리스트 | — |

## Meyer의 방법론 관점

### 경계해야 할 것들

Meyer는 **방법론 맹신**을 경계한다.

| 위험 | 착각 | 현실 |
|------|------|------|
| **체크리스트 맹신** | 체크리스트 통과 = 좋은 설계 | 진짜 문제는 체크리스트에 없을 수 있음 |
| **다이어그램 숭배** | 다이어그램 = 소프트웨어 | 코드와 다이어그램의 괴리 |
| **프로세스 숭배** | 프로세스 준수 = 성공 | 형식이 실질을 대체 |
| **도구 숭배** | 최신 도구 = 좋은 결과 | 도구는 수단일 뿐 |

### 좋은 방법론의 특징

| 특징 | 설명 |
|------|------|
| **원칙 기반** | 왜(Why)를 설명, 맹목적 규칙이 아닌 이해 가능한 지침 |
| **유연성** | 상황에 맞게 적용 가능, 맥락을 고려 |
| **점진적 적용** | 한 번에 모든 것이 아님, 단계적 개선 |
| **피드백 수용** | 실제 경험으로부터 학습, 지속적 개선 |

## OO 개발 과정

### 전통적 단계 모델

**폭포수 모델**: 분석 → 설계 → 구현 → 테스트 → 유지보수

| 문제점 | 설명 |
|--------|------|
| 피드백 어려움 | 단계 간 되돌아가기 힘듦 |
| 변경에 취약 | 요구사항 변경 대응 어려움 |
| 늦은 통합 | 문제를 늦게 발견 |

### 클러스터 모델

Meyer가 제안하는 접근법이다. 시스템을 클러스터로 나누어 각 클러스터를 독립적으로 개발한다.

| 단계 | 클러스터 1 | 클러스터 2 | 클러스터 3 |
|------|-----------|-----------|-----------|
| 분석 | ■ | | |
| 설계 | ■ | ■ | |
| 구현 | ■ | ■ | ■ |
| 테스트 | ■ | ■ | ■ |
| 통합 | | | ■ |

### 심리스 개발 (Seamless Development)

| 특징 | 설명 |
|------|------|
| 경계 흐릿함 | 분석, 설계, 구현이 연속적 |
| 도구 통일 | 같은 개념, 같은 도구, 같은 언어 |
| 자연스러운 연결 | 분석 모델 → 설계 모델 → 코드 |

Eiffel의 장점:

| 장점 | 설명 |
|------|------|
| 언어 통일 | 분석 = 설계 = 구현 언어 |
| 모델 일치 | 모델과 코드가 동일 |
| 역공학 불필요 | 코드가 곧 문서 |

## 반복과 점진

### 반복적 개발

| 원칙 | 설명 |
|------|------|
| **작게 시작** | 핵심 기능부터, 점진적 확장 |
| **자주 통합** | 매 반복마다 동작하는 시스템, 조기 피드백 |
| **지속적 개선** | 리팩토링, 설계 진화 |

### 가역성 (Reversibility)

소프트웨어의 특성:

| 특성 | 의미 |
|------|------|
| 물리적 제약 없음 | 재료 낭비 없음 |
| 변경 가능 | 언제든 수정 |
| 되돌릴 수 있음 | 이전 상태로 복귀 |

**가역성 원칙**: 결정을 번복하는 것이 비용이 들지만 불가능한 것은 아니다. 완벽한 사전 설계보다 적절한 시점의 수정이 낫다.

## 분석과 설계

### 분석의 목표

**분석**(Analysis)은 무엇(What)을 정의한다.

| 초점 | 질문 |
|------|------|
| 문제 영역 이해 | 시스템이 무엇을 해야 하는가? |
| 요구사항 추출 | 어떤 개념이 관련되는가? |
| 제약 조건 | 제약 조건은 무엇인가? |

### 설계의 목표

**설계**(Design)는 어떻게(How)를 정의한다.

| 초점 | 질문 |
|------|------|
| 솔루션 영역 구축 | 어떤 클래스가 필요한가? |
| 아키텍처 결정 | 클래스 간 관계는? |
| 알고리즘 선택 | 어떤 알고리즘을 사용하는가? |

### OO에서의 분석-설계 연속성

| 구분 | 전통적 분석 | OO 분석 |
|------|-----------|---------|
| 도구 | DFD, ERD | 클래스 다이어그램, 객체 상호작용 |
| 설계와 관계 | 언어가 다름 | 설계, 구현과 같은 개념 |

연속성의 이점:

| 이점 | 설명 |
|------|------|
| 변환 비용 없음 | 분석 → 설계 → 구현 직접 연결 |
| 추적 가능성 | 요구사항부터 코드까지 추적 |
| 모델 일관성 | 동일한 모델 유지 |

## 문서화

### 문서의 역할

| 목적 | 설명 |
|------|------|
| **의사소통** | 팀 간, 이해관계자 간 |
| **기록** | 결정 사항, 이유 |
| **유지보수** | 미래의 개발자를 위해 |
| **검증** | 리뷰, 승인 |

### 문서와 코드의 관계

| 문제 | Meyer의 해결책 |
|------|---------------|
| 문서와 코드의 불일치 | 코드가 곧 문서 |
| 문서 업데이트 누락 | Short Form(자동 생성된 인터페이스) |
| 이중 유지보수 | 계약이 명세 역할 |

### 자동 문서화

```eiffel
class ACCOUNT
    -- 은행 계좌를 나타낸다.
    -- 잔고 관리, 입출금, 이체 기능을 제공한다.

feature -- 접근
    balance: INTEGER
        -- 현재 잔고

feature -- 상태 변경
    deposit (amount: INTEGER)
        -- amount만큼 입금한다.
        require
            positive: amount > 0
        ensure
            increased: balance = old balance + amount
        end
end
```

```text
Short Form 자동 생성:

class interface ACCOUNT
feature
    balance: INTEGER
    deposit (amount: INTEGER)
        require positive: amount > 0
        ensure increased: balance = old balance + amount
end

→ 구현은 숨기고 계약만 노출
→ 항상 코드와 일치
```

## 품질 보증

### 검증 vs 검사

| 구분 | 검증(Verification) | 검사(Validation) |
|------|-------------------|-----------------|
| **질문** | 제대로 만들었는가? | 제대로 된 것을 만들었는가? |
| **영어** | Building the product right | Building the right product |
| **대상** | 명세와 일치하는가? | 사용자 요구에 맞는가? |
| **방법** | 정적 분석, 증명 | 테스트, 프로토타입 |

### Meyer의 품질 관점

| 방식 | 기법 |
|------|------|
| **정적 품질 보증** | Design by Contract, 타입 시스템, 정적 분석 |
| **동적 품질 보증** | 단언 검사(런타임), 단위 테스트, 통합 테스트 |
| **정적 + 동적** | 상호 보완적, 계약 = 테스트 오라클 |

## 재사용과 방법론

### 재사용 지향 개발

| 방향 | 설명 |
|------|------|
| **재사용 FOR 개발** | 기존 라이브러리 활용, 프레임워크 사용, 컴포넌트 조립 |
| **재사용 IN 개발** | 재사용 가능한 코드 작성, 라이브러리 구축, 일반화된 솔루션 |

### 방법론의 재사용

| 재사용 가능한 요소 | 주의사항 |
|------------------|---------|
| 설계 패턴 | 맹목적 적용 금지 |
| 아키텍처 패턴 | 맥락 고려 |
| 코딩 표준 | 적절한 변형 |
| 체크리스트 | — |

## 현실적 조언

### 방법론 선택

| 선택 기준 | 고려사항 |
|----------|---------|
| 팀 역량 | 경험과 기술 수준 |
| 프로젝트 특성 | 규모, 복잡성 |
| 도메인 특성 | 업무 영역 |
| 조직 문화 | 회사 환경 |
| 이해관계자 | 고객 요구 |

**만능 방법론은 없다!**

### 점진적 도입

| 단계 | 활동 |
|------|------|
| 1 | 파일럿 프로젝트로 시작 |
| 2 | 핵심 원칙부터 적용 |
| 3 | 점진적 확대 |
| 4 | 피드백 반영 |
| 5 | 조직에 맞게 조정 |

### Meyer의 실용주의

**원칙**: 방법론을 따르되, 방법론에 지배당하지 마라.

| 판단의 중요성 | 설명 |
|--------------|------|
| 규칙 | 가이드라인일 뿐 |
| 예외 | 항상 존재 |
| 맥락 | 결정의 핵심 |
| 경험 | 필수 요소 |

> "No silver bullet" — Fred Brooks

## 정리

- **방법론의 역할**: 가이드라인 제공, 판단 대체 불가
- **경계할 것**: 체크리스트/다이어그램/프로세스/도구 맹신
- **클러스터 모델**: 독립적 클러스터의 병렬 개발
- **심리스 개발**: 분석-설계-구현의 연속성
- **문서화**: 코드가 곧 문서, 자동 생성
- **가역성**: 변경 가능성 인정, 점진적 개선
- **실용주의**: 원칙 이해, 맥락 고려, 판단력
