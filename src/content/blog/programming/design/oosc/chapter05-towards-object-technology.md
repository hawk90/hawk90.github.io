## 한 줄 요약

> 전통적 방법(함수 분해, 데이터 흐름)은 분해에는 좋지만 재사용성·확장성이 부족하다. 객체지향이 필요한 이유다.

## 전통적 설계 방법들

객체지향 이전에도 소프트웨어 설계 방법론은 있었다. 왜 그것들로 충분하지 않은가?

### 함수 분해 (Functional Decomposition)

**하향식 설계**라고도 한다. 문제를 기능 단위로 분해:

1. 최상위 기능: "급여 시스템을 실행한다"
2. 하위 기능: 직원 데이터 읽기, 급여 계산, 세금 공제, 급여 명세서 출력
3. 더 분해: 급여 계산 → 기본급 계산, 초과근무 계산, 보너스 계산

**장점**: 직관적이고 체계적이다. 문제를 이해하기 쉽다.

**문제점**:

1. **최상위 함수 편향**: 시스템에 "메인 함수"가 하나 있다고 가정. 현실의 시스템은 여러 진입점이 있다.

GUI 애플리케이션은 버튼 클릭, 메뉴 선택, 타이머 등 여러 진입점이 있다. "메인 함수"가 없다.

2. **재사용 불가**: 분해된 함수들은 상위 문맥에 종속된다.

"급여 계산" 함수는 "급여 시스템"의 데이터 구조와 전역 변수에 의존하므로 다른 프로젝트에서 재사용할 수 없다.

3. **변경에 취약**: 최상위 구조가 바뀌면 하위 전체가 영향받는다.

"웹 인터페이스 추가" 같은 변경이 생기면 하위 로직을 두 인터페이스가 공유해야 한다. 대대적인 리팩토링이 필요하다.

### 데이터 흐름 설계 (Data Flow Design)

DFD(Data Flow Diagram)를 사용한 설계:

급여 시스템 예: 직원 데이터 → 급여 계산 → 세금 계산 → 명세서 생성 → 급여 명세서

**장점**: 데이터의 흐름을 시각화. 병목 파악 용이.

**문제점**:

1. **데이터와 처리의 분리**: 데이터 구조와 함수가 별개로 존재.

데이터(직원 레코드)와 함수(`calculatePay`)가 분리되면 직원 레코드 구조가 바뀔 때 `calculatePay`, `calculateTax`, `printStatement`를 모두 수정해야 한다.

2. **캡슐화 부재**: 데이터가 시스템 전체에 노출.

`직원.급여 = 100`처럼 아무 곳에서나 직접 수정하면 데이터 무결성이 위험해진다.

### 정보 흐름 기반의 구조 (Jackson/Warnier-Orr)

입력/출력 데이터 구조에서 프로그램 구조를 도출:

입력 구조에서 프로그램 구조를 도출하고, 다시 출력 구조로 대응시킨다. 그러나 입출력이 바뀌면 전체 구조가 바뀐다. 내부 로직과 외부 형식이 결합되어 있기 때문이다.

## "최상위 함수"의 문제

전통적 방법의 공통된 문제는 **"시스템의 최상위 기능이 무엇인가?"**라는 질문이다.

현실의 소프트웨어는 최상위 함수가 없다:

텍스트 편집기를 생각해 보라. 파일 열기, 텍스트 편집, 저장, 인쇄, 검색/치환 중 어느 것이 "최상위"인가? 사용자마다, 상황마다 다르다.

Meyer의 통찰: **시스템을 정의하는 것은 "무엇을 하는가"가 아니라 "무엇에 대해 하는가"다.**

| 질문 | 답 |
|------|-----|
| 급여 시스템의 메인 함수는? | 없음 |
| 급여 시스템이 다루는 것은? | 직원, 급여, 세금, 명세서 |

## 객체 중심의 사고

| 사고 방식 | 시스템 정의 | 연산 방식 |
|----------|------------|----------|
| 전통적 | 함수들의 집합 | 함수가 데이터를 조작 |
| 객체지향 | 객체들의 집합 | 객체가 메시지를 교환 |

**전환의 핵심**: 함수가 아닌 **데이터(객체)**를 중심에 둔다.

| 관점 | 표현 | 의미 |
|------|------|------|
| 함수 중심 | `calculatePay(employee)` | 직원에 대해 급여를 계산하라 |
| 객체 중심 | `employee.calculatePay()` | 직원아, 네 급여를 계산해라 |

이 전환이 왜 중요한가?

1. **데이터는 함수보다 안정적**: 급여 계산 방식은 바뀌어도 "직원"이라는 개념은 바뀌지 않는다.

2. **책임의 명확화**: 각 객체가 자신의 데이터를 책임진다.

3. **재사용 용이**: "직원" 클래스는 급여 시스템, HR 시스템, 출퇴근 시스템에서 재사용 가능.

## 객체 식별의 기준

"무엇이 객체가 되어야 하는가?"

Meyer의 가이드라인:

### 물리적 실체

직원, 제품, 건물, 차량 — 현실 세계의 물건.

### 역할

매니저, 고객, 공급자 — 누군가가 수행하는 역할.

### 이벤트

주문, 결제, 배송 — 발생하는 사건.

### 상호작용

계약, 거래, 미팅 — 둘 이상 간의 관계.

### 명세/규칙

세금 규칙, 가격 정책, 할인 규칙 — 비즈니스 로직.

## 전통적 방법과의 비교

| 측면 | 함수 분해 | 데이터 흐름 | 객체지향 |
|------|----------|------------|---------|
| **중심** | 함수 | 데이터 흐름 | 객체 |
| **분해 기준** | 기능 | 입출력 | 책임 |
| **재사용** | 어려움 | 어려움 | 용이 |
| **변경 대응** | 취약 | 취약 | 강함 |
| **캡슐화** | 없음 | 약함 | 강함 |

## 전환의 도전

전통적 방법에서 객체지향으로 전환하는 것은 쉽지 않다:

| 전환 영역 | 기존 | 객체지향 |
|----------|------|---------|
| 사고방식 | "무엇을 해야 하나?" (동사 중심) | "무엇에 대해 하나?" (명사 중심) |
| 설계 순서 | 기능 → 데이터 | 데이터(클래스) → 기능(메서드) |
| 프로젝트 구조 | 함수별 파일 분리 | 클래스별 파일 분리 |

## 정리

- **함수 분해의 한계**: 최상위 함수 편향, 재사용 불가, 변경 취약
- **데이터 흐름의 한계**: 데이터/처리 분리, 캡슐화 부재
- **"최상위 함수" 문제**: 현실 시스템에는 단일 최상위가 없다
- **객체 중심 사고**: 데이터(객체)를 중심에 두고 기능을 배치
- **데이터가 더 안정적**: 기능은 변해도 핵심 데이터 구조는 유지
- **객체 식별**: 물리적 실체, 역할, 이벤트, 상호작용, 명세
