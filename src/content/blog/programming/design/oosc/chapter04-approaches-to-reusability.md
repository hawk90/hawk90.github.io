## 한 줄 요약

> 소프트웨어 재사용은 산업의 오랜 꿈이지만, 기술적·심리적·경제적 장애물이 있다. 객체지향은 이를 해결하는 가장 유망한 방법이다.

## 재사용의 약속

하드웨어 산업을 보라. 반도체 칩, 저항, 커패시터는 표준화된 부품이다. 엔지니어는 이 부품들을 조합해 새로운 제품을 만든다. 처음부터 트랜지스터를 설계하지 않는다.

소프트웨어는 왜 그렇지 못한가?

| 산업 | 방식 | 재사용율 |
|------|------|----------|
| 하드웨어 | 표준 부품 조합 | 높음 |
| 소프트웨어 | 매번 새로 작성 | 낮음 |

재사용이 달성되면:

| 이점 | 설명 |
|------|------|
| **품질 향상** | 검증된 컴포넌트 사용 |
| **비용 절감** | 개발 시간 단축 |
| **일관성** | 표준화된 해결책 |
| **유지보수 용이** | 수정이 한 곳에 집중 |

## 재사용이 어려운 이유

### 기술적 장애물

**변형의 필요성**: 기존 컴포넌트가 *거의* 맞지만 *완전히* 맞지 않는 경우가 많다.

| 선택지 | 결과 |
|--------|------|
| 기존 것 수정 | 원본이 망가짐 |
| 복사 후 수정 | 코드 중복 |
| 처음부터 작성 | 재사용 포기 |

객체지향의 **상속**은 이 문제를 해결한다:

```eiffel
class SORTED_LIST [G -> COMPARABLE]
inherit
    LIST [G]
        redefine put end
feature
    put (x: G)
        -- 정렬 순서 유지하며 삽입
        do
            -- 원본 LIST는 변경 안 함
            -- SORTED_LIST만 다른 동작
        end
end
```

**타입 변형**: 같은 알고리즘이지만 타입이 다른 경우.

정수 정렬, 실수 정렬, 문자열 정렬 — 로직은 같은데 타입만 다르다.

객체지향의 **제네릭**이 해결:

```eiffel
class SORTER [G -> COMPARABLE]
feature
    sort (list: LIST [G])
        -- 어떤 COMPARABLE 타입도 정렬 가능
end
```

### 심리적 장애물

**NIH 증후군** (Not Invented Here): 남의 코드를 신뢰하지 않는 경향.

"이 라이브러리에 버그가 있을지 몰라", "내가 직접 작성하면 확실해", "남의 코드 이해하느니 새로 쓴다" — 이런 생각이 재사용을 막는다. 그러나 현실은 다르다:

| 항목 | 직접 작성 | 재사용 |
|------|----------|--------|
| 버그 확률 | 높음 (새 코드) | 낮음 (검증됨) |
| 개발 시간 | 길다 | 짧다 |
| 이해 비용 | 없음 | 있음 |
| 유지보수 | 내가 함 | 라이브러리 제공자 |

### 경제적 장애물

**일반화 비용**: 특정 문제 해결 코드를 일반화하려면 추가 노력이 필요하다.

| 개발 방식 | 시간 | 적합한 경우 |
|----------|------|------------|
| 특정 해결책 | 1일 | 프로젝트 A만 고려 |
| 일반화 컴포넌트 | 3일 | 프로젝트 A, B, C를 고려하면 이득 |

**발견 비용**: 재사용 가능한 컴포넌트가 있는지 찾는 데 시간이 걸린다.

"정렬된 리스트 구현이 있나?" 사내 라이브러리, 오픈소스, 상용 라이브러리를 검색하고 평가·비교·선택하는 과정이 직접 작성보다 오래 걸릴 수 있다.

## 재사용의 형태

### 인원 재사용 vs 코드 재사용

| 형태 | 설명 | 한계 |
|------|------|------|
| **인원 재사용** | 경험 있는 개발자를 여러 프로젝트에 투입 | 확장 불가, 인원 한계 |
| **코드 재사용** | 검증된 컴포넌트를 여러 프로젝트에서 사용 | 확장 가능, 무제한 |

인원 재사용은 **선형적**이다. 10개 프로젝트에는 10명의 전문가가 필요하다.
코드 재사용은 **지수적**이다. 하나의 좋은 라이브러리가 1000개 프로젝트에 쓰인다.

### 소스 코드 재사용 vs 바이너리 재사용

| 재사용 형태 | 장점 | 단점 |
|------------|------|------|
| 소스 코드 | 수정·이해 가능 | 빌드 필요, 버전 관리 복잡 |
| 바이너리 | 빠른 통합, 버전 명확 | 수정 불가, 디버깅 어려움 |

이상적인 형태는 **소스 레벨 접근 + 바이너리 배포**다. 객체지향은 이를 가능하게 한다.

## 재사용을 위한 조건

### 추상화 수준

재사용 가능한 컴포넌트는 **적절한 추상화 수준**을 가져야 한다:

| 추상화 수준 | 예시 | 문제점 |
|------------|------|--------|
| 너무 구체적 | `KoreanTaxCalculator2024` | 한국 2024년에만 사용 가능 |
| 너무 추상적 | `Calculator` | 뭘 계산하는지 모름 |
| 적절함 | `TaxCalculator[RULE]` | 규칙을 주입받아 어떤 국가든 처리 |

### 완전성

컴포넌트는 **자급자족**해야 한다:

| 완전성 | 예시 | 문제점 |
|--------|------|--------|
| 불완전 | `SortedList` + 별도 `Comparer` | 의존성이 문서에 없음 |
| 완전 | `SortedList[G -> COMPARABLE]` | 필요한 것이 명확함 |

### 문서화

재사용 가능한 컴포넌트는 **명확한 계약**을 문서화해야 한다:

```eiffel
class STACK [G]
feature
    push (x: G)
        -- x를 스택 최상위에 추가
        require
            not is_full  -- 사전조건
        ensure
            top = x  -- 사후조건: 최상위는 x
            count = old count + 1  -- 크기 증가

    pop
        require
            not is_empty
        ensure
            count = old count - 1
end
```

Design by Contract는 **실행 가능한 문서**다. 명세가 코드에 녹아 있어 항상 최신 상태다.

## 재사용 기술의 비교

### 서브루틴 라이브러리

C 표준 라이브러리(`strlen`, `strcpy`, `printf`)는 단순하고 잘 알려졌지만, 추상화 수준이 낮고 타입 안전성이 부족하며 확장이 어렵다.

### 모듈

Ada 패키지, Modula-2 모듈(STACK 패키지 등)은 캡슐화와 타입 안전을 제공하지만, 확장과 변형이 어렵다.

### 클래스

객체지향 클래스(`STACK[G]`)는 상속, 재정의, 제네릭이 모두 가능하다. 추상화·확장·변형을 한꺼번에 지원하지만 학습 곡선이 있다.

| 기술 | 캡슐화 | 타입 안전 | 확장성 | 변형성 |
|------|:------:|:--------:|:------:|:------:|
| 서브루틴 | ✗ | △ | ✗ | ✗ |
| 모듈 | ✓ | ✓ | ✗ | ✗ |
| 클래스 | ✓ | ✓ | ✓ | ✓ |

## 재사용의 단계

Meyer는 5단계 재사용 성숙도를 제시한다:

| 단계 | 설명 | 예 |
|------|------|-----|
| **0. 없음** | 매번 새로 작성 | |
| **1. 내부 재사용** | 프로젝트 내 공유 | 프로젝트 내 유틸리티 |
| **2. 팀 재사용** | 팀 내 공유 | 팀 공용 라이브러리 |
| **3. 조직 재사용** | 조직 내 공유 | 사내 프레임워크 |
| **4. 산업 재사용** | 산업 전체 공유 | 표준 라이브러리, 오픈소스 |

각 단계는 이전 단계의 문제를 해결하면서 새로운 도전을 제시한다:

팀 재사용에서 조직 재사용으로 나아가려면 문서화 수준을 높이고, 버전 관리를 강화하고, 지원 체계를 구축해야 한다.

## 객체지향의 해답

객체지향이 재사용 장애물을 어떻게 해결하는가:

| 장애물 | OO 해결책 |
|--------|----------|
| 변형 필요 | 상속으로 확장/재정의 |
| 타입 변형 | 제네릭 |
| 발견 어려움 | 클래스 카탈로그, 계층 구조 |
| 이해 어려움 | 캡슐화, 계약 |
| 일반화 비용 | 라이브러리 클래스 재사용 |

객체지향은 재사용의 **은탄환**은 아니지만, 현재까지 가장 유망한 방법이다.

## 자주 하는 실수

재사용 추구 시 흔히 빠지는 함정이다.

| 실수 | 증상 | 해결 |
|------|------|------|
| **NIH 증후군** | "남의 코드 못 믿어" → 이미 있는 것을 새로 작성 | 검증된 라이브러리가 새 코드보다 버그 적음. 이해 비용은 투자 |
| **너무 구체적** | `KoreanTaxCalculator2024` → 한국 2024년에만 사용 | 규칙을 매개변수화. `TaxCalculator[RULE]` |
| **너무 추상적** | `Calculator` → 뭘 계산하는지 모름 | 적절한 추상화 수준 찾기. 도메인 언어 반영 |
| **문서화 부족** | "코드가 곧 문서" → 사용법 모름 | 계약(사전/사후조건) 명시, 예제 제공 |
| **일반화 비용 무시** | "일단 만들고 나중에 일반화" → 결국 특정 프로젝트에 종속 | 재사용 고려하며 설계. 3배 비용 각오 |
| **의존성 지옥** | 컴포넌트가 10개 다른 라이브러리에 의존 | 의존성 최소화. 자급자족 컴포넌트 |

## 정리

- **재사용의 이점**: 품질, 비용, 일관성, 유지보수
- **기술적 장애물**: 변형 필요, 타입 변형
- **심리적 장애물**: NIH 증후군, 신뢰 부족
- **경제적 장애물**: 일반화 비용, 발견 비용
- **재사용 조건**: 추상화 수준, 완전성, 문서화
- **객체지향의 역할**: 상속, 제네릭, 캡슐화로 장애물 극복
