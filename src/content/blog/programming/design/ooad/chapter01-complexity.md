## 한 줄 요약

> 소프트웨어는 **본질적으로 복잡**하다. 이 복잡성을 다루는 열쇠는 **분해(Decomposition)**와 **추상화(Abstraction)**이며, 객체지향은 이 두 가지를 자연스럽게 지원한다.

## 소프트웨어 복잡성의 본질

### 왜 소프트웨어는 복잡한가

**Brooks의 주장 (No Silver Bullet)**:

| 복잡성 유형 | 특성 |
|------------|------|
| 본질적 복잡성 (Essential) | 문제 자체의 복잡성, 요구사항의 복잡성, 현실 세계의 복잡성. 제거할 수 없음 |
| 우발적 복잡성 (Accidental) | 도구, 언어, 플랫폼의 한계. 개선 가능. 점진적으로 감소 중 |

**결론**: "은탄환은 없다." 본질적 복잡성은 줄일 수 없다. 관리할 수 있을 뿐이다.

### 복잡성의 특성

| 소프트웨어 복잡성의 네 가지 원인 | 설명 |
|------------------------------|------|
| 문제 도메인의 복잡성 | 현실 세계는 복잡하다. 요구사항은 변한다. 이해관계자가 다양하다 |
| 개발 프로세스의 어려움 | 팀 간 의사소통, 변경 관리, 품질 보증 |
| 소프트웨어의 유연성 | "할 수 있으니까 한다." 제약이 없으면 복잡해진다 |
| 이산 시스템의 특성 | 연속적이지 않음. 작은 변경이 큰 영향. 테스트로 모든 상태를 커버할 수 없음 |

### 복잡한 시스템의 속성

**복잡한 시스템의 공통 특성 (Simon)**:

| 특성 | 설명 |
|------|------|
| 계층적 구조 | 시스템은 하위 시스템으로 구성. 하위 시스템도 다시 분해 가능 |
| 상대적 기본 요소 | 관찰자에 따라 기본 단위가 달라짐. 적절한 추상화 수준 선택 |
| 관심사 분리 | 컴포넌트 내부 상호작용 > 컴포넌트 간 상호작용. 느슨한 결합, 높은 응집 |
| 공통 패턴 | 다른 도메인에도 비슷한 구조. 재사용 가능한 패턴 |
| 중간 형태의 안정성 | 진화는 점진적. 안정된 중간 단계 존재 |

## 복잡성 다루기

### 분해 (Decomposition)

| 분해 방식 | 특징 |
|----------|------|
| 알고리즘 분해 | 절차 중심. "무엇을 하는가". 단계별 분해. 동사 중심 (process, calculate, validate) |
| 객체 분해 | 데이터 중심. "무엇이 있는가". 엔티티별 분해. 명사 중심 (Customer, Order, Product) |

### 왜 객체 분해인가

| 객체 분해의 장점 | 설명 |
|----------------|------|
| 현실 세계 모델링 | 도메인 전문가와 소통 용이. 개념이 코드에 직접 반영 |
| 변경 흡수 | 요구사항 변경 시 영향 국소화. 데이터가 동작보다 안정적 |
| 재사용 촉진 | 객체는 자기 완결적. 다른 맥락에서도 사용 가능 |
| 복잡성 관리 | 캡슐화로 내부 숨김. 인터페이스만 노출 |

### 추상화 (Abstraction)

**추상화란**: 본질적 특성에 집중하고 부수적 세부사항을 숨기는 것.

| 추상화 수준 | 예 |
|------------|-----|
| 높은 추상화 | "고객이 주문을 한다" |
| 중간 추상화 | `Customer.placeOrder(items)` |
| 낮은 추상화 | `INSERT INTO orders ...` |

| 좋은 추상화의 조건 |
|------------------|
| 의도가 명확 |
| 세부사항 숨김 |
| 일관된 추상화 수준 |
| 직관적인 인터페이스 |

## 복잡한 시스템의 구조

### 계층 (Hierarchy)

| 계층 종류 | 설명 | 예 |
|----------|------|-----|
| "is-a" 계층 (상속) | 일반화-특수화 | Vehicle → Car, Truck, Motorcycle |
| "part-of" 계층 (합성) | 전체-부분 | Car → Engine, Transmission, Chassis |

둘 다 복잡성 관리에 필수다.

### 모듈화 (Modularity)

| 모듈화 원칙 |
|------------|
| 시스템을 독립 모듈로 분리 |
| 각 모듈은 명확한 책임 |
| 모듈 간 의존성 최소화 |

| 목표 |
|------|
| 한 모듈의 변경이 다른 모듈에 영향 없음 |
| 병렬 개발 가능 |
| 개별 테스트 가능 |
| 이해 용이 |

**예**: GUI 모듈, 비즈니스 로직 모듈, 데이터 접근 모듈 → 각각 독립적으로 발전 가능.

### 모델의 역할

**모델이란**: 현실의 단순화된 표현. 이해와 소통을 위한 도구.

| 소프트웨어 모델 | 설명 |
|---------------|------|
| 논리적 모델 | 무엇을 하는가 |
| 물리적 모델 | 어떻게 구현되는가 |

| 모델의 가치 |
|------------|
| 복잡성 관리 |
| 의사소통 촉진 |
| 설계 결정 문서화 |
| 검증 가능 |

## 설계의 의미

### 좋은 설계란

| 설계의 목표 |
|------------|
| 기능 요구사항 충족 |
| 비기능 요구사항 충족 |
| 변경에 유연 |
| 이해 용이 |
| 구현 가능 |

**설계는 선택**: 트레이드오프의 연속이다. 완벽한 설계는 없으며, 맥락에 맞는 최선의 선택을 한다.

### 설계 품질 지표

| 좋은 설계의 징후 | 설명 |
|----------------|------|
| 높은 응집도 (Cohesion) | 각 모듈이 하나의 책임에 집중 |
| 낮은 결합도 (Coupling) | 모듈 간 의존성 최소화 |
| 적절한 캡슐화 | 구현 세부사항 숨김 |
| 명확한 인터페이스 | 사용법이 직관적 |

| 나쁜 설계의 징후 | 설명 |
|----------------|------|
| 경직성 | 변경이 어려움 |
| 취약성 | 한 곳 수정이 다른 곳 파괴 |
| 부동성 | 재사용 불가 |
| 점착성 | 올바른 방법보다 잘못된 방법이 쉬움 |

## 객체지향이 주는 해답

### 복잡성 관리 도구로서의 OO

| 객체지향이 복잡성을 다루는 방법 | 설명 |
|-----------------------------|------|
| 추상화 (Abstraction) | 클래스로 개념 표현. 인터페이스로 계약 정의 |
| 캡슐화 (Encapsulation) | 구현 숨김. 변경 영향 국소화 |
| 모듈성 (Modularity) | 클래스/패키지로 분리. 의존성 관리 |
| 계층 (Hierarchy) | 상속으로 공통성 추출. 합성으로 부분-전체 표현 |

### 관점 (Views)

**Booch의 뷰**:

| 뷰 | 내용 |
|----|------|
| 논리적 뷰 (Logical View) | 클래스, 객체, 관계. 기능적 요구사항 |
| 프로세스 뷰 (Process View) | 동시성, 동기화. 성능, 가용성 |
| 물리적 뷰 (Physical View) | 하드웨어 배치. 네트워크 토폴로지 |
| 개발 뷰 (Development View) | 소스 코드 조직. 빌드 의존성 |

각 뷰는 특정 이해관계자를 위한 것이다.

## 정리

- **본질적 복잡성**: 소프트웨어는 본질적으로 복잡하며, 이를 제거할 수 없다
- **분해**: 큰 문제를 작은 문제로 나눈다 (객체 분해 > 알고리즘 분해)
- **추상화**: 본질에 집중하고 세부사항을 숨긴다
- **계층**: is-a와 part-of로 구조화한다
- **모듈화**: 독립적인 단위로 분리한다
- **객체지향**: 복잡성 관리를 위한 효과적인 도구 세트
