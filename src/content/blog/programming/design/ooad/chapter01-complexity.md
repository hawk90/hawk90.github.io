## 한 줄 요약

> 소프트웨어는 **본질적으로 복잡**하다. 이 복잡성을 다루는 열쇠는 **분해(Decomposition)**와 **추상화(Abstraction)**이며, 객체지향은 이 두 가지를 자연스럽게 지원한다.

## 소프트웨어 복잡성의 본질

### 왜 소프트웨어는 복잡한가

프로젝트를 시작할 때는 단순해 보입니다. "사용자가 로그인하고, 상품을 보고, 주문한다." 요구사항 세 줄이면 될 것 같습니다. 그런데 6개월 후 코드베이스를 보면 클래스가 수백 개, 의존성은 스파게티처럼 얽혀 있습니다. 무엇이 잘못된 걸까요?

Fred Brooks는 1986년 논문 *No Silver Bullet*에서 이 질문에 답합니다. 소프트웨어의 복잡성은 두 종류로 나뉘고, 그중 하나는 원천적으로 제거할 수 없다는 것입니다.

| 복잡성 유형 | 특성 | 예시 |
|------------|------|------|
| 본질적 복잡성 (Essential) | 문제 자체의 복잡성. 현실 세계가 복잡하기 때문에 발생. 제거할 수 없음 | 세금 계산 규칙, 의료 진단 로직, 금융 규제 요건 |
| 우발적 복잡성 (Accidental) | 도구·언어·플랫폼의 한계에서 발생. 개선 가능 | 메모리 관리, 빌드 설정, 언어 문법의 장황함 |

**왜 중요한가?** 우발적 복잡성은 도구를 바꾸면 줄어듭니다. C에서 Java로, Java에서 Kotlin으로 옮기면 보일러플레이트가 줄어드는 식입니다. 하지만 본질적 복잡성은 다릅니다. 세금법이 복잡하면 세금 계산 코드도 복잡할 수밖에 없습니다. 아무리 좋은 언어를 써도 이 복잡성은 사라지지 않습니다.

Brooks의 결론은 유명합니다: **"은탄환은 없다."** 10배 생산성을 올려주는 마법의 도구는 없습니다. 본질적 복잡성을 제거할 수 없기 때문입니다. 할 수 있는 것은 *관리*뿐입니다. 그리고 객체지향은 그 관리 도구 중 하나입니다.

### 복잡성의 네 가지 원인

본질적 복잡성이 어디서 오는지 더 구체적으로 봅시다. Booch는 네 가지 원인을 제시합니다.

| 원인 | 설명 | 현실 예시 |
|------|------|----------|
| 문제 도메인의 복잡성 | 현실 세계 자체가 복잡함. 요구사항이 변하고 이해관계자가 다양함 | 병원 EMR 시스템 — 의사, 간호사, 행정, 보험사가 각자 다른 뷰를 원함 |
| 개발 프로세스의 어려움 | 팀 간 소통, 변경 관리, 품질 보증이 어려움 | 10명 팀의 코드 충돌, 기능 변경 시 사이드이펙트 추적 |
| 소프트웨어의 유연성 | 물리적 제약이 없어서 "할 수 있으니까 한다"가 됨 | "이 버튼 하나만 추가해주세요" → 구조 변경 없이 땜질 반복 |
| 이산 시스템의 특성 | 연속적이지 않아서 작은 변경이 큰 영향을 줌 | `if` 하나 잘못 넣어서 전체 배치 실패 |

네 번째 원인이 특히 중요합니다. 다리를 설계할 때는 하중이 5% 늘어나면 처짐도 대략 5% 늘어납니다. 연속적입니다. 그런데 소프트웨어는 다릅니다. 코드 한 줄이 시스템 전체를 멈출 수 있습니다. 이 비연속성 때문에 테스트로 *모든* 경우를 커버하는 것이 불가능합니다. 복잡성 관리가 필수인 이유입니다.

### 복잡한 시스템의 공통 속성

복잡성이 피할 수 없다면, 복잡한 시스템은 어떤 특성을 가질까요? Herbert Simon은 생물학, 사회학, 공학 등 다양한 분야의 복잡한 시스템을 연구하고 공통점을 발견했습니다. 흥미롭게도, 이 특성들은 소프트웨어 설계의 가이드라인이 됩니다.

| 특성 | 설명 | 소프트웨어에서의 의미 |
|------|------|---------------------|
| 계층적 구조 | 시스템은 하위 시스템으로 구성되고, 하위 시스템도 다시 분해됨 | 애플리케이션 → 모듈 → 클래스 → 메서드 |
| 상대적 기본 요소 | 관찰자에 따라 "기본 단위"가 달라짐 | 도메인 전문가에게 `Order`가 기본 단위, 개발자에게 `ArrayList`가 기본 단위 |
| 관심사 분리 | 컴포넌트 내부 상호작용이 컴포넌트 간 상호작용보다 많음 | 클래스 내부 응집도 > 클래스 간 결합도 |
| 공통 패턴 | 다른 도메인에서도 비슷한 구조가 반복됨 | Iterator, Observer, Factory 같은 디자인 패턴 |
| 중간 형태의 안정성 | 진화는 점진적이고, 안정된 중간 단계가 존재함 | 리팩터링은 작은 단계로, 각 단계에서 테스트 통과 |

다섯 번째 특성이 실무에서 중요합니다. 시계를 만든다고 가정합시다. 1000개 부품을 한 번에 조립하면, 중간에 방해받을 때마다 처음부터 다시 시작해야 합니다. 그런데 부품 10개로 하위 조립체를 만들고, 하위 조립체 10개로 상위 조립체를 만들면, 방해받아도 현재 단계만 다시 하면 됩니다. 소프트웨어도 마찬가지입니다. **안정된 중간 단계**(컴파일 성공, 테스트 통과)를 자주 만들어야 합니다.

## 복잡성 다루기

복잡성을 제거할 수 없다면 어떻게 해야 할까요? 핵심 도구는 두 가지입니다: **분해**와 **추상화**. 큰 문제를 작게 나누고, 세부사항을 숨기는 것입니다.

### 분해 (Decomposition)

분해는 큰 시스템을 작은 조각으로 나누는 것입니다. 그런데 *어떤 기준*으로 나눌지가 중요합니다. 역사적으로 두 가지 방식이 경쟁해왔습니다.

| 분해 방식 | 중심 질문 | 예시 |
|----------|----------|------|
| 알고리즘 분해 | "무엇을 *하는가*?" | `processOrder()` → `validateOrder()` → `calculateTotal()` → `saveOrder()` |
| 객체 분해 | "무엇이 *있는가*?" | `Order`, `Customer`, `Product`, `Payment` |

알고리즘 분해는 1960-70년대 구조적 프로그래밍의 핵심이었습니다. 문제를 단계별 절차로 쪼개는 것입니다. 직관적이지만, 요구사항이 바뀌면 절차 전체를 다시 짜야 하는 경우가 많습니다.

객체 분해는 다른 관점입니다. "주문 처리 시스템에 뭐가 있지?" → 주문, 고객, 상품, 결제. 이 *명사*들이 클래스가 됩니다. 각 클래스가 자기 데이터와 행동을 갖습니다.

### 왜 객체 분해인가

| 장점 | 설명 | 구체적 예 |
|------|------|----------|
| 현실 세계 모델링 | 도메인 전문가와 같은 언어로 소통 가능 | "주문이 결제를 기다리고 있다" → `order.awaitPayment()` |
| 변경 흡수 | 요구사항 변경 시 영향이 국소화됨 | 할인 정책 변경 → `DiscountPolicy` 클래스만 수정 |
| 재사용 촉진 | 객체가 자기 완결적이라 다른 맥락에서도 사용 가능 | `Money` 클래스를 주문, 환불, 정산에서 모두 사용 |
| 복잡성 관리 | 캡슐화로 내부 숨김, 인터페이스만 노출 | `Order`의 상태 전이 로직을 외부에서 몰라도 됨 |

핵심은 두 번째, **변경 흡수**입니다. 경험적으로 *데이터 구조*는 *동작*보다 안정적입니다. "주문에는 고객, 상품, 금액이 있다"는 10년이 지나도 비슷합니다. 그런데 "주문을 처리하는 절차"는 자주 바뀝니다. 할인 정책, 배송 방식, 결제 수단이 계속 추가되니까요. 객체 분해는 안정적인 것(데이터)을 축으로 삼기 때문에, 변경에 더 강합니다.

### 추상화 (Abstraction)

분해가 "나누기"라면, 추상화는 "숨기기"입니다. 본질적 특성에 집중하고 부수적 세부사항을 감춥니다.

지도를 생각해봅시다. 실제 서울에는 건물, 나무, 신호등, 사람이 있습니다. 그런데 지하철 노선도에는 역과 노선만 있습니다. 나머지는 *추상화*된 것입니다. 이 "생략"이 오히려 지도를 *유용*하게 만듭니다. 모든 것을 다 그리면 아무것도 볼 수 없으니까요.

```java
// 추상화 수준 예시

// 높은 추상화 — 비즈니스 언어
customer.placeOrder(cart);

// 중간 추상화 — 기술적 추상화
Order order = orderFactory.createFrom(cart);
orderRepository.save(order);
paymentGateway.charge(order.getTotal());

// 낮은 추상화 — 구현 세부사항
connection.execute(
    "INSERT INTO orders (customer_id, total) VALUES (?, ?)",
    customerId, total);
```

같은 동작을 세 수준으로 표현했습니다. 좋은 설계는 *각 계층에서 일관된 추상화 수준을 유지*합니다. `placeOrder()` 안에서 갑자기 SQL이 나오면 읽기 어렵습니다.

| 좋은 추상화의 조건 | 반대 상황 |
|------------------|----------|
| 의도가 명확함 | `doWork()`, `processData()` 같은 모호한 이름 |
| 세부사항 숨김 | 내부 자료구조를 그대로 노출 |
| 일관된 추상화 수준 | 고수준 메서드 안에 저수준 코드 혼재 |
| 직관적인 인터페이스 | 사용법이 복잡하거나 순서 의존성이 있음 |

## 복잡한 시스템의 구조

분해와 추상화를 *어떤 형태*로 조직할 것인가? 두 가지 구조적 도구가 있습니다: **계층**과 **모듈화**.

### 계층 (Hierarchy)

계층은 추상화를 *위아래로* 조직합니다. 두 종류가 있습니다.

| 계층 종류 | 관계 | 예시 | 언제 쓰는가 |
|----------|------|------|------------|
| "is-a" 계층 (상속) | 일반화-특수화 | `Vehicle` → `Car`, `Truck` | 공통 행동을 공유할 때 |
| "part-of" 계층 (합성) | 전체-부분 | `Car` → `Engine`, `Wheel` | 복잡한 객체를 조립할 때 |

```java
// is-a 계층 — 공통 행동 공유
abstract class Vehicle {
    abstract void move();
}
class Car extends Vehicle {
    @Override void move() { /* 바퀴로 이동 */ }
}

// part-of 계층 — 부품으로 조립
class Car {
    private Engine engine;      // Car has-a Engine
    private List<Wheel> wheels; // Car has-a Wheels
}
```

**둘 다 필요합니다.** 그런데 경험 많은 개발자는 "상속보다 합성을 선호하라"고 합니다. 상속은 강한 결합을 만들고, 부모 클래스 변경이 모든 자식에 영향을 주기 때문입니다. 합성은 더 유연합니다.

### 모듈화 (Modularity)

모듈화는 추상화를 *옆으로* 조직합니다. 시스템을 독립적인 조각으로 나누는 것입니다.

| 원칙 | 설명 |
|------|------|
| 독립 분리 | 각 모듈이 독립적으로 존재 |
| 명확한 책임 | 한 모듈은 한 가지 일을 함 |
| 최소 의존성 | 모듈 간 연결을 최소화 |

**왜 중요한가?** 모듈화가 잘 되어 있으면 다음이 가능합니다:

- **변경 격리**: 결제 모듈을 수정해도 주문 모듈에 영향 없음
- **병렬 개발**: 팀A가 UI 모듈, 팀B가 API 모듈을 동시에 개발
- **개별 테스트**: 모듈 단위로 테스트 가능
- **이해 용이**: 한 번에 한 모듈만 이해하면 됨

![E-commerce Module Decomposition](/images/blog/ooad/diagrams/ch01-module-decomposition.svg)

각 모듈은 자기 영역의 클래스들을 갖고, 다른 모듈과는 인터페이스로만 소통합니다.

### 모델의 역할

비행기를 설계할 때 실물 크기로 바로 만들지 않습니다. 먼저 축소 모형을 만들고, 풍동 실험을 하고, 시뮬레이션을 돌립니다. 소프트웨어도 마찬가지입니다. 코드로 바로 가기 전에 *모델*을 만듭니다.

| 모델 종류 | 질문 | 예시 |
|----------|------|------|
| 논리적 모델 | "무엇을 하는가?" | 클래스 다이어그램, 유스케이스 |
| 물리적 모델 | "어떻게 구현되는가?" | 배포 다이어그램, 컴포넌트 다이어그램 |

| 모델의 가치 | 설명 |
|------------|------|
| 복잡성 관리 | 전체를 한눈에 볼 수 있음 |
| 의사소통 촉진 | 개발자, 설계자, 이해관계자가 같은 그림을 봄 |
| 설계 결정 문서화 | 왜 이렇게 했는지 기록 |
| 검증 가능 | 코드 작성 전에 설계 검토 |

**주의**: 모델은 *현실의 단순화*입니다. 지도가 영토가 아니듯, 모델이 코드가 아닙니다. 모델이 코드와 동기화되지 않으면 오히려 혼란을 줍니다. "살아있는 문서"로 유지하거나, 의도적으로 특정 시점의 스냅샷으로 한정해야 합니다.

## 설계의 의미

### 좋은 설계란 무엇인가

"좋은 설계"를 어떻게 판단할까요? 기능이 동작하면 충분할까요? 아닙니다. 코드는 *한 번 쓰고 여러 번 읽고 수정*됩니다. 설계의 진짜 시험대는 *변경할 때*입니다.

| 설계의 목표 | 검증 질문 |
|------------|----------|
| 기능 요구사항 충족 | 원하는 대로 동작하는가? |
| 비기능 요구사항 충족 | 성능, 보안, 확장성을 만족하는가? |
| 변경에 유연 | 새 요구사항이 들어왔을 때 어디를 고쳐야 하는가? |
| 이해 용이 | 새 팀원이 합류했을 때 코드를 이해할 수 있는가? |
| 구현 가능 | 주어진 시간과 자원으로 만들 수 있는가? |

**설계는 선택입니다.** 모든 것을 완벽하게 할 수는 없습니다. 성능을 높이면 복잡해지고, 유연하게 만들면 이해하기 어려워집니다. 맥락에 맞는 최선의 트레이드오프를 찾는 것이 설계입니다.

### 설계 품질 지표

어떻게 좋은 설계와 나쁜 설계를 구분할까요? 몇 가지 징후가 있습니다.

| 좋은 설계의 징후 | 설명 | 확인 방법 |
|----------------|------|----------|
| 높은 응집도 (Cohesion) | 모듈 내 요소들이 밀접하게 관련됨 | 클래스 이름 하나로 책임을 설명할 수 있는가? |
| 낮은 결합도 (Coupling) | 모듈 간 의존성이 최소화됨 | 한 클래스 수정이 몇 개 파일에 영향을 주는가? |
| 적절한 캡슐화 | 구현 세부사항이 숨겨짐 | public 필드 없이 메서드로만 상호작용하는가? |
| 명확한 인터페이스 | 사용법이 직관적임 | 문서 없이 시그니처만 보고 사용할 수 있는가? |

| 나쁜 설계의 징후 | 설명 | 증상 |
|----------------|------|------|
| 경직성 (Rigidity) | 변경이 어려움 | 작은 수정에 연쇄 변경 필요 |
| 취약성 (Fragility) | 한 곳 수정이 다른 곳 파괴 | 고쳤는데 엉뚱한 곳에서 버그 발생 |
| 부동성 (Immobility) | 재사용 불가 | 다른 프로젝트에서 쓰려면 전부 복사해야 함 |
| 점착성 (Viscosity) | 올바른 방법보다 잘못된 방법이 쉬움 | "이거 제대로 하려면 너무 오래 걸려서 그냥 땜질했어요" |

마지막 징후, **점착성**이 특히 위험합니다. 개발자가 "올바른 방법"을 알지만, 그것이 너무 어려워서 편법을 선택하는 상황입니다. 이런 편법이 쌓이면 기술 부채가 됩니다.

## 객체지향이 주는 해답

지금까지 복잡성의 본질, 분해와 추상화, 계층과 모듈화를 봤습니다. 객체지향은 이 모든 것을 *언어 수준에서* 지원합니다.

### 복잡성 관리 도구로서의 OO

| 도구 | OO에서의 구현 | 효과 |
|------|-------------|------|
| 추상화 | 클래스로 개념 표현, 인터페이스로 계약 정의 | 세부사항 숨김, 의도 명확화 |
| 캡슐화 | `private` 필드, public 메서드 | 변경 영향 국소화 |
| 모듈성 | 패키지/모듈로 클래스 그룹화 | 의존성 관리, 병렬 개발 |
| 계층 | 상속으로 공통성 추출, 합성으로 조립 | 재사용, 확장 |

**핵심 통찰**: 이 네 가지는 *별개의 기법*이 아니라 *함께 작동*합니다. 클래스(추상화)가 내부를 숨기고(캡슐화), 패키지로 묶이며(모듈성), 상속/합성으로 연결됩니다(계층). 하나만 써서는 복잡성을 관리할 수 없습니다.

### 4+1 관점 (Views)

큰 시스템은 하나의 다이어그램으로 담을 수 없습니다. Booch는 네 가지 관점(+하나)을 제안합니다.

| 뷰 | 주요 관심사 | 이해관계자 |
|----|-----------|-----------|
| 논리적 뷰 (Logical) | 클래스, 객체, 관계 | 설계자, 개발자 |
| 프로세스 뷰 (Process) | 동시성, 동기화, 성능 | 시스템 통합자 |
| 물리적 뷰 (Physical) | 하드웨어 배치, 네트워크 | 시스템 엔지니어 |
| 개발 뷰 (Development) | 소스 코드 조직, 빌드 | 프로그래머 |
| (+1) 유스케이스 뷰 | 시스템 행동 시나리오 | 최종 사용자, 분석가 |

같은 시스템이지만, 관점에 따라 다르게 보입니다. 이 "다관점 접근"이 복잡성을 다루는 또 다른 전략입니다.

## 자주 하는 실수

복잡성을 다룰 때 흔히 빠지는 함정입니다.

| 실수 | 증상 | 해결 |
|------|------|------|
| **은탄환 기대** | "이 프레임워크 쓰면 문제 해결" | 본질적 복잡성은 도구로 사라지지 않음을 인정 |
| **과도한 추상화** | 실제 로직보다 추상 계층이 더 많음 | "지금 필요한가?"를 자문, YAGNI 적용 |
| **추상화 수준 혼합** | 고수준 메서드 안에 SQL, 저수준 API 호출 | 각 메서드의 추상화 수준을 일관되게 |
| **분해 없이 거대 클래스** | 2000줄짜리 `OrderService` | 책임별로 클래스 분리 |
| **과도한 분해** | 클래스당 메서드 1개, 한 줄짜리 클래스 | "이유 있는 분리"인지 검토 |
| **모델과 코드 불일치** | 문서의 다이어그램이 코드와 다름 | 모델을 업데이트하거나 버리거나, 중간 상태 금지 |

## 정리

- **본질적 복잡성은 제거할 수 없습니다.** 세금법이 복잡하면 세금 계산 코드도 복잡합니다. "은탄환은 없다."
- **분해로 큰 문제를 작게 나눕니다.** 알고리즘 분해(동사 중심)보다 객체 분해(명사 중심)가 변경에 강합니다.
- **추상화로 세부사항을 숨깁니다.** 각 계층에서 일관된 추상화 수준을 유지해야 합니다.
- **계층으로 위아래를 조직합니다.** is-a(상속)와 part-of(합성) 두 종류가 있습니다.
- **모듈화로 옆을 조직합니다.** 높은 응집도, 낮은 결합도가 목표입니다.
- **복잡한 시스템은 안정된 중간 단계가 필요합니다.** 리팩터링은 작은 단계로, 각 단계에서 테스트 통과.
- **객체지향은 이 모든 것을 언어 수준에서 지원합니다.** 추상화, 캡슐화, 모듈성, 계층이 함께 작동합니다.
