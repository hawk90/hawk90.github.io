## 이 챕터의 메시지

이 장은 "정책(policy)"과 "수준(level)"이라는 두 개념을 정의한다. 짧고 추상적이지만, 후속 장들의 토대다.

> **시스템은 정책들로 구성된다. 각 정책은 입력을 받아 출력을 만든다.**

좋은 아키텍처는 이 정책들을 **함께 변하는 것끼리 묶고**, **다른 시점에 변하는 것끼리 분리**한다.

## 정책이란

**정책(policy)** — 입력 → 출력 변환 규칙.

비즈니스 규칙은 모두 정책이다.

- "고객이 30일 이상 미납이면 연체 통보"
- "주문 합계 5만원 이상이면 무료 배송"
- "관리자 권한이 있어야 사용자 삭제 가능"

또한 데이터 변환도 정책이다.

- JSON → 도메인 객체
- DB row → 도메인 객체
- 도메인 객체 → HTML

시스템 전체는 정책의 그래프다. 입력이 한 정책을 통과해 다른 정책의 입력이 되고, 결국 어떤 출력으로 흘러간다.

## 수준이란

**수준(level)** — 입출력으로부터의 거리.

```
입력 ─→ [낮은 수준 정책] ─→ ... ─→ [높은 수준 정책] ─→ ... ─→ [낮은 수준 정책] ─→ 출력
```

- 입출력에 가까운 정책 = **낮은 수준** (디테일에 묶임)
- 입출력에서 먼 정책 = **높은 수준** (비즈니스 본질)

높은 수준의 정책은 더 일반적이고, 더 추상적이고, 더 안정적이다.

## ENCRYPT 예제

Martin이 책에서 드는 단순한 예. 카드 데이터를 암호화하는 프로그램.

```
입력 (카드 데이터) ─→ [translate] ─→ [encrypt] ─→ [write] ─→ 출력 (암호화된 데이터)
                       (낮은 수준)    (높은 수준)   (낮은 수준)
```

세 정책이 순서대로 실행된다. 가운데 `encrypt`가 핵심 비즈니스 로직 — **이 프로그램이 하는 일**. 양쪽의 `translate`와 `write`는 입출력 어댑터 — **디테일**.

수준의 정의에 따라:

- `translate`, `write` — 낮은 수준 (입출력 근처)
- `encrypt` — 높은 수준 (입출력에서 멈)

## 의존성 방향 — 다시

낮은 수준 정책이 높은 수준 정책에 의존해야 한다 (DIP).

```
[encrypt] (높은 수준)
   ↑                       ↑
   │ TranslateInterface    │ WriteInterface
   │                       │
[translate]              [write]
(낮은 수준)              (낮은 수준)
```

`encrypt`는 인터페이스만 알고, 어떤 구체적인 translate/write가 들어왔는지 모른다.

결과: 카드 데이터를 디스크에서 읽는 대신 네트워크에서 받아도, 결과를 파일이 아닌 DB에 써도, `encrypt`는 안 바뀐다.

## 정책의 묶기 — 함께 변하는 것

같은 수준의 정책들은 같이 묶을 수 있다. 다른 수준의 정책은 분리한다.

이게 12-14장의 컴포넌트 원칙들이다.

- **CCP** — 함께 변하는 정책들 (같은 수준 → 같은 컴포넌트)
- **CRP** — 따로 쓰이는 정책들 (다른 수준 → 다른 컴포넌트)

수준이 같은 정책들이 같은 변경 이유로 묶이는 경향이 있다.

## 정책의 컴포넌트화

수준이 비슷한 정책들을 같은 컴포넌트에 모으면, 컴포넌트의 수준이 그 정책들의 수준이 된다.

```
[높은 수준 컴포넌트] - 비즈니스 규칙
       ↑
       │
[중간 수준 컴포넌트] - use case orchestration
       ↑
       │
[낮은 수준 컴포넌트] - I/O, DB, UI
```

이 그림이 22장의 Clean Architecture 동심원으로 발전한다.

## 컴파일 시간과 수준

흥미로운 관찰 — 높은 수준 정책이 더 안정적이라는 것은 컴파일 시간에도 영향을 준다.

- 낮은 수준 정책 → 자주 변경 → 자주 재컴파일
- 높은 수준 정책 → 거의 안 변함 → 거의 재컴파일 안 함

만약 높은 수준이 낮은 수준에 의존한다면 — 낮은 수준 변경 시 높은 수준도 재컴파일된다. 의존성이 뒤집혀야 (DIP) 이 비효율이 사라진다.

## 정책의 진짜 분리

정책 분리의 진짜 기준은 **변경 이유**다.

- 두 정책이 같은 이유로 변하는가? → 같이 둔다
- 다른 이유로 변하는가? → 분리

이게 SRP + CCP의 통합 메시지다. 비슷해 보이는 정책도 다른 actor의 요구로 변한다면 분리해야 한다.

## 정리

- **정책** = 입력 → 출력 변환 규칙. 시스템은 정책의 그래프.
- **수준** = 입출력에서의 거리. 입출력 근처 = 낮은 수준, 비즈니스 본질 = 높은 수준
- 높은 수준 = 더 일반적, 더 안정적, 더 추상적
- **의존성은 낮은 수준 → 높은 수준** (DIP)
- 정책을 묶는 기준 = **변경 이유 동일성** (CCP, SRP)
