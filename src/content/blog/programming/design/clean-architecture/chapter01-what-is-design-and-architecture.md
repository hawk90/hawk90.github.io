## 이 챕터의 메시지

Martin은 책의 첫 챕터에서 단도직입적으로 묻는다 — **디자인과 아키텍처는 정말 다른 것인가?**

답은 "아니다"다. 둘 사이에 선을 긋는 행위는 인위적이다. 큰 그림 결정과 작은 디테일 결정은 같은 연속선 위에 놓여 있고, 어느 쪽도 다른 쪽 없이는 의미가 없다.

그렇다면 좋은 디자인 / 좋은 아키텍처를 어떻게 알 수 있는가? Martin은 단 하나의 지표를 제시한다.

> **시간이 지나도 변경 비용이 일정하게 유지된다.**

이 한 줄이 이 챕터의 전부다. 나머지는 이 지표를 증명하기 위한 사례와 그래프다.

## 핵심 내용

- 디자인과 아키텍처는 본질적으로 같다 — 인위적 구분이다
- 좋은 디자인의 지표는 단 하나, **변경 비용이 일정**
- 나쁜 디자인의 신호는 **시간이 지날수록 비용이 증가**
- "지금 빨리, 나중에 청소"는 거의 항상 거짓말이다
- 책임자(CTO/리드)는 일정이 아니라 **구조**를 지켜야 한다

## 디자인 vs 아키텍처

업계에서는 흔히 다음과 같이 구분한다.

- **아키텍처** — 큰 그림, 상위 결정 (모듈 경계, 통신 방식)
- **디자인** — 작은 결정, 하위 디테일 (클래스 구조, 함수 시그니처)

Martin은 이 구분이 의미가 없다고 본다. 이유는 단순하다.

- 큰 결정은 작은 결정들을 통해서만 실현된다
- 작은 결정들은 큰 결정의 의도를 따라야 한다
- 두 결정 사이를 가르는 명확한 경계선이 존재하지 않는다

집을 짓는 비유로 설명한다. 도면에는 "거실"이 있지만, 거실이 거실이 되려면 콘센트 위치, 문 크기, 조명, 가구 동선 같은 디테일이 받쳐 줘야 한다. 도면(아키텍처)과 디테일(디자인)을 분리해서 평가할 수 없다.

따라서 이 책에서 **아키텍처**는 "코드의 구조 전체"를 가리키는 단어로 쓰인다.

## 좋은 디자인의 유일한 지표

소프트웨어 시스템에서 좋은 디자인이란 무엇인가? Martin은 다음 사고 실험을 제시한다.

두 팀이 동일한 시스템을 만든다고 가정하자.

- **팀 A** — 코드가 어떻든 빠르게 기능을 찍어 낸다
- **팀 B** — 매번 깨끗하게 짜고, 테스트를 붙이고, 구조를 고려한다

초기에는 팀 A가 앞선다. 같은 시간에 두 배의 기능을 출시한다. 하지만 몇 달이 지나면 그래프가 뒤집힌다.

```
생산성
  │
  │  팀 A ─────╮
  │            ╲╲╲
  │            ╲╲╲╲
  │            ╲╲╲╲╲
  │  팀 B ─────────────────────
  │
  └─────────────────────────────→ 시간
```

팀 A는 시간이 갈수록 새 기능 하나를 만드는 비용이 커진다. 결국 같은 기능을 만들기 위해 팀 B의 10배, 100배 시간이 들게 된다.

**이게 나쁜 디자인이다.** 시간이 갈수록 비용이 증가한다.

좋은 디자인은 반대다. 새 기능을 추가하는 비용이 시스템이 커져도 거의 일정하게 유지된다. 이 한 가지 지표 외에 다른 지표는 부차적이다.

## 사례 — 어느 회사의 코드 베이스

Martin은 컨설팅에서 만난 회사 하나를 사례로 든다. 익명이지만 흔한 패턴이다.

- 첫 릴리스: 엔지니어 15명, 1.0 출시 — 회사 자랑거리
- 2년 후: 엔지니어 60명, 8.0 출시 — 점점 새 기능 출시가 느려진다
- 4년 후: 엔지니어 150명, 코드는 부풀어 올랐는데 신기능은 거의 안 나온다

같은 기간 엔지니어당 평균 비용은 계속 올랐다(인원이 늘었으니까). 새 기능당 비용을 그래프로 그리면 **지수적**으로 증가한다.

```
기능당 비용 ($)
  │                                      ╭─
  │                                  ╭───╯
  │                              ╭───╯
  │                        ╭─────╯
  │                  ╭─────╯
  │            ╭─────╯
  │       ╭────╯
  │  ─────╯
  └────────────────────────────────→ 릴리스 번호
```

여기서 회사는 답이 안 보인다. 엔지니어를 더 뽑아도 비용 곡선은 그대로 위로 향한다. 어느 시점에는 새 기능 하나 만드는 비용이 회사가 감당할 수 없는 수준이 된다.

**원인이 무엇이었나?** Martin은 늘 같은 답을 한다 — 초기에 "빨리 만들고 나중에 청소하자"는 결정.

## "빨리 만들고 나중에 청소"의 거짓말

이 문장은 거의 모든 팀에서 등장한다. Martin의 관찰에 따르면 이 약속은 거의 항상 거짓이다. 이유는 세 가지다.

**1. 청소할 시간은 영원히 오지 않는다.**

다음 릴리스, 다음 분기, 다음 해 — 압박은 줄어들지 않는다. 매번 새로운 "빨리"가 우선이 된다.

**2. 청소 비용은 시간이 갈수록 더 커진다.**

지저분한 코드 위에 새 기능이 쌓이면, 그 새 기능까지 함께 청소해야 한다. 청소를 미룰수록 청소가 더 어려워진다.

**3. 빠른 개발은 결국 느린 개발이다.**

지저분한 코드는 새 기능 개발도 느리게 만든다. "지저분하면 빠르다"는 가정 자체가 잘못됐다. 깨끗한 코드 위에서 새 기능을 만드는 속도가 항상 더 빠르다는 것이 Martin의 주장이다.

> "The only way to go fast is to go well."

빠르게 가는 유일한 방법은 잘 가는 것이다.

## 책임은 누구에게 있나

Martin은 챕터 끝에서 책임 소재를 다룬다. CTO, 엔지니어링 리드, 임원진은 흔히 다음과 같이 변명한다.

- "고객이 일정을 압박했다"
- "마케팅이 약속을 해 버렸다"
- "이사회가 출시를 못 미룬다고 했다"

Martin의 응답은 단호하다. **이런 일정 압박은 모든 조직에서 일어난다.** 차이를 만드는 것은 그 압박 앞에서 누군가가 구조의 가치를 지켜내는가 여부다.

엔지니어는 단순한 노동자가 아니다. 코드 베이스의 건강을 책임지는 **전문가**다. 외과의가 환자 보호자의 "빨리"라는 요구에 따라 손 씻기를 생략하지 않듯, 엔지니어도 일정 압박에 따라 구조의 무결성을 포기해서는 안 된다.

이 챕터의 마지막 문장은 그래서 다음과 같이 정리된다.

> 엔지니어가 코드의 구조를 지키지 않으면, 그 누구도 지키지 않는다.

## 정리

- **디자인과 아키텍처는 같다** — 크기만 다른 같은 활동이다
- 좋은 디자인의 지표는 단 하나 — **변경 비용이 일정**하게 유지된다
- 나쁜 디자인은 시간이 갈수록 비용이 **지수적으로 증가**한다
- "빨리 만들고 나중에 청소"는 거의 항상 거짓 — 청소는 오지 않고, 빠른 개발이 진짜 빠른 것도 아니다
- **빠르게 가는 유일한 방법은 잘 가는 것이다**
- 구조를 지키는 책임은 엔지니어 본인에게 있다
