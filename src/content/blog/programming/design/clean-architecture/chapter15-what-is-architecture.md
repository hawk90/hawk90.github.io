## 이 챕터의 메시지

Part V의 출발점. 이 챕터는 책 전체에서 가장 짧지만 가장 핵심적인 정의를 담는다.

> **The goal of software architecture is to minimize the human resources required to build and maintain the required system.**

소프트웨어 아키텍처의 목표는 시스템을 짓고 유지하는 데 드는 **인간 자원을 최소화**하는 것이다.

Martin의 정의가 의외로 단순하다. "성능"이나 "확장성"이 아니다. **사람의 시간과 노력**이다.

## 네 가지 비용

소프트웨어가 들이는 비용은 네 가지로 나뉜다.

1. **개발 (development)** — 처음 짓는 비용
2. **배포 (deployment)** — 사용자에게 전달하는 비용
3. **운용 (operation)** — 운영 환경에서 돌리는 비용
4. **유지보수 (maintenance)** — 변경하고 고치는 비용

좋은 아키텍처는 이 네 가지 모두에 좋아야 한다. 그런데 가장 큰 비용은 무엇일까?

**유지보수**다. 거의 항상.

- 개발은 한 번이지만 유지보수는 영원하다
- 한 시스템의 유지보수 비용 = 개발 비용의 10배 이상이 일반적

따라서 아키텍처는 **유지보수 비용 최소화**에 최우선을 둔다.

## 아키텍처의 진짜 목적 — 결정 연기

Martin이 반복해서 강조하는 핵심 아이디어.

> **A good architect maximizes the number of decisions not made.**

좋은 아키텍트는 **하지 않은 결정의 수를 최대화**한다.

무슨 뜻인가? 시스템에 대한 모든 결정을 가능한 한 늦게까지 미룬다는 것이다.

```
이른 결정              ↔        늦은 결정
─────────              ─        ────────
- 비즈니스 규칙                  - DB 선택
- 도메인 모델                    - UI 프레임워크
- 사용 사례                      - 웹 서버
                                 - 외부 API
```

이른 결정 — 비즈니스 규칙, 도메인 모델 — 은 시스템의 본질이라 일찍 결정해도 안 변한다. 늦은 결정 — DB, UI, 프레임워크 — 은 디테일이고 자주 변하므로 늦게 결정한다.

## 왜 결정을 연기하는가

결정을 늦출수록 다음이 가능해진다.

**1. 더 많은 정보를 가지고 결정**

프로젝트 시작 시점에는 모르던 것들 — 트래픽 패턴, 사용자 행동, 비즈니스 변화 — 을 알게 된다.

**2. 시장의 변화에 대응**

DB를 일찍 선택했는데 6개월 후 더 좋은 DB가 나오면 갈아 끼울 수 있어야 한다.

**3. 잘못된 결정의 비용 최소화**

결정을 늦추면 잘못된 결정의 영향이 작은 영역에 머문다.

## "선택지를 열어 두기"

아키텍처가 하는 일은 **선택지를 열어 두는 것**이다.

- DB를 정하지 않고 짤 수 있는가? → 그러면 나중에 정한다
- UI를 정하지 않고 짤 수 있는가? → 그러면 나중에 정한다
- Web framework를 정하지 않고 짤 수 있는가? → 그러면 나중에 정한다

이게 가능한 시스템은 다음과 같은 특성을 가진다.

- 핵심 정책이 디테일에서 분리되어 있다
- 디테일은 플러그인처럼 부착된다 (5장의 의존성 역전)
- 플러그인이 바뀌어도 핵심은 안 흔들린다

이게 Clean Architecture의 구조적 비전이다.

## 디바이스 / DB / Framework 독립

Martin은 세 가지 독립을 강조한다.

**1. 디바이스 독립**

옛날 컴퓨터에서는 어떤 장치(테이프, 카드)에 쓸지를 코드에서 직접 알아야 했다. 모던 OS의 파일 추상화가 이걸 해결했다 — 디바이스가 바뀌어도 코드는 그대로.

좋은 아키텍처도 마찬가지 — 어떤 디바이스 위에서 도는지 코드가 신경 쓰지 않는다.

**2. DB 독립**

비즈니스 로직이 어떤 DB를 쓰는지 알면 안 된다. SQL? NoSQL? 인메모리? 모두 같은 코드에서 동작해야 한다.

**3. Framework 독립**

Spring, Django, Rails — 프레임워크의 결정도 늦춰야 한다. 비즈니스 로직이 특정 프레임워크에 매여 있으면 그 프레임워크의 변화에 따라 시스템 전체가 흔들린다.

## 아키텍처의 균형 — 동작과 변경

2장의 두 가치가 다시 등장한다.

- **행동** — 시스템이 작동한다 (요구 충족)
- **구조** — 시스템을 변경할 수 있다

좋은 아키텍처는 둘 다 한다. 하지만 우선순위가 다르다 — **구조가 행동보다 더 중요**하다. 행동만 있고 구조가 없으면 시간이 지나며 행동도 못 한다.

따라서 아키텍트의 일은 다음을 동시에 만족시키는 것이다.

- **지금 작동하게** 만든다
- **앞으로 변경할 수 있게** 만든다

## 정리

- 아키텍처의 목표 — **사람의 시간과 노력 최소화**
- 네 비용 — 개발 / 배포 / 운용 / 유지보수. 그 중 **유지보수가 최대**
- 아키텍트의 일 — **결정을 가능한 한 연기**, 선택지를 열어 두기
- 핵심은 일찍 결정, 디테일은 늦게 결정
- 디바이스 / DB / Framework **모두 독립적**으로 짓는다
- 행동과 구조 둘 다 — 그러나 **구조가 더 중요**
