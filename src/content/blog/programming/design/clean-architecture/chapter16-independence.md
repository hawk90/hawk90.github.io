## 이 챕터의 메시지

좋은 아키텍처가 가능하게 해야 할 것은 무엇인가? Martin은 네 가지를 든다.

1. **Use case의 독립성** — 다른 use case를 안 건드리고 한 use case를 바꿀 수 있다
2. **운용의 독립성** — 한 모듈을 다른 모듈과 다른 환경에서 운용할 수 있다
3. **개발의 독립성** — 다른 팀이 다른 부분을 동시에 개발할 수 있다
4. **배포의 독립성** — 한 모듈만 따로 배포할 수 있다

## Use Case의 독립성

시스템의 핵심은 use case다. "고객이 주문을 생성한다", "사용자가 로그인한다" 같은 비즈니스 행위.

좋은 아키텍처에서는 각 use case가 **독립된 코드**로 표현된다. "주문 생성" use case의 코드가 한 곳에 모여 있고, 다른 use case와 분리되어 있다.

이게 가능하면 한 use case의 변경이 다른 use case에 영향을 주지 않는다. SRP의 컴포넌트 수준 적용이다.

## 운용의 독립성

같은 코드를 다른 환경에서 실행할 수 있어야 한다.

- 데스크톱 / 서버 / 모바일
- 단일 프로세스 / 멀티 프로세스 / 분산
- 모놀리스 / 마이크로서비스

비즈니스 로직이 특정 운용 환경에 매여 있으면 운용 모델을 바꿀 수 없다. 좋은 아키텍처는 운용 결정을 늦게 내릴 수 있게 만든다.

## 개발의 독립성

여러 팀이 동시에 개발할 수 있어야 한다.

- UI 팀 / 백엔드 팀 / DB 팀
- 마이크로서비스 팀들

이게 가능하려면 팀 사이에 명확한 **인터페이스 경계**가 있어야 한다. 한 팀이 다른 팀의 코드를 만지지 않고도 자기 일을 할 수 있게.

Conway's Law를 떠올리면 자연스럽다 — 시스템 구조는 조직 구조를 반영한다. 그 역도 성립한다 — 좋은 아키텍처는 좋은 조직 구조를 가능하게 한다.

## 배포의 독립성

한 모듈을 따로 배포할 수 있어야 한다.

- "오늘 결제 모듈만 새 버전을 올린다"
- "마케팅 페이지만 즉시 배포한다"
- "한 마이크로서비스만 롤백한다"

전체 시스템을 한 번에 배포해야 한다면, 작은 변경에도 큰 위험이 따른다. 모듈별 독립 배포가 가능하면 변경 단위를 작게 만들 수 있다.

## 디커플링의 세 모드

이 네 가지 독립성을 달성하는 메커니즘은 **디커플링**이다. Martin은 세 가지 모드를 든다.

### 1. Source level (소스 수준)

- 같은 코드 베이스, 같은 빌드, 같은 바이너리
- 다만 모듈을 잘 나누어 의존을 통제

가장 약한 형태. 한 바이너리이므로 배포 독립성은 없다. 개발 독립성도 제한적 — 모두가 같은 코드를 만진다.

다만 가장 단순하고 가장 빠르다. 작은 시스템에 적합.

### 2. Deployment level (배포 수준)

- 별도 모듈, 별도 바이너리 (jar, dll, .so)
- 같은 프로세스에서 실행
- 동적으로 로드

배포 독립성이 생긴다 — 한 모듈만 새로 컴파일/배포 가능. 그러나 같은 프로세스에서 도므로 운용 독립성은 아직 부분적.

중간 규모 시스템의 표준.

### 3. Service level (서비스 수준)

- 별도 프로세스, 별도 시스템
- 네트워크로 통신
- 마이크로서비스 / SOA

모든 독립성이 최대. 그러나 비용도 최대 — 네트워크 지연, 운영 복잡도, 분산 문제(eventual consistency, partition tolerance).

큰 규모 시스템에 적합. 작은 시스템에서는 과한 비용.

## 결정 연기로서의 디커플링

세 모드를 일찍 정할 필요는 없다. **좋은 아키텍처는 세 모드 중 어느 쪽으로든 갈 수 있게** 만든다.

```
초기:                나중:
Source Level         Deployment Level
                     또는 Service Level
```

처음에는 단순한 source level로 시작하되, 컴포넌트 경계를 잘 그어 둔다. 시스템이 자라면 deployment 또는 service level로 옮기는 것이 비교적 쉽다 — 경계가 이미 거기 있으니까.

이게 12장의 "마이크로서비스로 가는 길은 모듈러 모놀리스를 거친다"의 정확한 표현이다.

## 우연한 중복 vs 진짜 중복

7장 SRP에서 본 이슈가 다시 등장한다. 두 use case가 비슷한 코드를 쓴다고 합쳐서는 안 된다 — **다른 actor가 변경 요구를 한다면 분리가 맞다**.

"중복을 없애라(DRY)"는 단순한 권고가 아니다. **변경 이유가 같을 때만** 중복을 없앤다. 변경 이유가 다르면 같은 코드도 분리한다.

## 정리

- 좋은 아키텍처가 보장하는 4 독립성 — **use case / 운용 / 개발 / 배포**
- 디커플링의 3 모드 — **source / deployment / service**
- 일찍 결정하지 않는다 — 경계만 잘 그어 두고 모드는 나중에 선택
- **우연한 중복**을 합치지 않는다 — 변경 이유가 다르면 분리 유지
- 마이크로서비스는 마지막 모드 — 그 전에 source/deployment 모드를 거친다
