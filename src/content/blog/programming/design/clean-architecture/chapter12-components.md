## 이 챕터의 메시지

여기서부터 책의 시야가 클래스에서 **컴포넌트**로 확대된다. 컴포넌트는 무엇인가?

> **컴포넌트는 시스템 배포의 단위다.**

JAR, DLL, .so, gem, npm package — 한 단위로 빌드되고 한 단위로 배포되는 것. 시스템은 이런 컴포넌트들의 조립으로 만들어진다.

이 챕터는 컴포넌트가 어떻게 진화해 왔는지의 역사를 보여 준다. 그 진화의 끝에 **플러그인 아키텍처**가 있다.

## 컴파일과 링크의 역사

Martin은 자신의 경험을 인용한다. 1960~70년대의 프로그래밍은 다음과 같았다.

```
1. 소스 코드 카드 작성
2. 컴파일러로 입력 → object code 출력
3. 라이브러리 카드와 함께 링커에 입력
4. 실행 가능 바이너리 출력
5. 메모리에 로드해 실행
```

문제는 **메모리 주소가 컴파일 시점에 고정**된다는 것이었다. 한 모듈의 코드가 항상 같은 메모리 주소에 로드되어야 했다. 새 함수를 추가하면 모든 모듈의 주소가 밀려서 모든 코드를 다시 링크해야 했다.

작은 시스템은 괜찮았다. 큰 시스템은 빌드에 며칠이 걸렸다.

## Relocatable 바이너리

다음 진화는 **재배치 가능(relocatable) 바이너리**다. object code가 자기가 어디에 로드되는지 모르고, 로드 시점에 메모리 주소가 결정된다.

```
컴파일러 → relocatable object
              ↓
링커가 라이브러리들을 모음
              ↓
실행 시 로더가 메모리 주소 할당
```

이게 가능해지면서 한 모듈의 변경이 다른 모듈을 재컴파일하지 않아도 되게 됐다. 빌드 시간이 극적으로 줄었다.

여전히 모든 코드는 **빌드 시점**에 링크되어야 했다. 새 라이브러리를 추가하려면 다시 링크.

## 동적 링킹 — DLL / .so

진화의 다음 단계 — **동적 링킹**.

- Windows의 DLL
- Linux의 .so
- macOS의 .dylib
- Java의 .jar
- Python의 .whl, gem, npm packages

핵심 발견 — **실행 시점에 라이브러리를 로드**할 수 있다.

이게 가능해지면서 다음이 가능해졌다.

- 라이브러리를 빌드한 후 별도로 배포
- 실행 중에 새 모듈 로드
- 같은 인터페이스를 따르면 서로 교체 가능

이게 모던 **컴포넌트 시스템**의 토대다.

## 컴포넌트가 가져오는 진짜 가치

동적 링킹이 가능해진 결과는 단순히 "빌드가 빠르다"가 아니다. 더 깊은 결과가 있다.

**1. 독립적 개발**

서로 다른 팀이 다른 컴포넌트를 동시에 개발할 수 있다. 인터페이스만 일치하면 통합이 가능하다.

**2. 독립적 배포**

한 컴포넌트의 변경이 다른 컴포넌트를 재배포하게 강요하지 않는다.

**3. 플러그인 아키텍처**

5장에서 본 의존성 역전이 컴포넌트 수준에서 실현된다.

```
[핵심 정책 컴포넌트]
       ↑
       │ 인터페이스
       │
[플러그인 컴포넌트들] ← 실행 시점에 추가/교체 가능
```

핵심 정책은 인터페이스만 알고, 플러그인이 그 인터페이스를 구현한다. 실행 시점에 어떤 플러그인을 로드할지 결정한다.

## 컴포넌트와 모놀리스

"마이크로서비스 vs 모놀리스" 논쟁의 한 축이 여기 닿는다.

- **모놀리스** — 한 컴포넌트 (사실은 한 거대 바이너리)
- **모듈러 모놀리스** — 한 바이너리 내 여러 모듈
- **마이크로서비스** — 별도 바이너리, 별도 프로세스, 별도 배포

세 가지 모두 **컴포넌트 분리**의 다른 형태일 뿐이다. 본질은 같다 — 시스템을 독립 단위로 쪼개서 각각 별도로 발전 가능하게.

마이크로서비스의 부담을 모르는 게 아니라면, 모듈러 모놀리스가 종종 가장 실용적이다. 컴포넌트 경계만 명확하면 나중에 마이크로서비스로 쪼개기도 쉽다.

## 컴포넌트는 단순한 분리가 아니다

이 챕터의 메시지를 한 번 더 못 박는다.

컴포넌트는 단순히 "코드를 폴더로 나눈 것"이 아니다. **독립적으로 빌드, 배포, 진화할 수 있는 단위**다.

이 정의가 후속 챕터들의 기반이다. 13장(컴포넌트 응집도)과 14장(컴포넌트 결합)이 다룰 원칙들은 모두 이 "독립 단위로서의 컴포넌트"를 어떻게 만들 것인가의 문제다.

## 정리

- 컴포넌트 = **시스템 배포의 단위** (DLL, JAR, gem, npm package)
- 컴파일/링크의 역사 — 정적에서 **재배치 가능**으로, 그리고 **동적 링킹**으로
- 동적 링킹이 가능하게 만든 것 — **독립 개발, 독립 배포, 플러그인 아키텍처**
- 모놀리스 / 모듈러 모놀리스 / 마이크로서비스 — 모두 같은 본질의 다른 형태
- 컴포넌트는 단순한 폴더 분리가 아니라 **독립 진화 단위**
