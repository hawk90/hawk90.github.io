## 이 챕터의 메시지

아키텍처는 **선을 긋는 일**이다. 어디에 선을 그어 무엇을 무엇과 분리할 것인가.

Martin의 핵심 질문은 다음이다.

> **What kinds of decisions are premature?**

어떤 종류의 결정이 너무 이른가? 그 결정들에 우리 코드가 묶이지 않도록 선을 긋는다.

## 흔히 너무 이른 결정들

다음 결정들은 거의 항상 너무 이르다.

- **DB 선택** (MySQL? Postgres? MongoDB?)
- **웹 서버** (Apache? nginx?)
- **프레임워크** (Spring? Django? Rails?)
- **REST vs gRPC**
- **JSON vs Protobuf**
- **DI 컨테이너 선택**

이 결정들의 공통점 — **비즈니스 로직과 무관하다**. 우리 시스템이 무엇을 하는지와 별개의 결정이다.

따라서 이런 결정들은 시스템의 핵심(비즈니스 규칙)과 **분리된 곳**에 있어야 한다.

## FitNesse — 결정 연기의 사례

Martin은 자신이 만든 FitNesse(테스트 도구)의 예를 든다.

FitNesse를 만들 때 DB가 필요했다. 위키 페이지를 어디에 저장할까. MySQL, Oracle, ...

Martin은 결정을 **연기**했다. 대신 `WikiPagePersistence`라는 추상 인터페이스를 만들고, 그 뒤에 임시로 메모리 기반 구현을 두었다. "DB가 필요해질 때 다시 보자."

결과: **5년 동안 DB가 필요해지지 않았다**. 그리고 다른 사용자들이 다른 저장소 구현(파일, Git, 다른 위키 시스템과의 통합 등)을 추가했다. DB를 선택할 필요가 없었다.

> "We made the decision not to make the decision. We held it off until we knew more."

결정을 안 한 결정. 더 많이 알게 될 때까지 결정을 보류했다.

## 데이터베이스가 디테일이라는 것

이 챕터에서 가장 충격적일 수 있는 주장.

> **데이터베이스는 디테일이다.**

비즈니스 로직 입장에서 데이터가 메모리에 있든, MySQL에 있든, 파일에 있든, S3에 있든 상관없다. 그 결정은 디테일이고, 디테일은 비즈니스 로직 뒤에 와야 한다.

대부분의 시스템이 이걸 거꾸로 한다 — DB를 먼저 정하고, 그 위에 비즈니스 로직을 쌓는다. 결과적으로 DB 결정에 모든 코드가 매인다.

Clean Architecture의 메시지는 그 반대다 — **DB를 정하지 말고 시작**하라. 비즈니스 로직을 먼저 짓고, DB는 가장 마지막에 결정한다.

## 경계의 메커니즘 — 의존성 역전

경계를 만드는 도구는 11장의 DIP다.

```
[비즈니스 정책]
       ↑
       │ DataAccess Interface (정책 측이 소유)
       │
[데이터 액세스 구현]
   - MySqlDataAccess
   - MongoDbDataAccess
   - InMemoryDataAccess
```

비즈니스 정책은 인터페이스만 안다. 어떤 DB 구현체가 들어왔는지 모른다. 따라서 DB의 변경이 정책을 흔들지 못한다.

## 플러그인 아키텍처

이런 경계가 잘 그어진 시스템을 **플러그인 아키텍처**라 부른다.

```
        ┌──────────────────────┐
        │   비즈니스 로직 (안정) │
        └─────────▲────────────┘
                  │ 인터페이스
        ┌─────────┴────────────┐
        │      플러그인 영역      │
        │  ┌────┐ ┌────┐ ┌────┐ │
        │  │ DB │ │ UI │ │API │  │
        │  └────┘ └────┘ └────┘ │
        └──────────────────────┘
```

각 플러그인은 자유롭게 교체된다. 비즈니스 로직은 안 흔들린다.

## 흔한 비판 — "그러면 코드가 더 길어진다"

이 디자인의 비용은 인터페이스 추가의 보일러플레이트다. 한 줄짜리 데이터 접근이 인터페이스 + 구현체로 5줄이 된다.

Martin의 답: **그 비용은 미래의 변경 비용에 비하면 사소하다**. 한 번 짠 5줄 보일러플레이트는 그대로지만, DB를 바꾸는 한 번의 변경은 인터페이스 없으면 시스템 전체를 흔든다.

또 모든 곳에 경계를 그어야 하는 건 아니다. **변할 가능성이 있는 곳에만** 긋는다.

## 어디에 경계를 긋는가

Martin이 권하는 위치.

**확실한 곳**:
- 비즈니스 로직 ↔ 데이터베이스
- 비즈니스 로직 ↔ UI
- 비즈니스 로직 ↔ 외부 API

**상황에 따른 곳**:
- Use case 사이
- Microservice 사이
- 컴포넌트 사이

**보통 안 긋는 곳**:
- 클래스 내부의 작은 함수들
- 자주 함께 변하는 모듈 사이
- 일회용 스크립트

핵심은 **변경 가능성**이다. 변할 것 같으면 경계를 그어 격리한다.

## 정리

- 아키텍처 = **선을 긋는 일**
- 어떤 결정이 너무 이른가 — DB, framework, UI, ...
- **DB는 디테일** — 비즈니스 로직 뒤에 와야 한다
- 경계의 도구 — **의존성 역전 (DIP)**
- 결과 — **플러그인 아키텍처**
- 보일러플레이트 비용보다 **변경 비용 절감**이 크다
- 모든 곳에 경계를 그어야 하는 건 아니다 — **변할 곳에만**
