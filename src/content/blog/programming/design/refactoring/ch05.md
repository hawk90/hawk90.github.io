## 이 챕터의 메시지

5장은 짧은 안내서다. 책의 나머지(6~12장)는 약 60개의 리팩터링을 **카탈로그 형식**으로 정리한다. 그 카탈로그를 어떻게 읽고 사용할지를 이 짧은 장에서 설명한다.

핵심은 형식의 일관성이다. 모든 리팩터링은 같은 구조로 기술된다. 한 번 익히면 어떤 리팩터링이든 같은 방식으로 찾아 쓸 수 있다.

## 카탈로그 형식

각 리팩터링 항목은 다음 다섯 부분으로 구성된다.

### 1. 이름 (Name)

리팩터링을 부르는 이름. 이름이 곧 어휘다. 팀이 같은 이름을 공유하면 의사소통이 빨라진다.

> "Extract Function이 필요해" — 한 문장으로 의도 전달.

이름의 일관성을 위해 Fowler는 책 전반에서 같은 이름만 쓴다.

### 2. 스케치 (Sketch)

비포 / 애프터의 짧은 코드 시각화. 책의 가장 자주 보는 부분이다 — 카탈로그를 훑을 때 이름과 스케치만 봐도 어떤 변환인지 감이 온다.

```javascript
// Before
function printOwing(invoice) {
  printBanner();
  let outstanding = calculateOutstanding();
  console.log(`name: ${invoice.customer}`);
  console.log(`amount: ${outstanding}`);
}

// After
function printOwing(invoice) {
  printBanner();
  let outstanding = calculateOutstanding();
  printDetails(outstanding);
}

function printDetails(outstanding) {
  console.log(`name: ${invoice.customer}`);
  console.log(`amount: ${outstanding}`);
}
```

### 3. 동기 (Motivation)

**왜** 이 리팩터링을 하는가. 무엇이 더 좋아지는가. 언제는 하지 말아야 하는가.

이 부분이 카탈로그의 진짜 가치다. "어떻게"는 IDE가 해 주는 시대지만, "언제 / 왜"는 사람이 판단해야 한다.

### 4. 절차 (Mechanics)

리팩터링의 **단계별 절차**. 책의 다른 부분과 가장 다른 부분이다 — 매우 정확하게, 거의 기계적으로 따라할 수 있도록 기술된다.

```
1. 추출할 코드를 선택한다.
2. 추출한 코드를 담을 새 함수를 만든다.
3. 추출한 코드의 외부 변수를 찾는다.
   3.1. 읽기만 하는 변수는 매개변수로.
   3.2. 쓰는 변수가 있으면 처리를 별도로 고민한다.
4. 원래 위치에서 새 함수를 호출한다.
5. 테스트한다.
6. 다른 위치에서도 비슷한 코드가 있는지 확인한다.
   6.1. 있다면 같은 함수로 대체할 수 있는지 검토한다.
```

이 절차는 **숙련자용 체크리스트**다. 익숙해지면 머릿속에서 합쳐서 빠르게 실행한다. 다만 어려운 케이스에 부딪혔을 때 한 단계씩 정확하게 따르면 거의 항상 안전하게 나올 수 있다.

> "절차가 보수적인 이유는 큰 코드 베이스에서 안전하기 위해서다."

### 5. 예제 (Example)

실제 코드로 단계별 변환을 보여 준다. 1장의 워크스루와 같은 형식 — 한 단계씩, 매번 테스트.

## 카탈로그 읽는 법

Fowler가 권하는 사용법.

**1. 처음에는 전부 훑는다**

어떤 리팩터링이 있는지 이름과 스케치만 빠르게 본다. 모든 이름을 외울 필요는 없다 — "이런 게 있구나" 정도면 충분하다.

**2. 작업 중 필요할 때 다시 찾는다**

코드를 만지다가 "이거 어디선가 본 패턴인데" 싶을 때 카탈로그를 다시 펼친다. 이름이 떠오르면 해당 항목으로 바로 가고, 떠오르지 않으면 스케치를 훑어 본다.

**3. 절차는 어려울 때만 정독한다**

쉬운 케이스는 절차를 따로 보지 않아도 머릿속에서 처리된다. 어려운 케이스(상속이 얽혀 있거나, 부수 효과가 많거나) 일 때 절차를 한 단계씩 따른다.

## 카탈로그의 한계

Fowler는 카탈로그가 완전하지 않음을 인정한다.

**1. 모든 리팩터링을 다 담을 수 없다**

리팩터링은 무한히 다양하다. 책의 카탈로그는 자주 쓰이고, 일반적인 것들만 모았다. 도메인 특화 리팩터링은 각자 발견해야 한다.

**2. 이름의 정착이 완벽하지 않다**

Fowler가 1999년 1판에서 정한 이름이 2판(2018)에서도 유지된 게 대부분이지만, 그 사이 업계 용어가 달라진 부분도 있다. 책 안의 이름이 외부 세계의 이름과 100% 일치하진 않을 수 있다.

**3. 언어 의존성**

자바스크립트 예제 중심이지만, 일부 리팩터링은 다른 언어에서는 다르게 보인다(상속이 없는 언어, 정적 타입 언어, 함수형 언어 등).

## 카탈로그의 6개 그룹

6~12장은 다음과 같이 분류된다.

| 장 | 그룹 | 대표 리팩터링 |
|---|---|---|
| 6 | 첫 번째 묶음 | Extract Function, Inline Function, Extract Variable, Inline Variable, Change Function Declaration, ... |
| 7 | 캡슐화 | Encapsulate Record, Encapsulate Collection, Replace Primitive with Object, Extract Class, ... |
| 8 | 기능 이동 | Move Function, Move Field, Slide Statements, Split Loop, Replace Loop with Pipeline, ... |
| 9 | 데이터 조직화 | Split Variable, Rename Field, Change Reference to Value, ... |
| 10 | 조건문 단순화 | Decompose Conditional, Replace Nested Conditional with Guard Clauses, Replace Conditional with Polymorphism, ... |
| 11 | API 리팩터링 | Separate Query from Modifier, Remove Flag Argument, Replace Constructor with Factory Function, ... |
| 12 | 상속 다루기 | Pull Up Method, Push Down Method, Replace Type Code with Subclasses, Replace Subclass with Delegate, ... |

전체 60개. 이 묶음의 첫 번째 — 6장의 "첫 번째 묶음" — 이 가장 자주 쓰이는 핵심이다.

## 정리

- 카탈로그는 **이름 / 스케치 / 동기 / 절차 / 예제**의 5부 구조
- **이름**이 어휘 — 팀과의 의사소통 자산
- **동기**가 카탈로그의 진짜 가치 — IDE는 어떻게는 해 주지만 왜는 안 해 준다
- **절차**는 어려운 케이스용 체크리스트
- 처음엔 훑고, 필요할 때 찾고, 어려울 때만 정독
- 6장의 "첫 번째 묶음"이 가장 핵심
