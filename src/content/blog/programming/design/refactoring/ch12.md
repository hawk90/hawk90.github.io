## 이 챕터의 메시지

마지막 장은 **상속**을 다루는 리팩터링이다. 상속은 강력하지만 위험한 도구다. 잘못 쓰면 변경 비용을 지수적으로 키운다.

Fowler의 입장은 명확하다 — **상속을 처음부터 신중히 디자인하지 마라**. 단순하게 시작하고, 같은 일이 반복되는 게 보이면 그때 상속이나 위임으로 옮긴다. 그리고 어느 시점에 상속이 부담이 되면 다시 위임으로 바꾼다.

## Pull Up Method — 메서드 끌어 올리기

여러 자식 클래스에 같은 메서드가 있다면 부모로 올린다.

```javascript
// Before
class Employee {}
class Salesman extends Employee {
  get name() { return this._name; }
}
class Engineer extends Employee {
  get name() { return this._name; }
}

// After
class Employee {
  get name() { return this._name; }
}
```

**동기** — Duplicated Code 제거. 같은 코드가 여러 자식에 있으면 한 곳 고치면 다른 곳도 같이 고쳐야 한다 — 까먹기 쉽다.

## Pull Up Field — 필드 끌어 올리기

```javascript
// Before
class Employee {}
class Salesman extends Employee { _name; }
class Engineer extends Employee { _name; }

// After
class Employee { _name; }
```

**동기** — 같은 의미의 필드가 여러 자식에 있다.

## Pull Up Constructor Body — 생성자 본문 끌어 올리기

자식 생성자들이 공통으로 하는 일이 있다면 부모 생성자로 올린다.

```javascript
// Before
class Party {}
class Employee extends Party {
  constructor(name, id, monthlyCost) {
    super();
    this._id = id;
    this._name = name;        // 공통
    this._monthlyCost = monthlyCost;
  }
}
class Department extends Party {
  constructor(name, staff) {
    super();
    this._name = name;        // 공통
    this._staff = staff;
  }
}

// After
class Party {
  constructor(name) {
    this._name = name;
  }
}
class Employee extends Party {
  constructor(name, id, monthlyCost) {
    super(name);
    this._id = id;
    this._monthlyCost = monthlyCost;
  }
}
```

## Push Down Method / Push Down Field — 내려 보내기

반대 방향. 부모에 있지만 일부 자식만 쓰는 메서드/필드는 내려 보낸다.

```javascript
// Before
class Employee {
  get quota() { ... }  // 영업직만 사용
}

// After
class Employee {}
class Salesman extends Employee {
  get quota() { ... }
}
class Engineer extends Employee {}
```

**동기** — 부모에 있으면 모든 자식이 가진다는 신호다. 실제로는 일부만 쓰면 거짓 신호다.

## Replace Type Code with Subclasses — 타입 코드를 서브클래스로

```javascript
// Before
class Employee {
  constructor(name, type) {
    this._name = name;
    this._type = type;  // "engineer" / "manager" / "salesman"
  }
  get type() { return this._type; }
}

// After
class Employee {
  constructor(name) { this._name = name; }
}
class Engineer extends Employee {}
class Manager extends Employee {}
class Salesman extends Employee {}
```

**동기** — type code(`type === "engineer"`) 분기가 코드 도처에 흩어지는 패턴. 서브클래스로 만들면 다형성으로 그 분기를 제거할 수 있다 (10장의 Replace Conditional with Polymorphism으로 이어진다).

## Remove Subclass — 서브클래스 제거

서브클래스가 더 이상 의미를 갖지 않으면 제거한다.

```javascript
// Before
class Person { ... }
class Male extends Person { get isMale() { return true; } }
class Female extends Person { get isMale() { return false; } }

// After
class Person {
  constructor(name, genderCode) {
    this._name = name;
    this._genderCode = genderCode;
  }
  get isMale() { return "M" === this._genderCode; }
}
```

**동기** — 서브클래스가 거의 비어 있거나 단지 한 필드 값만 다르다면 굳이 클래스로 둘 필요 없다.

## Extract Superclass — 상위 클래스 추출

두 클래스가 비슷한 일을 한다면 공통 부모를 추출한다.

```javascript
// Before
class Department {
  get totalAnnualCost() { ... }
  get monthlyCost() { ... }
  get name() { ... }
}
class Employee {
  get annualCost() { ... }
  get monthlyCost() { ... }
  get name() { ... }
}

// After
class Party {
  get name() { ... }
  get annualCost() { ... }
}
class Department extends Party { get monthlyCost() { ... } }
class Employee extends Party { get monthlyCost() { ... } }
```

**동기** — 두 클래스의 메서드/필드가 비슷하면 공통 추상이 숨어 있을 가능성이 크다. 추출하면 그 추상이 명시된다.

다만 Fowler는 신중을 권한다. 추출은 강한 결합을 만든다 — 부모를 바꾸면 모든 자식이 영향을 받는다. **두 클래스가 정말 같은 본질**일 때만.

## Collapse Hierarchy — 계층 합치기

부모와 자식이 너무 비슷해서 따로 둘 의미가 없으면 합친다.

```javascript
// Before
class Employee {}
class Salesperson extends Employee {}  // 부모와 거의 같음

// After
class Employee {}  // Salesperson 흡수
```

**동기** — 한 계층의 진화 끝에서 종종 일어난다 — 추출했던 차이가 사라지거나, 자식이 더 이상 의미가 없어진 경우.

## Replace Subclass with Delegate — 서브클래스를 위임으로

가장 강력한(그리고 가장 복잡한) 리팩터링 중 하나.

```javascript
// Before — 상속
class Booking {
  constructor(show, date) { ... }
  get hasTalkback() { ... }
  get basePrice() { ... }
}
class PremiumBooking extends Booking {
  constructor(show, date, extras) { super(show, date); this._extras = extras; }
  get hasTalkback() { ... }  // override
  get basePrice() { ... }    // override
}

// After — 위임
class Booking {
  constructor(show, date) { ... }
  _bePremium(extras) { this._premiumDelegate = new PremiumBookingDelegate(this, extras); }
  get hasTalkback() {
    return (this._premiumDelegate)
      ? this._premiumDelegate.hasTalkback
      : ...;
  }
}
class PremiumBookingDelegate {
  constructor(host, extras) { this._host = host; this._extras = extras; }
  get hasTalkback() { ... }
  get basePrice() { ... }
}
```

**동기** — 상속은 단 한 축으로만 자식을 만들 수 있다. 두 축 이상의 변화(예: Booking의 premium 여부 + 다른 카테고리)가 필요하면 상속이 무너진다. 위임은 여러 축을 동시에 표현할 수 있다.

또 상속은 컴파일 타임에 고정된다. 위임은 런타임에 바꿀 수 있다 — 같은 booking 객체가 premium이 됐다가 안 됐다가.

GoF의 **"Favor composition over inheritance"** 원칙이 이 리팩터링의 정신이다.

## Replace Superclass with Delegate — 상위 클래스를 위임으로

```javascript
// Before
class List { ... }
class Scroll extends List {}

// After
class Scroll {
  constructor(...) { this._items = new List(); }
  get items() { return this._items; }
}
```

**동기** — Scroll이 List의 서브타입이 아닌데 단순히 List의 기능을 쓰기 위해 상속한 경우. is-a가 아니라 has-a라면 위임으로.

> "If `Bird` extends `Animal`, then a Bird **is an** Animal."

이 테스트에 떨어지면 상속이 아니다.

## 상속에 대한 Fowler의 입장

이 챕터를 통해 Fowler는 일관된 메시지를 전달한다.

- 상속을 너무 일찍 도입하지 마라
- 같은 일이 반복되면 그때 추출한다 (Extract Superclass / Pull Up)
- 상속이 부담이 되면 위임으로 옮긴다 (Replace Subclass with Delegate)
- 두 클래스가 정말 is-a 관계가 아니면 상속을 쓰지 않는다

상속과 위임은 흑백 선택이 아니다. 한 시점의 옳은 답이 다른 시점에는 틀린 답일 수 있다. 그래서 양방향 리팩터링이 모두 카탈로그에 있다.

## 정리

- **Pull Up / Push Down** — 메서드와 필드의 적절한 계층 위치
- **Replace Type Code with Subclasses** — type code 분기 제거의 발판
- **Extract Superclass** — 비슷한 클래스 둘의 공통 추상화
- **Collapse Hierarchy** — 부모/자식이 너무 비슷하면 합친다
- **Replace Subclass with Delegate** — 상속의 한계를 만나면 위임으로
- **Replace Superclass with Delegate** — is-a가 아니면 위임으로
- 상속은 **신중하게**, **늦게**, **되돌릴 수 있게**

## 시리즈 마무리

12장이 카탈로그의 마지막이다. 1장의 워크스루로 시작해, 원칙(2장), 냄새(3장), 테스트(4장), 카탈로그 소개(5장), 그리고 6~12장의 60여 개 리팩터링을 모두 거쳤다.

Fowler가 남긴 큰 메시지를 정리하면.

- 리팩터링은 **외부 동작 유지 + 작은 단계**
- 별도 시간이 아닌 **일상 작업**에 녹아 든다
- **테스트가 안전망** — 자신감 있는 변경의 전제
- **이름**이 가장 중요한 디자인 결정
- 큰 디자인을 미리 하지 말고, 변경이 필요할 때 리팩터링한다 (Evolutionary Design)
- 상속은 신중하게, 위임을 우선

이 메시지들이 모여 **모던 소프트웨어 엔지니어링의 한 축**을 만든다.
