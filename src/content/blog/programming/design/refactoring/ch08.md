## 이 챕터의 메시지

8장은 **코드의 위치**를 옮기는 리팩터링이다. 어떤 함수가 잘못된 클래스에 있다거나, 문장이 잘못된 순서로 있다거나, 한 루프가 두 가지 일을 하고 있다거나 — 이런 위치/구조의 문제를 푼다.

이름은 "Move"지만 본질은 **응집도**를 끌어올리는 작업이다. 같이 변하는 것은 같이 두고, 따로 변하는 것은 따로 둔다.

## Move Function — 함수 이동

함수를 다른 클래스로(또는 다른 모듈로) 옮긴다.

```javascript
// Before
class Account {
  get bankCharge() {
    let result = 4.5;
    if (this._daysOverdrawn > 0) result += this.overdraftCharge;
    return result;
  }
  get overdraftCharge() {
    if (this.type.isPremium) {
      const baseCharge = 10;
      if (this._daysOverdrawn <= 7) return baseCharge;
      else return baseCharge + (this._daysOverdrawn - 7) * 0.85;
    } else {
      return this._daysOverdrawn * 1.75;
    }
  }
}

// After — overdraftCharge가 AccountType의 책임
class AccountType {
  overdraftCharge(daysOverdrawn) {
    if (this.isPremium) { ... }
    else { return daysOverdrawn * 1.75; }
  }
}
class Account {
  get bankCharge() {
    let result = 4.5;
    if (this._daysOverdrawn > 0)
      result += this.type.overdraftCharge(this._daysOverdrawn);
    return result;
  }
}
```

**동기** — Feature Envy 냄새. `overdraftCharge`가 Account의 데이터보다 AccountType의 데이터에 더 의존한다. 그래서 AccountType의 책임이 더 자연스럽다.

## Move Field — 필드 이동

```javascript
// Before
class Customer {
  get plan() { return this._plan; }
  get discountRate() { return this._discountRate; }  // ⚠️ plan에 속함
}

// After
class Customer {
  get plan() { return this._plan; }
  get discountRate() { return this.plan.discountRate; }
}
class Plan {
  get discountRate() { return this._discountRate; }
}
```

**동기** — 필드가 한 클래스에 있지만 의미적으로 다른 클래스에 속한다. 필드 이동은 Move Function보다 까다롭다 — 데이터의 모든 접근 경로를 추적해야 한다.

## Move Statements into Function — 문장을 함수 안으로

함수 호출 직전(또는 직후)에 항상 같은 문장이 따라온다면, 그 문장을 함수 안으로 옮긴다.

```javascript
// Before
function renderPerson(outStream, person) {
  outStream.write(`<p>${person.name}</p>\n`);
  renderPhoto(outStream, person.photo);
  emitPhotoData(outStream, person.photo);
}

function listRecentPhotos(outStream, photos) {
  photos.forEach(p => {
    outStream.write("<div>\n");
    emitPhotoData(outStream, p);  // 동일 패턴
    outStream.write("</div>\n");
  });
}

// After
function emitPhotoData(outStream, photo) {
  outStream.write(`<p>title: ${photo.title}</p>\n`);
  outStream.write(`<p>location: ${photo.location}</p>\n`);
  outStream.write(`<p>date: ${photo.date.toDateString()}</p>\n`);
}
// 호출처가 함수에 흡수
```

**동기** — 중복 제거.

## Move Statements to Callers — 문장을 호출자로

반대 방향. 함수 안에서 일부 호출자만 필요로 하는 일을 한다면 그 문장을 호출자로 빼낸다.

```javascript
// Before
function renderPerson(outStream, person) {
  renderPhoto(outStream, person.photo);
  outStream.write(`<p>title: ${person.photo.title}</p>\n`);
  emitPhotoData(outStream, person.photo);  // 호출자가 두 가지 처리를 한다
}

// After — 분리
function renderPerson(outStream, person) {
  renderPhoto(outStream, person.photo);
  emitPhotoData(outStream, person.photo);
}
```

**동기** — 함수의 책임이 일관되지 않을 때.

## Replace Inline Code with Function Call — 인라인 코드를 함수 호출로

```javascript
// Before
let appliesToMass = false;
for(const s of states) {
  if (s === "MA") appliesToMass = true;
}

// After
appliesToMass = states.includes("MA");
```

**동기** — 표준 라이브러리에 이미 같은 기능이 있는데 손으로 다시 짠 경우. 표준이 더 빠르고, 더 정확하고, 다른 사람에게 의도가 더 명확하다.

## Slide Statements — 문장 슬라이드

문장의 순서를 바꾼다. 연관된 문장들을 가까이 둔다.

```javascript
// Before
const pricingPlan = retrievePricingPlan();
const order = retreiveOrder();
let charge;
const chargePerUnit = pricingPlan.unit;

// After — 연관 묶기
const pricingPlan = retrievePricingPlan();
const chargePerUnit = pricingPlan.unit;
const order = retreiveOrder();
let charge;
```

**동기** — Extract Function의 사전 작업. 또는 단순히 가독성. 한 변수의 선언과 사용 사이의 거리를 좁힌다.

## Split Loop — 루프 분리

한 루프가 두 가지 일을 한다면 분리한다.

```javascript
// Before
let totalSalary = 0;
let youngestAge = Number.MAX_VALUE;
for (const p of people) {
  if (p.age < youngestAge) youngestAge = p.age;
  totalSalary += p.salary;
}

// After
let totalSalary = 0;
for (const p of people) totalSalary += p.salary;

let youngestAge = Number.MAX_VALUE;
for (const p of people) if (p.age < youngestAge) youngestAge = p.age;
```

성능 손해처럼 보인다 — 루프를 두 번 돈다. Fowler의 답: **측정 전에는 걱정하지 마라**. 분리하면 각 루프의 의도가 명확해지고, 필요시 Extract Function이 쉬워진다.

> "Frequently, splitting a loop allows me to recognize additional refactoring opportunities."

## Replace Loop with Pipeline — 루프를 파이프라인으로

```javascript
// Before
const names = [];
for (const i of input) {
  if (i.job === "programmer")
    names.push(i.name);
}

// After
const names = input
  .filter(i => i.job === "programmer")
  .map(i => i.name);
```

**동기** — Loops 냄새. 파이프라인이 의도를 더 잘 드러낸다. 필터, 변환, 집계 같은 단계가 이름으로 표시된다.

다만 모든 루프가 파이프라인이 되는 건 아니다. early return / break / 상태가 누적되는 복잡한 루프는 그대로 두는 게 낫다.

## Remove Dead Code — 죽은 코드 제거

```javascript
// Before
if (false) {
  doSomethingThatUsedToMatter();
}

// After
// (지움)
```

**동기** — 죽은 코드는 거짓말한다. 누군가 그게 의미 있다고 생각하고 시간을 쓴다. 버전 관리(git)에 항상 남아 있으니 망설일 필요 없이 지운다.

## 위치 이동의 가치

이 챕터의 리팩터링들은 코드의 모양을 거의 안 바꾼다 — 같은 코드가 다른 곳에 있게 한다. 그런데 그 위치 차이가 **응집도**를 결정한다.

좋은 코드는 **함께 변하는 것이 함께 있는 코드**다.

- 같이 변하는 함수와 데이터 → 같은 클래스
- 같이 변하는 문장들 → 인접한 위치
- 다른 책임 → 다른 곳

Move Function / Move Field가 이걸 가능하게 한다.

## 정리

- 위치 이동의 본질은 **응집도** — 같이 변하는 건 같이 둔다
- **Move Function / Move Field** — Feature Envy 해결
- **Move Statements** 양방향 — 호출자와 피호출자의 책임 분배
- **Slide Statements** — 변수 선언과 사용을 가까이
- **Split Loop** — 한 루프 한 책임
- **Replace Loop with Pipeline** — 의도를 단계 이름으로 드러낸다
- **Remove Dead Code** — 버전 관리가 있으니 망설이지 마라
