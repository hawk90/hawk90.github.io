## 이 챕터의 메시지

9장은 **데이터의 모양**을 다듬는 리팩터링이다. 변수 하나에 두 가지 의미가 섞여 있다거나, 필드 이름이 의도와 어긋난다거나, reference로 다룰 것을 value로 다루고 있다거나 — 이런 데이터 차원의 문제를 다룬다.

핵심은 **하나의 데이터는 하나의 의미**다.

## Split Variable — 변수 분리

한 변수가 두 가지 다른 값을 담는다.

```javascript
// Before
let temp = 2 * (height + width);
console.log(temp);
temp = height * width;
console.log(temp);

// After
const perimeter = 2 * (height + width);
console.log(perimeter);
const area = height * width;
console.log(area);
```

**동기** — temp라는 변수가 두 가지 일에 재사용되면 어디서 어떤 의미인지 추적해야 한다. 분리하면 각각의 의미가 이름으로 드러난다.

루프 변수(`i`)나 누적 변수(`sum`)는 의도된 재사용이므로 분리 대상이 아니다.

## Rename Field — 필드 이름 변경

```javascript
// Before
class Organization {
  get name() { ... }
}

// After
class Organization {
  get title() { ... }  // 도메인에서 더 정확한 용어
}
```

**동기** — 코드를 읽는 사람의 정신 모델은 이름을 통해 만들어진다. 이름이 부정확하면 잘못된 가정이 누적된다.

발행된 데이터 구조(API, DB 스키마, 파일 포맷)에서의 rename은 점진적 마이그레이션 필요. 내부 필드는 IDE의 rename으로 끝.

## Replace Derived Variable with Query — 파생 변수를 질의로

다른 데이터에서 계산할 수 있는 변수는 변수로 두지 말고 함수로 만든다.

```javascript
// Before
class ProductionPlan {
  constructor() {
    this._production = 0;
    this._adjustments = [];
  }
  get production() { return this._production; }  // ⚠️ 파생 가능
  applyAdjustment(adjustment) {
    this._adjustments.push(adjustment);
    this._production += adjustment.amount;
  }
}

// After
class ProductionPlan {
  constructor() {
    this._adjustments = [];
  }
  get production() {
    return this._adjustments.reduce((sum, a) => sum + a.amount, 0);
  }
  applyAdjustment(adjustment) {
    this._adjustments.push(adjustment);
  }
}
```

**동기** — 파생 데이터는 동기화 위험을 가진다. 한 곳에서 업데이트를 깜빡하면 일관성이 깨진다. 매번 계산하면 그 위험이 사라진다 — 성능 손실은 보통 측정 가능한 수준이 아니다.

## Change Reference to Value — 참조를 값으로

같은 데이터에 대한 변경이 한 곳에서만 일어나도록 — 즉 reference로 공유하던 것을 value로 바꾼다.

```javascript
// Before
class Person {
  constructor() {
    this._telephoneNumber = new TelephoneNumber();
  }
  get officeAreaCode() { return this._telephoneNumber.areaCode; }
  set officeAreaCode(arg) { this._telephoneNumber.areaCode = arg; }  // 내부 객체 수정
}

// After
class Person {
  get officeAreaCode() { return this._telephoneNumber.areaCode; }
  set officeAreaCode(arg) {
    this._telephoneNumber = new TelephoneNumber(arg, this.officeNumber);
  }
}
class TelephoneNumber {
  constructor(areaCode, number) {  // 불변
    this._areaCode = areaCode;
    this._number = number;
  }
  get areaCode() { return this._areaCode; }
  get number() { return this._number; }
  equals(other) { /* areaCode와 number가 같으면 equal */ }
}
```

**동기** — Value Object의 가치. 불변이라 공유해도 안전하고, equality가 의미 있고, 코드 추적이 쉽다. C++의 가이드라인 22(값 의미론)와 같은 정신.

## Change Value to Reference — 값을 참조로

반대 방향. 같은 데이터의 인스턴스가 여러 곳에 흩어져 있으면 reference로 공유한다.

```javascript
// Before
let order1 = new Order({customer: "alice"});  // customer를 각자 보유
let order2 = new Order({customer: "alice"});

// After
let alice = new Customer("alice");
let order1 = new Order({customer: alice});
let order2 = new Order({customer: alice});
// 한 곳에서 customer 갱신하면 모두 반영
```

**동기** — 한 실체에 대한 정보가 여러 인스턴스에 흩어져 있으면 일관성이 깨진다. 공유 reference로 만들면 한 곳에서 갱신한다.

다만 reference 공유는 디버깅을 어렵게 하기도 한다 — 의도된 변경이 다른 곳에 영향을 줄 수 있다. 둘 사이의 선택은 도메인이 결정한다.

## Value vs Reference

이 둘은 9장의 핵심 짝이다. 같은 데이터를 어떻게 다룰지 결정한다.

| | Value | Reference |
|---|---|---|
| 동일성 | 필드 값 비교 | 같은 인스턴스 |
| 불변 | 보통 불변 | 보통 mutable |
| 공유 | 안전 (불변이라) | 신중 (변경 영향) |
| 예 | 좌표, 돈, 날짜 범위 | Customer, Account, Project |

도메인이 답을 준다. 좌표는 값이다 — `(3, 4)`는 어디서든 `(3, 4)`다. 고객은 reference다 — 같은 고객의 주소가 바뀌면 모든 주문이 갱신을 봐야 한다.
