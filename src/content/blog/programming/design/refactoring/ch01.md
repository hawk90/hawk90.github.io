## 이 챕터의 메시지

Fowler는 책의 첫 챕터를 통째로 **하나의 워크스루**에 할애한다. 이론을 늘어놓는 대신, 그저 한 함수를 가지고 리팩터링을 끝까지 따라가게 만든다. 이유는 단순하다 — 리팩터링은 **리듬**이 본질이고, 리듬은 코드를 한 줄 한 줄 옮겨 가면서만 익혀진다.

핵심 메시지는 세 가지다.

- **작게**, **자주** 옮긴다 — 한 번에 큰 변경을 하지 않는다
- 매 단계마다 **테스트**가 통과해야 한다
- 매 단계마다 **커밋**할 수 있어야 한다

이 챕터를 다 읽고 나서 남는 것은 어떤 특정 기법의 이름이 아니다. "변경은 이렇게 작게 쪼개도 되는구나"라는 감각이다.

## 출발점 — 연극 청구서

극단이 손님(고객)에게 공연 청구서를 발급한다. 비극과 희극 두 장르가 있고, 각각의 가격 계산 규칙이 다르다. 또 관객 수에 따른 보너스 크레딧이 단골 할인용으로 따로 누적된다.

원본 코드는 자바스크립트로 짜인 단일 함수다.

```javascript
function statement(invoice, plays) {
  let totalAmount = 0;
  let volumeCredits = 0;
  let result = `Statement for ${invoice.customer}\n`;
  const format = new Intl.NumberFormat("en-US",
                        { style: "currency", currency: "USD",
                          minimumFractionDigits: 2 }).format;

  for (let perf of invoice.performances) {
    const play = plays[perf.playID];
    let thisAmount = 0;

    switch (play.type) {
      case "tragedy":
        thisAmount = 40000;
        if (perf.audience > 30) {
          thisAmount += 1000 * (perf.audience - 30);
        }
        break;
      case "comedy":
        thisAmount = 30000;
        if (perf.audience > 20) {
          thisAmount += 10000 + 500 * (perf.audience - 20);
        }
        thisAmount += 300 * perf.audience;
        break;
      default:
        throw new Error(`unknown type: ${play.type}`);
    }

    // add volume credits
    volumeCredits += Math.max(perf.audience - 30, 0);
    // add extra credit for every ten comedy attendees
    if ("comedy" === play.type) volumeCredits += Math.floor(perf.audience / 5);

    // print line for this order
    result += ` ${play.name}: ${format(thisAmount / 100)} (${perf.audience} seats)\n`;
    totalAmount += thisAmount;
  }
  result += `Amount owed is ${format(totalAmount / 100)}\n`;
  result += `You earned ${volumeCredits} credits\n`;
  return result;
}
```

40줄짜리 함수다. 짧다. "딱히 안 좋아 보이지도 않는다"가 첫인상이다. 그러나 곧 두 가지 요구가 들어온다.

- **HTML 청구서**가 필요하다 — 같은 계산 결과를 다른 형식으로 출력해야 한다
- **새 장르**(역사극, 목가극 등)를 추가해야 한다 — 가격과 보너스 규칙이 또 다르다

이 두 변경이 들어오는 순간, 위 함수는 곧바로 부담스러워진다. 출력 형식과 계산 로직이 한 덩어리이고, 장르별 분기가 switch 하나에 묶여 있다. **변경 비용이 폭발할 위치**가 그대로 드러난다.

## 가장 먼저 — 테스트

Fowler가 한 줄도 손대기 전에 한 일은 **테스트를 만드는 것**이다.

```javascript
test("statement", () => {
  expect(statement(invoice, plays)).toEqual(expected);
});
```

리팩터링은 외부 동작을 유지한 채 내부 구조만 바꾸는 작업이다. 외부 동작을 유지했는지를 알려 줄 도구가 없으면 그건 그냥 "코드를 만지다가 깨진" 것이다. 테스트가 안전망 역할을 한다.

이 챕터의 모든 단계가 같은 패턴을 반복한다.

1. 아주 작은 변경을 한다
2. 테스트를 돌린다
3. 통과하면 커밋한다

## 첫 추출 — switch를 함수로

가장 먼저 눈에 들어오는 덩어리는 `switch` 블록이다. 이걸 별도 함수로 빼낸다 — **Extract Function**.

```javascript
function amountFor(perf, play) {
  let thisAmount = 0;
  switch (play.type) {
    case "tragedy":
      thisAmount = 40000;
      if (perf.audience > 30) {
        thisAmount += 1000 * (perf.audience - 30);
      }
      break;
    case "comedy":
      thisAmount = 30000;
      if (perf.audience > 20) {
        thisAmount += 10000 + 500 * (perf.audience - 20);
      }
      thisAmount += 300 * perf.audience;
      break;
    default:
      throw new Error(`unknown type: ${play.type}`);
  }
  return thisAmount;
}
```

원래 함수는 이 함수를 호출하기만 한다.

```javascript
const thisAmount = amountFor(perf, play);
```

테스트 통과, 커밋. 끝.

여기서 중요한 디테일이 하나 있다. `thisAmount`라는 이름은 원래 함수의 문맥에선 의미 있었지만, 새 함수 안에서는 무의미하다. 책은 곧바로 변수명을 `result`로 바꾼다. **함수에서 반환되는 값의 변수명은 항상 `result`** — Fowler의 작은 규칙 하나다.

## 인라이닝 — play 변수 제거

`amountFor` 안에서 `play`는 매번 `plays[perf.playID]`로 다시 계산할 수 있는 값이다. 즉 함수의 입력에서 **유도 가능한 변수**다.

이런 경우 **Replace Temp with Query**를 쓴다. play를 변수로 두지 않고, 필요할 때 조회하는 함수로 바꾼다.

```javascript
function playFor(perf) {
  return plays[perf.playID];
}
```

이제 `amountFor`는 `play` 매개변수가 필요 없다.

```javascript
function amountFor(perf) {
  let result = 0;
  switch (playFor(perf).type) {
    // ...
  }
  return result;
}
```

매개변수가 하나 줄었다. 한 번 더 작은 단계. 한 번 더 테스트, 커밋.

이게 챕터 전체의 진짜 메시지다 — "매개변수 하나 줄이려고 이렇게까지?"가 아니라, "이렇게 작은 단계로 옮겨도 안전하고, 옮긴 만큼 다음 변경이 쉬워진다"는 것.

## 형식과 계산을 분리

여러 단계의 Extract Function과 Inline Variable을 반복하면 `statement`는 점점 얇아진다. 결국 다음과 같은 구조가 된다.

```javascript
function statement(invoice, plays) {
  let result = `Statement for ${invoice.customer}\n`;
  for (let perf of invoice.performances) {
    result += ` ${playFor(perf).name}: ${usd(amountFor(perf))} (${perf.audience} seats)\n`;
  }
  result += `Amount owed is ${usd(totalAmount())}\n`;
  result += `You earned ${totalVolumeCredits()} credits\n`;
  return result;
}
```

이 시점이 결정적이다. **출력**과 **계산**이 분리됐다. 처음 들어왔던 요구사항 "HTML 청구서"가 자연스럽게 풀린다.

```javascript
function htmlStatement(invoice, plays) {
  return renderHtml(createStatementData(invoice, plays));
}

function statement(invoice, plays) {
  return renderPlainText(createStatementData(invoice, plays));
}
```

같은 데이터를 두 가지 방식으로 출력한다. 새 형식이 들어와도 계산 로직은 한 번도 건드리지 않는다.

## 장르 추가 — 다형성 도입

두 번째 요구였던 "새 장르 추가"는 switch가 가로막는다. 비극·희극·역사극·목가극을 한 switch로 묶는 것은 OCP 위반(가이드라인 5의 그 OCP)이다.

해법은 **Replace Conditional with Polymorphism**이다.

```javascript
class PerformanceCalculator {
  constructor(perf, play) { this.performance = perf; this.play = play; }
  get amount() { throw new Error("subclass responsibility"); }
  get volumeCredits() { return Math.max(this.performance.audience - 30, 0); }
}

class TragedyCalculator extends PerformanceCalculator {
  get amount() {
    let result = 40000;
    if (this.performance.audience > 30) {
      result += 1000 * (this.performance.audience - 30);
    }
    return result;
  }
}

class ComedyCalculator extends PerformanceCalculator {
  get amount() {
    let result = 30000;
    if (this.performance.audience > 20) {
      result += 10000 + 500 * (this.performance.audience - 20);
    }
    result += 300 * this.performance.audience;
    return result;
  }
  get volumeCredits() {
    return super.volumeCredits + Math.floor(this.performance.audience / 5);
  }
}
```

이제 새 장르를 추가하려면 새 calculator를 한 개 만들면 된다. switch가 사라지는 게 아니라 **하나로 모인다** — 객체를 생성하는 factory 한 곳에만 남는다.

```javascript
function createPerformanceCalculator(perf, play) {
  switch (play.type) {
    case "tragedy": return new TragedyCalculator(perf, play);
    case "comedy":  return new ComedyCalculator(perf, play);
    default: throw new Error(`unknown type: ${play.type}`);
  }
}
```

이 switch는 받아들일 만하다. 어디서 분기가 일어나는지 한 곳에 모여 있고, 새 장르 추가가 곧 새 클래스 추가다.

## 리듬

이 챕터 전체에서 Fowler가 강조하는 건 **리듬**이다.

- 작은 변경 → 테스트 → 커밋
- 큰 변경은 **여러 개의 작은 변경**의 합
- 중간 단계에서도 코드는 항상 동작 가능한 상태

이상하게 들리는 부분 하나 — Fowler는 "이런 작은 단계로 옮기는 것은 학습용 과장이 아니다. **내가 실제로 이렇게 작업한다**"라고 못 박는다. 숙련자일수록 단계가 더 작다는 게 그의 주장이다.

이유는 단순하다. 작은 단계는 **실패 시 되돌리기가 쉽다**. 한 시간 작업한 큰 변경이 깨졌을 때는 어디서 깨졌는지 찾기 어렵지만, 5분짜리 변경이 깨지면 직전 커밋으로 되돌리고 다시 시작하면 된다.

## 흔한 오해

이 챕터를 읽고 사람들이 자주 빠지는 오해 둘.

**오해 1 — 모든 코드를 이렇게까지 쪼개야 한다**

아니다. 이 챕터의 worked example은 **요구사항 변경**이라는 동기에서 출발했다. 변경이 들어오지 않을 코드는 그대로 두는 게 맞다. 리팩터링은 **변경을 쉽게 만드는 준비 작업**이다.

**오해 2 — 한 번에 다 리팩터링하고 끝낸다**

아니다. 보통은 **변경을 하려고 보니 어렵다 → 변경하기 쉽게 만든다 → 변경한다** 순서다. 챕터의 예제도 두 요구사항(HTML / 새 장르)을 동기로 두고 진행했다.

> "Make the change easy, then make the easy change." — Kent Beck

## 정리

- Fowler의 첫 챕터는 **이론이 아닌 워크스루**다 — 리팩터링의 리듬을 코드로 보여 준다
- 변경은 **작게**, **자주**, 매 단계마다 테스트 통과 + 커밋
- 동기는 보통 **외부 변경 요구** — 그걸 쉽게 만들기 위한 준비 작업이다
- **출력과 계산 분리**, **다형성으로 switch 교체**가 이 챕터에서 등장하는 큰 흐름이다
- 숙련자일수록 **단계가 더 작다**
