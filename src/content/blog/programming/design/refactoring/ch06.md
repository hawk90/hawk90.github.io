## 이 챕터의 메시지

6장은 카탈로그의 첫 번째 그룹 — Fowler가 "가장 빈번하게 쓰는" 것들로 묶은 약 12개의 리팩터링이다. 이걸 익히면 일상의 리팩터링 작업의 80%가 커버된다.

이 그룹의 특징은 **짝**이 많다는 점이다. Extract / Inline, Encapsulate / Inline, Move into / Move out of. 한 방향으로 갔다가 필요하면 되돌아온다. 리팩터링은 한 번 옳은 답이 정해지는 게 아니라 코드의 흐름에 따라 왔다 갔다 한다.

## Extract Function — 함수 추출

가장 자주 쓰이는 리팩터링.

```javascript
// Before
function printOwing(invoice) {
  printBanner();
  let outstanding = calculateOutstanding();
  console.log(`name: ${invoice.customer}`);
  console.log(`amount: ${outstanding}`);
}

// After
function printOwing(invoice) {
  printBanner();
  let outstanding = calculateOutstanding();
  printDetails(invoice, outstanding);
}

function printDetails(invoice, outstanding) {
  console.log(`name: ${invoice.customer}`);
  console.log(`amount: ${outstanding}`);
}
```

**동기** — Fowler의 규칙은 단순하다.

> "If you have to spend effort looking at a fragment of code and figuring out what it's doing, then you should extract it into a function and name the function after the 'what'."

코드 한 조각을 보고 "이게 뭘 하는 거지?"라고 한 번이라도 생각했다면 → 추출한다. 이름이 그 설명이 된다.

**언제 안 하는가** — 한 줄짜리 함수가 보일러플레이트로 느껴진다면? Fowler의 답: **여전히 한다**. 함수 호출의 비용은 거의 0이고, 이름의 가치는 크다.

## Inline Function — 함수 인라인

Extract의 반대. 함수가 너무 잘게 쪼개져서 오히려 흐름을 가린다면 합친다.

```javascript
// Before
function getRating(driver) {
  return moreThanFiveLateDeliveries(driver) ? 2 : 1;
}
function moreThanFiveLateDeliveries(driver) {
  return driver.numberOfLateDeliveries > 5;
}

// After
function getRating(driver) {
  return driver.numberOfLateDeliveries > 5 ? 2 : 1;
}
```

**동기** — 함수의 이름이 본문보다 더 길거나 의미를 더 모호하게 만든다면 인라인한다.

## Extract Variable — 변수 추출

함수 단위가 아니라 식(expression) 단위에서 같은 일을 한다.

```javascript
// Before
return order.quantity * order.itemPrice
    - Math.max(0, order.quantity - 500) * order.itemPrice * 0.05
    + Math.min(order.quantity * order.itemPrice * 0.1, 100);

// After
const basePrice = order.quantity * order.itemPrice;
const quantityDiscount = Math.max(0, order.quantity - 500) * order.itemPrice * 0.05;
const shipping = Math.min(basePrice * 0.1, 100);
return basePrice - quantityDiscount + shipping;
```

**동기** — 식의 각 부분에 이름을 붙여 의미를 드러낸다.

## Inline Variable — 변수 인라인

```javascript
// Before
let basePrice = anOrder.basePrice;
return basePrice > 1000;

// After
return anOrder.basePrice > 1000;
```

**동기** — 변수 이름이 식보다 더 정보를 주지 않으면 그냥 식을 쓴다.

## Change Function Declaration — 함수 선언 변경

이름이나 매개변수를 바꾼다. 이름은 가장 중요한 디자인 결정 중 하나이고, 매개변수 변경은 외부 인터페이스 변경이다.

```javascript
// Before
function circum(radius) { return 2 * Math.PI * radius; }

// After (이름 개선)
function circumference(radius) { return 2 * Math.PI * radius; }

// After (매개변수 추가)
function chargeForReservation(customer, dateRange) { ... }
```

**절차** — 단순한 변경(한 곳에서만 호출)은 IDE의 rename으로 끝. 복잡한 변경(많은 호출처, 외부에 발행된 함수)은 점진적 마이그레이션 필요.

## Encapsulate Variable — 변수 캡슐화

변수를 함수 뒤로 숨긴다.

```javascript
// Before
let defaultOwner = { firstName: "Martin", lastName: "Fowler" };

// 다른 코드
spaceship.owner = defaultOwner;

// After
let defaultOwnerData = { firstName: "Martin", lastName: "Fowler" };
export function defaultOwner() { return defaultOwnerData; }
export function setDefaultOwner(arg) { defaultOwnerData = arg; }
```

**동기** — 변수 접근에 어떤 일이 더 일어나야 한다면(로깅, 검증, 변경 통지) 함수로 감싸는 게 좋다. 게다가 모든 mutable data는 캡슐화하는 게 좋다는 게 Fowler의 일반 원칙이다.

## Rename Variable — 변수 이름 변경

```javascript
// Before
let a = height * width;

// After
let area = height * width;
```

이름 잘 짓는 게 가장 중요한 프로그래밍 활동이라는 게 Fowler의 입장이다. "이름이 잘못된 코드를 두면 다음 사람이 잘못된 정신 모델을 가지게 된다." 이름 변경의 리스크는 낮고 이득은 크다.

## Introduce Parameter Object — 매개변수 객체 도입

매개변수 3~4개가 같이 다닌다면 객체로 묶는다.

```javascript
// Before
function amountInvoiced(startDate, endDate, customerId) { ... }
function amountReceived(startDate, endDate, customerId) { ... }
function amountOverdue(startDate, endDate, customerId) { ... }

// After
function amountInvoiced({startDate, endDate, customerId}) { ... }
// 호출
amountInvoiced({startDate, endDate, customerId});
```

**동기** — Data Clumps(3장의 냄새) 해결. 또 객체로 묶고 나면 그 객체 자체가 더 자라기도 한다 — 거기 메서드가 붙으면서 새 도메인 개념이 등장한다.

## Combine Functions into Class — 함수를 클래스로 묶기

같은 데이터를 다루는 함수들을 클래스로 묶는다.

```javascript
// Before — 함수 여러 개가 reading을 받는다
function base(reading) { ... }
function taxableCharge(reading) { ... }
function calculateBaseCharge(reading) { ... }

// After
class Reading {
  constructor(data) { Object.assign(this, data); }
  get base() { ... }
  get taxableCharge() { ... }
  get calculateBaseCharge() { ... }
}
```

**동기** — 함수들이 공통 데이터를 공유한다면, 그 데이터가 자기 자신의 책임을 가진다는 뜻이다. 객체로 만든다.

## Combine Functions into Transform — 변환 함수로 묶기

함수형 스타일의 대안. 객체 대신 새 데이터 구조로 변환한다.

```javascript
function enrichReading(original) {
  const result = _.cloneDeep(original);
  result.baseCharge = calculateBaseCharge(result);
  result.taxableCharge = calculateTaxableCharge(result);
  return result;
}
```

**동기** — 함수형 / 불변 스타일을 선호한다면. 원본을 안 건드리고 새 객체를 만든다는 점이 다르다.

## Split Phase — 단계 분리

서로 다른 일을 하는 두 단계가 한 함수에 섞여 있으면 분리한다.

```javascript
// Before
function priceOrder(product, quantity, shippingMethod) {
  const basePrice = product.basePrice * quantity;
  const discount = Math.max(quantity - product.discountThreshold, 0)
                  * product.basePrice * product.discountRate;
  const shippingPerCase = (basePrice > shippingMethod.discountThreshold)
                  ? shippingMethod.discountedFee : shippingMethod.feePerCase;
  const shippingCost = quantity * shippingPerCase;
  return basePrice - discount + shippingCost;
}

// After
function priceOrder(product, quantity, shippingMethod) {
  const priceData = calculatePricingData(product, quantity);
  return applyShipping(priceData, shippingMethod);
}

function calculatePricingData(product, quantity) {
  const basePrice = product.basePrice * quantity;
  const discount = Math.max(quantity - product.discountThreshold, 0)
                  * product.basePrice * product.discountRate;
  return { basePrice, quantity, discount };
}

function applyShipping(priceData, shippingMethod) {
  const shippingPerCase = (priceData.basePrice > shippingMethod.discountThreshold)
                  ? shippingMethod.discountedFee : shippingMethod.feePerCase;
  const shippingCost = priceData.quantity * shippingPerCase;
  return priceData.basePrice - priceData.discount + shippingCost;
}
```

**동기** — 가격 계산과 배송비 계산이 서로 다른 책임이다. 첫 단계의 출력이 두 번째 단계의 입력이 되는 파이프라인 구조로 만든다. 한 단계를 변경할 때 다른 단계는 안 건드린다.

## 핵심 짝 — Extract / Inline

이 묶음에서 가장 자주 쓰는 도구는 Extract 와 Inline 짝이다. 두 방향 모두 한다.

- **Extract**가 더 자주 — 일반적으로 함수를 짧게 유지하는 방향
- **Inline**도 필요 — 잘못 추출했거나, 추출 후 더 잘 묶을 게 보이면 다시 합쳐서 재추출

리팩터링은 한 방향의 종착점이 아니다. **왔다 갔다 하면서 형태를 찾는다**.

## 정리

- 6장은 가장 자주 쓰는 12개 리팩터링 묶음
- **Extract Function**이 가장 빈번 — "뭐 하는 거지?" 싶으면 추출
- **Extract / Inline 짝**으로 왔다 갔다 하면서 형태를 찾는다
- **Change Function Declaration**, **Rename Variable** — 이름이 가장 중요한 디자인 결정
- **Encapsulate Variable**, **Introduce Parameter Object** — mutable data 감싸기
- **Split Phase** — 한 함수에 두 책임이 섞여 있을 때
