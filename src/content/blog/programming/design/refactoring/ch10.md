## 이 챕터의 메시지

조건문은 코드 이해를 가장 어렵게 만드는 요소다. 분기마다 다른 일을 하고, 그 분기들이 어떻게 합쳐지는지 한눈에 안 보인다. 10장은 조건문을 더 읽기 좋게 만드는 리팩터링들이다.

세 가지 큰 흐름이 있다.

1. **분해** — 복잡한 조건을 의미 단위로 쪼갠다 (Decompose / Consolidate)
2. **납작화** — 중첩을 풀어낸다 (Guard Clauses)
3. **객체화** — 분기 자체를 다형성으로 옮긴다 (Polymorphism / Special Case)

## Decompose Conditional — 조건문 분해

조건과 분기 본문을 함수로 추출해 의미 단위로 분해한다.

```javascript
// Before
if (!aDate.isBefore(plan.summerStart) && !aDate.isAfter(plan.summerEnd))
  charge = quantity * plan.summerRate;
else
  charge = quantity * plan.regularRate + plan.regularServiceCharge;

// After
if (summer())
  charge = summerCharge();
else
  charge = regularCharge();

function summer() {
  return !aDate.isBefore(plan.summerStart) && !aDate.isAfter(plan.summerEnd);
}
function summerCharge() {
  return quantity * plan.summerRate;
}
function regularCharge() {
  return quantity * plan.regularRate + plan.regularServiceCharge;
}
```

**동기** — 조건과 본문 모두에 이름이 붙는다. 코드를 읽을 때 "여름인지 검사" → "여름 요금 계산" → "기본 요금 계산"으로 단계적으로 이해된다.

## Consolidate Conditional Expression — 조건식 통합

여러 조건이 모두 같은 결과를 낸다면 하나로 합친다.

```javascript
// Before
if (anEmployee.seniority < 2) return 0;
if (anEmployee.monthsDisabled > 12) return 0;
if (anEmployee.isPartTime) return 0;

// After
if (isNotEligibleForDisability()) return 0;

function isNotEligibleForDisability() {
  return ((anEmployee.seniority < 2)
       || (anEmployee.monthsDisabled > 12)
       || (anEmployee.isPartTime));
}
```

**동기** — 같은 결과를 내는 조건들은 한 개념을 표현한다. 이름을 주면 그 개념이 코드에 명시된다.

## Replace Nested Conditional with Guard Clauses — 중첩 조건을 가드절로

```javascript
// Before
function getPayAmount() {
  let result;
  if (isDead)
    result = deadAmount();
  else {
    if (isSeparated)
      result = separatedAmount();
    else {
      if (isRetired)
        result = retiredAmount();
      else
        result = normalPayAmount();
    }
  }
  return result;
}

// After
function getPayAmount() {
  if (isDead) return deadAmount();
  if (isSeparated) return separatedAmount();
  if (isRetired) return retiredAmount();
  return normalPayAmount();
}
```

**동기** — 중첩된 if-else는 "정상 경로"가 어디인지 모호하다. Guard Clause는 예외 케이스를 일찍 처리하고, 정상 경로를 가장 아래에 둔다.

Fowler가 좋아하는 패턴이다. 코드를 위에서 아래로 읽으면서 "이건 다 예외 처리고, 진짜 일은 마지막 줄에서 일어난다"가 명확해진다.

## Replace Conditional with Polymorphism — 조건문을 다형성으로

```javascript
// Before
switch (bird.type) {
  case "EuropeanSwallow":
    return "average";
  case "AfricanSwallow":
    return (bird.numberOfCoconuts > 2) ? "tired" : "average";
  case "NorwegianBlueParrot":
    return (bird.voltage > 100) ? "scorched" : "beautiful";
  default:
    return "unknown";
}

// After
class Bird {
  get plumage() { return "unknown"; }
}
class EuropeanSwallow extends Bird {
  get plumage() { return "average"; }
}
class AfricanSwallow extends Bird {
  get plumage() { return (this.numberOfCoconuts > 2) ? "tired" : "average"; }
}
class NorwegianBlueParrot extends Bird {
  get plumage() { return (this.voltage > 100) ? "scorched" : "beautiful"; }
}
```

**동기** — Repeated Switches 냄새. 같은 분기가 코드 여러 곳에 흩어져 있으면 새 케이스 추가가 위험하다. 다형성은 새 케이스를 새 클래스 추가로 격리한다 — OCP의 정확한 형태.

Strategy / State / Template Method 같은 GoF 패턴이 이 리팩터링의 변형들이다.

## Introduce Special Case — 특수 케이스 도입

특정 값에 대해 모든 클라이언트가 같은 처리를 한다면, 그 특수 케이스를 객체로 만든다 — **Null Object** 패턴의 일반화.

```javascript
// Before
if (customer === "unknown") customerName = "occupant";
if (customer === "unknown") plan = registry.billingPlans.basic;

// After
class UnknownCustomer {
  get isUnknown() { return true; }
  get name() { return "occupant"; }
  get billingPlan() { return registry.billingPlans.basic; }
}
// 클라이언트는 그냥 customer.name, customer.billingPlan 호출
```

**동기** — Null 체크나 특수값 체크가 코드 도처에 흩어진다. 특수 케이스를 객체로 만들면 그 체크들이 모두 사라진다.

## Introduce Assertion — 단언 도입

```javascript
// Before
if (this.discountRate)
  base = base - (this.discountRate * base);

// After
assert(this.discountRate >= 0);
if (this.discountRate)
  base = base - (this.discountRate * base);
```

**동기** — 코드가 암묵적으로 가정하는 조건을 명시한다. 가정이 깨지면 즉시 에러로 드러난다. 또 다음 사람에게 "이 코드는 X를 전제로 한다"가 명확히 전달된다.

다만 assertion은 **버그를 찾기 위한 도구**다. 비즈니스 로직(예: 입력 검증)은 assertion이 아니라 정식 예외 처리로 한다.

## 조건문 단순화의 흐름

이 챕터의 리팩터링들은 보통 다음 순서로 적용된다.

1. **Decompose Conditional** — 조건과 본문에 이름 붙이기
2. **Replace Nested with Guard Clauses** — 예외 케이스를 먼저
3. **Consolidate** — 같은 결과의 조건들 통합
4. **Replace with Polymorphism** — 같은 분기가 반복되면 다형성
5. **Introduce Special Case** — Null / 특수값 체크 제거

순서가 중요한 건 아니지만, 작은 정리(1-3)를 먼저 하고 큰 구조 변경(4-5)을 나중에 하는 게 안전하다.

## 정리

- 조건문은 코드 이해의 가장 큰 적
- **Decompose Conditional** — 조건과 분기에 이름 붙이기
- **Consolidate** — 같은 결과 조건들을 한 개념으로
- **Guard Clauses** — 예외를 일찍, 정상 경로를 명확하게
- **Polymorphism** — Repeated Switches 해결, OCP 달성
- **Special Case** — Null Object 일반화
- **Assertion** — 암묵 가정을 명시
