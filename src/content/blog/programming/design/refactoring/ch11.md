## 이 챕터의 메시지

11장은 **함수 시그니처**, **인터페이스 모양**을 다듬는 리팩터링들이다. 함수가 두 가지 일을 하지 않게, 불필요한 매개변수를 받지 않게, 그리고 호출 방식이 의도를 정확히 드러내게 만든다.

API는 호출자와의 계약이다. 계약이 명확할수록 양쪽 모두 편하다.

## Separate Query from Modifier — 질의와 수정의 분리

함수는 값을 반환하거나(query) 상태를 바꾸거나(modifier) — 둘 중 하나만 한다.

```javascript
// Before
function getTotalOutstandingAndSendBill() {
  const result = customer.invoices.reduce(...);
  sendBill();
  return result;
}

// After
function totalOutstanding() {  // query
  return customer.invoices.reduce(...);
}
function sendBill() {  // modifier
  emailGateway.send(formatBill(customer));
}
```

**동기** — Command-Query Separation (CQS) 원칙. query는 부수 효과 없이 호출 가능해야 한다. 그래야 디버깅, 테스트, 캐싱이 안전하다.

> "Asking a question should not change the answer."

## Parameterize Function — 함수 매개변수화

같은 일을 약간씩 다른 값으로 하는 함수 여러 개가 있다면 매개변수로 합친다.

```javascript
// Before
function tenPercentRaise(person) { person.salary *= 1.10; }
function fivePercentRaise(person) { person.salary *= 1.05; }

// After
function raise(person, factor) { person.salary *= (1 + factor); }
```

**동기** — 함수의 본질이 같은데 상수만 다른 경우. 매개변수화하면 코드 중복이 사라진다.

## Remove Flag Argument — 플래그 인자 제거

함수의 동작을 boolean 인자로 분기하는 패턴을 제거한다.

```javascript
// Before
bookConcert(aCustomer, true);   // premium
bookConcert(aCustomer, false);  // regular

// After
premiumBookConcert(aCustomer);
regularBookConcert(aCustomer);
```

**동기** — `bookConcert(c, true)`를 봤을 때 `true`가 뭔지 모른다. 호출처 옆에 주석을 달아야 한다. 분리하면 함수 이름이 직접 설명한다.

특히 한 함수 안에 if(flag) {...} else {...}가 거의 모든 본문을 가른다면 — 그건 사실 두 함수다.

## Preserve Whole Object — 객체 전체 전달

```javascript
// Before
const low = aRoom.daysTempRange.low;
const high = aRoom.daysTempRange.high;
if (aPlan.withinRange(low, high)) { ... }

// After
if (aPlan.withinRange(aRoom.daysTempRange)) { ... }
```

**동기** — 객체의 필드 몇 개만 추출해서 넘기지 말고 객체 자체를 넘긴다. 그러면 호출자와 피호출자가 객체의 구조에 결합되지 않고, 메서드 시그니처도 짧아진다.

## Replace Parameter with Query — 매개변수를 질의로

매개변수가 다른 매개변수에서 유도 가능하면 빼낸다.

```javascript
// Before
const basePrice = this.quantity * this.itemPrice;
const discountLevel = (this.quantity > 100) ? 2 : 1;
const finalPrice = this.discountedPrice(basePrice, discountLevel);

// After
const finalPrice = this.discountedPrice(basePrice);

discountedPrice(basePrice) {
  return basePrice - (this.discountLevel * basePrice / 10);
}
get discountLevel() {
  return (this.quantity > 100) ? 2 : 1;
}
```

**동기** — 호출자가 매번 같은 계산을 해서 넘겨 줘야 하는 매개변수는 함수 안에서 직접 조회한다.

## Replace Query with Parameter — 질의를 매개변수로

반대 방향. 함수가 내부에서 어떤 데이터를 조회하는데, 호출자별로 그 데이터가 달라야 한다면 매개변수로 받는다.

```javascript
// Before
targetTemperature(aPlan) {
  currentTemperature = thermostat.currentTemperature;  // 글로벌 의존
  if (aPlan.targetWithin(currentTemperature)) { ... }
}

// After
targetTemperature(aPlan, currentTemperature) {
  if (aPlan.targetWithin(currentTemperature)) { ... }
}
```

**동기** — 함수의 외부 의존(글로벌, 싱글톤, 동적 데이터)을 명시적 매개변수로. 함수가 더 testable, more predictable해진다. 가이드라인 37(Singleton)과 같은 정신.

## Remove Setting Method — 설정자 제거

생성 시점에 한 번만 정해지는 값에 setter를 두지 않는다.

```javascript
// Before
class Person {
  set name(arg) { this._name = arg; }
}
const p = new Person();
p.name = "Alice";

// After
class Person {
  constructor(name) { this._name = name; }
  // setter 없음
}
const p = new Person("Alice");
```

**동기** — setter가 있으면 누구나 언제든 바꿀 수 있다고 신호를 보낸다. 사실 한 번만 정해지는 값이라면 그 신호가 거짓이다. 불변에 가깝게 만든다.

## Replace Constructor with Factory Function — 생성자를 팩토리 함수로

```javascript
// Before
const aPayment = new Payment("monthly", 12);

// After
const aPayment = createMonthlyPayment(12);
```

**동기** — 생성자는 제약이 많다 (정확히 그 클래스의 인스턴스를 반환해야 함, 이름이 클래스명 고정). 팩토리는 더 유연하다 — 적절한 서브클래스 반환, 캐싱, 이름으로 의도 표현.

## Replace Function with Command — 함수를 커맨드로

복잡한 함수를 객체로 승격한다. 객체로 만들면 매개변수가 필드가 되고, 중간 상태를 둘 수 있고, 작업의 분해가 가능해진다.

```javascript
// Before
function score(candidate, medicalExam, scoringGuide) {
  let result = 0;
  let healthLevel = 0;
  // ... 50줄
  return result;
}

// After
class Scorer {
  constructor(candidate, medicalExam, scoringGuide) {
    this._candidate = candidate;
    this._medicalExam = medicalExam;
    this._scoringGuide = scoringGuide;
  }
  execute() {
    this._result = 0;
    this._healthLevel = 0;
    this._scoreSmoking();
    this._scoreFitness();
    // ...
    return this._result;
  }
  _scoreSmoking() { ... }
  _scoreFitness() { ... }
}
```

**동기** — 큰 함수 / 많은 매개변수 / 복잡한 중간 상태 — 이 셋이 합쳐지면 함수보다 커맨드 객체가 더 자연스럽다. GoF Command 패턴의 응용.

## Replace Command with Function — 커맨드를 함수로

반대 방향. 단순한 처리는 함수가 더 가볍다. 커맨드 객체를 만드는 게 과한 경우 함수로 되돌린다.

## API 디자인의 일반 원칙

이 챕터의 리팩터링들을 관통하는 원칙들.

- **하나의 함수 = 하나의 일** (CQS, Flag Removal)
- **명확한 의도** (Parameterize, Factory)
- **최소한의 결합** (Preserve Whole Object, Replace Query with Parameter)
- **불변 우선** (Remove Setting Method)
- **복잡함에 비례한 무게** (Function ↔ Command 양방향)

좋은 API는 호출자가 의도를 표현하기 쉽게 만들고, 호출자가 잘못 쓰기 어렵게 만든다.

## 정리

- **Separate Query from Modifier** — 한 함수가 두 가지 일 안 한다 (CQS)
- **Remove Flag Argument** — boolean 분기 함수는 두 함수
- **Preserve Whole Object** — 필드 추출 대신 객체 전달
- **Parameter / Query 양방향** — 외부 의존을 명시화하거나, 중복 계산을 함수로
- **Constructor → Factory** — 더 유연한 생성
- **Function ↔ Command 양방향** — 복잡함이 크면 객체화
