## 이 챕터의 메시지

리팩터링은 외부 동작을 유지하는 활동이다. 그런데 "외부 동작이 유지됐다"를 어떻게 알 수 있는가? 답은 단순하다 — **테스트**.

Fowler는 이 챕터에서 두 가지를 동시에 주장한다.

1. **테스트가 없으면 리팩터링은 위험하다.**
2. **테스트는 리팩터링뿐 아니라 코드 작성 자체를 더 빠르게 만든다.**

두 번째 주장이 핵심이다. 테스트가 시간을 잡아먹는다는 인식은 잘못됐다. 충분히 빠르게 코드를 짜기 위해 테스트가 필요하다는 게 그의 주장이다.

## 자기 테스트 코드의 가치

Fowler는 자신의 경험을 인용한다. 그는 컴파일이 오래 걸리던 시절을 보냈다. 그 시절에는 컴파일 / 실행 / 수동 검증의 한 사이클이 매우 길었다. 그래서 한 번에 많이 바꾸고, 그 다음에 깨진 곳을 찾아 헤매는 패턴이었다.

자기 테스트 코드(self-testing code) — 자동으로 자기 자신을 검증하는 테스트 — 가 생기면서 사이클이 짧아졌다. 작은 변경 → 테스트 → 결과를 초 단위로 확인 → 다음 변경. 이 짧은 사이클이 **자신감 있는 변경**을 만든다.

> "If it ain't broke, don't fix it" 의 반대 — **자주 깨뜨려 봐야 빨리 배운다**.

테스트는 깨뜨리는 비용을 0에 가깝게 만든다. 그래서 더 많이 시도하고 더 많이 배운다.

## 테스트가 빠른 개발의 도구인 이유

테스트가 시간을 절약하는 메커니즘은 세 가지다.

**1. 디버깅 시간이 사라진다**

테스트가 없으면, 변경 후 어딘가 깨졌을 때 어디서 깨졌는지 찾는 데 시간이 든다. 테스트가 있으면, 마지막 변경이 곧 원인이다. **변경 → 즉시 발견** 사이클.

**2. 작은 단계로 진행할 수 있다**

테스트 없이는 5분짜리 변경 후에 30분 동안 모든 동작을 손으로 확인해야 한다. 그래서 변경을 모아서 한 번에 한다. 큰 변경은 위험하다. 테스트가 있으면 5분짜리 변경을 매번 검증할 수 있다 — 위험이 분산된다.

**3. 두려움이 사라진다**

테스트가 없는 코드는 만지기 무섭다. 그래서 안 만진다. 안 만진 코드는 더 더러워진다. 악순환. 테스트가 있으면 만질 수 있고, 만지면 깨끗해진다.

## JUnit / Jest 스타일 테스트

책은 자바스크립트 예제를 쓴다. Jest 스타일이다.

```javascript
import { Producer } from "./producer.js";

describe("province", () => {
  let asia;
  
  beforeEach(() => {
    asia = sampleProvinceData();
  });

  test("shortfall", () => {
    expect(asia.shortfall).toBe(5);
  });

  test("profit", () => {
    expect(asia.profit).toBe(230);
  });
});
```

핵심 요소.

- **fixture** — 테스트마다 시작점이 같도록 준비. `beforeEach`로 매번 새로 만든다.
- **assertion** — 기대값과 실제값 비교. `expect(...).toBe(...)`.
- **격리** — 한 테스트가 다른 테스트에 영향을 주지 않는다.

언어가 달라도(JUnit, pytest, Catch2, ...) 구조는 같다.

## fixture를 매번 새로 — 공유하지 마라

흔한 실수: fixture를 전역으로 한 번 만들어 두고 모든 테스트가 공유한다.

```javascript
// ❌ 위험
const asia = sampleProvinceData();  // 전역

test("a modifies asia", () => {
  asia.totalIncome = 999;  // 변경
  // ...
});

test("b reads asia", () => {
  // a가 먼저 돈 상태에 의존한다
});
```

테스트 순서에 따라 결과가 달라진다. 순서 의존 = 깨지기 쉬운 테스트.

해법은 `beforeEach`로 매번 새로 만드는 것이다.

```javascript
// ✅
let asia;
beforeEach(() => {
  asia = sampleProvinceData();  // 매번 새로
});
```

비용이 좀 들지만 (이 객체를 매번 만든다), 격리의 가치가 훨씬 크다.

## 한 테스트는 한 가지만 검증

```javascript
// ❌ 너무 많이
test("province operations", () => {
  asia.totalIncome = 999;
  expect(asia.shortfall).toBe(...);
  asia.producers.push(new Producer(...));
  expect(asia.profit).toBe(...);
  asia.demand = 0;
  expect(asia.shortfall).toBe(...);
});
```

테스트가 깨졌을 때 어디서 깨졌는지 추적이 어렵다. 또 한 부분에서 깨지면 그 뒤로 못 가서 검증이 멈춘다.

```javascript
// ✅ 단위별로
test("shortfall after income change", () => { ... });
test("profit after producer added", () => { ... });
test("shortfall when demand zero", () => { ... });
```

**한 테스트 = 한 시나리오 = 한 검증.**

## 경계 조건 — 가장 가치 있는 테스트

평범한 입력에 대한 테스트는 시작점이다. 진짜 가치는 **경계 조건**에 있다.

- 빈 컬렉션 (`[]`)
- null / undefined
- 음수 / 0
- 매우 큰 수 / 매우 작은 수
- 중복 요소
- 한 요소만 있는 컬렉션

```javascript
test("empty producers", () => {
  asia.producers = [];
  expect(asia.shortfall).toBe(30);
  expect(asia.profit).toBe(0);
});

test("negative demand", () => {
  asia.demand = -1;
  // 어떻게 동작해야 하는가?
});
```

테스트를 짜면서 처음으로 "음수 demand는 어떻게 처리해야 하지?" 같은 질문이 떠오른다. 이게 테스트의 두 번째 가치 — **요구사항을 명확히 한다**.

## 무엇을 테스트할까

Fowler는 두 가지 가이드를 준다.

**1. 가장 두려운 부분**

"이 부분이 잘못되면 가장 큰 문제가 될 것 같다" 싶은 곳부터 테스트한다. 직관이 가장 정확한 안내자다.

**2. 새 기능을 짤 때마다**

기능을 짜기 전에 테스트를 짠다 — TDD의 핵심. 짜기 전에 한 번 더 그 기능을 어떻게 쓸지 생각해 본다.

다만 Fowler는 TDD 광신도는 아니다. "테스트 먼저"를 항상 따를 필요는 없다 — 자신감을 주는 정도면 충분하다고 본다.

## 100% 커버리지의 함정

테스트 커버리지가 100%여야 한다는 강박은 잘못됐다. 커버리지는 어디서 테스트가 빠졌는지 알려 주는 지표일 뿐이다. 100% 커버리지여도 잘못된 단언(assertion)으로 가득하면 의미가 없다.

> "Imperfect tests, run frequently, are much better than perfect tests that are never written at all."

자주 돌리는 불완전한 테스트가, 결코 작성되지 않는 완벽한 테스트보다 훨씬 낫다.

## 테스트 자체의 리팩터링

테스트 코드도 코드다. 더러워지면 리팩터링한다.

- 중복되는 셋업 → `beforeEach`로 추출
- 같은 단언 패턴 → 헬퍼 함수로 추출
- 의미 없는 이름의 테스트 → 무엇을 검증하는지 드러나도록 rename

테스트는 **두 번째 클라이언트**다. 본 코드의 가장 첫 번째 사용자이며, 그래서 본 코드의 API가 좋은지 나쁜지를 가장 먼저 알려 준다. 테스트가 짜기 어려운 함수는 보통 디자인이 잘못됐다.

## 잘못된 의심 — "이건 너무 단순해서 테스트 안 해도 돼"

흔한 변명. Fowler는 단호하다 — 의심이 들면 테스트한다.

게터/세터처럼 정말 단순한 코드는 굳이 테스트할 필요 없을 수 있다. 그러나 "이건 단순하니까"라는 핑계로 안 하다 보면 점점 더 많은 코드가 테스트 없이 쌓인다.

작은 테스트가 100개 있는 게, 큰 테스트 5개보다 낫다.

## 정리

- **테스트는 리팩터링의 안전망**이다 — 외부 동작 유지를 검증한다
- 테스트는 **빠른 개발의 도구**이기도 하다 — 디버깅 시간 / 단계 크기 / 두려움 모두 줄여 준다
- fixture는 매번 새로 만든다 (`beforeEach`)
- 한 테스트 = 한 시나리오
- **경계 조건**에 가장 큰 가치가 있다
- **불완전하지만 자주 돌리는** 테스트가 완벽하지만 안 짠 테스트보다 낫다
- 테스트도 코드 — 리팩터링한다
