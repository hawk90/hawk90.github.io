## 이 챕터의 메시지

리팩터링은 도구다. 도구를 언제 꺼낼지를 알아야 한다. 3장은 그 신호 — **코드의 냄새**(code smell) — 를 카탈로그로 정리한다.

이 단어를 만든 사람은 Kent Beck이다. "냄새"라는 비유는 의도적이다.

- 명확한 결함이 아니다 — 동작은 멀쩡할 수 있다
- 그러나 무언가 잘못됐다는 **직관적 신호**다
- 자세히 보면 보통 진짜 문제가 있다

코드 냄새는 측정 가능한 지표가 아니다. **감각**이다. 경험이 쌓일수록 더 빨리 알아챈다. 이 챕터는 그 감각을 빠르게 학습하기 위한 카탈로그다.

## 핵심 내용

- 코드 냄새는 리팩터링이 필요하다는 **직관적 신호**다
- Fowler/Beck가 정리한 카탈로그 — 약 24개
- 냄새 → 추천 리팩터링 매핑이 핵심 가치
- 모든 냄새가 항상 문제는 아니다 — **맥락**이 결정한다

## 대표 냄새 — 가장 흔한 것들

24개 전부를 외울 필요는 없다. 가장 자주 등장하는 것들을 안다.

### Mysterious Name (수수께끼 이름)

이름이 의도를 드러내지 않는다.

```javascript
function calc(x, y) { return x * 0.875 + y * 0.125; }
```

`calc`가 뭘 계산하는지, `x`와 `y`가 뭔지 모른다. 좋은 이름이 떠오르지 않는다는 건 그 함수의 책임이 명확하지 않다는 신호이기도 하다.

**추천**: Rename Variable / Rename Function / Change Function Declaration

### Duplicated Code (중복 코드)

같은 코드가 여러 곳에 있다. 한 곳을 고치면 다른 곳도 같이 고쳐야 한다 — 까먹기 쉽다.

**추천**: Extract Function (같은 클래스 내) / Pull Up Method (같은 부모 자식) / Slide Statements (인접 코드 그룹화)

### Long Function (긴 함수)

Fowler의 기준은 명확하다. **짧을수록 좋다.** 그가 책에서 권하는 한계는 보통 10줄 이하다.

긴 함수의 가장 큰 문제는 **이름이 모자라다**는 점이다. 5단계 처리를 한 함수로 묶으면 그 처리들에 각각 이름을 붙일 기회를 잃는다. 짧은 함수 5개로 쪼개면 이름이 5개 생긴다.

**추천**: Extract Function (가장 많이 쓰이는 리팩터링) / Replace Temp with Query / Introduce Parameter Object / Decompose Conditional

### Long Parameter List (긴 매개변수 목록)

매개변수가 5개 이상이면 의심한다. 매개변수가 많으면 호출이 어렵고, 매개변수 사이의 관계도 알기 어렵다.

**추천**: Replace Parameter with Query (다른 값에서 유도 가능하면) / Preserve Whole Object (여러 필드가 한 객체에서 옴) / Introduce Parameter Object (관련 매개변수 묶음)

### Global Data (전역 데이터)

어디서든 수정 가능한 데이터. 디버깅의 악몽이다.

**추천**: Encapsulate Variable (게터/세터로 감싸기)

### Mutable Data (변경 가능한 데이터)

값을 자유롭게 바꿀 수 있으면 어디서 바뀌었는지 추적이 어렵다. 함수형 프로그래밍의 영향 — 가능한 한 불변으로.

**추천**: Encapsulate Variable / Split Variable / Slide Statements / Replace Derived Variable with Query

### Divergent Change (산만한 변경)

한 모듈이 여러 가지 다른 이유로 바뀐다. SRP(Single Responsibility) 위반의 다른 이름이다. UI를 바꿀 때도 만지고, DB를 바꿀 때도 만지고, 비즈니스 규칙이 바뀔 때도 만진다면 그 모듈은 여러 책임을 졌다는 뜻이다.

**추천**: Split Phase (단계 분리) / Move Function / Extract Class

### Shotgun Surgery (산탄총 수술)

한 가지 변경이 여러 모듈을 동시에 건드린다. Divergent Change의 반대 형태다 — 한 책임이 여러 곳에 흩어져 있다.

**추천**: Move Function / Move Field / Combine Functions into Class / Inline Function

### Feature Envy (기능 욕심)

함수가 자기 클래스보다 **다른 클래스의 데이터에 더 관심**이 많다. 보통 그 함수는 다른 클래스로 옮겨야 한다.

```javascript
function discount(customer, amount) {
  if (customer.isPremium && customer.yearsSubscribed > 5 && customer.purchases > 100) {
    // customer의 필드 3개 + amount
    return amount * 0.2;
  }
  return amount * 0.1;
}
```

이 함수는 amount보다 customer의 필드를 더 많이 본다. customer 클래스로 옮기는 게 자연스럽다.

**추천**: Move Function / Extract Function

### Data Clumps (데이터 뭉치)

같은 데이터 묶음이 여러 곳에서 함께 등장한다. `(start, end)`, `(latitude, longitude)`, `(firstName, lastName)` 등.

**추천**: Extract Class / Introduce Parameter Object / Preserve Whole Object

### Primitive Obsession (기본형 집착)

전화번호를 string으로, 돈을 number로, 좌표를 두 number로 표현한다. 그 결과 같은 검증 로직이 도처에 흩어진다.

**추천**: Replace Primitive with Object / Replace Type Code with Subclasses

### Repeated Switches (반복되는 switch)

같은 조건의 switch / if-else가 코드 전반에 흩어져 있다. 새 케이스 추가 시 모두 찾아 고쳐야 한다.

**추천**: Replace Conditional with Polymorphism

### Loops (반복문)

루프 자체가 냄새다 — 함수형 연산이 더 의도를 드러낸다.

```javascript
// 루프
const result = [];
for (const x of items) {
  if (x.active) result.push(x.name);
}

// pipeline
const result = items.filter(x => x.active).map(x => x.name);
```

**추천**: Replace Loop with Pipeline

### Lazy Element (게으른 요소)

존재 이유가 약한 클래스나 함수. 한 줄짜리 함수가 무수히 있거나, 멤버 하나뿐인 클래스. 만들 때는 의미가 있었지만 진화하면서 비어 갔다.

**추천**: Inline Function / Inline Class / Collapse Hierarchy

### Speculative Generality (추측성 일반화)

"언젠가 필요할지 모르니까" 미리 만든 유연성. YAGNI 위반. 추상 클래스 하나에 자식 하나, 매개변수 안 쓰는 것, 사용처가 없는 메서드.

**추천**: Collapse Hierarchy / Inline Function / Change Function Declaration / Remove Dead Code

### Temporary Field (임시 필드)

특정 상황에서만 값이 채워지는 필드. 다른 상황에선 null이거나 의미가 없다. 이런 필드는 별도 클래스나 매개변수로 분리해야 한다.

**추천**: Extract Class / Introduce Special Case / Move Function

### Message Chains (메시지 체인)

`a.b().c().d().e()` 같은 긴 호출 체인. 클라이언트가 객체 그래프의 구조를 알게 된다 — 그 구조가 바뀌면 클라이언트도 바뀐다.

**추천**: Hide Delegate / Extract Function + Move Function

### Middle Man (중개자)

위임만 하는 클래스. 메서드의 절반 이상이 다른 객체로 그대로 전달되기만 한다. 위임 자체가 나쁜 건 아니지만, 거의 모든 메서드가 그러면 그 클래스의 존재 이유를 의심해야 한다.

**추천**: Remove Middle Man / Inline Function

### Insider Trading (내부자 거래)

모듈 사이의 비공식적 정보 교환. A 모듈이 B 모듈의 사정을 너무 잘 안다. 공식 인터페이스를 통하지 않고 내부 구조에 의존한다.

**추천**: Move Function / Move Field / Hide Delegate / Replace Subclass with Delegate

### Large Class (큰 클래스)

너무 많은 일을 하는 클래스. 필드가 많고 메서드가 많다. 책임이 응집되어 있는지 다시 검토한다.

**추천**: Extract Class / Extract Superclass / Replace Type Code with Subclasses

### Alternative Classes with Different Interfaces (인터페이스가 다른 대안 클래스)

같은 일을 하는 두 클래스가 인터페이스가 다르다. 같은 일이면 같은 이름이어야 한다. Strategy로 통합할 수 있게.

**추천**: Change Function Declaration / Move Function / Extract Superclass

### Data Class (데이터 클래스)

필드와 게터/세터만 있는 클래스. 행동이 없다. Data가 자기 자신을 다루는 행동을 가지지 못하고, 다른 곳에서 그 데이터를 만져 댄다.

**추천**: Encapsulate Record / Remove Setting Method / Move Function

### Refused Bequest (거절된 유산)

부모 클래스의 기능을 자식이 별로 쓰지 않거나, 일부만 쓴다. 상속이 잘못된 신호다.

**추천**: Push Down Method / Push Down Field / Replace Subclass with Delegate / Replace Superclass with Delegate

### Comments (주석)

주석이 많은 코드는 의심한다. **주석은 종종 변명**이다. 이름을 잘 짓고 함수를 잘 쪼개면 주석이 줄어든다.

물론 좋은 주석도 있다.

- **왜**(why) 설명 — 이 코드가 왜 이렇게 됐는지
- 라이센스, 참조 링크
- TODO

나쁜 주석은 **무엇**(what)을 설명한다 — 그건 코드가 직접 말해야 한다.

**추천**: Extract Function (긴 주석을 함수 이름으로) / Change Function Declaration / Introduce Assertion

## 냄새는 신호일 뿐

이 카탈로그를 외울 필요는 없다. 더 중요한 건 **냄새를 알아채는 감각**이다.

Fowler 본인도 "이 냄새가 있으면 무조건 리팩터링하라"고 말하지 않는다. 모든 냄새는 **맥락 의존**이다.

- 짧은 함수가 모든 경우에 옳은 건 아니다 — 도메인이 복잡한 알고리즘 한 줄짜리는 그 자체로 의미를 가진다
- 주석이 항상 나쁜 건 아니다 — 왜를 설명하는 주석은 좋다
- 중복이 항상 나쁜 건 아니다 — 우연한 일치는 강제로 합치면 안 된다 (Premature Abstraction)

판단은 항상 사람의 몫이다.

## 정리

- 코드 냄새는 리팩터링이 필요하다는 **직관적 신호**다
- 카탈로그가 있지만 **외우는 것보다 감각이 중요**하다
- 가장 자주 만나는 냄새: Long Function, Duplicated Code, Long Parameter List, Feature Envy, Primitive Obsession, Mysterious Name
- 모든 냄새는 맥락 의존 — 무조건 고치는 게 아니라 **상황에 따라 판단**한다
- 다음 장의 테스트가 있어야 이 모든 리팩터링이 안전해진다
