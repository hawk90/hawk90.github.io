## 이 챕터의 메시지

캡슐화는 객체지향의 핵심 아이디어다. 외부에 노출되는 표면을 좁히면 내부 변경이 자유로워진다. 7장은 이 캡슐화를 강화하는 리팩터링들을 모은다.

핵심 메시지는 단순하다 — **데이터를 직접 노출하지 마라**.

## Encapsulate Record — 레코드 캡슐화

raw record(객체 리터럴, 딕셔너리)를 클래스로 감싼다.

```javascript
// Before
const organization = {name: "Acme Gooseberries", country: "GB"};

// After
class Organization {
  constructor(data) {
    this._name = data.name;
    this._country = data.country;
  }
  get name() { return this._name; }
  set name(arg) { this._name = arg; }
  get country() { return this._country; }
  set country(arg) { this._country = arg; }
}
```

**동기** — record를 직접 만지면 코드의 어디서 어떤 필드가 읽히고 써지는지 추적이 어렵다. 함수 뒤로 숨기면 단순 변경(필드 이름, 계산 방식)이 다른 코드를 안 깨고 가능하다.

다만 **불변 레코드**(immutable record)는 캡슐화 압박이 약하다 — 누가 만져도 똑같으므로.

## Encapsulate Collection — 컬렉션 캡슐화

게터가 컬렉션을 그대로 반환하면, 호출자가 그 컬렉션을 마음대로 수정할 수 있다.

```javascript
// Before — 위험
class Person {
  get courses() { return this._courses; }
}

person.courses.push(newCourse);  // Person이 모르는 수정
```

해법은 추가/제거 메서드를 제공하고 게터는 복사본을 반환하는 것이다.

```javascript
// After
class Person {
  get courses() { return this._courses.slice(); }  // 복사
  addCourse(course) { this._courses.push(course); }
  removeCourse(course, fnIfAbsent = () => { throw new RangeError(); }) {
    const i = this._courses.indexOf(course);
    if (i === -1) fnIfAbsent();
    else this._courses.splice(i, 1);
  }
}
```

**동기** — 클래스가 자기 상태의 변경을 통제해야 한다. 컬렉션은 mutable이라 게터로 그대로 줘 버리면 통제가 깨진다.

## Replace Primitive with Object — 기본형을 객체로

전화번호를 string, 돈을 number, 좌표를 두 number로 표현하는 게 Primitive Obsession이다. 점점 복잡해지면 객체로 승격한다.

```javascript
// Before
function priorityString(priority) {  // 단순 string
  if (priority === "low") return 1;
  if (priority === "normal") return 2;
  if (priority === "high") return 3;
}

// After
class Priority {
  constructor(value) {
    if (Priority.legalValues().includes(value)) this._value = value;
    else throw new Error(`<${value}> is invalid for Priority`);
  }
  toString() { return this._value; }
  get _index() { return Priority.legalValues().indexOf(this._value); }
  static legalValues() { return ["low", "normal", "high", "rush"]; }
  
  equals(other) { return this._index === other._index; }
  higherThan(other) { return this._index > other._index; }
  lowerThan(other) { return this._index < other._index; }
}
```

**동기** — primitive에 행동을 붙일 곳이 없다. 객체로 만들면 검증, 비교, 변환 같은 행동을 한 곳에 모을 수 있다.

## Replace Temp with Query — 임시 변수를 질의로

값을 계산해서 변수에 담고, 그 변수를 여러 번 쓴다면 — 그 계산을 함수로 만들어 쓴다.

```javascript
// Before
class Order {
  get price() {
    let basePrice = this._quantity * this._itemPrice;
    let discountFactor = 0.98;
    if (basePrice > 1000) discountFactor -= 0.03;
    return basePrice * discountFactor;
  }
}

// After
class Order {
  get price() {
    return this.basePrice * this.discountFactor;
  }
  get basePrice() { return this._quantity * this._itemPrice; }
  get discountFactor() {
    let discountFactor = 0.98;
    if (this.basePrice > 1000) discountFactor -= 0.03;
    return discountFactor;
  }
}
```

**동기** — 계산을 함수로 빼면 다른 곳에서도 재사용 가능하고, 더 이상 함수에 매개변수로 전달할 필요가 없어진다. 또 함수 추출의 사전 준비 단계로 자주 쓰인다.

## Extract Class — 클래스 추출

큰 클래스에서 응집된 일부 책임을 별도 클래스로 빼낸다.

```javascript
// Before
class Person {
  get officeAreaCode() { return this._officeAreaCode; }
  set officeAreaCode(arg) { this._officeAreaCode = arg; }
  get officeNumber() { return this._officeNumber; }
  set officeNumber(arg) { this._officeNumber = arg; }
}

// After
class Person {
  get telephoneNumber() { return this._telephoneNumber; }
}

class TelephoneNumber {
  get areaCode() { return this._areaCode; }
  set areaCode(arg) { this._areaCode = arg; }
  get number() { return this._number; }
  set number(arg) { this._number = arg; }
}
```

**동기** — Large Class 냄새. 같이 변하고 같이 쓰이는 필드 묶음이 보이면 별도 클래스로 분리한다. 이름이 주어지면서 새 도메인 개념이 등장한다.

## Inline Class — 클래스 인라인

Extract Class의 반대. 책임이 너무 작아져서 별도 클래스로 둘 의미가 없으면 합친다.

```javascript
// Before
class Person { /* ... */ }
class TrackingInformation { /* shippingCompany 한 필드 */ }

// After
class Person {
  // shippingCompany를 직접 가짐
}
```

**동기** — Lazy Element 냄새. 한 필드짜리 클래스, 한 메서드짜리 클래스는 인라인을 검토한다.

## Hide Delegate — 위임 숨기기

```javascript
// Before
manager = aPerson.department.manager;
// 클라이언트가 department의 존재를 알게 된다

// After
manager = aPerson.manager;
// person.manager가 내부에서 department.manager를 호출
```

**동기** — Message Chains 냄새. 클라이언트가 객체 그래프의 구조를 알면, 그 구조 변경 시 클라이언트도 깨진다. 위임을 숨기면 그 의존이 끊긴다.

## Remove Middle Man — 중개자 제거

Hide Delegate의 반대. 클래스가 위임만 하는 메서드로 가득하면 중개자 역할만 한다. 그 경우 클라이언트가 직접 호출하게 한다.

```javascript
// Before
class Person {
  get manager() { return this._department._manager; }  // 단순 위임
}

// After
person.department.manager;  // 직접
```

**동기** — Middle Man 냄새. 위임 자체가 나쁜 게 아니라, 한 클래스가 거의 모든 메서드를 위임만 한다면 그 클래스의 존재 이유를 다시 묻는다.

Hide Delegate와 Remove Middle Man은 반대 방향이다. 어느 쪽이 더 맞는지는 맥락 의존 — 일반적으로 시작은 위임을 숨기는 쪽이고, 위임이 너무 많아지면 줄인다.

## Substitute Algorithm — 알고리즘 교체

```javascript
// Before
function foundPerson(people) {
  for(let i = 0; i < people.length; i++) {
    if (people[i] === "Don") return "Don";
    if (people[i] === "John") return "John";
    if (people[i] === "Kent") return "Kent";
  }
  return "";
}

// After
function foundPerson(people) {
  const candidates = ["Don", "John", "Kent"];
  return people.find(p => candidates.includes(p)) || "";
}
```

**동기** — 더 명확한 알고리즘으로 통째로 갈아 끼운다. 단계별 리팩터링이 아니라 한 번에 큰 변경이다. 그래서 위험하다 — 테스트가 견고해야 한다.

## 캡슐화의 일반 원칙

이 그룹 전체를 관통하는 원칙.

- **데이터 직접 노출 최소화** — 가능하면 함수 뒤로
- **불변성 우선** — mutable data의 캡슐화 압박이 더 크다
- **객체가 자기 상태를 통제** — 외부에서 마음대로 만지면 일관성 무너진다

객체지향의 본질이기도 하다 — 데이터와 그 데이터를 다루는 행동을 한 곳에 모으는 것.

## 정리

- **Encapsulate Record/Collection** — raw 데이터 직접 노출을 막는다
- **Replace Primitive with Object** — Primitive Obsession 해결
- **Replace Temp with Query** — 계산을 함수로, 재사용 가능하게
- **Extract Class / Inline Class** — 클래스 단위로 책임 재분배
- **Hide Delegate / Remove Middle Man** — 위임 표면의 적절한 균형
- **Substitute Algorithm** — 큰 변경, 테스트가 무기
